## 2. Strategy: lrlex + lrpar (grmtools)

### 2.1 Why grmtools?

[grmtools](https://softdevteam.github.io/grmtools/) is the Rust equivalent of Yacc/Lex:
- **lrlex**: lexer generator compatible with the `.l` format
- **lrpar**: LALR(1) parser generator compatible with the `.y` format
- Static compilation via `build.rs` (LR table generated at compile-time)
- Built-in error recovery (Yelp error recovery)
- Well maintained (Softdev Lab, Kings College London)

### 2.2 Migration process

```
faustlexer.l  ──[adaptation]──→  faust.l      ──[lrlex/build.rs]──→  lexer
faustparser.y ──[adaptation]──→  faust.y      ──[lrpar/build.rs]──→  parser
                                 actions.rs   ←── Rust actions (replace C++ actions)
```

### 2.3 Necessary adaptations of .l

The Faust lexer uses some features specific to Flex:

| Feature Flex | lrlex adaptation |
|-------------|-----------------|
| `%x` (exclusive states) | lrlex supports states via `%s` / `%x` |
| `yylval` union | Return of typed tokens via the lrlex system |
| `#include` guidelines | Managed by `SourceReader` before lexing |
| Nested comments `(* *)` | Regex or dedicated state |
| Multi-line strings | Status `STRING` |

Main tokens to migrate:
```
INT      [0-9]+
FLOAT    [0-9]+\.[0-9]* | [0-9]*\.[0-9]+
IDENT    [a-zA-Z_][a-zA-Z_0-9]*
STRING   \"[^\"]*\"
FSTRING  [fF]\"[^\"]*\"
// Operators
SEQ      ":"
PAR      ","
SPLIT    "<:"
MERGE    ":>"
REC      "~"
// Keywords
PROCESS  "process"
IMPORT   "import"
DECLARE  "declare"
// ... ~50 tokens
```

### 2.4 Necessary adaptations of .y

Yacc actions in C++ build trees via `boxSeq`, `boxPar`, etc.

C++ example:
```yacc
expression : expression SEQ expression  { $$ = boxSeq($1, $3); }
           | expression PAR expression  { $$ = boxPar($1, $3); }
```

Becomes in Rust (grmtools actions):
```rust
// In the actions file
expression -> Result<TreeId, ()>:
    expression 'SEQ' expression {
        let a = $1?;
        let b = $3?;
        Ok(builder.seq(a, b))
    }
    | expression 'PAR' expression {
        let a = $1?;
        let b = $3?;
        Ok(builder.par(a, b))
    }
```

### 2.5 build.rs structure

```rust
// build.rs
use cfgrammar::yacc::YaccKind;
use lrlex::CTLexerBuilder;

fn main() {
    CTLexerBuilder::new()
        .lrpar_config(|ctp| {
            ctp.yacckind(YaccKind::Grmtools)
               .grammar_in_src_dir("faust.y")
               .unwrap()
        })
        .lexer_in_src_dir("faust.l")
        .build()
        .unwrap();
}
```

## 6. Known pitfalls

### 6.1 Yacc actions and mutable state
Yacc actions in C++ access `gGlobal` directly. In Rust, actions must receive a mutable context:

```rust
// Parser state passed to actions
struct ParseContext<'a> {
    arena: &'a mut TreeArena,
    boxes: BoxBuilder<'a>,
    reader: &'a mut SourceReader,
    diag: &'a mut DiagnosticCollector,
    // Local environment for let-expressions
    local_env: Vec<(String, TreeId)>,
}
```

### 6.2 LALR(1) conflicts
The Faust grammar has some ambiguities resolved by precedence rules (`%left`, `%right`) in Yacc. It will be necessary to verify that grmtools manages these precedences in the same way.

Known conflicts:
- `:` (sequence) vs `:` in other contexts
- `<:` (split) vs `<` (comparison)
- Dangling-else in the `if/then/else`

### 6.3 Preprocessor and imports
C++ does the preprocessing (imports, includes) **before** lexing via `SourceReader`. This separation must be preserved: `SourceReader` resolves the imports, concatenates the sources, then passes the complete text to the lexer.

### 6.4 Format strings (fstring)
Faust supports `f"signal {expr}"` format strings. The lexer must handle them as a special state.

### 6.5 Quick compatibility test
**Significant risk**: check very early that lrpar manages the Faust grammar. Recommended prototype:
1. Convert the first 10 rules of `faustparser.y`
2. Check that the parser compiles and parses `process = _;`
3. If it blocks → fallback to `lalrpop` or `tree-sitter`
