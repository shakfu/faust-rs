## feat(eval): preserve AD wrappers after recursive evaluation

Commit date: 2026-04-13
Source commit: pending (recorded by the step commit that includes this journal entry)

### Context

In the C++ compiler, `fad(expr)` and `rad(expr)` do not lower directly to
signals during evaluation. `eval.cpp` recursively evaluates the wrapped
expression first, then reconstructs `boxForwardAD(...)` /
`boxReverseAD(...)` around the normalized residual box.

### What was done

Added explicit `BoxMatch::ForwardAD` and `BoxMatch::ReverseAD` arms in
`crates/eval/src/lib.rs::eval_value_uncached(...)`.

Each arm:

- recursively evaluates the wrapped expression in the current environment
- forces the result back to box form with `force_value_to_box(...)`
- rebuilds the explicit AD wrapper with `BoxBuilder`

Extended the lightweight evaluator-local arity inference so residual
`ForwardAD`/`ReverseAD` wrappers remain transparent for application probing.

Added evaluator integration tests proving `eval_process(...)` preserves the
explicit AD wrapper for both `fad(...)` and `rad(...)` while still
normalizing the inner expression to first-order box form.

### Decision

- Eval follows the C++ contract exactly: AD wrappers survive as box nodes.
- Reverse-mode remains accepted and preserved at eval time even though the
  propagation step is still phase-gated later in the plan.

### Validation

- `cargo test -p eval`
- `cargo fmt --all --check`

- C++ parity for `fad(1, 2)`, which parses as `fad((1, 2))` because the inner
  argument is a full Faust expression rather than an arity-checked argument

- Parser behavior follows the C++ grammar as source of truth.

  was intentionally not applied because it would diverge from reference parser

## feat(boxes): add forward/reverse AD box wrappers

Commit date: 2026-04-13
Source commit: pending (recorded by the step commit that includes this journal entry)

### Context

The AD port starts at the box layer: `fad(expr)` and `rad(expr)` must survive
parsing and evaluation as explicit structural wrappers before propagation can
lower them to signal-space transforms. The existing Rust `boxes` crate already
exposes the canonical `BoxBuilder`/`BoxMatch` surface used by parser, eval,
and propagate, so this was the correct insertion point for the new primitives.

### What was done

Added crate-private tags `BOXFAUTODIFF` and `BOXRAUTODIFF` in
`crates/boxes/src/tags.rs`.

Extended `BoxMatch` with `ForwardAD(BoxId)` and `ReverseAD(BoxId)` plus
`match_box(...)` decode arms, documenting the corresponding C++ provenance from
`compiler/boxes/boxes.cpp`.

Added `BoxBuilder::forward_ad(...)` and `BoxBuilder::reverse_ad(...)`, keeping
the public construction API aligned with the existing canonical builder style.

Extended the `boxes` integration test coverage so wrapper round-trips now
explicitly include `Inputs`, `Outputs`, `ForwardAD`, `ReverseAD`, `Ondemand`,
`Upsampling`, and `Downsampling`.

### Decision

- AD wrappers stay as explicit unary box nodes instead of being normalized away
  early.
- Arity transparency for these wrappers is deferred to the propagation step,
  where `box_arity_typed(...)` already owns the flat-box arity contract.

### Validation

- `cargo test -p boxes`
- `cargo fmt --all --check`

## docs(porting): add automatic differentiation forward-mode port plan

### Context

Automatic Differentiation (AD) is a major feature in the C++ Faust compiler,
providing `fad(expr)` (forward-mode) and `rad(expr)` (reverse-mode) language
constructs. These allow users to compute derivatives of DSP programs with
respect to UI controls — enabling sensitivity analysis, gradient-based
optimization, Newton solvers, and differentiable DSP for machine learning.

The C++ implementation uses a signal-to-signal transformation architecture:
the eval stage resolves `fad(expr)` / `rad(expr)` into concrete boxes, the
propagate stage lowers them to signals, then a `ForwardADSignalTransform`
(or `ReverseADSignalTransform`) applies calculus rules to produce an expanded
signal list containing both primal outputs and their derivatives.

### What was done

Analyzed both reference documents (`Faust_Autodifferentiation_report.pdf` and
`Automatic_Differentiation_in_the_Faust_Compiler_(v1).pdf`) and the full C++
implementation across parser, boxes, eval, propagate, and transform stages.

Explored the Rust codebase architecture (TreeArena-based boxes, BoxMatch enum,
SigMatch enum, SigBuilder, FlatNodeKind, propagate_inner dispatch) to identify
precise integration points.

Wrote a comprehensive 8-step port plan covering:

1. **Box primitives** — `BOXFAUTODIFF`/`BOXRAUTODIFF` tags, `BoxMatch` variants, `BoxBuilder` methods
2. **Parser** — `fad`/`rad` lexer tokens and grammar rules
3. **Evaluator** — recursive eval arms for `ForwardAD`/`ReverseAD`
4. **Flat-box + propagation bridge** — `FlatNodeKind` variants, validation, arity, dispatch
5. **AD control collector** — signal visitor collecting differentiable UI controls with `[autodiff:false]` support
6. **Forward AD transform** — dual-number transformation with 30+ differentiation rules and DAG-preserving memoization
7. **End-to-end integration** — pipeline tests and corpus DSP files
8. **Future work** — reverse-mode `rad` propagation deferred

### Decision

- Both `fad(expr)` and `rad(expr)` are added to parser and eval in this phase.
- Only `fad` is implemented in propagation; `rad` returns a clear error.
- The AD transform lives in `propagate_inner` dispatch, avoiding the need for
  the C++ `boxSignal` bridge mechanism.
- Memoization cache in the transform is critical for DAG preservation and
  preventing expression swell.
