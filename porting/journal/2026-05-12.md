## Document BRA scheduling and tape semantics

Commit date: 2026-05-12
Source commit: this commit (`Document BRA scheduling and tape semantics`)

Expanded the in-code documentation for the BlockReverseAD/BRA path across
propagation, Signal-IR construction, and FIR lowering. The docs now spell out
that `SigBlockReverseAD` is a semantic carrier, while
`crates/transform/src/signal_fir/module.rs` owns the concrete loop placement,
forward tape arrays, carry resets, and typed tape-storage boundary.

The pass documents the surprising adaptive case where a BRA gradient projection
is consumed inside a forward-time recursive update. In that shape the generated
C++ may contain no separate public backward loop: the same local adjoint sweep
is emitted inline while lowering the causal forward loop. The notes also
separate this placement rule from the dormant LTI/IIR `ReverseTimeRec`
transpose path; inline BRA does not mean a linear subgraph was analytically
transposed.

The BRA tape-store boundary now skips non-real structural tape candidates
instead of silently casting them at FIR lowering. This matters for DSPs with
integer/discrete subgraphs upstream of a `FloatCast`, such as LCG noise
recursions. The normal Signal pipeline still performs the needed promotion:
the real use site is represented by an explicit `FloatCast` inserted by
normalform / `signalPromotion`. BRA does not run a second promotion pass for
its synthesized `fBraTapeN` stores. It records only values that are already
real after that promotion boundary; upstream integer nodes keep their integer
semantics, and `propagate_bra_adj` stops the adjoint at the int→real cast.
Actual tape stores must still lower to `real_ty`.

- `cargo test -p compiler --test rad_runtime bra_reverse_sweep_minmax_conditions_compile`

## Persist BRA constants used by generated reverse sweeps

Commit date: 2026-05-12
Source commit: this commit (`Persist BRA constants used by reverse sweeps`)

Fixed `rad_stress3.dsp`, where a RAD/BRA body containing `fi.resonbp` generated
SR-derived `fConst*` values as stack-local temporaries inside
`instanceConstants()`, then reused those names from the synthesized BRA reverse
sweep in `compute()`. FIR verification rejected the generated module because
those stack locals are not declared in `compute`.

The placement analysis now treats `Konst` descendants of `BlockReverseAD`
carriers as escaping constants. This is intentionally conservative, not a
minimal lifetime proof: BRA lowering synthesizes forward tape stores and a
reverse adjoint program whose compute-time uses are not represented as ordinary
parent edges in the original signal DAG. Persistent struct storage for these
constants is therefore the local robust boundary. The code comments in
`crates/transform/src/signal_fir/placement.rs` document this tradeoff and the
future alternative of lifecycle-aware FIR caches.

Added a C++ compilation regression test for a small `fi.resonbp` RAD bank that
exercises the same SR-derived constant path as `rad_stress3.dsp`.

## Fix BRA min/max comparison FIR typing

Commit date: 2026-05-12
Source commit: this commit (`Fix BRA min/max comparison FIR typing`)

Fixed the BlockReverseAD reverse sweep for `min` and `max` subgradients so
the generated FIR comparison conditions are declared as `Int32` instead of the
DSP real type. The previous lowering emitted `Le`/`Ge` nodes with `Float32`
results, which made FIR verification reject recursive RAD programs whose BRA
sweep crossed clamped soft-clippers or parameter clamps.

Added a regression test that compiles a recursive nonlinear soft-clip learning
case with both inner `max(-1,min(1,...))` saturation and outer parameter
clamping. This covers the same FIR typing failure reported by
`rad_softclip1.dsp`.

## Update supported subset RAD status

Commit date: 2026-05-12
Source commit: this commit (`Update supported subset RAD status`)

Updated the RAD/FAD documentation set with the latest RAD boundary changes:
the disabled reverse-time-recursive RAD dispatcher fast path, the BRA
real-valued tape-store typing boundary, and the shared FAD/RAD wrapper
input-slicing fix.

The pass also corrected stale wording that still described temporal or
recursive `rad(...)` bodies as user-visible rejections. Current public RAD
dispatch keeps the local symbolic reverse sweep feed-forward only, but routes
temporal/recursive kinds (`delay`, `prefix`, recursion, IIR carriers) through
`BlockReverseAD`. `RadUnsupportedNode` remains the internal dispatch signal for
those families and the public diagnostic for hard unsupported families such as
mutable tables, soundfiles, and unsupported foreign functions.

The RAD supported-subset note now documents that `rad(*, (_,_,_))` exposes a
three-input wrapper while propagating the body `*` with only its two required
inputs. The expected output bundle is:

1. primal: `x0 * x1`
2. `d(body)/d(seed0)`: `d(x0*x1)/d(x0) = x1`
3. `d(body)/d(seed1)`: `d(x0*x1)/d(x1) = x0`
4. `d(body)/d(seed2)`: `d(x0*x1)/d(x2) = 0`

## Fix AD wrapper input slicing

Commit date: 2026-05-12
Source commit: this commit (`Fix AD wrapper input slicing`)

Fixed propagation for `fad(body, seed)` and `rad(body, seeds)` when the seed
box consumes a wider upstream bus than the differentiated body. The typed
arity of these wrappers is the maximum of the child input arities, but each
child must still be propagated with only the prefix of that bus matching its
own input arity.

Previously the seed side already used this sliced input bus, while the body
side received the full wrapper input bus. That made a valid expression such
as `fad(*, (_,_,_))` fail during propagation because `*` expects two inputs
but received the three-input wrapper bus. The same mismatch existed in
`rad(*, (_,_,_))`.

The propagation lowering now computes the body arity and passes a sliced
`body_inputs` vector to both FAD and RAD body propagation, matching the seed
handling and the wrapper arity contract.

For the concrete wide seed bus case:

```faust
process = fad(*, (_,_,_));
```

with wrapper inputs `x0`, `x1`, `x2`, the differentiated body is still only
`x0 * x1`. The expected outputs are:

1. primal: `x0 * x1`
2. `d(body)/d(seed0)`: `d(x0*x1)/d(x0) = x1`
3. `d(body)/d(seed1)`: `d(x0*x1)/d(x1) = x0`
4. `d(body)/d(seed2)`: `d(x0*x1)/d(x2) = 0`

For the analogous reverse-mode case:

```faust
process = rad(*, (_,_,_));
```

the visible output contract is the same for this scalar body:

1. primal: `x0 * x1`
2. `d(body)/d(seed0)`: `d(x0*x1)/d(x0) = x1`
3. `d(body)/d(seed1)`: `d(x0*x1)/d(x1) = x0`
4. `d(body)/d(seed2)`: `d(x0*x1)/d(x2) = 0`

Internally, `fad` obtains these lanes by propagating one tangent direction per
seed output, while `rad` starts from the scalar body output cotangent and
accumulates adjoints back to each seed output. The arity slicing fix is shared
because both wrappers expose the maximum child input arity externally, while
each child must still be evaluated at its own arity internally.

## Fix BRA tape stores for integer forward values

Commit date: 2026-05-12
Source commit: this commit (`Fix BRA tape stores for integer forward values`)

Fixed the remaining `cargo test` failure after disabling the `ReverseTimeRec`
RAD dispatcher path. `debug_softclip_gradient_values` failed in the
interpreter backend with:

```text
FBC runtime error [stack_underflow] opcode=StoreIndexedReal
```

The generated BRA forward tape stores always declare tape arrays as
`Array(real_ty, MAX_BRA_TAPE_BLOCK_SIZE)`, but some taped forward values can
still lower as integer FIR values. The first softclip debug tape store tried
to write an integer `LoadInt` directly into a real tape array, so the
interpreter emitted `StoreIndexedReal` and then found no real value on the
runtime stack.

`ensure_bra_tape_stores` now casts every taped value to `real_ty` before
storing when its FIR type is not already the tape element type. This preserves
the real-valued tape contract used by the BRA backward sweep and removes the
interpreter stack-type mismatch.
