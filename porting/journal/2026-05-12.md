## Update supported subset RAD status

Commit date: 2026-05-12
Source commit: this commit (`Update supported subset RAD status`)

Updated the RAD/FAD documentation set with the latest RAD boundary changes:
the disabled reverse-time-recursive RAD dispatcher fast path, the BRA
real-valued tape-store typing boundary, and the shared FAD/RAD wrapper
input-slicing fix.

The pass also corrected stale wording that still described temporal or
recursive `rad(...)` bodies as user-visible rejections. Current public RAD
dispatch keeps the local symbolic reverse sweep feed-forward only, but routes
temporal/recursive kinds (`delay`, `prefix`, recursion, IIR carriers) through
`BlockReverseAD`. `RadUnsupportedNode` remains the internal dispatch signal for
those families and the public diagnostic for hard unsupported families such as
mutable tables, soundfiles, and unsupported foreign functions.

The RAD supported-subset note now documents that `rad(*, (_,_,_))` exposes a
three-input wrapper while propagating the body `*` with only its two required
inputs. The expected output bundle is:

1. primal: `x0 * x1`
2. `d(body)/d(seed0)`: `d(x0*x1)/d(x0) = x1`
3. `d(body)/d(seed1)`: `d(x0*x1)/d(x1) = x0`
4. `d(body)/d(seed2)`: `d(x0*x1)/d(x2) = 0`

## Fix AD wrapper input slicing

Commit date: 2026-05-12
Source commit: this commit (`Fix AD wrapper input slicing`)

Fixed propagation for `fad(body, seed)` and `rad(body, seeds)` when the seed
box consumes a wider upstream bus than the differentiated body. The typed
arity of these wrappers is the maximum of the child input arities, but each
child must still be propagated with only the prefix of that bus matching its
own input arity.

Previously the seed side already used this sliced input bus, while the body
side received the full wrapper input bus. That made a valid expression such
as `fad(*, (_,_,_))` fail during propagation because `*` expects two inputs
but received the three-input wrapper bus. The same mismatch existed in
`rad(*, (_,_,_))`.

The propagation lowering now computes the body arity and passes a sliced
`body_inputs` vector to both FAD and RAD body propagation, matching the seed
handling and the wrapper arity contract.

For the concrete wide seed bus case:

```faust
process = fad(*, (_,_,_));
```

with wrapper inputs `x0`, `x1`, `x2`, the differentiated body is still only
`x0 * x1`. The expected outputs are:

1. primal: `x0 * x1`
2. `d(body)/d(seed0)`: `d(x0*x1)/d(x0) = x1`
3. `d(body)/d(seed1)`: `d(x0*x1)/d(x1) = x0`
4. `d(body)/d(seed2)`: `d(x0*x1)/d(x2) = 0`

For the analogous reverse-mode case:

```faust
process = rad(*, (_,_,_));
```

the visible output contract is the same for this scalar body:

1. primal: `x0 * x1`
2. `d(body)/d(seed0)`: `d(x0*x1)/d(x0) = x1`
3. `d(body)/d(seed1)`: `d(x0*x1)/d(x1) = x0`
4. `d(body)/d(seed2)`: `d(x0*x1)/d(x2) = 0`

Internally, `fad` obtains these lanes by propagating one tangent direction per
seed output, while `rad` starts from the scalar body output cotangent and
accumulates adjoints back to each seed output. The arity slicing fix is shared
because both wrappers expose the maximum child input arity externally, while
each child must still be evaluated at its own arity internally.

## Fix BRA tape stores for integer forward values

Commit date: 2026-05-12
Source commit: this commit (`Fix BRA tape stores for integer forward values`)

Fixed the remaining `cargo test` failure after disabling the `ReverseTimeRec`
RAD dispatcher path. `debug_softclip_gradient_values` failed in the
interpreter backend with:

```text
FBC runtime error [stack_underflow] opcode=StoreIndexedReal
```

The generated BRA forward tape stores always declare tape arrays as
`Array(real_ty, MAX_BRA_TAPE_BLOCK_SIZE)`, but some taped forward values can
still lower as integer FIR values. The first softclip debug tape store tried
to write an integer `LoadInt` directly into a real tape array, so the
interpreter emitted `StoreIndexedReal` and then found no real value on the
runtime stack.

`ensure_bra_tape_stores` now casts every taped value to `real_ty` before
storing when its FIR type is not already the tape element type. This preserves
the real-valued tape contract used by the BRA backward sweep and removes the
interpreter stack-type mismatch.

- `cargo test -p compiler --test rad_runtime debug_softclip_gradient_values -- --nocapture`
- `cargo test -p transform signal_fir::tests::bra_body_with_integer_float_cast_compiles`
- `cargo test --workspace --all-targets`
- `cargo clippy --workspace --all-targets -- -D warnings`
