## Fix BRA tape stores for integer forward values

Commit date: 2026-05-12
Source commit: this commit (`Fix BRA tape stores for integer forward values`)

Fixed the remaining `cargo test` failure after disabling the `ReverseTimeRec`
RAD dispatcher path. `debug_softclip_gradient_values` failed in the
interpreter backend with:

```text
FBC runtime error [stack_underflow] opcode=StoreIndexedReal
```

The generated BRA forward tape stores always declare tape arrays as
`Array(real_ty, MAX_BRA_TAPE_BLOCK_SIZE)`, but some taped forward values can
still lower as integer FIR values. The first softclip debug tape store tried
to write an integer `LoadInt` directly into a real tape array, so the
interpreter emitted `StoreIndexedReal` and then found no real value on the
runtime stack.

`ensure_bra_tape_stores` now casts every taped value to `real_ty` before
storing when its FIR type is not already the tape element type. This preserves
the real-valued tape contract used by the BRA backward sweep and removes the
interpreter stack-type mismatch.

Validation:

- `cargo fmt --all`
- `cargo test -p compiler --test rad_runtime debug_softclip_gradient_values -- --nocapture`
- `cargo test -p transform signal_fir::tests::bra_body_with_integer_float_cast_compiles`
- `cargo test --workspace --all-targets`
- `cargo clippy --workspace --all-targets -- -D warnings`
