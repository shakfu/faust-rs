## Bound the BRA forward-tape index (W5)

Commit date: 2026-06-20
Source commit: this commit

Acted on W5 / rewriting-calculus §8.5: BRA forward tapes are fixed-size
`Array(real_ty, MAX_BRA_TAPE_BLOCK_SIZE)` struct fields indexed by the raw loop
counter `i0`, so a host calling `compute(count > MAX_BRA_TAPE_BLOCK_SIZE)` with a
`SigBlockReverseAD` carrier wrote and read out of bounds (silent UB).

`MAX_BRA_TAPE_BLOCK_SIZE` is a power of two, so the tape index is now masked via a
shared `bra_tape_index` helper (`i0 & (MAX - 1)`), applied at both the forward
store and the reverse load. For the supported block size the mask is a no-op
(`i0 < MAX`); for an over-long block the access wraps within the array
(aliased/approximate gradients for the tail) instead of overflowing. A
`const _: () = assert!(MAX_BRA_TAPE_BLOCK_SIZE.is_power_of_two())` protects the
masking invariant. Exact arbitrary-length support remains a larger change
(chunked TBPTT or a dynamically sized tape).

Updated the W5 framing in both companion documents accordingly.

- `cargo clippy -p transform --all-targets`
- `cargo fmt -p transform -- --check`

## Reframe W8 as an intentional preparation superset

Commit date: 2026-06-20
Source commit: this commit

Documentation correction. While attempting the "reject the W8 families early in
`verify`" resolution, two existing tests failed —
`prepare_signals_for_fir_accepts_filter_carrier_children` and
`prepare_signals_for_fir_recovers_shared_zero_pad_amount_from_float_context` —
which assert that preparation *keeps* `Fir`/`ZeroPad` carriers. Preparation is
therefore a deliberate superset of the fast-lane lowerer, and W8 is violated by
design, not by accident; the rejection attempt was reverted (nothing landed).

Corrected the framing in the rewriting-calculus companion (§8.2, the §8.6 payoff
table, and §10 item 2) and added a consistency note to the analysis document's
W8 entry:

- the coverage obligation is `L_prep ⊆ dom(lower) ∪ Deferred`;
- the `lowering_coverage` classifier is a drift guard, not an equality assertion
  (`{verify-accepted} == {lower-handled}` is intentionally false);
- the right resolution is to lower a family when a DSP needs it, not to reject
  it; the residual defect is only the late, generic `UnsupportedSignalNode`
  failure (W13).

## Enforce P and D1 at the prepare verifier boundary

Commit date: 2026-06-20
Source commit: this commit

Acted on §8.1 of the rewriting-calculus formalisation: the staging verifier
(`signal_prepare::verify`) enforced a predicate strictly weaker than the lowering
precondition `L_prep` — it checked neither `P` (the promotion invariant) nor `D1`
(one-sample delays canonicalized to `Delay1`). `P` was caught only lazily inside
`lower_binop`; `D1` was never re-guarded after canonicalization.

Added `verify_promotion_invariant`, run per reachable node during `verify`,
checking the documented invariant over the reduced `SimpleSigType` domains:

- `D1`: reject `Delay(_, Int 1)`;
- arithmetic / Div / Rem / bitwise / shift `BinOp` operands share the node's
  result domain;
- comparison `BinOp` operands share a numeric domain;
- `Delay` / `RdTbl` / `WrTbl` / `Select2` / `Enable` integer-context operands are
  `Int`;
- `Prefix(init, value)` has matching init/value domains.

The boundary gate now matches the precondition the FIR lowering rules assume, so
a staging regression is reported close to its source instead of surfacing late
(or silently) during FIR emission. `lower_binop`'s per-node check becomes a
backstop rather than the only guard.

Validation:

- the full documented invariant holds across the real-DSP corpus with no
  regression: `cargo test -p compiler` (91 + 2), `cargo test -p transform` (103,
  including 3 new boundary tests)

- `cargo fmt -p transform -- --check`

## Make the lowering-coverage obligation executable

Commit date: 2026-06-20
Source commit: this commit

Acted on the first "cheap win" of the rewriting-calculus formalisation (§8.2 /
analysis W8): turned the `L_prep ⊆ dom(lower)` coverage obligation into an
executable test in `crates/transform/src/signal_fir/tests.rs`.

`lowering_coverage(&SigMatch)` classifies every signal constructor as `Direct`
(own `lower_signal` arm), `ViaParent` (reachable only through `Proj`, e.g.
`BlockReverseAD`/`ReverseTimeRec`), or `Unsupported` (falls through to the
`UnsupportedSignalNode` arm). The match has no wildcard, so adding a new
`SigMatch` variant will not compile until it is consciously classified — the
drift guard. The `Unsupported` arm is the executable record of the W8 gap
(`Gen`, `AssertBounds`, `TempVar`, `PermVar`, `Seq`, `ZeroPad`, `Clocked`,
`OnDemand`, `Upsampling`, `Downsampling`, `Fir`, `Iir` are accepted by
`signal_prepare::verify` but have no lowering arm).

Behavioural tests confirm a representative supported program lowers to a module,
and that an unsupported family (`Gen`) fails with the stable
`UnsupportedSignalNode` code instead of silently emitting FIR.

This characterises the current boundary (the tests are green); resolving the W8
gap itself — rejecting the unsupported families earlier in `verify`, or adding
their lowering — remains a separate decision.
