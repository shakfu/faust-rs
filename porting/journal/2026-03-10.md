### `transform` / `compiler` / `codegen` — Restore polymorphic integer `min/max` and `abs` through recursive typing, promotion, FIR lowering, and backend calls

Commit date: 2026-03-10

Source commit: pending

#### Context

The fast-lane had a parity bug on recursive integer programs using polymorphic
numeric operators such as `min`/`max` and `abs`. Minimal cases such as:

```faust
process = 1 : (+ : min(3)) ~ _;
process = 1 : (+ : abs) ~ _;
```

should keep its recursive carrier in integer state, matching Faust C++.
Instead, `faust-rs` widened the recursion to floating-point because the
prepared typing and promotion path treated `SIGMIN`/`SIGMAX`/`SIGABS` as
always-real operations.

That was not just a local lowering issue:

- reduced typing always classified `min/max/abs` as `Real`
- promotion always inserted float casts for their operands
- FIR lowering emitted either floating-point math calls or a temporary
  integer `Select2` synthesis instead of preserving the function-form parity

So integer recursive feedback loops using these operators lost parity before
codegen.

#### Changes

- [signal_prepare.rs](../../crates/transform/src/signal_prepare.rs)
  - changed reduced typing so `SIGMIN` / `SIGMAX` unify their operand reduced
    types instead of forcing `Real`
  - changed `SIGABS` typing so it preserves integer operands instead of forcing
    `Real`
  - added dedicated promotion logic that preserves integer cases and only
    widens mixed/real numeric operands when needed
  - added a recursion-focused non-regression test for
    `1 : (+ : min(3)) ~ _`
  - added the symmetric recursion-focused non-regression test for
    `1 : (+ : abs) ~ _`

- [module.rs](../../crates/transform/src/signal_fir/module.rs)
  - changed fast-lane lowering so integer `min/max` no longer go through
    floating-point `fmin/fmax` helpers
  - changed integer `abs` lowering so it no longer goes through floating-point
    `fabs`
  - integer cases now stay explicit FIR function calls:
    - `min_i`
    - `max_i`
    - `abs`
  - real cases keep using the existing math-call path
  - restored matching FIR prototypes for these integer calls so the FIR checker
    still accepts the resulting module

- [mod.rs](../../crates/transform/src/signal_fir/mod.rs)
  - added structural non-regressions asserting that recursive integer `min`
    and `abs` allocate `Int32` two-slot recursion arrays and do not emit float
    helper calls
  - these tests now also assert the preserved explicit integer call form in the
    dumped FIR

- [signal_fir_lane.rs](../../crates/compiler/tests/signal_fir_lane.rs)
  - added end-to-end C++ fast-lane regressions on the minimal DSPs above,
    asserting that generated code uses `int[2] iRec...` instead of a float
    recursion carrier and keeps explicit `std::min<int>` / `std::abs` calls

- [mod.rs](../../crates/codegen/src/backends/cpp/mod.rs)
  - mapped FIR `min_i` / `max_i` / `abs` calls to the expected C++ spellings:
    `std::min<int>`, `std::max<int>`, `std::abs`

- [mod.rs](../../crates/codegen/src/backends/c/mod.rs)
  - mapped FIR `min_i` / `max_i` calls to local `faustmini` / `faustmaxi`
    helpers, matching the C backend style used in Faust C++

- [mod.rs](../../crates/codegen/src/backends/cranelift/mod.rs)
  - added direct lowering/import support for integer `abs`, `min_i`, and
    `max_i`
  - extended the subset matcher so these calls are accepted by the same
    contract as other supported FIR function calls

#### Validation

- `cargo test -p transform prepare_signals_for_fir_keeps_integer_recursive_min_feedback_int`
- `cargo test -p transform prepare_signals_for_fir_keeps_integer_recursive_abs_feedback_int`
- `cargo test -p transform integer_recursive_min_lowers_to_int_recursion_and_min_i_call`
- `cargo test -p transform integer_recursive_abs_lowers_to_int_recursion_and_abs_call`
- `cargo test -p compiler --test signal_fir_lane fastlane_cpp_keeps_integer_recursive_min_feedback_in_int_state`
- `cargo test -p compiler --test signal_fir_lane fastlane_cpp_keeps_integer_recursive_abs_feedback_in_int_state`
- `cargo run -p compiler -- -lang cpp /tmp/rec_int_min.dsp`
- `cargo run -p compiler -- -lang cpp /tmp/rec_int_abs.dsp`
- `cargo run -p compiler -- -lang cranelift /tmp/rec_int_min.dsp`
- `cargo run -p compiler -- -lang cranelift /tmp/rec_int_abs.dsp`

#### Outcome

The fast-lane now treats `min/max` and `abs` as genuinely polymorphic numeric
operators for the active reduced-type domain. Integer recursive loops using
these operators stay in integer state through prepared typing, promotion, FIR
lowering, and backend emission. They also keep explicit function-call form,
matching the Faust C++ strategy instead of being rewritten as conditional FIR
selections.

### `eval` / `compiler` — Match C++ partial-application fallback for symbolic arguments

Commit date: 2026-03-10

Source commit: pending

#### Context

`karplus.dsp` exposed a parity gap in the evaluator’s non-closure partial
application path.

The problematic shape was:

- `*(button("play") : trigger(size))`

Faust C++ accepts this because `applyList(...)` treats it as an under-applied
binary primitive and rewrites it to the equivalent of:

- `(_, button("play") : trigger(size)) : *`

The crucial detail is the helper used during argument counting:

- C++ `boxlistOutputs(...)` falls back to counting one output when local
  `getBoxType(...)` cannot determine the width of an already-evaluated
  argument.

Rust was stricter:

- `list_outputs(...)` delegated to the lightweight `infer_box_arity(...)`
  helper,
- if that helper returned `None`, Rust skipped the implicit-wire rewrite and
  left the raw `arg : *` shape intact,
- `propagate` then failed later with a spurious sequential composition
  mismatch `1 != 2`.

This was not a real `karplus` recursion bug. It was an evaluator parity issue
in partial application of binary primitives with residual symbolic/recursive
arguments.

#### Changes

- [lib.rs](../../crates/eval/src/lib.rs)
  - changed `list_outputs(...)` to match C++ `boxlistOutputs(...)`
  - when local arity inference cannot determine an evaluated argument width,
    it now counts that argument as a single output instead of aborting the
    partial-application rewrite
  - documented the provenance and the exact reason for the fallback in Rustdoc,
    including the `*(button("play") : trigger(n))` repro and why it must be
    rewritten to `(_, arg) : *`

- [core_eval.rs](../../crates/eval/tests/core_eval.rs)
  - added a structural evaluator regression covering the symbolic-trigger case
  - asserts that under-applied `*` prepends an implicit wire even when the
    explicit argument defeats the lightweight local arity probe

- [signal_pipeline.rs](../../crates/compiler/tests/signal_pipeline.rs)
  - added an end-to-end compiler regression for the reduced repro
    `*(button("play") : trigger(128))`
  - asserts that the signal pipeline now lowers the result to a `Mul` signal
    with one operand coming from the implicit input wire

#### Validation

- `cargo fmt --all`
- `cargo test -p eval --test core_eval`
- `cargo test -p compiler --test signal_pipeline`
- `cargo run -p compiler -- --dump-sig karplus.dsp`

#### Outcome

`karplus.dsp` no longer fails in `propagate` on a fake `1 != 2` sequential
composition mismatch. The pipeline now advances to the separate existing
fast-lane limitation on non-constant `SIGDELAY` amounts, which confirms the
partial-application parity bug is fixed.

    invariant in debug builds

  - strengthened recursion structural tests to assert that recursive FIR
    lowering no longer inserts cast wrappers around recursion-array accesses

### `transform` / `compiler` — Lower recursive feedback to C++-style 2-slot arrays and keep fixed delays on `fVec/iVec`

- but `delay1(proj(symref(...)))` in the recursion path could still allocate a
  separate scalar shadow state.

That diverged from the Faust C++ compiler contract, where first-order
recursion uses a 2-slot `fRec*/iRec*` array and shifts `[1] = [0]` after the
sample, while fixed delays greater than one sample remain separate circular
buffers (`fVec*/iVec*`) addressed through `fIOTA`.

`tests/corpus/rep_38_sine_phasor.dsp` exposed the first-order case, and
`tests/corpus/rep_55_sine_phasor_echo_feedback.dsp` remains the concrete
example for recursion plus fixed delay `@(10000)`.

#### Changes

- [module.rs](../../crates/transform/src/signal_fir/module.rs)
  - changed `SIGREC` / `SIGPROJ` lowering to use explicit 2-slot recursion
    arrays as the canonical carrier
  - made symbolic recursion references load the current slot `[0]`
  - taught `delay1(proj(symref(...)))` in the active recursion path to reuse
    the recursion array previous slot `[1]` instead of allocating a separate
    scalar state variable
  - kept fixed `SIGDELAY(x, N)` with `N > 1` on the existing circular delay
    line path (`fVec*/iVec*` plus `fIOTA`)

- [mod.rs](../../crates/transform/src/signal_fir/mod.rs)
  - updated recursion tests to expect a 2-slot array declaration
  - added a structural non-regression test ensuring recursive
    `delay1(proj(symref(...)))` reuses one `fRec*[2]` array and only writes
    slots `0` and `1`

- [signal_fir_lane.rs](../../crates/compiler/tests/signal_fir_lane.rs)
  - tightened the `rep_38_sine_phasor` regression to assert the emitted
    `float[2] fRec*` update pattern
  - aligned the `rep_56_noise_smoo_slider` regression with the new integer
    array form `int[2] iRec*`

#### Validation

- `cargo fmt --all`
- `cargo test -p transform`
- `cargo test -p compiler --test signal_fir_lane`

#### Outcome

The fast-lane now matches Faust C++ more closely for recursive state:
one-sample recursion uses `fRec*/iRec*[2]`, while fixed delays greater than one
sample still lower through circular `fVec*/iVec*` buffers indexed by `fIOTA`.
