### `cranelift-ffi` / `codegen` — Make the Cranelift factory/instance runtime standalone and audible

Commit date: 2026-03-10

Source commit: this commit (`Make Cranelift runtime standalone`)

#### Context

The Cranelift backend had reached a problematic hybrid state:

- Cranelift JIT handled `compute`,
- but `cranelift-ffi` still depended on interpreter-side sidecars for
  `buildUserInterface`, `metadata`, and parts of instance initialization.

That temporary architecture caused a real runtime bug on synthesized corpus
cases such as:

- `tests/corpus/rep_38_sine_phasor.dsp`
- `tests/corpus/rep_55_sine_phasor_echo_feedback.dsp`

Both DSPs compiled and lowered through Cranelift, but could remain silent at
runtime because UI/control values lived in the sidecar heap while JIT `compute`
read only the native `dsp*` state buffer.

The desired target was a fully standalone Cranelift runtime:

- one FIR-derived native state layout,
- one native runtime descriptor for UI/metadata/init/reset,
- no interpreter-side execution dependency in the Cranelift factory/instance
  path.

#### Changes

- [mod.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/backends/cranelift/mod.rs)
  - extended the Cranelift `dsp*` layout contract to accept
    `DeclareVar(..., Array(_, N), AccessType::Struct)` as inline table-style
    fields
  - added Int32 bitwise/remainder lowering (`Rem`, `And`, `Or`, `Xor`) so
    fixed-delay ring-buffer masks such as `fIOTA & 16383` no longer trigger
    subset-matcher drift
  - added backend structural regressions for:
    - struct array vars (`fRec*[2]`)
    - Int32 `And`

- [runtime.rs](/Users/letz/Developpements/RUST/faust-rs/crates/cranelift-ffi/src/runtime.rs)
  - added a FIR-derived native runtime descriptor for standalone Cranelift
    instances
  - records:
    - struct field initial values,
    - field-aware clear/reset policy,
    - control defaults,
    - UI widget declarations,
    - metadata declarations,
    - sample-rate field names

- [factory.rs](/Users/letz/Developpements/RUST/faust-rs/crates/cranelift-ffi/src/factory.rs)
  - changed file/string/boxes/signals factory creation to compile FIR once and
    derive both:
    - the Cranelift JIT module,

  - removed interpreter-side factory creation from the Cranelift runtime path

    descriptor from source/FIR

- [instance.rs](/Users/letz/Developpements/RUST/faust-rs/crates/cranelift-ffi/src/instance.rs)

    - `instanceResetUserInterface`

    to operate directly on the native Cranelift `dsp_state`
  - kept `compute` as a pure Cranelift JIT call using the same `dsp_state`
  - added end-to-end runtime regressions asserting non-silent output for:
    - `rep_38_sine_phasor`
    - `rep_55_sine_phasor_echo_feedback`

### `eval` / `compiler` — Match C++ partial-application fallback for symbolic arguments

Commit date: 2026-03-10

Source commit: pending

#### Context

`karplus.dsp` exposed a parity gap in the evaluator’s non-closure partial
application path.

The problematic shape was:

- `*(button("play") : trigger(size))`

Faust C++ accepts this because `applyList(...)` treats it as an under-applied
binary primitive and rewrites it to the equivalent of:

- `(_, button("play") : trigger(size)) : *`

The crucial detail is the helper used during argument counting:

- C++ `boxlistOutputs(...)` falls back to counting one output when local
  `getBoxType(...)` cannot determine the width of an already-evaluated
  argument.

Rust was stricter:

- `list_outputs(...)` delegated to the lightweight `infer_box_arity(...)`
  helper,
- if that helper returned `None`, Rust skipped the implicit-wire rewrite and
  left the raw `arg : *` shape intact,
- `propagate` then failed later with a spurious sequential composition
  mismatch `1 != 2`.

This was not a real `karplus` recursion bug. It was an evaluator parity issue
in partial application of binary primitives with residual symbolic/recursive
arguments.

#### Changes

- [lib.rs](/Users/letz/Developpements/RUST/faust-rs/crates/eval/src/lib.rs)
  - changed `list_outputs(...)` to match C++ `boxlistOutputs(...)`
  - when local arity inference cannot determine an evaluated argument width,
    it now counts that argument as a single output instead of aborting the
    partial-application rewrite
  - documented the provenance and the exact reason for the fallback in Rustdoc,
    including the `*(button("play") : trigger(n))` repro and why it must be
    rewritten to `(_, arg) : *`

- [core_eval.rs](/Users/letz/Developpements/RUST/faust-rs/crates/eval/tests/core_eval.rs)
  - added a structural evaluator regression covering the symbolic-trigger case
  - asserts that under-applied `*` prepends an implicit wire even when the
    explicit argument defeats the lightweight local arity probe

- [signal_pipeline.rs](/Users/letz/Developpements/RUST/faust-rs/crates/compiler/tests/signal_pipeline.rs)
  - added an end-to-end compiler regression for the reduced repro
    `*(button("play") : trigger(128))`
  - asserts that the signal pipeline now lowers the result to a `Mul` signal
    with one operand coming from the implicit input wire

#### Validation

- `cargo fmt --all`
- `cargo test -p eval --test core_eval`
- `cargo test -p compiler --test signal_pipeline`
- `cargo run -p compiler -- --dump-sig karplus.dsp`

#### Outcome

`karplus.dsp` no longer fails in `propagate` on a fake `1 != 2` sequential
composition mismatch. The pipeline now advances to the separate existing
fast-lane limitation on non-constant `SIGDELAY` amounts, which confirms the
partial-application parity bug is fixed.

### `transform` — Remove redundant recursion-load casts after proving the prepared typing invariant

Commit date: 2026-03-10

Source commit: pending

#### Context

After switching recursive FIR state to C++-style 2-slot arrays, the recursion
load paths still carried defensive casts:

- `proj(symref(...))` loaded the recursion array slot and could cast it to the
  projection result type,
- `delay1(proj(symref(...)))` did the same for the previous-slot feedback load.

For the active fast-lane recursion subset, prepared typing already guarantees
these types are equal:

- `infer_proj` returns the recursion-group slot type directly,
- recursion arrays are allocated from that same reduced slot type,
- recursive `delay1` preserves the carried sample type.

That made the emitted casts redundant on the hot recursive load path.

#### Changes

- [module.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_fir/module.rs)
  - removed the recursion-specific cast branches from `lower_proj` and the
    recursive `delay1` reuse path
  - replaced them with direct `load_table(...)` returns
  - added `debug_assert_eq!` checks documenting and guarding the typing
    invariant in debug builds

- [mod.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_fir/mod.rs)
  - strengthened recursion structural tests to assert that recursive FIR
    lowering no longer inserts cast wrappers around recursion-array accesses

#### Validation

- `cargo test -p transform`
- `cargo test -p compiler --test signal_fir_lane`

#### Outcome

Recursive FIR loads are now emitted directly without redundant casts, while the
debug assertions keep the prepared typing contract explicit and checked during
development.

### `transform` / `compiler` — Lower recursive feedback to C++-style 2-slot arrays and keep fixed delays on `fVec/iVec`

Commit date: 2026-03-10

Source commit: pending

#### Context

The fast-lane still lowered recursive feedback with a mixed state shape:

- `SIGREC` / `SIGPROJ` paths had been converted to `fRec*/iRec*[2]`,
- but `delay1(proj(symref(...)))` in the recursion path could still allocate a
  separate scalar shadow state.

That diverged from the Faust C++ compiler contract, where first-order
recursion uses a 2-slot `fRec*/iRec*` array and shifts `[1] = [0]` after the
sample, while fixed delays greater than one sample remain separate circular
buffers (`fVec*/iVec*`) addressed through `fIOTA`.

`tests/corpus/rep_38_sine_phasor.dsp` exposed the first-order case, and
`tests/corpus/rep_55_sine_phasor_echo_feedback.dsp` remains the concrete
example for recursion plus fixed delay `@(10000)`.

#### Changes

- [module.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_fir/module.rs)
  - changed `SIGREC` / `SIGPROJ` lowering to use explicit 2-slot recursion
    arrays as the canonical carrier
  - made symbolic recursion references load the current slot `[0]`
  - taught `delay1(proj(symref(...)))` in the active recursion path to reuse
    the recursion array previous slot `[1]` instead of allocating a separate
    scalar state variable
  - kept fixed `SIGDELAY(x, N)` with `N > 1` on the existing circular delay
    line path (`fVec*/iVec*` plus `fIOTA`)

- [mod.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_fir/mod.rs)
  - updated recursion tests to expect a 2-slot array declaration
  - added a structural non-regression test ensuring recursive
    `delay1(proj(symref(...)))` reuses one `fRec*[2]` array and only writes
    slots `0` and `1`

- [signal_fir_lane.rs](/Users/letz/Developpements/RUST/faust-rs/crates/compiler/tests/signal_fir_lane.rs)
  - tightened the `rep_38_sine_phasor` regression to assert the emitted
    `float[2] fRec*` update pattern
  - aligned the `rep_56_noise_smoo_slider` regression with the new integer
    array form `int[2] iRec*`

#### Validation

- `cargo fmt --all`
- `cargo test -p transform`
- `cargo test -p compiler --test signal_fir_lane`

#### Outcome

The fast-lane now matches Faust C++ more closely for recursive state:
one-sample recursion uses `fRec*/iRec*[2]`, while fixed delays greater than one
sample still lower through circular `fVec*/iVec*` buffers indexed by `fIOTA`.
