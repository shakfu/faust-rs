## Plan read-only generated table effects (Phase E0)

Commit date: 2026-07-17
Source commit: this commit

Added the implementation plan for the compute-time effect identity of
`rdtable` generated tables, after live diagnosis disproved the assumption that
scheduled `waveform5` into Phase E4.

`waveform5` stops at `FRS-VEC-FALLBACK-EVENTS` with
`vector execution reverses scalar dependence 14 -> 402`. That dependence is the
anti-dependence `ReadTable(12) @ Loop(0) sample 0 -> WriteTable(12) @ Loop(1)
sample 1`, and vector loop fission runs `Loop(1)` before `Loop(0)` and reverses
it. The write does not exist. `rdtable` and `rwtable` both lower to `SIGWRTBL`
and differ only in their write arguments, but `direct_effects` matches
`SigMatch::WrTbl(_, _, _, _)` and gives every one of them a `WriteTable` atom.
Probing the corpus separates the shapes exactly: `waveform5` signal 12 and
`table1` signal 49 carry a nil write index and nil write value, `table1`
signal 55 carries real ones. The emitted code agrees with the probe and not
with the model: `waveform5` emits `const static int iTbl12[5]` and certified
`waveform6` emits `const static int iVecTbl13[6]` in scalar and vector alike,
with no runtime table store in either. The pure vector lowerer already draws
this distinction - `lower_readonly_generator` admits a generator only when both
write arguments are nil - so the effect model and the lowerer contradict each
other on the same signal, and the lowerer is the one that matches emitted code.
Removing the atom from the nil-write arm certifies `waveform5` (86 -> 87,
`FRS-VEC-FALLBACK-EVENTS` 2 -> 1) with `cargo test -p transform --lib` at
375/375; that experiment was reverted and is not part of this commit.

`waveform5` differs from certified `waveform6` only in indexing its table with
a UI slider rather than an input signal. That index is block-invariant, which
splits the program into two loops (`loops=2 data_edges=1 effect_edges=0`) where
`waveform6` keeps one. `waveform6` is therefore not safer, only unfissionable.
Dropping the atom loses no ordering obligation because fill-before-read is
carried by the data edge from the read to its table operand, and that program
plans with no effect edges at all.

The plan records one property that separates this phase from C and D: their
independence came from a producer and a checker reconstructing the same facts
from disjoint code, but `direct_effects` is shared by both, so correcting it
moves them together and their agreement can no longer catch a mistake. Both
currently believe `rdtable` writes, so they jointly reject and the bug costs
coverage rather than correctness; the correction inverts that risk. The
independent evidence therefore has to come from the assembled FIR, and the plan
makes "no `StoreTable` to any table classified read-only" the load-bearing
obligation, to land with the effect change rather than after it.

The phase is numbered E0 rather than E4a because it precedes E1: `table1`
carries both a phantom write and a real `rwtable` write, so sizing E1's
write-effect resource before this correction would target a conflict surface
that partly does not exist. E4 keeps only `mixer`, whose reversal involves UI
writes from `component()` vumeters and needs its own diagnosis. This planning
step also marks the Phase D plan complete, qualified in `2d0a2a49`. No compiler
behavior changes here.

## Qualify general fused serial group certificates (Phase D3)

Commit date: 2026-07-17
Source commit: this commit

Phase D1-D2 (`7f9f7678`) explicitly deferred the full 16-mode corpus, native
C++, golden, compile-budget, and workspace qualification to D3. That gate had
never run: it failed on first contact in four independent ways, all
pre-existing in `7f9f7678` and none of them visible from the certificate
reasoning alone.

The coverage evidence reproduces exactly. Regenerating all sixteen
`count_vector_corpus --json` reports from the D1-D2 code and merging them
yields `corpus-baseline.json` and `certified-dspfiles.txt` byte-identical to
the uncommitted working-tree copies: 86 certified, 6 fallback, 1 error over 93
DSPs, identical in every mode, and no DSP certified at `12fcdd14` lost its
certification. Phase D's delta is 62 -> 86 (+24 DSPs); Phase C's +13 (49 -> 62)
was already committed.

Two correctness-adjacent defects were fixed. `build_fused_serial_groups`
assigned `close_effect_users` after `Candidate::default()`, which
`cargo clippy -D warnings` rejects, so the workspace gate could not have passed
against `7f9f7678`. The
`recursive_short_delay_cpp_has_one_fused_read_compute_write_loop` compiler test
still required a second physical `i0` loop for the safe pure tail; D1-D2 fuses
that tail into the single sample body, which is what the test's own name
describes. The assertion now requires exactly one `i0` loop and orders delayed
read before state write before pure tail, and the earlier expectation is
confirmed stale rather than the lowering regressed: the generated body keeps
its scalar transport, its read/write order, and its 60,000-frame C++ parity.

Two compile-cost regressions were the substantive finding. Phase C recorded a
green basket at `spectral_level` 871/3,057 ms and `cubic_distortion` 572/1,163
ms scalar/vector. The same basket on `7f9f7678` measured `spectral_level`
885/26,924 ms against a 5,424 ms ratio budget - an 8.8x vector regression with
scalar unchanged - and `cubic_distortion` 663/4,487 ms. Rebuilding `12fcdd14`
in a worktree and measuring it on the same machine under the same load
reproduced 956/3,199 ms, so the budget baseline was not simply set on a quieter
runner: the regression is real and belongs to D1-D2. Stage timing attributed it
to two stages rather than to the certificate model:

- `fused-groups` 21.179 s. The component fixpoint resolved each `start -> end`
  path by scanning all 553 loops and probing `PlanReachability::reaches` twice
  per loop, and `reaches` costs two `BTreeMap` lookups. Nested under
  `effect_users x carrier_owners x components x iterations` that reached
  455,886,564 inner iterations. Every loop on the path is reachable from
  `start` and reaches `end`, so the forward closure row of `start` intersected
  with a transposed reverse row of `end` answers the whole query in one pass
  over the bitset. The transpose is built once, O(V^2) bit tests. Two smaller
  recomputations were hoisted alongside: `loop_effects` is pure over the
  immutable placement state, and the carrier-absorption predicate is
  component-independent except for its owning-loop test.
- `fir-assembly` 0.047 s -> 2.439 s. `verify_assembled_fused_serial_groups`
  called `fir_contains` per delayed-read and per state-write signal, and each
  call walked the whole physical body allocating a fresh `BTreeSet`. The body
  is fixed per group, so its reachable set is now computed once as
  `fir_reachable` and every query is a set membership test.

Both rewrites are semantics-preserving by construction and were held to that
standard by measurement, not by argument: the generated C++ for
`spectral_level` stayed byte-identical after each step, and the full 16-mode
merge from the optimized compiler reproduces the baseline and certified list
byte-identically.

Measured on this machine, scalar/vector, against Phase C for reference:

| case | Phase C | `7f9f7678` | this commit |
| --- | --- | --- | --- |
| `cubic_distortion` | 599/1,201 ms | 577/4,487 ms | 599/1,383 ms |
| `spectral_level` | 956/3,199 ms | 885/26,924 ms | 918/4,530 ms |
| `reverb_designer` | 7,583/15,195 ms | - | 7,482/11,351 ms |

`spectral_level` keeps about 1.2 s over Phase C: that is the fused-group
analysis itself, which is new required work and now fits the ratio budget.
`reverb_designer` ends up faster than Phase C because the `fir_reachable` hoist
benefits every group.

Two observations for later work, recorded because they weaken gates rather than
code. `golden-check` compares source hashes: all 198 `tests/golden/rust/*`
cases hold a five-line `faust-rs-golden-v1` header over the `.dsp` bytes and no
generated code, so it is a corpus-integrity and no-crash gate and contributes
nothing to codegen regression coverage. The `make`-driven impulse matrix
reports success from cached `.ir` files: an initial run of all eight vector
modes exited 0 having executed zero comparisons, because outputs from an
earlier run were newer than their inputs. Both matrix runs below therefore
delete the target `.ir` files first and assert the `filesCompare` invocation
count.
