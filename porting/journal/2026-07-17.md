## Close the FIR checker gap on undeclared table accesses

Commit date: 2026-07-17
Source commit: this commit

E1.1 found `verify_fir_module` silent on `LoadTable`/`StoreTable` names no
declaration backs: mid-implementation, a vector module stored to a struct
field that was never declared or initialized - invalid C++ behind a green
report. The silence was double. `check_store_table_types` returned early for
every `Struct` access, and both table paths ran their declaration checks
inside `if let Some(entry) = self.resolve(...)`, so a failed resolution
produced nothing at all. `FIR-T03` also could not fire for struct fields
because `resolve` returns a `Void` placeholder type for them.

Both table paths now resolve through one `table_access_elem_type`: `Struct`
accesses resolve against the real `dsp_struct` field types and every other
class through the scope/global resolution. A name with no declaration anywhere
is the new `FIR-T04` error - an access no declaration backs cannot compile in
any backend, so it must fail the report rather than pass it. A declared
non-indexable name keeps the `FIR-T03` warning, now including struct fields,
and `Struct`-access stores gain the `FIR-T02` element-type check they
previously skipped entirely.

The check runs on both verified paths - the scalar `maybe_verify_fir_module`
and the vector final module - so the corpus sweep doubles as the
no-false-rejection gate: 90/93 unchanged. Regression tests pin the exact E1
shape (a `Struct` store to an undeclared `fVecMutTbl55` is an error), an
undeclared stack load, a declared struct array accepted clean, and a declared
scalar struct field warning `FIR-T03` without `FIR-T04`.

## Implement mutable table vector lowering (Phase E1.1)

Commit date: 2026-07-17
Source commit: this commit

`rwtable` is admitted to the checked vector pipeline. The admission gate gains
a mutable arm beside `readonly_table_signal`, and the table definition lowering
splits on the shared `wrtbl_is_readonly` predicate: the read-only path is
unchanged, and a live-port table becomes a DSP-struct array field
(`fVecMutTbl{id}`/`iVecMutTbl{id}`), element-wise initialization in
`instanceConstants`, and one `StoreTable` per sample in the writer's loop. Both
classes evaluate their content through one extracted `table_initializers`
(constant size, element type, SIGGEN-interpreted generator), and the `Gen`
placeholder acceptance generalizes to any owning table, since `Gen` is a
lifecycle boundary whichever class owns it. Reads consult the read-only
registry first and the mutable registry second, differing only in access.
`table1` and `table2` certify; the corpus stands at 90/93 single-mode with
`sound`, `math`, and the `subcontainer1` error remaining.

The same-sample ordering contract is carried structurally. `rwtable` reads
must observe the current sample's write, so the store cannot ride the
bargraph's end-of-loop append; it is head-inserted into the region statement
list instead, and shared-value materialization - which inserts every
definition immediately before its first use - places the write's index and
value above the store while every dependent read materializes below. The
emitted body reproduces the scalar shape exactly, including the
`((i % n) + n) % n` write-index wrap, and the native oracle confirmed
`table1` and `table2` sample-for-sample over 60,000 frames before any
qualification matrix ran.

One finding mid-implementation: after the lowering landed but before the
module wiring, the pipeline certified `table1` while emitting `compute` code
that stored to a struct field no declaration or initialization backed -
invalid C++ behind a green certificate. `verify_fir_module` does not check
`StoreTable`/`LoadTable` names against any declaration, a generic checker gap
now recorded. For tables it is closed by the new obligations:
`verify_mutable_table_attribution` derives the mutable-table set from the
emitted DSP-struct declarations alone, requires physical `StoreTable` counts
in `compute` to equal claimed direct `WriteTable` performers in both
directions, and requires `instanceConstants` to initialize every cell of every
mutable table exactly once at constant indices. E0's static-table check is
untouched and still rejects any store into a statically declared table.
Initial cell values are numeric claims the certificate does not carry; the
60,000-frame oracle matrix is their arbiter, stated in the check's contract.

Rejecting mutations cover a claimed writer with no emitted store, an emitted
store with no claimed writer, two claimed writers for one store, and an
initialization that skips a cell; E0's forged store into a read-only table is
retained, and the two-performer fission rejection continues to cover
writer/reader ordering at the event level. A corpus-independent fixture packs
one `rdtable`, one float `rwtable`, and one int `rwtable` with a recursive
generator behind one recursive index, certified and scalar/vector bit-exact
under both loop variants, all four scheduling strategies, and a chunk size
that does not divide the block; admission rejected that shape outright before
this change, so certification alone discriminates. Full 16-mode corpus,
baseline refresh, byte-identity, complete oracle matrices, and compile-budget
remain E1.2 work and are not claimed by this entry.

## Plan mutable table vector lowering (Phase E1.0)

Commit date: 2026-07-17
Source commit: this commit

Added the implementation plan for admitting `rwtable` to the checked vector
pipeline, the `FRS-VEC-FALLBACK-PURE` class (`table1`, `table2`). The
admission gate accepts table atoms only through `readonly_table_signal`, so a
live-port `WrTbl` is rejected wholesale before lowering is even attempted.

Diagnosis established the target shape from the scalar backend: the mutable
table is a DSP-struct array field initialized element-wise in
`instanceConstants` and written once per sample in `compute`, ordered before
its same-sample reads. The decisive measurement is that both cases const-fold
their generators at compile time - including `table2`'s recursive integrators,
which the SIGGEN interpreter reduces to per-element constants
(`iTbl79[0] = 1; ...`) - so no runtime init loop exists anywhere in the class
and the vector lowering can reuse the same evaluator the read-only path
already calls. Unlike delay rings, tables are absolutely addressed, so the
struct field crosses chunks with no copy-in/copy-out.

The plan discharges the two obligations earlier phases recorded against E1:
E0's `verify_readonly_table_stores` ("no store into any declared table",
sound only while every vector-module table is read-only) is replaced by a
classification derived from the emitted FIR - statically declared tables
reject stores, struct-field tables require store counts equal to claimed
direct `WriteTable` performers in both directions - and E4's per-kind carrier
argument, which explicitly excluded mutable tables, is extended: writer and
readers are direct performers whose conflicting events the certificate orders
exactly as it orders bargraph writes. The corpus-independent fixture cannot be
vacuous the way E4's was dropped for being: admission rejects the shape
outright today, so certification alone discriminates. No compiler behavior
changes in this planning step.

## Implement direct effect event attribution (Phases E4.1-E4.2)

Commit date: 2026-07-17
Source commit: this commit

Effect events are now attributed to the signal that performs the effect, not to
every transitive carrier of it. E4.1 carries the existing projection through:
`SignalUses::direct_effects` becomes public, `DecorationRecord` and the plan
`signalRecord` gain the field, the certificate moves v3 -> v4
(`porting/schemas/vector-verification-certificate-v4.schema.json`), and both
checkers reject a `direct_effects` that is unsorted or not a subset of
`effects`. E4.1 alone changes no certification outcome; the corpus was verified
unchanged at 87/5/1 with emission still reading the accumulated set.

E4.2 flips both event emission sites - the producer templates and the
independent checker reconstruction - to the direct projection. The
event-count precheck flips with them: its lower-bound property relied on a root
emitting its whole transitive set, so counting transitive atoms against direct
emission would have turned the bound into a spurious rejection.
`verify_required_dependencies` deliberately stays on the accumulated set:
demanding that a carrier's definition be ordered after the state reads it
inherits is a strictly stronger obligation, fail-closed, and corpus-green.
`duplicable` also keeps reading the accumulated set, which is what keeps
performers unique.

The per-kind survival argument for removed carrier events: unmanaged state,
table, UI, and output atoms keep their performer events, and a carrier is
ordered after its performer by the data and execution-condition edges the
accumulated fixed point propagated along, which effect removal does not touch.
Managed state was already filtered from events. Mutable tables and foreign
calls do not reach checked vector modules today, so E1 and E3 must make their
own argument when they admit those classes.

Producer and checker both derive the projection from one `direct_effects`
function, so their agreement proves nothing about it. The independent evidence
reads the emitted FIR: `verify_ui_write_attribution` counts claimed `WriteUi`
performers per control against physical `StoreVar` zone stores in the assembled
`compute` body, in both directions - a carrier promoted to performer is
rejected because no store backs it, and a performer demoted to carrier is
rejected because a store has no claimed writer. FIR nodes are interned, so a
body duplicated across chunk drivers contributes one node per physical store.
This is the diagnosed kind; read-only tables are already store-free under
`verify_readonly_table_stores` and outputs under output-store coverage.

`mixer` certifies, eliminating `FRS-VEC-FALLBACK-EVENTS` entirely: 88/93 with
the remaining fallbacks in three classes - mutable tables (`table1`,
`table2`), soundfile (`sound`), foreign functions (`math`), plus the
`subcontainer1` transform error. Its event table falls from 1,702 to 1,222
events, the 480 removed being exactly the carrier effect events, and the
certificate builds in 5 ms.

One honest omission: the plan's corpus-independent P2 fixture could not be made
meaningful. A mixer-shaped program at 4 and 8 loops certifies even with the old
attribution - fission at that size never lands two carriers in reversing
loops - so a committed test would pass without the fix and was dropped as
vacuous. The corpus `mixer` case is the regression instance, and E4.3 pins it
with the native oracle matrix. The carrier/performer distinction is instead
covered at the unit level: the pre-existing two-performer rejection keeps its
`FissionSafeViolation`, and the new mirror fixture with one performer and one
carrier is accepted with the effect event attributed to the performer alone.

Full 16-mode corpus, coverage baseline refresh, byte-identity against the
pre-change compiler, the `mixer` oracle matrix, and compile-budget remain E4.3
work and are not claimed by this entry.

## Plan direct effect event attribution (Phase E4)

Commit date: 2026-07-17
Source commit: this commit

Added the implementation plan for which signal an effect event is attributed
to, after diagnosing the last `FRS-VEC-FALLBACK-EVENTS` case.

`mixer` stops at `vector execution reverses scalar dependence 155 -> 747`. Both
events are `WriteUi(2)`: one in `Loop(0)` at sample 0 for signal 1661, one in
`Loop(18)` at sample 1 for signal 1657. The model reports a write-after-write on
one UI zone and vector fission reverses it. Control 2 has one writer. Probing
the effect analysis reports one direct writer of `WriteUi(2)`, signal 1514, and
38 transitive carriers; neither 1661 nor 1657 is 1514, and both merely contain
the vumeter in their subtree. The emitted code agrees: `mixer` declares 10
distinct bargraph zones and stores each exactly once, so `fVbargraph2` has
nothing to reorder against.

The cause is attribution rather than a wrong atom, which is what separates this
from E0. `independently_expected_event_keys` builds the event table from
`plan.signals[].effects`, the conservative transitive projection documented as
"including non-`Gen` children", so every materialized carrier of an effect emits
an event for it. Two carriers in different loops then manufacture a conflict on
an operation that happens once; `mixer` has 48 loops and 38 carriers of one zone
and finds one. Like E0 this over-reports operations and is fail-closed: it costs
coverage, not correctness.

`SignalUses` already carries the needed projection. Its `direct_effects` field
is documented as "effects performed by this node itself, excluding child
effects", existing so "scalar scheduling [can] orient actual effect operations".
Scalar scheduling therefore already orients on real operations while the vector
event model orients on carriers. The field is private and reaches neither
`DecorationRecord` nor the plan, so the phase carries an existing projection
through rather than adding analysis. The accumulated set stays: `duplicable`
legitimately needs it, and it is what keeps performers unique.

Unlike E0, no experiment proves the fix. Plumbing the projection through the
decoration record, the plan signalRecord, and a v3 -> v4 certificate is too
large for a throwaway probe, so this entry claims a diagnosis, not a proven
correction.

The plan records two obligations it must not assume. Removing carrier events
removes ordering constraints, so the argument that a carrier still executes
after the performer it inherits from - via the data or execution-condition path
the accumulated fixed point propagated along - has to be established per effect
kind before the attribution flips, exactly as E0 established that
fill-before-read survived as a data edge. And the producer and checker would
both call the same `direct_effects`, so their agreement proves nothing about the
projection; the independent evidence has to come from the assembled FIR, by
requiring that a claimed direct performer in a loop matches a physical operation
on that resource in that loop, and that the per-sample operation counts match.
`mixer` fails today precisely because 38 claimed operations correspond to one
physical store. No compiler behavior changes in this planning step.

Added the implementation plan for the compute-time effect identity of
`rdtable` generated tables, after live diagnosis disproved the assumption that
scheduled `waveform5` into Phase E4.

`waveform5` stops at `FRS-VEC-FALLBACK-EVENTS` with
`vector execution reverses scalar dependence 14 -> 402`. That dependence is the

`SigMatch::WrTbl(_, _, _, _)` and gives every one of them a `WriteTable` atom.

with the model: `waveform5` emits `const static int iTbl12[5]` and certified

this distinction - `lower_readonly_generator` admits a generator only when both
write arguments are nil - so the effect model and the lowerer contradict each
other on the same signal, and the lowerer is the one that matches emitted code.
Removing the atom from the nil-write arm certifies `waveform5` (86 -> 87,
`FRS-VEC-FALLBACK-EVENTS` 2 -> 1) with `cargo test -p transform --lib` at

`waveform5` differs from certified `waveform6` only in indexing its table with
a UI slider rather than an input signal. That index is block-invariant, which
splits the program into two loops (`loops=2 data_edges=1 effect_edges=0`) where
`waveform6` keeps one. `waveform6` is therefore not safer, only unfissionable.
Dropping the atom loses no ordering obligation because fill-before-read is
carried by the data edge from the read to its table operand, and that program
plans with no effect edges at all.

The plan records one property that separates this phase from C and D: their
independence came from a producer and a checker reconstructing the same facts
from disjoint code, but `direct_effects` is shared by both, so correcting it
moves them together and their agreement can no longer catch a mistake. Both
currently believe `rdtable` writes, so they jointly reject and the bug costs
coverage rather than correctness; the correction inverts that risk. The
independent evidence therefore has to come from the assembled FIR, and the plan
makes "no `StoreTable` to any table classified read-only" the load-bearing
obligation, to land with the effect change rather than after it.

The phase is numbered E0 rather than E4a because it precedes E1: `table1`
carries both a phantom write and a real `rwtable` write, so sizing E1's
write-effect resource before this correction would target a conflict surface

writes from `component()` vumeters and needs its own diagnosis. This planning
step also marks the Phase D plan complete, qualified in `2d0a2a49`. No compiler
behavior changes here.

## Qualify general fused serial group certificates (Phase D3)

Commit date: 2026-07-17
Source commit: this commit

Phase D1-D2 (`7f9f7678`) explicitly deferred the full 16-mode corpus, native
C++, golden, compile-budget, and workspace qualification to D3. That gate had
never run: it failed on first contact in four independent ways, all
pre-existing in `7f9f7678` and none of them visible from the certificate
reasoning alone.

The coverage evidence reproduces exactly. Regenerating all sixteen
`count_vector_corpus --json` reports from the D1-D2 code and merging them
yields `corpus-baseline.json` and `certified-dspfiles.txt` byte-identical to
the uncommitted working-tree copies: 86 certified, 6 fallback, 1 error over 93
DSPs, identical in every mode, and no DSP certified at `12fcdd14` lost its
certification. Phase D's delta is 62 -> 86 (+24 DSPs); Phase C's +13 (49 -> 62)
was already committed.

Two correctness-adjacent defects were fixed. `build_fused_serial_groups`
assigned `close_effect_users` after `Candidate::default()`, which
`cargo clippy -D warnings` rejects, so the workspace gate could not have passed
against `7f9f7678`. The
`recursive_short_delay_cpp_has_one_fused_read_compute_write_loop` compiler test
still required a second physical `i0` loop for the safe pure tail; D1-D2 fuses
that tail into the single sample body, which is what the test's own name
describes. The assertion now requires exactly one `i0` loop and orders delayed
read before state write before pure tail, and the earlier expectation is
confirmed stale rather than the lowering regressed: the generated body keeps
its scalar transport, its read/write order, and its 60,000-frame C++ parity.

Two compile-cost regressions were the substantive finding. Phase C recorded a
green basket at `spectral_level` 871/3,057 ms and `cubic_distortion` 572/1,163
ms scalar/vector. The same basket on `7f9f7678` measured `spectral_level`
885/26,924 ms against a 5,424 ms ratio budget - an 8.8x vector regression with
scalar unchanged - and `cubic_distortion` 663/4,487 ms. Rebuilding `12fcdd14`
in a worktree and measuring it on the same machine under the same load
reproduced 956/3,199 ms, so the budget baseline was not simply set on a quieter
runner: the regression is real and belongs to D1-D2. Stage timing attributed it
to two stages rather than to the certificate model:

- `fused-groups` 21.179 s. The component fixpoint resolved each `start -> end`
  path by scanning all 553 loops and probing `PlanReachability::reaches` twice
  per loop, and `reaches` costs two `BTreeMap` lookups. Nested under
  `effect_users x carrier_owners x components x iterations` that reached
  455,886,564 inner iterations. Every loop on the path is reachable from
  `start` and reaches `end`, so the forward closure row of `start` intersected
  with a transposed reverse row of `end` answers the whole query in one pass
  over the bitset. The transpose is built once, O(V^2) bit tests. Two smaller
  recomputations were hoisted alongside: `loop_effects` is pure over the
  immutable placement state, and the carrier-absorption predicate is
  component-independent except for its owning-loop test.
- `fir-assembly` 0.047 s -> 2.439 s. `verify_assembled_fused_serial_groups`
  called `fir_contains` per delayed-read and per state-write signal, and each
  call walked the whole physical body allocating a fresh `BTreeSet`. The body
  is fixed per group, so its reachable set is now computed once as
  `fir_reachable` and every query is a set membership test.

standard by measurement, not by argument: the generated C++ for
`spectral_level` stayed byte-identical after each step, and the full 16-mode
merge from the optimized compiler reproduces the baseline and certified list
byte-identically.
