## Adapt `ma.SR` across multirate clock domains

Commit date: 2026-07-18
Source commit: this commit

Ported the C++ `propagate.cpp` foreign-constant rule for sampling rate. While
propagating `fSamplingFreq` or `fSamplingRate` inside a clocked body, Rust now
walks the active `ClockDomainTable` parent chain, multiplies the runtime sample
rate by each `upsampling` clock, divides it by each `downsampling` clock, and
leaves it unchanged across `ondemand`. Nested domains compose from the
innermost domain to the outermost, matching the C++ clock-environment stack.

This is an adapted internal representation of the C++ structural clock
environment: Rust follows typed side-table ids instead of decoding a cons
tree. It changes no public Rust, CLI, C, or C++ API. Runtime regressions assert
that a 48 kHz instance observes 144 kHz under `US(3)`, 16 kHz under `DS(3)`,
and 32 kHz under nested `US(2)`/`DS(3)`. All three cases also match the pinned
C++ clock-domain reference sample-for-sample.

## Canonicalize Cranelift factory identity across FIR allocations

Commit date: 2026-07-18
Source commit: this commit

Fixed the Cranelift FFI cache-identity mismatch between source, boxes, and
signals constructors. The shared factory builder used the diagnostic FIR dump
as its semantic fingerprint; that dump intentionally exposes arena-local
`TreeId` values, so semantically identical FIR modules built through different
paths could receive different SHA keys. The first failing assertion also
poisoned the crate's shared test mutex and caused the remaining failures to
cascade.

The cache identity is the stable key for a compiled factory exposed through
the Cranelift C/C++ FFI. It lets the process-wide factory registry deduplicate
equivalent compilations and retrieve an existing factory through
`getCCraneliftDSPFactoryFromSHAKey`; it also binds a serialized source-backed
bitcode payload to the factory rebuilt from that payload. Its inputs therefore
cover every compilation choice that can change factory behavior: canonical FIR
semantics, optimization level, canonicalized compiler options, and the
registered foreign-function set. The intended contract is “same DSP semantics
and compilation environment, same key”; changing either must change the key.
This identity does not affect DSP samples or generated machine-code behavior
directly, but an unstable key defeats cache deduplication and constructor
equivalence, while an insufficiently discriminating key could return the wrong
factory or accept a mismatched serialized payload.

The `fir` crate now provides a canonical structural fingerprint that traverses
the complete reachable tree, assigns local preorder labels, writes interned tag
names instead of numeric tag ids, and preserves observable sharing. Unreachable
allocation history and arena-local ids cannot affect this encoding. Diagnostic
`dump_fir` output remains unchanged. The Cranelift common factory builder owns
fingerprint generation so all constructors necessarily use the same rule.

This public Rust API is an adapted internal mapping: there is no corresponding
C++ fingerprint entry point, but it implements the stable factory/cache
identity contract used by the C++-compatible Cranelift FFI constructors.
Compatibility impact: newly created factories have allocation-independent
keys, while source-backed V1/V2 bitcode carrying an exact legacy dump-based SHA
is still accepted and retains that serialized identity.

Added structural regression coverage for equivalent FIR stores with different
allocation histories, semantic-change discrimination, source/boxes/signals SHA
parity, and legacy source-backed SHA restoration.

## Trace the vector-plan contexts divergence (Phase X1)

Commit date: 2026-07-18
Source commit: this commit

Traced classes B and C of the contexts divergence to their mechanisms and
corrected the class A diagnostic. All three classes concentrate on symbolic
multi-projection recursion groups, and the unified statement sharpens: the
certificate's dependency edges carry three semantics - value use,
scheduling-only, and delayed - and the plan flattens all three into one
context/transport model, each wrongly.

**B is a `Delayed` edge nobody serves.** On the one-line reproducer
`process = pm.ks(200, 0.5)`, consumer `SIGPROJ(0, SYMREF(W1))` in loop 1
depends `Delayed { amount: 1 }` on `SIGPROJ(1, SYMREC(W2))` in loop 0 - and
that target's decoration record carries `max_delay = 0, delay_reads = 0`. The
certificate is self-inconsistent: delay facts and dependency edges come from
different projections that disagree on cross-group symbolic reads, and no
checker states the obligation that a delayed edge target a record with delay
facts. Mechanically nobody serves the read - transports derive only from
`delay == 0` occurrences and `DelayTransition`s only from `max_delay > 0`
records - so the lowering resolves the raw value cross-loop (backtraced
through `lower_raw -> resolve_in_loop`) and fails `MissingTransport`.
Synthetic stdlib-free reductions with cross-group delayed reads certify; the
karplus termination/chain structure resists reduction, so the one-liner is
the regression instance.

**C is a scheduling-only edge transported as a value**, reproduced stdlib-free
in one line: `attach(_, aOut)` where `aOut` projects a two-projection
recursion group. `attach(x, y)` only forces `y`'s computation, but the plan
flattens the `Immediate` edge into `cross_uses` and plans a value transport;
the `SIGATTACH` lowering discards the attached value, so the consumer body
never loads it and the body check rejects - correctly, which is why the class
fails closed rather than miscompiling.

The class A diagnostic no longer lies: the two failure sites inside
`add_dependency_edges` raised `MissingRecord` ("dependency names missing
signal N") when the record exists and only the execution context is missing.
They now raise a dedicated `MissingContext` whose message states exactly
that - "placement traversal never reached signal N: it has a record but no
execution context" - verified end-to-end on `clarinetMIDI.dsp`. The
`MissingRecord` sites that really do look up records keep their variant. No
other behavior change; the corpus stays at 91/93.

## Plan the vector-plan contexts divergence fix (Phase X0)

Commit date: 2026-07-18
Source commit: this commit

Diagnosed why `-vec` silently produces scalar code for `clarinetMIDI.dsp` and
added the correction plan. The command exits 0 and `-time` prints
`fir-vector-certification` as usual, but the emitted C++ contains no `vindex`:
the pipeline fell back to scalar without saying so. The reason, read from
`vector_pipeline_detail` which the `faust-rs` binary discards, is
`Fallback(VectorPlan)`: `vector-plan dependency names missing signal 2196`.

Measured against the external 197-DSP directory rather than the corpus: 159
vectorize, 34 fall back, 4 fail to compile. **24 of the 34 are one bug** under
three manifestations, and the 93-DSP corpus - at 91/93 - exercises none of
them. All three read or write `contexts` in `build_vector_plan`, the plan's
model of which loops execute which signal: transports derive from it while the
lowering materializes by walking the expression tree, so each direction of
disagreement surfaces differently. Class A (11 files, MIDI physical models):
context absent. Class B (8 files, Karplus family): context too narrow, so no
transport is planned for a use the lowering resolves. Class C (5 files,
virtual-analog family): context too wide, so a planned transport is never
loaded.

Class A is traced to its mechanism. `contexts` is populated only inside
`visit`, which returns at a structural carrier without recursing into its
children - the opposite of the module's stated contract that "their executable
children, rather than the containers, enter the sample closure". Separately,
`Owned` placements are pre-seeded from the `owner` map without touching
`contexts`. A sample-required signal reachable only through a structural
carrier therefore carries a placement and no context, and
`add_dependency_edges`, which iterates every certificate dependency rather
than only the traversed ones, trips on it. On `clarinetMIDI` the pruned
traversal covers 256 of 305 signals; signal 2196's only parents are 2202
(structural) and 2205 (not itself reached). The diagnostic is also wrong: the
record exists, only the context is missing.

Classes B and C are characterized but not traced to a line, and the plan says
so. Two negative results were worth the time: C is **not** the typed-walker
blindness fixed for soundfiles in E2, since a generic child enumeration does
not find the load either, and it is **not** a state-plan consumption, since
the signal appears in no prefix, delay, recursion, or waveform transition of
the consumer loop. Both classes involve non-duplicable `Scal` signals carrying
38 and 4 effects respectively - the region where duplicability and effects
decide materialization.

The plan keeps the existing routing and body checks as the obligations that
already make B and C fail closed rather than miscompile, adds the missing
placement/context agreement check that class A's accidental failure stands in
for, and defers the choice of correction shape to a first step that must trace
B and C before any fix is designed. It also records that the external
directory cannot become a CI gate as it stands, so promoting one
representative of each affected family into the committed corpus is the
durable protection. No compiler behavior change in this planning step; all
diagnosis probes were reverted and the corpus re-verified at 91/93.

## Implement soundfile vector lowering (Phase E2.1)

Commit date: 2026-07-18
Source commit: this commit

Soundfile data reads are admitted to the checked vector pipeline. The early
guard is removed and four P5.2 arms mirror the scalar template: the handle
loads its `Sound` struct field through the kind-checked zone path the sliders
use, and length, rate, and buffer reads emit the existing
`LoadSoundfile*` FIR on the zone name resolved from their `SIGSOUNDFILE`
operand. `sound.dsp` certifies; single-mode coverage reaches 91/93 with
`FRS-VEC-FALLBACK-UI` eliminated, and spot oracle runs at `-lv 0 -ss 0` and
`-lv 1 -ss 3` match the 60,000-frame reference.

Two real boundaries had to learn the handle type, and both fixes follow the
shape-derivation precedent the prepared boundary set:

- The plan vocabulary had no way to say `Sound`, so the plan/prepared type
  check rejected the handle as `planned Int`. `build_vector_plan` is
  deliberately certificate-pure, so the fact belongs to the decoration
  certificate: `CanonicalSigType::Sound` is now derived from the signal shape
  symmetrically by the producer and the independent checker - exactly as
  `derive_simple_types` derives `SimpleSigType::Sound` - and `ValueType::Sound`
  flows through plan, state, route, lower, and assemble. A handle can never be
  a delay or transport payload; those arms stay hard errors. The v4 schema's
  `signalType` vocabulary gains `sound` in place: the schema was born in this
  stream days ago and has no external consumer yet.
- The routed-body verifier reported the index transport "absent from its
  consumer body": `fir_children` and `fir_reachable` had no
  `LoadSoundfile*` arms, so a transport load buried in a buffer read's index
  operand was invisible to the traversal - the same blind-walker family as
  the FIR checker's table-name gap. Both walkers now descend soundfile
  operands.

The new obligation `verify_sound_field_immutability` rejects any compute-time
`StoreVar` into a `Sound`-typed DSP-struct field, deriving the field set from
the emitted struct declarations as the table checks do, with a forged store as
its rejecting mutation. A corpus-independent fixture pins the `sound.dsp`
shape with a live length read, certified across both loop variants and all
four strategies; certification alone discriminates since the guard rejected
every soundfile program outright. Numeric coverage belongs to the native
matrix - the interpreter lane does not run soundfiles - and full 16-mode
corpus, baseline refresh, byte-identity, the complete oracle matrix, and
compile-budget remain E2.2 work, not claimed by this entry.
