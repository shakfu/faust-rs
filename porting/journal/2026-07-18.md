# 2026-07-18

## Canonicalize pure FIR `Drop` scaffolding before backend lowering

Commit date: 2026-07-18
Source commit: this commit

Moved elimination of pure `Drop` roots from individual backend emitters to a
mandatory FIR canonicalization pass at the signal-to-FIR boundary. The pass
rebuilds the verified module without root drops whose values are structurally
effect-free (constants, loads, casts, selects, and pure arithmetic), while
retaining calls and unknown nodes conservatively. It runs even with
`--no-fir-verify`, so every compiler-produced FIR artifact supplied to C/C++,
AssemblyScript, Julia, Wasm, Cranelift, and the interpreter has the same
canonical shape.

Removed the duplicated backend purity predicate and its backend-local elision
branches. Direct backend APIs still faithfully emit manually constructed raw
FIR; canonical compiler output is the contract used by normal compilation.

Validated with the full `fir`, `codegen`, and `compiler` unit suites plus
Clippy across their targets.

## Align Rust backend runtime contract and validate against Faust C++ architectures

Commit date: 2026-07-18
Source commit: this commit

Aligned the module-first Rust emitter with the public Rust contract emitted by
Faust C++: host-supplied `F32`/`F64`/`FaustFloat`, `ParamIndex`, `UI`, and
`FaustDsp`; C++ lifecycle names and ordering; index-based parameter access;

local mutability from FIR stores, eliminating `unused_mut` diagnostics without
changing generated semantics. Soundfile UI is supported as a documented
faust-rs extension through `ParamIndex` while scalar parameter accessors skip
the non-scalar soundfile fields; the impulse fixture supplies the standard
sinusoidal test soundfile.

Regenerated and migrated `examples/rust/dx7-piano` to the same `ParamIndex` /
`set_param` contract.

## Implement soundfile vector lowering (Phase E2.1)

Commit date: 2026-07-18
Source commit: this commit

Soundfile data reads are admitted to the checked vector pipeline. The early
guard is removed and four P5.2 arms mirror the scalar template: the handle
loads its `Sound` struct field through the kind-checked zone path the sliders
use, and length, rate, and buffer reads emit the existing
`LoadSoundfile*` FIR on the zone name resolved from their `SIGSOUNDFILE`
operand. `sound.dsp` certifies; single-mode coverage reaches 91/93 with
`FRS-VEC-FALLBACK-UI` eliminated, and spot oracle runs at `-lv 0 -ss 0` and
`-lv 1 -ss 3` match the 60,000-frame reference.

Two real boundaries had to learn the handle type, and both fixes follow the
shape-derivation precedent the prepared boundary set:

- The plan vocabulary had no way to say `Sound`, so the plan/prepared type
  check rejected the handle as `planned Int`. `build_vector_plan` is
  deliberately certificate-pure, so the fact belongs to the decoration
  certificate: `CanonicalSigType::Sound` is now derived from the signal shape
  symmetrically by the producer and the independent checker - exactly as
  `derive_simple_types` derives `SimpleSigType::Sound` - and `ValueType::Sound`
  flows through plan, state, route, lower, and assemble. A handle can never be
  a delay or transport payload; those arms stay hard errors. The v4 schema's
  `signalType` vocabulary gains `sound` in place: the schema was born in this
  stream days ago and has no external consumer yet.
- The routed-body verifier reported the index transport "absent from its
  consumer body": `fir_children` and `fir_reachable` had no
  `LoadSoundfile*` arms, so a transport load buried in a buffer read's index
  operand was invisible to the traversal - the same blind-walker family as
  the FIR checker's table-name gap. Both walkers now descend soundfile
  operands.

The new obligation `verify_sound_field_immutability` rejects any compute-time
`StoreVar` into a `Sound`-typed DSP-struct field, deriving the field set from
the emitted struct declarations as the table checks do, with a forged store as
its rejecting mutation. A corpus-independent fixture pins the `sound.dsp`
shape with a live length read, certified across both loop variants and all
four strategies; certification alone discriminates since the guard rejected
every soundfile program outright. Numeric coverage belongs to the native
matrix - the interpreter lane does not run soundfiles - and full 16-mode
corpus, baseline refresh, byte-identity, the complete oracle matrix, and
compile-budget remain E2.2 work, not claimed by this entry.

## Plan soundfile vector lowering (Phase E2.0)

Commit date: 2026-07-18
Source commit: this commit

Added the implementation plan for soundfile data reads in the checked vector
pipeline, the `FRS-VEC-FALLBACK-UI` class (`sound.dsp`), rejected today by an
explicit guard before any vector stage runs.

Diagnosis found the phase unusually contained: the effect model already treats
soundfile signals as pure reads - correctly, since compute never writes
soundfile state - so unlike E0 no model correction is needed and the
read-only-table ordering argument carries over unchanged. The FIR vocabulary,
the scalar lowering template in `module/ui_lowering.rs`, the vector UI
lifecycle (the `Sound` struct field and `addSoundfile` registration, pinned by
exact block equality), and the `FIR-SF01` field check all exist. The scalar
C++ harness passes `sound.dsp` against its 60,000-frame reference today, so
the vector oracle matrix inherits a real numeric arbiter. What is missing is
exactly the guard removal and four P5.2 arms mirroring the scalar template.

The one new obligation is read-only evidence in the emitted FIR: no `StoreVar`
into any `Sound`-typed DSP-struct field anywhere in the assembled compute
body, with the field set derived from the emitted struct declarations as the
table checks derive theirs, and a forged store as its rejecting mutation. No
compiler behavior change in this planning step.
