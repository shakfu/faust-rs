#### Context

to a structural parser/import mismatch:

- Rust `parse_file_with_imports(...)` still expands imported source text before

rejected as the final approach because it weakens the rule instead of restoring
the C++ boundary.

   `porting/parser-import-format-definitions-cpp-parity-plan-2026-03-19-en.md`.
3. Expanded the plan beyond the initial note so it now covers:
   - the exact Rust/C++ boundary mismatch,
   - target architecture options,

   - risks/watchpoints,
   - validation and success criteria,
   - explicit removal of parser-side workaround logic.

### `eval`: align `seq(i,0,body)` with C++ `neutralExpSeq`

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/eval/src/lib.rs`, `crates/eval/tests/core_eval.rs`

#### Context

`mth_octave_filterbank_alt_test` failed in `propagate` with a false sequential
composition mismatch `2 != 1`.

The root cause was earlier, in `eval`:
- Rust expanded `iseq(i,0,body)` to the universal empty route
  `route(0,0,par(0,0))`,
- C++ does not do that for `seq`,
- `compiler/evaluate/eval.cpp::neutralExpSeq(...)` evaluates `body` once with
  `i = 0`, lowers it through `a2sb`, and if `ins == outs` returns an identity
  bus `_, ..., _` of matching width.

This matters for filterbank helpers such as:
- `apchain(i) = seq(j, N-1-i, ap(j+1))`

When `N-1-i == 0`, Rust incorrectly dropped the chain to `0 -> 0`, while C++
kept the expected `1 -> 1` identity. That shrank one branch inside
`mth_octave_filterbank_alt` and later surfaced as a bogus `propagate`
composition error.

#### Changes

1. Added `neutral_seq_body(...)` in `eval` as the Rust port of C++
   `neutralExpSeq(...)`.
2. Updated `iterate_seq(...)` so the zero-iteration case now:
   - evaluates `body` with `i = 0`,
   - lowers the result with `a2sb(...)`,
   - infers local arity,
   - returns an identity bus of width `outs` when `ins == outs`,
   - only falls back to the empty `route(0,0,par(0,0))` for a true `0 -> 0`
     body.
3. Added a non-regression covering the sibling-helper reproducer that exposed
   the bug before the full filterbank case.

### `eval`: memoize `eval(expr, env)` like C++ and bound matcher-state caching

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/eval/src/lib.rs`, `crates/eval/tests/core_eval.rs`, `crates/propagate/src/lib.rs`, `crates/sigtype/src/rules.rs`, `porting/MEMOIZATION.md`

#### Context

`jprev_demo_test` no longer failed structurally after the waveform fix, but
Rust still spent tens of seconds inside `eval` while the C++ compiler finished
in well under one second.

Profiling reduced the hotspot to evaluator reduction itself:
- `parse` completed quickly,
- `eval` kept revisiting the same higher-order box subgraphs in `jpverb`,
- C++ `eval.cpp` caches `eval(exp, env)` through `getEvalProperty(...)` /
  `setEvalProperty(...)`,
- Rust had no equivalent cache and only memoized `a2sb(...)` and case-rule
  automata.

Porting that cache directly exposed one representational difference:
- C++ stores immutable tree values in the cache,
- Rust stores host-side `EvalValue`,
- `EvalValue::PatternMatcher` still carries mutable per-rule environments,
  so caching it verbatim caused incorrect `PatternMatchFailed` reuse.

#### Changes

1. Added `LoopDetector.eval_cache` keyed by `(TreeId, EnvFrameKey)` and taught
   `EnvFrameKey` to include the source-context identity alongside the lexical
   environment/store identity.
2. Split `eval_value(...)` into a memoized wrapper and an uncached core,
   mirroring C++ `eval(...)` vs `realeval(...)`.
3. Restricted cacheable results to `EvalValue::Box` and `EvalValue::Closure`;
   `EvalValue::PatternMatcher` is intentionally excluded until its state can be
   represented immutably enough for C++-style property caching.
4. Documented this new cache and its current pattern-matcher limitation in
   `porting/MEMOIZATION.md`.
5. Kept the flat-box validation optimization in `propagate`:
   `try_build_flat_box(...)` now walks one shared DAG with a visitation set
   instead of recursively re-validating identical subtrees.
6. Adjusted one stale `eval` arity regression expectation and fixed two
   `clippy` issues in recursive `sigtype` convergence code so the workspace
   gates remain green.

### `sigtype`/`interval`: stabilize recursive canonical typing and restore `cubic_distortion.dsp`

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/sigtype/src/types.rs`, `crates/sigtype/src/rules.rs`, `crates/interval/src/lib.rs`, `crates/interval/src/ops/arithmetic.rs`, `crates/interval/src/ops/logic.rs`, `crates/interval/src/ops/trig.rs`

#### Context

After the first structural `sigtype` refactor steps, the remaining regressions
were no longer promotion bugs but failures in the canonical typing core:
- interval precision arithmetic could overflow on large recursive products and
  shifts,
- recursive typing could fail to converge because Rust `SigType` equality was
  still stricter than the C++ `operator==(Type, Type)` semantics,
- and even after recursive convergence, the canonical annotation map could
  remain incomplete for reachable projection nodes, which then broke promotion
  consumers such as `cubic_distortion.dsp`.

This step finishes the parity-critical part of the refactor by making the
recursive solver stable enough for the current fast-lane and by aligning the
type equality model with the C++ compiler.

#### Changes

1. Added saturating helpers for interval precision arithmetic and used them in
   arithmetic, logic, and trig interval operations so recursive typing no
   longer panics on precision overflow in Rust.
2. Fixed a parity bug in interval right-shift precision tracking:
   Rust now follows the C++ rule and uses the upper shift bound for `rsh`
   precision updates.
3. Realigned `SigType` equality with the C++ `operator==(Type, Type)` rules:
   - `SimpleType` equality now ignores `res` and interval `lsb`,
   - `TableType` equality compares only `content`,
   - `TupletType` equality compares only `components`.
4. Updated the recursive solve loop so convergence is tested on the widened
   approximation that is fed back into the next iteration, instead of the raw
   body re-inference value. This is a Rust value-semantics adaptation of the
   C++ recursive driver, not a 1:1 control-flow port.
5. Added a reachable-node population pass after recursive solving so the final
   canonical `SigId -> SigType` map is complete for all reachable nodes,
   including projections that can otherwise be skipped by memoized recursive
   inference.
6. Added focused regression tests covering:
   - saturating interval multiplication precision,
   - `rsh` precision parity,
   - C++-style `SigType` equality semantics,
   - recursive multiplication convergence and projection-type population.

#### Limitation

The Rust recursive typing driver is now substantially closer to the C++
semantics, but it is still not a structural 1:1 port of
`typeAnnotation(...)` / `updateRecTypes(...)`.

Current adapted mechanism:
- explicit recursive-group discovery and global solve phase,
- value-based `SigType` maps instead of pointer-attached mutable type cells,
- convergence judged on widened approximations re-injected into the next
  iteration,
- explicit reachable-type population after solving to restore a complete
  canonical map.

This limitation remains documented in Rustdoc because it matters for future
maintenance and for remaining parity work.

### `sigtype`: switch `annotate()` to a global recursive-group driver

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/sigtype/src/rules.rs`

#### Context

The recursive parity plan cannot be reached while recursion is still solved
locally inside `infer_sym_rec(...)`. The next step was therefore to move the
active path toward the C++ control structure:
- solve recursive groups globally first,
- then type bodies and outputs under those solved approximations.

#### Changes

1. Switched `TypeAnnotator::annotate(...)` to:
   - discover symbolic recursive groups first,
   - build a `RecTypingState`,
   - solve recursive approximations before ordinary output typing,
   - then seed the solved recursive groups back into the main annotation pass.
2. Added an explicit `update_rec_types(...)` helper in the Rust driver, mirroring
   the C++ `updateRecTypes(...)` structure:
   - seed all recursive groups up front,

### `sigtype`/`normalize`/`transform`: make canonical typing the single fast-lane source and realign recursive inference

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/sigtype/src/rules.rs`, `crates/normalize/src/normalform.rs`, `crates/transform/src/signal_prepare.rs`, `crates/fir/src/checker.rs`

#### Context

`goertzelOpt_test` exposed two coupled issues in the fast-lane:
- promotion lived in `transform::signal_prepare` instead of the canonical
  normalization/type pipeline,
- `signal_prepare` still ran its own reduced recursive typer/fixpoint, which
  diverged from the newer `sigtype`-based path and caused contradictory
  integer/real recursion decisions.

Subsequent regressions on `signal_fir_lane` showed that simply switching to
canonical `SigType` was not enough: the Rust `sigtype` recursion port still
used an oversimplified recursive algorithm that did not match the C++ `TREC`
seed semantics and did not re-evaluate memoised body subtrees under updated
recursive approximations.

#### Changes

1. Moved the detailed fast-lane promotion pass from `transform` into
   `normalize`, so the active preparation path now consumes canonical
   `sigtype::TypeAnnotator` results before lowering.
2. Simplified `transform::signal_prepare` so it derives its reduced
   `SimpleSigType` view from canonical `SigType` instead of running a second
   local recursive typer.
3. Realigned recursive `sigtype` inference with the C++ intent:
   - recursive groups are now seeded from the C++-style `TREC` approximation
     instead of a single maximal scalar,
   - recursive symbol references read the current approximation attached to the
     recursion variable,
   - memoised recursive body subtrees are invalidated between fixpoint steps so
     the next iteration is re-inferred under the updated approximation,
   - recursive interval updates now mirror the default C++ widening behaviour:
     after recomputing one recursive body, the new interval is first reunited
     with the previous approximation and any drifting bound is widened
     immediately to the `TRECMAX` carrier bound.
4. Realigned `Abs` typing with the C++ primitive semantics: it now preserves
   the input nature and only updates the interval, instead of being forced to
   `Real`.
5. Kept one explicit fast-lane-only adaptation in `transform`: a completely
   unconstrained self-recursion still closes to `Real` in the reduced
   `SimpleSigType` view so FIR lowering preserves the historical fallback used
   by the existing fast-lane tests.
6. Updated FIR verification expectations so comparison binops declared as
   `Int32` (“boolean int”) match the fast-lane/C++ signal typing model.
7. Restored canonical typing for `SIGFFUN` so the shared `sigtype` pass now
   uses the declared foreign return type and the argument-dependent
   variability/computability instead of falling back to a maximal real type.
8. Updated the affected `propagate` flat-box regression test to build
   `with_rec_def` using the current parser-shape definition lists and to expect
   the eager `withlocaldef` lowering introduced earlier in the day.

#### Limitation

The Rust recursive typing path is now aligned with C++ on the initial
approximation (`TREC`) and on iterative re-evaluation intent, but it is still
not a structural port of `typeAnnotation(...)` / `updateRecTypes(...)`.

Current Rust mechanism:
- memoised `SigId -> SigType` environment,
- local invalidation of the recursive body sub-DAG between fixpoint steps,
- re-inference under the updated recursive approximation,
- C++-style interval widening on recursive bounds.

Current C++ mechanism:
- global recursive-group discovery,
- explicit vectors of recursive group types,
- repeated `updateRecTypes(...)` passes over those vectors.

This limitation is now documented in Rustdoc in `sigtype::rules` and in the
fast-lane preparation docs in `transform::signal_prepare`.

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/boxes/src/lib.rs`, `crates/boxes/tests/core_api.rs`, `crates/eval/src/lib.rs`, `crates/eval/tests/core_eval.rs`, `crates/parser/tests/structural_cpp_differential.rs`

#### Context

`spectralCentroid_test` in `faustlibraries/tests/analyzers_tests.dsp` exposed a

C++ lowers `letrec` eagerly in `boxWithRecDef(...)` from

- one projection definition per recursive binding.

not C++-equivalent and led to evaluation failures on real library code.

#### Changes

1. Moved `letrec` parity to the correct layer: `boxes::node_with_rec_def(...)`

3. Corrected the expression extraction step of the lowering to respect the Rust
   parser’s normalized definition payload instead of treating the whole payload
   as the expression.
4. Removed the active `eval` fallback for `WithRecDef`: reaching
   `BOXWITHRECDEF` in `eval` now raises an explicit internal invariant error.
5. Added structural regressions proving:
   - parser production no longer leaves `BOXWITHRECDEF` in the output tree,
   - parser-lowered `letrec` evaluates successfully without any `eval`
     fallback,
   - manually injected legacy `BOXWITHRECDEF` nodes are rejected explicitly,
   - `BoxBuilder::with_rec_def(...)` now expands eagerly to `WithLocalDef(...)`
     and exposes the synthetic `LETRECBODY` binding.

   explicit: `boxes` owns `letrec` lowering; `eval` must never repair it.
