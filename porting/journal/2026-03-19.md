### `sigtype`: switch `annotate()` to a global recursive-group driver

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/sigtype/src/rules.rs`

#### Context

The recursive parity plan cannot be reached while recursion is still solved
locally inside `infer_sym_rec(...)`. The next step was therefore to move the
active path toward the C++ control structure:
- solve recursive groups globally first,
- then type bodies and outputs under those solved approximations.

#### Changes

1. Switched `TypeAnnotator::annotate(...)` to:
   - discover symbolic recursive groups first,
   - build a `RecTypingState`,
   - solve recursive approximations before ordinary output typing,
   - then seed the solved recursive groups back into the main annotation pass.
2. Added an explicit `update_rec_types(...)` helper in the Rust driver, mirroring
   the C++ `updateRecTypes(...)` structure:
   - seed all recursive groups up front,
   - infer each recursive body list under that seed,
   - merge intervals by intersection or reunion depending on the phase.
3. Replaced the old local recursive solver in `infer_sym_rec(...)` with a
   seeded lookup / conservative fallback.
4. Moved widening to a dedicated `apply_recursive_widening(...)` helper so the
   global recursive driver owns the recursive interval update policy.

#### Current Status

This is a major structural step toward parity:
- the active path is no longer driven by subtree invalidation inside
  `infer_sym_rec(...)`,
- recursive groups are now solved as an explicit set before final annotation.

Known remaining blocker after this step:
- `cubic_distortion.dsp` no longer fails with missing canonical recursive
  types, but now exposes an overflow in interval arithmetic
  (`crates/interval/src/ops/arithmetic.rs`) during deeper canonical typing.
- That is a new lower-level blocker and should be handled next; it is not
  committed away in promotion.

### `sigtype`: scaffold explicit recursive-group typing state for `updateRecTypes` parity

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/sigtype/src/rules.rs`

#### Context

The new recursive parity plan requires moving away from the current purely
node-driven recursive typing loop. Before switching the active algorithm, Rust
needs the same kind of explicit recursive-group state that C++
`typeAnnotation(...)` prepares before calling `updateRecTypes(...)`.

#### Changes

1. Added a private `RecGroup` scaffold in `sigtype` to record:
   - the `SYMREC` node,
   - its recursive variable,
   - its body list,
   - and its arity.
2. Added a private `RecTypingState` scaffold that mirrors the main C++ driver
   vectors:
   - `groups`,
   - `current`,
   - `upper`,
   - `age_min`,
   - `age_max`.
3. Added deterministic recursive-group discovery helpers:
   - DFS over the output forest,
   - DAG-aware deduplication,
   - explicit body-list arity validation.
4. Documented the new scaffolding in Rustdoc as a preparation step for the
   future structural port of `typeAnnotation/updateRecTypes`.

No behavioural switch was made yet: the active recursive typing path still uses
the existing `infer_sym_rec(...)` logic until the explicit driver is ported in
the next step.

### `porting`: add a dedicated recursive `sigtype` parity refactor plan

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `porting/recursive-sigtype-parity-plan-2026-03-19-en.md`

#### Context

The recent fast-lane regressions made the remaining gap clearer: the current
Rust recursive typing path still differs structurally from the C++
`typeAnnotation(...)` / `updateRecTypes(...)` algorithm.

Local fixes improved behaviour, but they did not remove the architectural
weakness:
- recursive convergence is still encoded in a node-driven memoized typer,
- canonical type maps can still become incomplete under recursive retyping,
- promotion then ends up compensating for typing instability instead of simply
  consuming a stable canonical annotation.

Before continuing with more local fixes, the port needs an explicit parity plan
for the recursive typing algorithm itself.

#### Changes

Added a dedicated porting plan:
- `porting/recursive-sigtype-parity-plan-2026-03-19-en.md`

The plan documents:
- the exact structural difference between the current Rust implementation and
  the C++ recursive typing driver,
- the target architecture for a global recursive-group typing state,
- the intended Rust equivalent of `updateRecTypes(...)`,
- the migration phases,
- the validation matrix and exit criteria,
- and the temporary nature of promotion-side fallbacks introduced only to
  survive incomplete canonical type maps.

#### Validation

- documentation-only change

### `sigtype`/`normalize`/`transform`: make canonical typing the single fast-lane source and realign recursive inference

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/sigtype/src/rules.rs`, `crates/normalize/src/normalform.rs`, `crates/transform/src/signal_prepare.rs`, `crates/fir/src/checker.rs`

#### Context

`goertzelOpt_test` exposed two coupled issues in the fast-lane:
- promotion lived in `transform::signal_prepare` instead of the canonical
  normalization/type pipeline,
- `signal_prepare` still ran its own reduced recursive typer/fixpoint, which
  diverged from the newer `sigtype`-based path and caused contradictory
  integer/real recursion decisions.

Subsequent regressions on `signal_fir_lane` showed that simply switching to
canonical `SigType` was not enough: the Rust `sigtype` recursion port still
used an oversimplified recursive algorithm that did not match the C++ `TREC`
seed semantics and did not re-evaluate memoised body subtrees under updated
recursive approximations.

#### Changes

1. Moved the detailed fast-lane promotion pass from `transform` into
   `normalize`, so the active preparation path now consumes canonical
   `sigtype::TypeAnnotator` results before lowering.
2. Simplified `transform::signal_prepare` so it derives its reduced
   `SimpleSigType` view from canonical `SigType` instead of running a second
   local recursive typer.
3. Realigned recursive `sigtype` inference with the C++ intent:
   - recursive groups are now seeded from the C++-style `TREC` approximation
     instead of a single maximal scalar,
   - recursive symbol references read the current approximation attached to the
     recursion variable,
   - memoised recursive body subtrees are invalidated between fixpoint steps so
     the next iteration is re-inferred under the updated approximation,
   - recursive interval updates now mirror the default C++ widening behaviour:
     after recomputing one recursive body, the new interval is first reunited
     with the previous approximation and any drifting bound is widened
     immediately to the `TRECMAX` carrier bound.
4. Realigned `Abs` typing with the C++ primitive semantics: it now preserves
   the input nature and only updates the interval, instead of being forced to
   `Real`.
5. Kept one explicit fast-lane-only adaptation in `transform`: a completely
   unconstrained self-recursion still closes to `Real` in the reduced
   `SimpleSigType` view so FIR lowering preserves the historical fallback used
   by the existing fast-lane tests.
6. Updated FIR verification expectations so comparison binops declared as
   `Int32` (“boolean int”) match the fast-lane/C++ signal typing model.
7. Restored canonical typing for `SIGFFUN` so the shared `sigtype` pass now
   uses the declared foreign return type and the argument-dependent
   variability/computability instead of falling back to a maximal real type.
8. Updated the affected `propagate` flat-box regression test to build
   `with_rec_def` using the current parser-shape definition lists and to expect
   the eager `withlocaldef` lowering introduced earlier in the day.

#### Limitation

The Rust recursive typing path is now aligned with C++ on the initial
approximation (`TREC`) and on iterative re-evaluation intent, but it is still
not a structural port of `typeAnnotation(...)` / `updateRecTypes(...)`.

Current Rust mechanism:
- memoised `SigId -> SigType` environment,
- local invalidation of the recursive body sub-DAG between fixpoint steps,
- re-inference under the updated recursive approximation,
- C++-style interval widening on recursive bounds.

Current C++ mechanism:
- global recursive-group discovery,
- explicit vectors of recursive group types,
- repeated `updateRecTypes(...)` passes over those vectors.

This limitation is now documented in Rustdoc in `sigtype::rules` and in the
fast-lane preparation docs in `transform::signal_prepare`.

### `boxes`/`eval`/`parser`: make `letrec` parity eager and remove the eval fallback

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/boxes/src/lib.rs`, `crates/boxes/tests/core_api.rs`, `crates/eval/src/lib.rs`, `crates/eval/tests/core_eval.rs`, `crates/parser/tests/structural_cpp_differential.rs`

#### Context

`spectralCentroid_test` in `faustlibraries/tests/analyzers_tests.dsp` exposed a
remaining Rust/C++ divergence around `letrec`.

C++ lowers `letrec` eagerly in `boxWithRecDef(...)` from
`compiler/boxes/boxes.cpp` into a `with_local_def(...)` carrying:
- one synthetic `LETRECBODY = rec(...)` definition,
- one projection definition per recursive binding.

Rust still preserved an adapted `BOXWITHRECDEF` node on the parser/boxes
production path and relied on `eval` to expand or interpret it later. That was
not C++-equivalent and led to evaluation failures on real library code.

#### Changes

1. Moved `letrec` parity to the correct layer: `boxes::node_with_rec_def(...)`
   now performs eager C++-style lowering instead of emitting a persistent
   `BOXWITHRECDEF` node for normal parser output.
2. Added dedicated helpers in `boxes` for the `LETRECBODY` expansion and
   documented their parser-shape invariant:
   definitions must arrive as normalized parser cells
   `cons(name, cons(args, expr))`.
3. Corrected the expression extraction step of the lowering to respect the Rust
   parser’s normalized definition payload instead of treating the whole payload
   as the expression.
4. Removed the active `eval` fallback for `WithRecDef`: reaching
   `BOXWITHRECDEF` in `eval` now raises an explicit internal invariant error.
5. Added structural regressions proving:
   - parser production no longer leaves `BOXWITHRECDEF` in the output tree,
   - parser-lowered `letrec` evaluates successfully without any `eval`
     fallback,
   - manually injected legacy `BOXWITHRECDEF` nodes are rejected explicitly,
   - `BoxBuilder::with_rec_def(...)` now expands eagerly to `WithLocalDef(...)`
     and exposes the synthetic `LETRECBODY` binding.
6. Expanded Rustdoc/comments to make the new parity point and invariant
   explicit: `boxes` owns `letrec` lowering; `eval` must never repair it.

### `transform`/`codegen`: lower foreign math calls through fast-lane FIR and backend runtimes

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/transform/src/signal_prepare.rs`, `crates/transform/src/signal_fir/module.rs`, `crates/transform/src/signal_fir/mod.rs`, `crates/codegen/src/backends/cranelift/mod.rs`

#### Context

`dsp/math.dsp` failed in the fast-lane with an unsupported `SIGFFUN` node for
`isnanf(...)`. The prepared signal forest already preserved foreign function
calls, but the FIR lowerer still rejected them in Step 2C.

Once `SIGFFUN` lowering was added, `-lang interp` worked immediately because the
interpreter backend already mapped `isnanf`/`isinf`/`copysign`-style calls to
FBC opcodes. `-lang cranelift` still fell back to the stub because its subset
matcher and lowering path did not accept these direct foreign math calls.

#### Changes

1. Corrected `signal_prepare` reduced typing for `SIGFFUN`: the result type now
   follows the `FFUN(signature, ...)` return code instead of defaulting every
   foreign call to `Real`.
2. Added fast-lane `SIGFFUN` lowering in Step 2C by decoding raw `FFUN`
   descriptors from `TreeArena`, selecting the precision-appropriate symbol
   name (`isnanf` vs `isnan`), and emitting FIR `FunCall` + extern prototypes.
3. Added transform regressions covering:
   - foreign-function return typing in `prepare_signals_for_fir`,
   - `SIGFFUN -> Fir::FunCall` lowering and prototype emission.
4. Extended Cranelift subset lowering/import registration for the foreign math
   calls exercised by `math.dsp`:
   - `isnanf` / `isnan`
   - `isinff` / `isinf`
   - `copysignf` / `copysign`
   - `acoshf` / `acosh`
   - `asinhf` / `asinh`
   - `atanhf` / `atanh`
   - `coshf` / `cosh`
   - `sinhf` / `sinh`
   - `tanhf` / `tanh`
5. Added a Cranelift regression proving that one supported foreign subset call
   lowers all the way to JIT codegen instead of triggering subset-gap fallback.

### `transform`: memoize unary symbolic recursion group discovery on shared DAGs

Commit date: 2026-03-19

Source commit: `pending (recorded with this journal update)`

Source: `crates/transform/src/signal_prepare.rs`, `porting/MEMOIZATION.md`

#### Context

The unary recursive projection canonicalization added for `Birds.dsp` introduced
an unbounded traversal cost on large shared signal DAGs. On
`dsp/cubic_distortion.dsp`, `faust-rs` appeared to hang in
`prepare_signals_for_fir(...)`.

Sampling confirmed the hot path was entirely inside
`collect_unary_sym_groups(...)`, recursively revisiting the same shared
subgraphs while discovering unary symbolic recursion groups.

#### Changes

1. Added a `HashSet<SigId>` visited set to `collect_unary_sym_groups(...)` so
   each signal node is analyzed at most once per prepared forest.
2. Kept the existing structural list expansion semantics while ensuring shared
   DAGs no longer trigger exponential traversal during unary-group discovery.
3. Added a focused regression test that builds a deep shared "diamond DAG"
   around a unary recursive projection and verifies preparation completes.
4. Documented the new traversal memoization explicitly in `MEMOIZATION.md`.
