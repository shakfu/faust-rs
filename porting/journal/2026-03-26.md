## feat(codegen): lower mono passthrough `compute` in the WASM subset

Commit date: 2026-03-26
Source commit: working tree after `fcd6bb5`

Implemented the next Step-5 slice of the WASM backend by replacing the no-op
`compute` body with a real lowering path for the smallest useful FIR subset:
mono passthrough-style DSPs built from I/O aliases and a forward sample loop.

### Scope

- Added a dedicated partial `compute` subset lowerer in
  `crates/codegen/src/backends/wasm/mod.rs`.
- The lowered subset now handles:
  - `Block`
  - `DeclareVar(kStack)`
  - forward `SimpleForLoop`
  - `LoadVar(kFunArgs=count | kLoop | kStack)`
  - `LoadTable(kFunArgs=inputs/outputs | kStack aliases)`
  - `StoreTable(kStack aliases)`
- The lowerer pre-collects local stack and loop variables, assigns WASM locals,
  and emits a real `block/loop/br_if` structure.
- Mono `input0` / `output0` aliases created from `inputs[0]` / `outputs[0]`
  now lower to pointer locals, then to real sample `f32.load` / `f32.store`
  operations inside `compute`.

### Deliberate limitation

- This is still a strict subset bring-up, not generic FIR-to-WASM lowering.
- Unsupported `compute` shapes still fall back to the valid no-op body instead
  of aborting the whole module emission.

### Regression coverage

Added a code-body inspection test in
`crates/codegen/src/backends/wasm/tests.rs`:

- `wasm_compute_passthrough_lowers_loop_and_sample_io`

The test decodes the generated WASM and asserts that `compute` now contains a
real loop plus `f32.load` / `f32.store` sample traffic for the passthrough
fixture.

## feat(codegen): implement initial WASM memory layout engine

Commit date: 2026-03-26
Source commit: working tree after `905d1d9`

Implemented the Step-2 memory layout engine for the WASM backend so the
scaffold no longer reports a placeholder memory description.

### Scope

- Replaced the dummy `WasmMemoryLayout::scaffold(...)` usage with a real
  `WasmMemoryLayout::from_module(...)` derivation.
- The layout now walks FIR module sections and computes:
  - struct-backed field offsets,
  - static-table offsets,
  - `tables_offset`,
  - `io_zone_offset`,
  - raw `total_bytes`,
  - required page count.
- The engine currently accepts:
  - `DeclareVar { access: Struct }`
  - `DeclareVar { access: Struct, typ: Array(..) }`
  - `DeclareTable { access: Struct }`
  - `DeclareTable { access: Static }`
  - helper `DeclareFun { body: None }` entries, which are ignored

### Layout rule implemented

The current layout follows the WASM plan’s C++-parity rule that each field slot
is at least one audio-sample slot wide:

- single precision: minimum slot size = 4 bytes
- double precision: minimum slot size = 8 bytes

For Rust FIR compatibility, `f64`-typed storage in single-precision mode is
still widened to 8 bytes instead of being truncated to a 4-byte slot.

- `wasm_layout_pads_i32_fields_to_audio_slot_in_double_mode`
- `wasm_layout_places_static_tables_after_struct_region`

## fix(transform): lower `SIGFVAR(count)` in signal FIR Step 2C

Commit date: 2026-03-26
Source commit: working tree after `b601eaa`

Fixed the `faust-rs -pn bsmooth_test tests/signals_tests.dsp` failure reported
from `faustlibraries`, where the transform fast lane aborted with
`[FRS-SFIR-0004] unsupported signal node in Step 2C:
FVar(... sym("count"), sym("<math.h>"))`.

### Root cause

`crates/transform/src/signal_fir/module.rs` already handled `SIGFCONST` and
`SIGFFUN` in Step 2C, but `SIGFVAR` still fell through to the generic
unsupported-node error. In the failing DSP, Faust lowers the smoothing window
expression through the foreign variable `count`, which C++ treats as the
`compute(int count, ...)` function argument.

### Fix

- Added Step 2C lowering for `SigMatch::FVar`.
- Matched C++ `InstructionsCompiler::generateFVar` parity:
  `count` now lowers to `LoadVar("count", AccessType::FunArgs, Int32)`.
- Added deduplicated extern-global declarations for other foreign variables so
  non-`count` `SIGFVAR` nodes follow the same reference path instead of
  remaining unsupported.

### Regression coverage

- Added `foreign_var_count_lowers_to_compute_funarg` in
  `crates/transform/src/signal_fir/tests.rs` to assert that
  `SIGFVAR(int(0), sym("count"), sym("<math.h>"))` lowers to a compute-argument
  load in the emitted FIR.

### Validation

## fix(eval): fold numeric case arguments before matcher dispatch

Fixed the `faust-rs -pn encoder3D_test tests/hoa_tests.dsp` stack overflow from
`faustlibraries`, and the related `optimBasic3D_test` failure on the same 3D HOA
path. The overflow reduced to the local recursive `p/pcalcul` helper inside
`hoa.lib`, where Rust repeatedly selected the catch-all recursive `case` branch
instead of the `0/1/2` base cases.

### Root cause

Rust already simplified `case` patterns before compiling the matcher automaton,
but `apply_pattern_matcher_value(...)` still matched against the raw evaluated
argument tree. In the HOA selector path, the first argument to `pcalcul(...)`
was a residual arithmetic box like `((l != 0) & ...) + ... * 2 + ... * 3`
instead of a folded numeric literal.

Because the matcher only saw that unfurled arithmetic tree, it could not match
the constant branches `0`, `1`, or `2`, so only the catch-all recursive rule
remained active. That sent `p(...)` back through `pcalcul(...)` indefinitely
until the Rust process overflowed its stack.

### Fix

- `apply_pattern_matcher_value(...)` now simplifies each consumed argument with
  `box_simplification(...)` before passing it to the pattern matcher.
- This restores the same effective behavior as the reference C++ compiler for
  numeric `case` dispatch: selector expressions are reduced to literal ints/reals
  before rule matching, so constant rules win before catch-all recursive clauses.

### Regression coverage

Added two eval tests:

- `eval_process_preserves_parser_case_rule_priority_for_recursive_local_cases` —
  self-contained regression covering the recursive `p/pcalcul` shape reduced
  from `hoa.lib`.
- `eval_process_parser_case_rule_priority_matches_cpp_reference` — file-backed
  differential test that compiles the same fixture with the C++ reference
  compiler when `FAUST_CPP_BIN` (or `/usr/local/bin/faust`) is available, then
  checks Rust eval on the same source.

### Validation

- `cargo test -p eval eval_process_matches_exact_integer_real_case_arguments_like_cpp -- --nocapture`
- `cargo test -p eval eval_process_preserves_parser_case_rule_priority_for_recursive_local_cases -- --nocapture`
- `cargo test -p eval eval_process_parser_case_rule_priority_matches_cpp_reference -- --nocapture`

## fix(eval): preserve captured environments for unapplied local case values

Fixed a second eval parity bug exposed by `faust-rs -pn u_transformer_test tests/wdmodels_tests.dsp`
and `u_transformerActive_test` against `faustlibraries`. The failing path passed a local
multi-clause function (`s`) through another higher-order function (`matrix`) before applying it.
Rust then raised `undefined symbol R0`, even though the reference C++ compiler kept `R0` and `R1`
captured correctly.

### Root cause

Rust already represented evaluated `case` values as `EvalValue::PatternMatcher`, but
`force_value_to_box(...)` still had one non-parity shortcut: for an unapplied matcher
(`state == 0`, no consumed arguments), it returned the original raw `case` tree instead of a
runtime `boxPatternMatcher(...)` node.

That shortcut dropped the captured matcher environments. As soon as a local `case` value escaped
its lexical site before first application, later calls re-entered a bare `case` tree with no
captured bindings, so free variables like `R0`/`R1` became undefined.

### Fix

- `force_value_to_box(...)` now always stores `EvalValue::PatternMatcher` in the side-table and
  reifies it as `boxPatternMatcher(key)`, even before the first application.
- This aligns the Rust representation with the reference compiler model where evaluated `case`
  values remain closure-like matcher runtime objects carrying their lexical environment.

### Regression coverage

Added two eval tests:

- `eval_process_preserves_unapplied_local_case_environment_through_higher_order_call` —
  in-memory regression for a local `transformer_scatter` case function passed through
  `apply_twice(...)`.
- `eval_process_unapplied_local_case_higher_order_matches_cpp_reference` — self-contained
  file-backed differential that compiles the same fixture with the C++ reference compiler when
  `FAUST_CPP_BIN` (or `/usr/local/bin/faust`) is available, then checks Rust eval on the same
  source.

### Validation
