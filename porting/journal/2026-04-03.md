## fix(transform): preserve previous-step semantics in SIGGEN Delay1 fallback

Commit date: 2026-04-03
Source commit: working tree after `1300754`

Fixed a latent bug in the compile-time SIGGEN interpreter used to expand
generated fast-lane tables. The non-recursive `Delay1(x)` fallback was reading
the current-step value of `x` instead of the previous-step value, so table
generators that relied on unary delay semantics were shifted by one sample.
This bug predated the `simplify` integration, but became visible on
`table.dsp` once the fast-lane path exercised the generator interpreter more
often.

### Scope

- updated
  [crates/transform/src/signal_fir/module.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_fir/module.rs)
  so the SIGGEN interpreter keeps explicit previous/current maps for
  non-recursive `Delay1` sources and returns the previous-step sample instead
  of re-evaluating the current step
- added a non-regression in
  [crates/transform/src/signal_fir/tests.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_fir/tests.rs)
  asserting that `Delay1(waveform(...))` expands to `[0, x0, x1, ...]`

- `cargo test -p transform signal_fir -- --nocapture`

## fix(transform/codegen): restore wasm UI resets and FBC float round-tripping

Commit date: 2026-04-03
Source commit: working tree after `a1f5d02`

Two regressions surfaced after wiring `simplify` into the fast-lane path.
First, wasm `instanceResetUserInterface` could be materialized before the UI
program was emitted, so controls that survived only in UI metadata stopped
getting reset stores and `norm1.dsp` failed the standalone wrapper defaults
check. Second, interpreter `.fbc` serialization was truncating `f32` literals
to a human-readable precision instead of round-trip precision, which changed
`1.0 / 48000.0` into `0.0000208` and made the interpreter diverge from the
Cranelift runtime on `rep_61_fmin_sr.dsp`.

### Scope

- updated
  [crates/transform/src/signal_fir/module.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_fir/module.rs)
  so `emit_ui_program()` runs before `instanceResetUserInterface` is frozen,
  ensuring UI-only controls still contribute reset stores
- added a non-regression in
  [crates/transform/src/signal_fir/tests.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_fir/tests.rs)
  asserting that a slider present only in the UI program still appears in both
  `buildUserInterface` and `instanceResetUserInterface`
- updated
  [crates/codegen/src/backends/interp/compiler/tests.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/backends/interp/compiler/tests.rs)
  to serialize the foreign-function-registry tests behind one mutex, avoiding
  cross-test races while validating the workspace
- updated
  [crates/codegen/src/backends/interp/serial.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/backends/interp/serial.rs)
  so `.fbc` real literals use round-trip formatting instead of truncated

- added a non-regression in
  [crates/codegen/src/backends/interp/serial/tests.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/backends/interp/serial/tests.rs)
  covering the `1.0 / 48000.0` `f32` round-trip that previously shifted the

## fix(transform): canonicalize one-sample delays before FIR lowering

Commit date: 2026-04-03
Source commit: working tree after `32e0c26`

Moved the `Delay(x, 1)` handling out of the fast-lane FIR lowerer and into the
signal-preparation boundary. `normalize` still exposes one-sample delays in the
generic `Delay(_, 1)` form, but `prepare_signals_for_fir` now rewrites that
shape back to `Delay1(_)` immediately after `simplify` and before the final
typing/promotion snapshot. This restores a single unary-delay representation
for downstream fast-lane consumers and removes the local SIGGEN/interpreter
special case that had been added for `comb_delay1.dsp`.

### Scope

  [crates/transform/src/signal_prepare.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_prepare.rs)
  to:
  - run a memoized `Delay(_, Int(1)) -> Delay1(_)` canonicalization pass after
    `simplify`
  - re-type and re-promote the canonicalized forest before exporting
    `PreparedSignals`
  - document `Delay1` as the fast-lane preparation boundary form for
    one-sample delays
- updated
  [crates/transform/src/signal_prepare/tests.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_prepare/tests.rs)
  with a dedicated non-regression asserting that literal `Delay(_, 1)` becomes
  `Delay1`, and aligned existing preparation tests with the new canonical form
- updated
  [crates/transform/src/signal_fir/module.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_fir/module.rs)
  to remove the fast-lane-local `Delay(_, 1)` compatibility branch and the
  associated SIGGEN-only regression test

## fix(normalize): preserve konst order-0 factors in mterm normalization

Commit date: 2026-04-03
Source commit: working tree after `e75c231`

Removed the temporary fast-lane-specific simplification workaround and fixed the
actual normalization bug exposed by `simplify`. Rust could reach
`Mterm::normalized_tree()` with a non-numeric `Konst` factor in the order-0
bucket, such as `float(fSamplingFreq)`, then overwrite that bucket with the
numeric coefficient and collapse delay amounts like `int(0.5 * SR / 440)` to
`int(0)`. The C++ reference in
`compiler/normalize/mterm.cpp::normalizedTree()` assumes this situation cannot
happen and asserts that the order-0 buckets are empty before installing `fCoef`.
The Rust port now adapts that step by merging `fCoef` into the existing order-0
numerator so the simplification path stays semantically correct even when the
stronger C++ invariant is not yet reproduced upstream.

### Scope

- updated
  [crates/normalize/src/mterm.rs](/Users/letz/Developpements/RUST/faust-rs/crates/normalize/src/mterm.rs)
  so `normalized_tree()` combines the coefficient with any pre-existing
  order-0 numerator instead of overwriting it
- added a non-regression test in
  [crates/normalize/src/mterm.rs](/Users/letz/Developpements/RUST/faust-rs/crates/normalize/src/mterm.rs)
  covering `0.5 * float(fSamplingFreq)`
- added a pipeline non-regression in
  [crates/transform/src/signal_prepare/tests.rs](/Users/letz/Developpements/RUST/faust-rs/crates/transform/src/signal_prepare/tests.rs)
  asserting that a simplified delay amount still contains `fSamplingFreq`
- left the generic fast-lane `simplify` entry point unchanged; no
  `simplify_fastlane_preserving_*` helper remains

## fix(normalize): avoid numeric-only assumption in aterm constant bucket

Commit date: 2026-04-03
Source commit: working tree after `25f7280`

Checked the C++ reference `compiler/normalize/aterm.cpp` and confirmed that
`aterm::normalizedTree()` starts with `subNums(P[0], N[0])`, which assumes the
order-0 buckets are already numeric literals. After wiring `simplify` into the
fast-lane preparation path, Rust can now reach `aterm::normalized_tree` with
order-0 non-literal terms, which triggered `sub_nums: not numeric` on
`comb_delay1.dsp`. The Rust port now combines the order-0 buckets through the
generic signed combiner instead of the numeric-only helper.

### Scope

  so `normalized_tree()` initializes its accumulator with
  `add_terms_with_sign(true, P[0], false, N[0])`
- removed the now-unused `sub_nums` import from the same file
- verified against the C++ source in
  `/Users/letz/Developpements/RUST/faust/compiler/normalize/aterm.cpp`

### Validation

## fix(transform): accept simplified one-sample delays in the FIR fast-lane

Commit date: 2026-04-03
Source commit: working tree after `89db136`

Wired `prepare_signals_for_fir` to run algebraic `simplify` before FIR
lowering, then taught the fast-lane to accept the normalized `Delay(x, 1)`
shape as equivalent to `Delay1(x)` instead of preserving the original delay
form artificially. The lowering path now keeps single-sample state lowering
for literal `1` delays, still allocates proper delay lines for interval-sized
variable delays, and tolerates simplification failures in the fast-lane entry
point by falling back to the original prepared node.
