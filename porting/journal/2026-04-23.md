## feat(propagate): collapse recursive FAD outputs onto one unified transform

Commit date: 2026-04-23

Source commit: pending

### Context

After the previous carrier refactor, `ForwardADTransform` could already carry
several tangent lanes internally, but `generate_fad_signals_multi(...)` still
instantiated one transformer per seed and stitched the rows together at the
end.

That kept the old recursive-codegen shape alive:

- one exposed primal recursion,
- plus one AD-local primal shadow recursion per seed.

The visible optimization target of the plan was to stop doing that and make
all primal/tangent outputs project into the same differentiated recursive
group.

### Implementation

Updated `crates/propagate/src/forward_ad.rs`:

- `generate_fad_signals_multi(...)` now creates one
  `ForwardADTransform::new(arena, seeds)` instead of one transformer per seed
- the output bundle is now assembled directly from one `Dual` per propagated
  output:
  - `result.push(dual.primal)`
  - `result.extend(dual.tangents)`
- Rustdoc now documents the unified-recursion behavior instead of the old
  temporary per-seed orchestration model

Added regression coverage:

- new corpus fixture:
  - `tests/corpus/fad_recursive_multi_seed_shared.dsp`
- structural tests in `crates/compiler/tests/signal_pipeline.rs` now assert:
  - `fad_recursive_parametric_self.dsp`: primal and tangent project from the
    same recursive group
  - `fad_recursive_multi_seed_shared.dsp`: all three output lanes project from
    the same recursive group
- runtime regression in `crates/compiler/tests/fad_recursive_runtime.rs` checks
  the two-seed recursive case against central finite differences for both
  tangent lanes
- `crates/propagate/tests/core_api.rs` was updated to the new invariant:
  primal and tangent projections now share the same interleaved
  `DEBRUIJNREC`

Concrete generated-code result:

- `fad_bug1.dsp` now lowers to one primal state plus one tangent state:
  the duplicate primal shadow recursion is gone from the emitted C++

### Verification

- `cargo fmt --all`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo test --workspace --all-targets`
- `cargo run -q -p compiler -- fad_bug1.dsp`

Outcome:

- the redundant recursive primal duplication is removed;
- recursive single-seed and multi-seed FAD cases remain numerically correct;
- the new observable invariant is "shared recursive group, distinct projected
  slots".

## refactor(propagate): generalize forward AD carrier to multi-lane tangents

Commit date: 2026-04-23

Source commit: pending

### Context

The unified-recursion FAD plan needs the internal transformer to carry
`1 + N` lanes (`primal + one tangent per seed`) before the public
multi-seed entry point can collapse the current "one transformer per seed"
orchestration.

The pre-existing implementation hard-coded a one-seed carrier:

- `Dual { primal, tangent }`
- `diff_seed: SigId`
- de Bruijn recursion interleaving fixed to `[primal, tangent]`

That shape forced every later optimization to duplicate most derivative
rules, because the carrier itself only knew how to represent one tangent
lane.

### Implementation

Updated `crates/propagate/src/forward_ad.rs`:

- introduced a multi-lane carrier:
  - `Dual { primal, tangents: SmallVec<[SigId; 2]> }`
- generalized `ForwardADTransform` to:
  - `diff_seeds: Vec<SigId>`
  - `diff_seed_index: AHashMap<SigId, SmallVec<[usize; 2]>>`
- generalized the node rules so they now build one tangent lane per seed
  internally:
  - arithmetic
  - unary chain rules
  - delays
  - `select2` / `prefix`
  - helper pass-through nodes
  - known differentiable `FFun` cases
- generalized de Bruijn recursion handling to interleave
  `[primal, tangent_0, tangent_1, ...]` for the active seed set
- kept `generate_fad_signals_multi(...)` on the old external orchestration
  shape for this step:
  - still one transformer per seed
  - still one tangent row gathered at a time

Supporting dependency update:

- added `smallvec` to `crates/propagate/Cargo.toml`

### Verification

- `cargo test -p propagate`
- `cargo test -p compiler --test signal_pipeline fad_recursive -- --nocapture`

Outcome:

- recursive FAD coverage still passes;
- the transformer now has the internal lane model required for the next
  step, without changing the public output expansion strategy yet.

## docs(porting): tighten unified-recursion FAD refactor plan

Commit date: 2026-04-23

Source commit: pending

### Context

`porting/fad-n-lanes-unified-rec-plan-2026-04-23-en.md` already pointed at
the right optimization target: collapse the current "one recursive AD shadow
per seed" model into one unified recursive group carrying the primal and all
tangent lanes.

The initial draft was directionally correct, but a few invariants were still
too implicit for a parity-sensitive refactor:

- seed lookup cost and shape were under-specified once `diff_seed` becomes a
  vector;
- the temporary recursive placeholder invariant needed a stronger statement;
- duplicate seed handling was not documented;
- "one recursion accumulator" could be read too literally at the backend code
  emission level;
- validation criteria leaned too much on structural C++ shape and not enough
  on numeric checks.

### Implementation

Refined the plan in
`porting/fad-n-lanes-unified-rec-plan-2026-04-23-en.md`:

- added an explicit `diff_seed_index` recommendation for multi-seed lookup;
- clarified the `DEBRUIJNREC` placeholder invariant and the required
  regression coverage around back-edge resolution;
- documented duplicate-seed expectations and recommended preserving duplicate
  tangent lanes in the first implementation;
- clarified that success means one unified recursion group at the
  signal/FIR boundary, not necessarily one literal backend array variable;
- strengthened Step 4/5/Validation to require numeric regression checks
  alongside structural assertions.

### Verification

Documentation-only update. No code or tests changed in this step.

## test(compiler): validate recursive forward-mode AD numerically

Commit date: 2026-04-23

Source commit: pending

### Context

`fad_bug1.dsp` raised a concern that the generated C++ for a seeded
self-recursive system was wrong because the AD lowering materialized a second
recursive state next to the exposed primal state.

The produced code does duplicate the primal recurrence inside the AD row, but
that alone is not evidence of a semantic bug. For a seeded recursion such as:

```text
y[n] = p * y[n-1] + 2
```

forward-mode AD legitimately needs both:

- the process-visible primal recurrence `y[n]`,
- and an AD-local primal shadow feeding the tangent recurrence
  `dy/dp[n] = y[n-1] + p * dy/dp[n-1]`.

What was missing was hard evidence across more difficult recursive families:
single-state feedback, nested recursions, multi-output recursive groups, and
genuine mutual recursion.

### Implementation

Added Rustdoc and regression coverage:

- `crates/propagate/src/forward_ad.rs`
  - documented the current recursive codegen model explicitly:
    generated AD code may duplicate primal recursion state while remaining
    semantically correct.
- `tests/corpus/`
  - added:
    - `fad_recursive_parametric_self.dsp`
    - `fad_recursive_nested_parametric.dsp`
    - `fad_recursive_multi_output_parametric.dsp`
    - `fad_recursive_mutual_crossed_parametric.dsp`
- `crates/compiler/tests/signal_pipeline.rs`
  - added structural acceptance tests for the new recursive corpus fixtures.
- `crates/compiler/tests/fad_recursive_runtime.rs`
  - added runtime checks through the interpreter fast lane:
    - closed-form recurrence validation for the `fad_bug1`-style self-recursive
      case,
    - central finite-difference validation for nested recursion,
    - central finite-difference validation for multi-output recursion,
    - central finite-difference validation for mutual recursion.

### Verification

- `cargo fmt --all`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo test --workspace --all-targets`

Outcome:

- the `fad_bug1` pattern is numerically correct under the current lowering;
- the current concern is redundancy/optimization, not a proven semantic error;
- the new recursive fixtures now guard against regressions in
  `crates/propagate/src/forward_ad.rs`.
