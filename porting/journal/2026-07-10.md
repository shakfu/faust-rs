## Harden FIR recursive split for vector mode

Commit date: 2026-07-10
Source commit: this commit

Implemented the FIR-level hardening plan for the `-vec` recursive split:

- `collect_var_loads` is now conservative and covers nested FIR value forms used
  by the split, including `LoadTable`, `LoadSoundfileBuffer`,
  `LoadSoundfileLength/Rate`, `ValueArray`, `LoadVarAddress`, calls, casts,
  selects, arithmetic, and `TeeVar`;
- unsupported statement/value shapes now make `partition_recursive_body` fall
  back to one serial loop instead of speculatively hoisting;
- `rewrite_var_loads` now returns `Option<FirId>` and descends through the same
  nested forms, so accepted cross-boundary reads are actually rewritten inside
  table/soundfile indexes and other value operands;
- unrepresentable boundary uses, such as taking the address of a boundary temp
  or treating it as a table, reject the split and preserve scalar ordering;
- `split_bodies` falls back to the fused serial loop when tail rewriting cannot
  prove the split safe.

Regression coverage added in `loop_graph`:

- APF-like direct `LoadTable(Struct)` reads in a candidate tail are rejected;
- soundfile buffer indexes depending on a serial carrier are collected,
  boundary-buffered, and rewritten to `vbufN[i0 - vindex]`.

are backend support gaps for the already-correct vector FIR shape and should be
handled by a follow-up backend-lowering task rather than hidden behind a broad
known-failure list.

## Plan FIR hardening for vector-mode split correctness

Source commit: this commit

The plan records the immediate correction strategy:

- keep the current faust-rs FIR-level vector architecture for the short-term
  fix instead of starting a full C++ `VectorCompiler` signal-level port;
- introduce total FIR value traversal/mapping helpers so dependency collection

  FIR split is correctness-safe.

## S-D — recursive-slice split, the vector-mode payoff

The vector substrate (P6 V1–V5b) chunked only state-free slices; a recursive
slice stayed one plain serial loop, so SIMD never touched a stateful DSP. S-D
delivers the first real per-statement split (vector doc §5.4, "pure tail").

### What lands

A recursive sample-loop slice reaches the FIR as a fused loop-carried chain, e.g.
for `process = (_ : + ~ _) * 0.5`:

```c
float fRecCur = (float)input0[i0] + fRec;   // serial: reads state
output0[i0]   = 0.5f * fRecCur;             // state-free tail: reads fRecCur
fRec          = fRecCur;                     // serial: writes state
```

`-vec` now emits:

```c
float vbuf0[32];
for (int vindex = 0; vindex < count; vindex += 32) {
    int vend = ((vindex + 32 < count) ? vindex + 32 : count);
    for (int i0 = vindex; i0 < vend; i0++) {        // serial core
        float fRecCur = (float)input0[i0] + fRec;
        fRec = fRecCur;
        vbuf0[i0 - vindex] = fRecCur;                // buffer the carrier
    }
    for (int i0 = vindex; i0 < vend; i0++) {        // vectorizable tail
        output0[i0] = 0.5f * vbuf0[i0 - vindex];
    }
}
```

The output scaling is now a self-contained inner loop with no loop-carried
dependency, a contiguous buffer read and a contiguous output write — the C
compiler can auto-vectorize it. The recursion stays serial.

### How

Three pure pieces (S-A/B/C) assembled into the emission:

- **`partition_recursive_body`** (`loop_graph.rs`) — splits a flat body into a
  **serial core** (statements touching `Struct` state, plus, by fixpoint, every
  producer of a temp a serial statement reads) and a **vectorizable tail** (the
  state-free rest). Boundary temps = serial-produced temps read by the tail. The
  fixpoint guarantees no serial statement reads a tail-produced temp, so hoisting
  the tail past the whole serial core is order-safe. Returns `None` (→ single-loop
  fallback) for unsupported statement shapes (nested `If`/loops — clocked islands)
  or when nothing is state-free to hoist.
- **`ChunkBuffer`** (S-C) — `vbufN[i0 - vindex]` chunk-local store/load.
- **`rewrite_var_loads`** — rebuilds the tail statements, redirecting boundary-temp
  `LoadVar`s to the buffer load, preserving interned identity elsewhere.
- **`emit_split_sample_loop` / `build_split_chunk_driver`** (`module/build.rs`) —
  emit `[buf decls…, chunk driver{ serial inner loop, tail inner loop }]`.
  `emit_sample_loop` now returns `Vec<FirId>` (buffers + driver) and only takes
  this path for a forward `Recursive` slice in vector mode.

### Why bit-exact

The serial core runs the whole chunk first — state evolves exactly as in the
fused loop, buffering each per-sample carrier — then the tail reads those values
back at the same global `i0`. Only storage/loop structure changed, not the
arithmetic.

## Vector loop variants `-lv 0` / `-lv 1` (Faust C++ parity)

`ComputeMode::Vector { loop_variant }` was plumbed since V1 but ignored — both
values produced the same chunk driver, and that driver was the runtime-bound
"simple" form (Faust's `-lv 1`), not the default. Now the two variants are real,
selected by `-lv` exactly as Faust C++ (`0 = fastest (default)`, `1 = simple`):

- **`-lv 1` "simple"** — one loop, runtime `vend = min(vindex+vs, count)` bound
  (the previous behaviour, now explicitly variant 1).
- **`-lv 0` "fastest" (default)** — a **constant-trip** main loop over
  `[0, count - count%vs)` with inner bound `vindex + vs` (the C compiler proves
  trip count = vs, so it unrolls/SIMDs the vectorizable inner loop), plus a scalar
  remainder over `[count - count%vs, count)`. This is the autovectorization-
  friendly form and the missing half of real SIMD for S-D's tail loop.

### Fixes / design

- **Inverted labels corrected**: the `-lv` doc in `args.rs` and `ComputeMode` said
  `0 = simple, 1 = faster` — backwards vs Faust. Now `0 = fastest (default)`.
- **Empty-loop guard (interp parity)**: the interpreter runs a loop body once
  before its first condition check (both loop opcodes are `do/while`); the whole
  codebase had relied on `count > 0` so loops were never empty. The fastest
  remainder is empty when `vs | count`, and the main loop is empty when
  `count < vs` — so each is wrapped in an `if` (`vlimit > 0`, `vlimit < count`).
  The C `for` would skip an empty range anyway, but the guard keeps interp and C
  identical (this was caught by the V6 oracle as an `io_oob` at sample 64).
- **No `vlimit` variable**: `count - count%vs` is emitted as a reused interned
  expression, and the remainder's `int vindex = vlimit` lives inside its `if`
  block, so multiple sample-loop slices never collide on a function-scope name.
- Emission refactored: `emit_sample_loop` → `plain_sample_loop` / `split_bodies`
  / `build_chunk_driver` → `build_simple_driver` | `build_fastest_driver`, sharing
  `chunk_inner_loops` (one inner `for` per body: 1 for a vectorizable slice, 2
  for a split serial+tail).

### Validation

- **V6 oracle** now runs every case under **both** `-lv 0` and `-lv 1` (scalar vs
  `-vec`, byte-identical via interp). 5/5.
- **C cross-check** across block sizes `{1,2,7,16,31,32,33,64,65,100}` — scalar vs
  `-lv 0` vs `-lv 1`, all bit-identical (exercises `count < vs`, `vs | count`, and
  remainders — the edge cases the count-64 oracle can't reach).
- **`-lv` exposed** to the user (help text + legacy `-lv`/`--lv`), default 0.
- 190 goldens byte-identical (scalar path untouched); clippy `-D warnings` clean.

## S-E — measure: the split tail actually vectorizes (and when it pays)

Confirmed on arm64 / Apple clang 14 (NEON) that the C compiler auto-vectorizes
the split's state-free tail loop, while the recursive core stays serial — the
whole point of S-D. Method: `-Rpass=loop-vectorize` remarks + `-S` asm + a
microbenchmark (N=4096, 40k iters, `-O3 -ffast-math`), scalar vs `-vec -lv 0`.

### Vectorization confirmed

| Loop | clang verdict |
|---|---|
| scalar single fused loop | **not vectorized** (loop-carried `fRec` dependency) |
| `-vec` serial core (main + remainder) | **not vectorized** (recursion — correct) |
| `-vec` tail loop, `… * 0.5` | **vectorized, width 4, interleaved 4** (NEON, 16 samples/iter) |
| `-vec` tail loop, abs/min/max shaper | **vectorized, width 2** (main + remainder) |

The asm shows `fmul.4s v1..v4` (4-wide single-precision NEON multiplies, 4
registers interleaved) in the `*0.5` tail — concrete SIMD.

### When the split pays (net speedup)

The split adds one store + one load per sample (the carrier through `vbufN`). So
the net win depends on the tail's arithmetic intensity vs that buffering overhead
and vs the serial recursion that runs regardless:

| DSP (recursion + tail) | speedup `-lv 0` vs scalar |
|---|---|
| `(_:+~_) * 0.5` (cheap tail) | **0.92×** — buffering overhead > SIMD gain |
| `(_:+~_) : *0.5:+0.1:*0.9:…` (affine chain) | 0.92× — chain folds to ~1 op under `-ffast-math` |
| `(_:+~_) : abs/min/max shaper` (≈5 vector ops) | **1.15×** — SIMD gain > overhead |

So loop separation is a **conditional** optimization, exactly as in Faust C++:
it pays for compute-heavy tails and is a small regression for memory-bound
trivial ones. It never changes results (bit-exact, oracle-locked) — only speed.

### Takeaways for the next steps

- The mechanism is proven end-to-end (vectorizes + can net-win). 
- A cost heuristic (only split when the tail's op count clears a threshold, à la
  Faust's vectorization policy) would avoid the cheap-tail regression — a natural
  follow-up before making the split unconditional.
- `x^n` waveshapers fold to `pow()` (a scalar libm call that blocks
  vectorization); polynomial tails would need `-fno-math-errno` + a vectorizable
  `pow`, orthogonal to loop separation.
