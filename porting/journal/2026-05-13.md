## Fix Julia UI zones and double precision alias

Commit date: 2026-05-13
Source commit: this commit (`Fix Julia UI zones and double precision alias`)

Fixed the Rust Julia backend output used by the Faust impulse-test Julia
architecture.

The generated UI callbacks now pass Faust Julia runtime zone identifiers as
symbols (`:fHslider0`, `:fButton0`, etc.) instead of DSP field values
(`dsp.fHslider0`). This matches the C++ Julia backend and the installed
`/usr/local/share/faust/julia/gui/UI.jl` and `MapUI.jl` signatures, where active
widgets, bargraphs, and soundfiles receive `param::Symbol`.

The backend also propagates the compiler precision mode into the generated
Julia `REAL` alias, so `-double` emits `const REAL = Float64`. This keeps
`impulsejulia.jl` inputs/outputs aligned with its `const FAUSTFLOAT = Float64`
and avoids a later `Matrix{Float32}`/`Matrix{Float64}` method mismatch.

## Allow Julia architecture wrapping

Commit date: 2026-05-13
Source commit: this commit (`Allow Julia architecture wrapping`)

Extended CLI architecture wrapping to `-lang julia` so Faust-compatible
commands such as:

```sh
faust-rs --timeout 20 -lang julia APF.dsp -a ../archs/impulsejulia.jl -I dsp -double
```

can pass the generated Julia module through the existing enrobage marker
pipeline instead of being rejected as a non-C/C++ backend.

The change keeps `-a/--architecture` rejected for FIR/interp/cranelift/WASM
modes, reuses the existing architecture search, class-name rewrite, and
recoverable include-error handling, and adds fixture coverage for Julia
architecture templates plus a CLI smoke test for `faust-rs -lang julia -a`.

## Document BRA tape type boundary

Commit date: 2026-05-13
Source commit: this commit (`Document BRA tape type boundary`)

Expanded `porting/rad-disable-reverse-time-rec-fast-path-plan-2026-05-10-en.md`
with the implementation boundary between promoted Signal-IR values and
backend-local BRA tape storage.

The new section explains why `SigBlockReverseAD` does not create `fBraTapeN`
nodes during RAD propagation: the carrier records the semantic need for a
block-local reverse sweep, while `signal_fir` owns the backend scheduling
decision about which values must be stored, where stores occur relative to
delay/state updates, and which concrete `real_ty` tape layout is used.

It also documents the 2026-05-12 cast fix: integer-typed Signal-IR values can
legitimately be selected for the BRA real tape after normal promotion, so the
FIR boundary must cast `Int32` lowered values to `real_ty` before
`store_table(fBraTapeN, ...)`.
