## fix(wasm-ffi): unblock rust-branch faustwasm compiler module

Commit date: 2026-04-29

Source commit: this commit

The raw `faustwasm` compiler-module path was trapping under Node before
parsing user DSP code. The panic came from `std::time::Instant::now()`
inside the compiler facade timing helper: `wasm32-unknown-unknown`
does not provide a host clock for this raw ABI, and `Instant::now()`
lowers to a panic in that target.

The compiler facade now skips `Instant` entirely when no timing sink is
installed. Native CLI `-time` behavior is unchanged because timing still
uses `Instant` when a sink exists; the `wasm-ffi` path uses the no-sink
mode and can compile without touching unsupported clock APIs.

The same validation exposed the next rust-branch integration issue:
project-local imports such as `ad.lib` cannot be passed with `-I` because
the raw compiler module does not expose an Emscripten-style filesystem.
The `wasm-ffi` build script now supports embedding multiple `.lib` roots
from `FAUST_LIB_PATH`, preserving search order and first-root-wins
semantics. This allows a compiler module to bundle both a local project
library root and the standard Faust libraries.

### Validation

- `cargo test -p compiler timing_helper_without_sink_runs_without_measuring`
- `cargo test -p wasm-ffi`
- `cargo clippy -p compiler -p wasm-ffi --all-targets -- -D warnings`
- `FAUST_LIB_PATH=/Users/letz/Developpements/RUST/faust-rs:/Users/letz/faust/libraries cargo run -p xtask -- build-faustwasm-compiler-module`
- `node scripts/faust2wasm.js /Users/letz/Developpements/RUST/faust-rs/fad_pendulum_cello4.dsp test/fad_pendulum_cello4 -pwa -rust-compiler /Users/letz/Developpements/RUST/faust-rs/target/wasm32-unknown-unknown/release/faust_wasm_ffi.wasm`

## feat(codegen,compiler): rad_gradient_descent example

Commit date: 2026-04-29

Source commit: 658909e

Adds two minimal public APIs on `FbcDspInstance` for host-driven
training loops:

- `ui_instructions()` exposes the UI-instruction list collected at
  compile time so the host can discover slider labels and the bound
  real-heap offsets.
- `get_real_zone(offset)` / `set_real_zone(offset, value)` provide
  bounds-checked R/W on the real heap.

Then ships `crates/compiler/examples/rad_gradient_descent.rs`, a
self-contained host loop that fits the `rad_gain_bias_train` fixture
against a synthetic target via plain SGD over 512-sample batches.
The example recovers `gain` and `bias` to within ~1e-6 of the true
values in 400 iterations.

This is the concrete answer to "what is RAD good for in `faust-rs`
today": any feed-forward DSP whose parameters live in `hslider` /
`vslider` / `numentry` controls is fittable by a few lines of host
Rust + the existing `rad(...)` surface.

Run with: `cargo run -p compiler --example rad_gradient_descent`.

## bench(compiler): rad_vs_fad_perf comparison example

Commit date: 2026-04-29

Source commit: f9f775b

Side-by-side RAD-vs-FAD harness covering five trainable shapes:

- `gain·x + bias` (1 in, 2 seeds),
- polynomial waveshaper (3 in, 4 seeds),
- static softclip with `tanh` (1 in, 2 seeds),
- FIR taps with host-fed delays (4 in, 4 seeds),
- deep multiplicative chain (1 in, 6 seeds — adjoint-sum stress).

For each pair the binary reports compile time, interp bytecode size,
and per-frame compute averaged over 200 reps on a 4096-sample buffer.

Concrete result on the current pipeline (release builds): bytecode
size and per-frame compute are within ~5 % between RAD and FAD on
every shape, **including** the deep multiplicative chain. The
signal-prepare + CSE pipeline already absorbs the symbolic-growth
cost of reverse-mode accumulation that plan §17 risk #2 warned
about. At the scales explored here, RAD is essentially free relative
to FAD for feed-forward seeds.

Run with: `cargo run --release -p compiler --example rad_vs_fad_perf`.

## feat(corpus,compiler): rad_adaptive_notch demo (LMS on FIR notch)

Commit date: 2026-04-29

Source commit: this commit

A recognisable adaptive-filtering use case demonstrating that phase-1
RAD already covers a real DSP problem end-to-end.

### Implementation

- New corpus fixture
  `tests/corpus/rad_adaptive_notch_omega.dsp` — a 3-tap FIR notch
  with the radian frequency `omega` exposed as the single `rad(...)`
  seed:

  ```faust
  notch(xn, xn1, xn2) = xn - 2.0 * cos(omega) * xn1 + xn2;
  process = rad(notch, omega);
  ```

  The transfer function `H(z) = 1 - 2·cos(ω)·z⁻¹ + z⁻²` places zeros
  on the unit circle at `e^(±j·ω)`. The host buffers
  `x[n], x[n-1], x[n-2]` and feeds them as three audio channels so
  the differentiated body stays strictly feed-forward.

- New example
  `crates/compiler/examples/rad_adaptive_notch.rs` — synthesises a
  noisy sinusoid at `ω* = 1.3 rad/sample`, runs the rad-compiled DSP
  block-by-block, accumulates `loss = E[y²]` and
  `grad = E[2·y·∂y/∂ω]`, and SGDs `ω`. Starting from `ω = 0.4` the
  loop converges to `|ω − ω*| ≈ 2e-4` in under 50 iterations of
  512-sample blocks; the residual loss matches the additive noise
  floor `σ²` as an LMS adaptive notch should.

- Updated `docs/rad-usage-en.md` with a "Pattern: LMS adaptive
  filtering (host-managed delay line)" section that walks through the
  loop and links the example.

### Why it matters

This is the recognisable adaptive-filtering demonstration recommended
in the previous step's recap. It transforms the RAD chantier from
"works on a 2-parameter linear map" to "drives a real DSP adaptation
problem to the noise floor in <50 iterations". Importantly, it pins
the practical scope of phase-1 RAD: any FIR or memoryless nonlinear
filter parameterised by one or more sliders is fittable today, with
the host buffering whatever state the differentiated body cannot
carry.

The lasting friction point this exposes is not in the RAD pass
itself — it is the manual buffering of the delay line in the host.
The phase-E1 transposition path (plan §19.2) addresses this directly
by accepting LTI feedback inside `rad(...)` once the block/tape
convention is wired into the FIR backend.

### Validation

- `cargo run --release -p compiler --example rad_adaptive_notch`
  converges and asserts `|Δω| < 1e-2`.
- `cargo test --workspace` remains green.
