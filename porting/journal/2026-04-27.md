## docs(porting): plan reverse-mode AD implementation

Commit date: 2026-04-27

Source commit: this commit

Added `porting/reverse-ad-rad-implementation-plan-2026-04-27-en.md`, a
phase-gated implementation plan for reverse-mode automatic differentiation
(`rad`) in the Rust compiler.

The plan defines the proposed source contract:

```text
rad(expr, seeds) -> expr_0, expr_1, ..., d sum(expr_i)/d seed_0, ...
```

Key decisions:

- `rad` follows the explicit-seed model already used by `fad`;
- multi-output `expr` is supported from the start with an implicit all-ones
  output cotangent;
- a future VJP-style API can add custom output cotangents;
- the first implementation is signal-level and feed-forward-first;
- exact reverse-through-time for delays and recursive state is deferred to a
  separate BPTT/tape design.

No Rust code was changed in this entry.

## docs(propagate): synthesis note on FAD over de Bruijn recursive signals

Commit date: 2026-04-27

Source commit: pending

### Context

The forward-mode AD pass in `crates/propagate/src/forward_ad.rs` handles
recursive signal groups expressed in de Bruijn form, but the bookkeeping
involved (de Bruijn levels, `debruijn_depth` traversal counter, projection
slot arithmetic on the interleaved `1 + N` layout) was only documented
inside the module's Rustdoc. There was no standalone note explaining the
mechanism in one place, and in particular no formal description of the
rewrite rules used by `transform_uncached` and the `SigMatch::Proj` arm.

### Implementation

Added [docs/fad-debruijn-recursion-en.md](docs/fad-debruijn-recursion-en.md):

- Distinguishes the three index notions (de Bruijn level on `DEBRUIJNREF`,
  dynamic `debruijn_depth` counter, slot index on `Proj`).
- Describes the full sequence in the `match_de_bruijn_rec` branch of
  `transform_uncached`: cycle-breaking placeholder, depth bump, seed
  lifting via `lift_de_bruijn`, body differentiation, interleaved
  reconstruction via `de_bruijn_rec`.
- Documents the `GroupKind` classifier (`BoundRec` / `UnboundRef` /
  `Other`) used by `SigMatch::Proj` and the slot arithmetic
  `i · L + 1 + j` with `L = 1 + N`.
- Formalises the rewriter as inference rules `(Seed)`, `(Ref)`, `(Rec)`,
  `(Proj-Bound)`, `(Proj-Unbound)`, `(Proj-Other)`.
- Walks through the `fad((2 : + ~ *(p)), p)` example end-to-end and lists
  the invariants preserved by the rules (layout, depth, seed lifting,
  cycle safety, soundness boundary at unbound references).

Code is unchanged; this is purely a documentation addition.

## test(propagate,compiler): strengthen FAD de-Bruijn invariant coverage

Commit date: 2026-04-27

Source commit: pending

### Context

The synthesis note added earlier today identified the de Bruijn index
arithmetic in `forward_ad.rs` (`transform_uncached` REC arm and the
`SigMatch::Proj` classifier) as the highest-risk zone of the FAD pass.
The existing runtime corpus already covers numerical correctness through
central finite differences, but multi-seed coverage was thin on deeply
nested recursions, and there was no structural fuzzing of randomly-shaped
de-Bruijn expressions.

### Implementation

Three complementary additions:

1. `debug_assert!` invariants in `crates/propagate/src/forward_ad.rs`:
   - cached and freshly-computed `Dual`s have `tangents.len() == seed_count()`,
   - `debruijn_depth` is balanced across `transform_uncached`,
   - every REC body element has the right tangent lane count,
   - the rebuilt interleaved REC body has length `k · (1 + N)`.

2. New multi-seed central-difference harness in
   `crates/compiler/tests/fad_recursive_runtime.rs`
   (`assert_multi_seed_fad_matches_central_difference`) plus two new tests:
   - multi-seed × nested recursion (2 levels),
   - multi-seed × nested mutual recursion.
   Both compare each tangent lane to a per-seed central finite difference
   on the corresponding primal source.

3. Randomised structural test module at the bottom of `forward_ad.rs`
   (`#[cfg(test)] mod tests`). A small seeded LCG drives a generator that
   builds closed expressions over `+`, `*`, `sin`, `delay1`, `DEBRUIJNREC`
   (with valid `DEBRUIJNREF` levels and slot indices). Three tests:
   - `fad_invariants_hold_on_random_expressions` — 64 random expressions
     × varied seed configurations; the embedded `debug_assert!`s verify
     layout invariants on every visit;
   - `fad_of_seed_is_one_hot` — `fad([s0, s1], [s0, s1])` yields the
     identity Jacobian;
   - `fad_rec_proj_uses_interleaved_layout` — `Proj(0, REC([s, c]))`
     against seed `s` rewrites to `Proj(0·L, R')` / `Proj(0·L+1, R')`
     with `R'` body of length `k · (1 + N) = 4`.

No external proptest dependency is added: deterministic seeded
generation is enough for this scope and keeps the dev-deps unchanged.

### Validation

`cargo test -p propagate --lib` and `cargo test -p compiler` both pass
locally (3 new propagate unit tests, 2 new compiler runtime tests, no
regressions in the 200+ pre-existing tests).

## feat(propagate,compiler): RAD phase B — feed-forward reverse-mode core

Commit date: 2026-04-27

Source commit: pending

### Context

Second step of the reverse-AD plan. Phase B implements the
feed-forward reverse-mode core in
`crates/propagate/src/reverse_ad.rs` and wires it into the
`FlatNodeKind::ReverseAD` arm of propagation. RAD now produces the
documented `[primals…, gradient(seeds)…]` output bundle, with a sum
cotangent for multi-output primals.

### Implementation

- New module `crates/propagate/src/reverse_ad.rs`:
  - `ReverseADTransform` carrying the seed set, postorder of the active
    subgraph, and the per-`SigId` adjoint map.
  - Three-pass algorithm: (1) postorder DFS from each primal output,
    stopping descent at any seed; (2) initialize each primal's adjoint to
    `1.0` and walk the postorder in reverse, emitting local transpose
    contributions for every visited node; (3) re-emit primals and append
    the accumulated adjoint per seed (`0.0` when the seed is unreachable).
  - Adjoint contributions implement the rule table for the
    feed-forward differentiable subset that overlaps with FAD:
    arithmetic `BinOp` (Add/Sub/Mul/Div/Rem and discrete bitwise/
    comparison), `Pow`, `Atan2`, `Fmod`, `Remainder`, unary trig and
    inverse-trig, `Exp`, `Log`, `Log10`, `Sqrt`, `Abs`, `Min`, `Max`,
    `Select2`, `IntCast`/`FloatCast`, and bargraphs (zero contribution).
  - Out-of-scope families (delays, prefix, recursion, projection, tables,
    waveform, foreign functions, soundfile, pass-through wrappers,
    representation casts) raise the new `RadUnsupportedNode` error in
    line with the strict-by-default policy from §12 of the plan.
- `crates/propagate/src/lib.rs`:
  - new `PropagateError::RadUnsupportedNode { node, kind }` variant with
    `Display` and a `PROP_UNSUPPORTED_BOX` structured diagnostic that
    points at the offending family;
  - the `FlatNodeKind::ReverseAD` arm now propagates body and seed
    children onto the shared input bus, calls
    `reverse_ad::generate_rad_signals`, and emits the bundled output
    list. Phase A's arity validation runs first, so empty body/seed
    diagnostics are unchanged.

Tests:

- `crates/propagate/tests/core_api.rs`:
  - phase-A unsupported test rewritten as a positive
    `propagate_reverse_ad_succeeds_on_feed_forward_unary_call`,
  - new `propagate_reverse_ad_feed_forward_returns_primal_then_gradients`
    asserting the `[primal, da, db]` bundle layout for `rad(x*y, (x, y))`,
  - new `propagate_reverse_ad_temporal_node_emits_unsupported_diagnostic`
    asserting that `rad(x', x)` errors with `RadUnsupportedNode`.
- `crates/compiler/tests/rad_runtime.rs`: 8 new runtime tests via the
  interpreter fast lane:
  - five **central-finite-difference** checks
    (polynomial, trig composition, repeated seeds, absent seed,
    multi-output implicit-ones cotangent),
  - three **RAD-vs-FAD parity** checks (polynomial, min/max trig,
    pow/select2). Multi-output FAD parity sums the FAD lanes per seed
    to match the RAD sum-cotangent contract.
- Corpus migration:
  - `tests/corpus/rad_parse_only.dsp` → `tests/corpus/rad_basic.dsp` (now
    a positive RAD fixture, gold regenerated),
  - new `tests/corpus/err_rad_delay_temporal_unsupported.dsp` exercising
    the `rad(x', x)` strict-failure path.
- `crates/compiler/tests/diagnostic_errors.rs`: the legacy
  `reverse_ad_fixture_fails_at_propagate_stage_with_unsupported_box_code`
  test is rewritten against the new temporal-error fixture, asserting the
  `FRS-PROP-0001` code and that the message names `rad`.

### Validation

`cargo test --workspace` passes locally. Highlights:

- 8/8 new RAD runtime tests green (RAD vs FAD parity and RAD vs
  central-difference);
- 13/13 existing FAD recursive runtime tests unaffected;
- the temporal-unsupported diagnostic surfaces cleanly through
  `compile_source_to_signals`.

## feat(propagate,compiler): RAD phase C — extended primitive parity

Commit date: 2026-04-27

Source commit: pending

### Context

Third step of the reverse-AD plan. Phase C extends the RAD rule table
to cover the same primitive families that FAD already supports outside
the temporal/recursive subset:

- read-only table reads (`rdtbl(T, idx)` with `T` either a `Waveform` or
  a write-once `WrTbl(_, _, nil, nil)` — same classifier as FAD),
- unary foreign functions (`tanh` family, `sinh` family, `cosh` family,
  inverse-trig hyperbolic),
- pass-through wrappers (`Attach`, `Enable`, `Control`) and `Output`.

Mutable tables, non-unary or unrecognized FFuns, and remaining opaque
families keep raising `RadUnsupportedNode`.

### Implementation

`crates/propagate/src/reverse_ad.rs`:

- new module-level helpers `ffun_is`, `ffun_unary_kind`,
  `ffun_unary_supported`, `is_readonly_table_source` mirroring FAD;
- new `FFunUnaryKind` enum precomputes the family before constructing
  the `SigBuilder`, sidestepping a `&self.arena` / `&mut self.arena`
  borrow conflict;
- `active_children` now recognises `RdTbl(T, ridx)` with read-only
  `T`, unary `FFun` calls, `Attach` / `Enable` / `Control` (descend
  through the signal-carrying operand only), and `Output` (transparent);
- `propagate_adjoint` emits the matching adjoint contributions:
  - `rdtbl`: `idx_bar += y_bar * (rdtbl(T, idx + 1) - rdtbl(T, idx - 1)) / 2`,
  - `tanh`: `1 - tanh²(x)`, with the primal reused from `b.ffun(...)`,
  - `sinh`: `cosh(x) = sqrt(1 + sinh²(x))` rebuilt from the primal,
  - `cosh`: `sinh(x) = (exp(x) - exp(-x)) / 2`,
  - `atanh` / `asinh` / `acosh`: same chain rule as FAD,
  - pass-through: `x_bar += y_bar`,
  - `Output`: `inner_bar += y_bar`.

Tests:

- new `rad_vs_fad_parity_on_tanh_ffun` and
  `rad_vs_fad_parity_on_readonly_table_index` in
  `crates/compiler/tests/rad_runtime.rs`,
- the test harness now spawns the interpreter run on a 64 MB stack
  thread (same pattern as `signal_pipeline.rs` /
  `zita_pipeline.rs`); without it, FFun-heavy expressions trigger a
  default-stack overflow during deep tree traversal.

### Validation

`cargo test --workspace` is green; the RAD runtime suite is now 10
tests covering finite-difference and FAD parity across arithmetic,
trig, transcendentals, min/max/select2, pow, repeated/absent seeds,
multi-output sum cotangent, read-only tables, and unary FFun.

## feat(propagate): RAD phase D — conservative temporal boundary

Commit date: 2026-04-27

Source commit: pending

### Context

Final phase of the initial RAD plan. Phase D refines the strict
diagnostics around the temporal boundary: any signal family whose
reverse transpose is non-causal raises a tailored
`RadUnsupportedNode` diagnostic that explicitly explains why phase 1
RAD does not implement reverse-through-time (BPTT is reserved for a
later phase per plan §10.3).

### Implementation

`crates/propagate/src/lib.rs`:

- `IntoDiagnostic` for `RadUnsupportedNode` now switches on `kind` and
  emits family-specific notes and help:
  - `delay-or-prefix`: explains the non-causal transpose
    `adj_x[n] += adj_y[n+1]` and points at FAD as the causal escape
    hatch;
  - `recursive-projection`: explains that the recursive feedback would
    require an anti-causal transpose over an infinite stream;
  - `writable-table` / `writable-table-or-waveform-direct`: directs the
    user to read tables via `rdtable(...)`;
  - `ffun`: documents the recognized unary chain-rule set;
  - `soundfile`: states that soundfile content is not differentiated;
  - other kinds keep the generic note + help.

`crates/propagate/src/reverse_ad.rs`:

- new "Why temporal nodes refuse adjoint" docstring section explains
  the causal forward rule vs. anti-causal transpose, and links the
  policy to plan §10.3 (BPTT mode reserved for a future phase).

Tests:

- `propagate_reverse_ad_temporal_node_emits_unsupported_diagnostic`
  now also asserts that the structured diagnostic mentions both
  "non-causal transpose" and "BPTT", so the help text doesn't drift
  silently;
- new `propagate_reverse_ad_variable_delay_emits_temporal_diagnostic`
  exercises the `delay(x, d)` rejection path;
- new `propagate_reverse_ad_recursive_body_emits_temporal_diagnostic`
  builds the closed recursive primal `2 : + ~ *(0.5)` and confirms
  it is rejected with the `recursive-projection` kind.

### Validation

`cargo test --workspace` remains green. The RAD diagnostic surface
now distinguishes the temporal boundary from the merely-unimplemented
boundary, so users see actionable guidance instead of a generic
"unsupported node" message.
