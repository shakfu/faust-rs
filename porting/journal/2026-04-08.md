## refactor(signal_fir): represent sample-loop work with explicit emission phases

Commit date: 2026-04-08

Source commit: pending

### Problem

The delay/module boundary had improved, but sample-loop ordering was still
represented with implementation buckets rather than explicit phases:

- `sample_statements`
- `deferred_shift_writes`
- `compute_updates`

That made the intent harder to read, and it leaked `Shift`-specific timing
details into `module.rs`.

### Implementation

Introduced an explicit `SamplePhases` model in:

- `crates/transform/src/signal_fir/module.rs`

with three ordered stages:

- `immediate`
- `post_output`
- `sample_end`

The lowering now works as follows:

- ordinary per-sample work and direct writes go to `immediate`
- `Shift` copy loops and other post-output state advances go to `post_output`
- generic delay maintenance (`fIOTA`, `IfWrapping` counters) goes to
  `sample_end`

`DelayLoweringCtx` in `delay.rs` was updated accordingly so strategy emitters
no longer talk about raw `deferred_shift_writes`.

This also removed the old `compute_updates` / `deferred_shift_writes`
structural split from `SignalToFirLower`.

### Tests

Added structural tests in:

- `crates/transform/src/signal_fir/tests.rs`

to lock the new phase ordering:

- shift-copy updates for `@(2)` occur after the output store
- `fIOTA` bump for circular delays occurs after the output store

Adjusted one compiler integration test in:

- `crates/compiler/tests/signal_fir_lane.rs`

to avoid depending on the exact generated loop-variable suffix in `instanceClear`.

### Validation

- `cargo fmt --all`
- `cargo clippy -p transform --lib -- -D warnings`
- `cargo test -p transform --lib signal_fir`
- `cargo test -p compiler --test signal_fir_lane fastlane_delay_echo_uses_circular_delay_line_and_iota_in_c_and_cpp -- --exact`

- `CircularPow2Model` already fits an index-based delay-line model well
- `IfWrapping` is conceptually in the same family

The intended ownership boundary is:

- `delay.rs` owns delay strategy implementations and delay-only emission logic
- `module.rs` keeps orchestration, recursion integration, and
  `lower_signal(...)`-dependent decisions

## feat(signal_fir): plan recursion carriers from accumulated delay analysis

Commit date: 2026-04-08

Source commit: pending

### Problem

The new read-only delay analysis existed, but the actual fast-lane planning and
lowering still mostly followed the older local heuristics:

- recursion arrays were sized from the legacy narrow merge pattern map
- `Delay1^k(Proj(...))` chains could still fall back to standalone delay-state
  lowering
- top-level delayed recursion projections (`Proj(..., SYMREC)` outside the
  recursion body) were not yet recognized as recursion-owned delay chains
- `APF.dsp` still emitted an upsized recursion carrier plus dead `fVec` buffers

### Implementation

Integrated the accumulated delay analysis into the active planning and lowering path:

- `prepare_delay_lines` now runs `DelayManager::analyze_signals(...)` before the
  existing delay-line scan
- `ensure_recursion_array_for_group` now sizes recursion carriers from
  `rec_output_analysis`
- generalized recursion-delay reuse from the narrow
  `Delay1(Proj(SYMREF))` case to arbitrary `Delay1^k(Proj(...))` chains
- `lower_delay_state` now reads recursion carriers at offset `k + 1`
- `lower_fixed_delay` now reads recursion carriers at offset `amount + k`
- the delay pre-scan now treats both `SYMREF` and `SYMREC` projection chains as
  recursion-owned, so it no longer preallocates dead `fVec` buffers for those
  cases
- added a carrier lookup path that can reuse recursion arrays for top-level
  `Proj(..., SYMREC)` delayed accesses after forcing `lower_proj` scheduling

### Structural result

This step changes the APF-class shape materially:

- `APF.dsp` now lowers to a single recursion carrier `fRec...[4]`
- the auxiliary `fVec` delay buffers previously emitted for the output-side
  delay chain disappear

This is still not byte-for-byte C++ parity, because the fast lane keeps its
power-of-two circular strategy for merged recursion carriers larger than 2, so
`APF` uses `fRec[4]` instead of the reference scalar compiler's exact-size
`fRec[3]`. But the history ownership and buffer compaction now follow the same
carrier-level analysis.

## feat(signal_fir): add pure accumulated delay-analysis pass

Commit date: 2026-04-08

Source commit: pending

### Problem

The fast-lane delay planning still relied on a narrow local pre-scan:

- immediate `SIGDELAY(value, amount)` carriers,
- standalone non-recursive `Delay1(value)`,
- one dedicated recursion merge pattern.

That was enough to plan current delay lines, but not enough to reproduce the
reference C++ compiler's global delay analysis. In particular, chained delay
paths such as `Delay1(Delay1(Proj(...)))` or `Delay(Delay1(Proj(...)), N)` were
not available as reusable metadata before lowering.

### Implementation

Added a new read-only accumulated delay-analysis layer in
`crates/transform/src/signal_fir/delay.rs`:

- `DelayAnalysisEntry` records `max_delay` and `delay_count`
- `DelayManager` now owns:
  - `delay_analysis: HashMap<SigId, DelayAnalysisEntry>`
  - `rec_output_analysis: HashMap<(u32, usize), DelayAnalysisEntry>`
- `DelayManager::analyze_signals(...)` walks the prepared signal DAG with an
  accumulated delay counter and records:
  - per-signal delayed access metadata
  - per-recursion-output delayed access metadata keyed by symbolic variable id
    and projection index

Traversal semantics in this first step:

- `SIGDELAY(value, amount)` adds the proven delay bound
- `SIGDELAY1(value)` adds 1
- `SIGPREFIX(init, value)` adds 1 on the carried state edge
- non-delay computation nodes reset the accumulator to 0
- symbolic recursion groups entered through `Proj(..., SYMREC(...))` recurse
  into bodies with accumulator 0, while delayed `Proj(..., SYMREF(...))`
  accesses are attributed to the owning recursion output

This step is intentionally analysis-only: no FIR lowering or buffer planning
behavior changed yet.

### Tests

Added targeted unit tests in `crates/transform/src/signal_fir/tests.rs`:

- `delay_analysis_attributes_nested_delay1_chain_to_recursion_output`
- `delay_analysis_attributes_fixed_delay_over_feedback_delay1_to_recursion_output`

Validated with:

- `cargo test -p transform --lib signal_fir::tests::delay_analysis_attributes_nested_delay1_chain_to_recursion_output -- --exact`
- `cargo test -p transform --lib signal_fir::tests::delay_analysis_attributes_fixed_delay_over_feedback_delay1_to_recursion_output -- --exact`
- `cargo clippy -p transform --lib -- -D warnings`

## docs(porting): add C++ delay-analysis parity plan for signal_fir

Commit date: 2026-04-08

Source commit: pending

### Context

Analysis of `APF.dsp` against the reference Faust C++ compiler showed that the
remaining structural drift is no longer mainly about local delay-line strategy
selection, but about the absence of a true C++-style delay analysis pass in the
Rust fast lane.

Faust C++ computes delay metadata globally per signal carrier and uses it to
size compact recursion or delay buffers from the effective maximum observed
delay. The current Rust fast lane still relies on a narrower pre-scan plus a
small number of local merge patterns.

### Added plan

Added a new porting plan:

- `porting/cpp-delay-analysis-parity-plan-2026-04-08-en.md`

The document defines the next parity step:

- reproduce the C++ `maxDelay` / `delayCount` analysis model in Rust
- make `prepare_delay_lines` consume a canonical delay-analysis map
- replace narrow recursion-delay merge pattern matching with carrier-based
  accumulated-delay attribution
- drive recursion-array and delay-line sizing from analysis rather than from
  opportunistic lowering-time decisions

### Intended outcome

This plan is meant to unlock APF-class compaction in the Rust fast lane, where
Faust C++ emits one recursion carrier such as `fRec[3]` while Rust still emits
multiple auxiliary 2-slot buffers.

## fix(signal_fir): lower simple recursion groups as direct 2-slot carriers

Commit date: 2026-04-08

Source commit: 1083acf

### Problem

Simple recursion groups still used the `fIOTA`-indexed circular path in the
fast-lane lowering, even when the recursion carrier size was only 2.

That diverged from the Faust C++ reference generated for cases such as
`process = + ~ _;` and the sine phasor fixture:

- C++ uses direct two-slot carriers:
  `fRec[0] = current; out = fRec[0]; fRec[1] = fRec[0];`
- Rust still emitted `fIOTA`-based indexing for those same 2-slot recursion
  arrays.

The generated semantics remained correct, but the implementation strategy did
not match the reference compiler for simple recursion.

### Fix

Special-cased recursion arrays of size 2 in the fast-lane lowering:

- active recursion-group projection now reads the current value from slot `[0]`
  instead of `fIOTA & 1`
- recursion-feedback `Delay1(Proj(...))` now reads the previous value from slot
  `[1]`
- recursion body lowering now writes the current sample to slot `[0]`
- a deferred post-output copy stores slot `[0]` into slot `[1]`

Merged recursion-delay buffers with size greater than 2 still use the existing
`fIOTA`-indexed circular path.

### Integration test fallout

This change invalidated two `compiler` integration-test assumptions:

- the sine phasor fixture no longer always exposes `fIOTA & 1` recursion access
- the delay-echo fixture test relied on an exact loop-variable spelling in
  `instanceClear`

Updated those assertions to check the generated structure robustly instead of
requiring overly specific emitted names.

## refactor(signal_fir): pre-scan standalone Delay1 lines before lowering

Commit date: 2026-04-08

Source commit: d161b8a

### Problem

Delay-line geometry selection was split across two phases:

- `prepare_delay_lines` pre-scanned `SIGDELAY(value, amount)` and allocated one
  canonical line per carried signal.
- `lower_shift_delay1` still called `ensure_delay_line_decl(value, 1)` during
  lowering for standalone `Delay1(value)` nodes.

That mixed two responsibilities in one API:

- pre-allocating the canonical delay line,
- looking up the chosen `DelayLineInfo` during FIR emission.

It also meant standalone `Delay1` buffers were not part of the same up-front
strategy selection pass as `SIGDELAY`, even though `Delay1(x)` and `Delay(x, N)`
can share one underlying line.

### Fix

Extended `DelayManager::scan_signals` so the pre-scan also records standalone
non-recursive `Delay1(value)` nodes when `max_copy_delay >= 1`. This makes the
pre-pass responsible for all shareable delay-line geometry decisions.

Then replaced lowering-side opportunistic allocation with strict lookup:

- `lower_fixed_delay` now reads the pre-allocated line metadata via
  `delay_line_info(value)`.
- `lower_shift_delay1` now also uses `delay_line_info(value)` and assumes the
  pre-scan already chose the canonical shared geometry.

### Why this boundary is cleaner

Delay-line strategy selection is now centralized in one pass before lowering,
while FIR emission only consumes that decision. This removes the previous
"allocate-or-lookup" ambiguity from the lowering path and makes missing delay
resources fail explicitly as an internal fast-lane error.

Recursive `Delay1(Proj(...))` remains outside this mechanism because that path
can reuse recursion arrays instead of `fVec`/`iVec` delay lines.
