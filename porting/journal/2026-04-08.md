Commit date: 2026-04-08

### Problem

## refactor(signal_fir): represent sample-loop work with explicit emission phases

Commit date: 2026-04-08

Source commit: pending

### Problem

The delay/module boundary had improved, but sample-loop ordering was still
represented with implementation buckets rather than explicit phases:

- `sample_statements`
- `deferred_shift_writes`
- `compute_updates`

That made the intent harder to read, and it leaked `Shift`-specific timing
details into `module.rs`.

### Implementation

Introduced an explicit `SamplePhases` model in:

- `crates/transform/src/signal_fir/module.rs`

with three ordered stages:

- `immediate`
- `post_output`
- `sample_end`

The lowering now works as follows:

- ordinary per-sample work and direct writes go to `immediate`
- `Shift` copy loops and other post-output state advances go to `post_output`
- generic delay maintenance (`fIOTA`, `IfWrapping` counters) goes to
  `sample_end`

`DelayLoweringCtx` in `delay.rs` was updated accordingly so strategy emitters
no longer talk about raw `deferred_shift_writes`.

This also removed the old `compute_updates` / `deferred_shift_writes`
structural split from `SignalToFirLower`.

### Tests

Added structural tests in:

- `crates/transform/src/signal_fir/tests.rs`

to lock the new phase ordering:

- shift-copy updates for `@(2)` occur after the output store
- `fIOTA` bump for circular delays occurs after the output store

Adjusted one compiler integration test in:

- `crates/compiler/tests/signal_fir_lane.rs`

to avoid depending on the exact generated loop-variable suffix in `instanceClear`.

### Validation

- `cargo fmt --all`
- `cargo clippy -p transform --lib -- -D warnings`
