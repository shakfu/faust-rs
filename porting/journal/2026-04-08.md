## docs(porting): add C++ delay-analysis parity plan for signal_fir

Commit date: 2026-04-08

Source commit: pending

### Context

Analysis of `APF.dsp` against the reference Faust C++ compiler showed that the
remaining structural drift is no longer mainly about local delay-line strategy
selection, but about the absence of a true C++-style delay analysis pass in the
Rust fast lane.

Faust C++ computes delay metadata globally per signal carrier and uses it to
size compact recursion or delay buffers from the effective maximum observed
delay. The current Rust fast lane still relies on a narrower pre-scan plus a
small number of local merge patterns.

### Added plan

Added a new porting plan:

- `porting/cpp-delay-analysis-parity-plan-2026-04-08-en.md`

The document defines the next parity step:

- reproduce the C++ `maxDelay` / `delayCount` analysis model in Rust
- make `prepare_delay_lines` consume a canonical delay-analysis map
- replace narrow recursion-delay merge pattern matching with carrier-based
  accumulated-delay attribution
- drive recursion-array and delay-line sizing from analysis rather than from
  opportunistic lowering-time decisions

### Intended outcome

This plan is meant to unlock APF-class compaction in the Rust fast lane, where
Faust C++ emits one recursion carrier such as `fRec[3]` while Rust still emits
multiple auxiliary 2-slot buffers.

## fix(signal_fir): reuse shared ring-buffer strategy for `Delay1` at the `-mcd` boundary

Commit date: 2026-04-08

Source commit: 5942057

### Problem

After aligning delay-strategy thresholds with the Faust C++ reference,
`Delay(x, mcd)` now switches to the circular ring-buffer strategy at the exact
`-mcd` boundary.

That exposed a remaining assumption in `lower_shift_delay1`: the code still
expected any standalone `Delay1(x)` to resolve to a `Shift` line when
`max_copy_delay >= 1`. This stopped being true when the same carried signal
also appeared in `Delay(x, mcd)`.

The resulting failure showed up immediately on larger FIR-like impulse tests
such as `par_fir_32.dsp`, where the shared carried signal spans both `Delay1`
and `Delay(..., 16)` uses.

### Fix

Updated `lower_shift_delay1` to dispatch on the canonical pre-scanned strategy
of the shared delay line instead of assuming `Shift`:

- `Shift` keeps the existing immediate-write plus deferred-shift behavior
- `CircularPow2` now writes at `fIOTA & mask` and reads at `(fIOTA - 1) & mask`
- `IfWrapping` now writes via the per-line counter and reads through the
  wrapped `amount = 1` path

This keeps `Delay1(x)` coherent with any larger `Delay(x, N)` that shares the
same carried signal and buffer geometry.

Added a targeted structural regression test:

## fix(signal_fir): lower simple recursion groups as direct 2-slot carriers

Commit date: 2026-04-08

Source commit: 1083acf

### Problem

Simple recursion groups still used the `fIOTA`-indexed circular path in the
fast-lane lowering, even when the recursion carrier size was only 2.

That diverged from the Faust C++ reference generated for cases such as
`process = + ~ _;` and the sine phasor fixture:

- C++ uses direct two-slot carriers:
  `fRec[0] = current; out = fRec[0]; fRec[1] = fRec[0];`
- Rust still emitted `fIOTA`-based indexing for those same 2-slot recursion
  arrays.

The generated semantics remained correct, but the implementation strategy did
not match the reference compiler for simple recursion.

### Fix

Special-cased recursion arrays of size 2 in the fast-lane lowering:

- active recursion-group projection now reads the current value from slot `[0]`
  instead of `fIOTA & 1`
- recursion-feedback `Delay1(Proj(...))` now reads the previous value from slot
  `[1]`
- recursion body lowering now writes the current sample to slot `[0]`
- a deferred post-output copy stores slot `[0]` into slot `[1]`

Merged recursion-delay buffers with size greater than 2 still use the existing
`fIOTA`-indexed circular path.

### Integration test fallout

This change invalidated two `compiler` integration-test assumptions:

- the sine phasor fixture no longer always exposes `fIOTA & 1` recursion access
- the delay-echo fixture test relied on an exact loop-variable spelling in
  `instanceClear`

Updated those assertions to check the generated structure robustly instead of
requiring overly specific emitted names.

## fix(signal_fir): align `-mcd` and `-dlt` thresholds with Faust C++ delay selection

Commit date: 2026-04-08

Source commit: 197bded

### Problem

The Rust fast-lane delay strategy selection used inclusive comparisons:

- `delay <= max_copy_delay` for the Shift strategy
- `delay <= delay_line_threshold` for the circular power-of-two strategy

But the Faust C++ reference in `/Users/letz/faust/compiler/generator/instructions_compiler.cpp`
uses strict comparisons in the active delay-lowering path:

- `mxd < gMaxCopyDelay`
- `mxd < gMaskDelayLineThreshold`

Observed generated C++ confirmed the boundary behavior:

- `@(4)` with `-mcd 4` already uses `IOTA` + masked ring access
- `@(8)` with `-mcd 4 -dlt 8` already uses the wrapping-index `if` strategy

So the Rust implementation was off by one at both thresholds.

### Fix

Aligned the Rust strategy selection with the observed C++ behavior:

- Shift for `delay < max_copy_delay`
- CircularPow2 for `max_copy_delay <= delay < delay_line_threshold`
- IfWrapping for `delay >= delay_line_threshold`

Also updated the public option documentation to describe the actual threshold
behavior instead of the previous inclusive wording.

### Tests

Added targeted FIR structure tests for the four strategy boundaries:

- `fixed_delay_two_uses_unrolled_shift_copies`
- `fixed_delay_three_uses_shift_loop`
- `fixed_delay_at_mcd_boundary_uses_circular_pow2`
- `fixed_delay_at_dlt_boundary_uses_if_wrapping`

Validated with:

- `cargo clippy -p transform --all-targets -- -D warnings`
- `cargo test -p transform --lib signal_fir::tests::fixed_delay_two_uses_unrolled_shift_copies -- --exact`
- `cargo test -p transform --lib signal_fir::tests::fixed_delay_three_uses_shift_loop -- --exact`

## refactor(signal_fir): pre-scan standalone Delay1 lines before lowering

Commit date: 2026-04-08

Source commit: d161b8a

### Problem

Delay-line geometry selection was split across two phases:

- `prepare_delay_lines` pre-scanned `SIGDELAY(value, amount)` and allocated one
  canonical line per carried signal.
- `lower_shift_delay1` still called `ensure_delay_line_decl(value, 1)` during
  lowering for standalone `Delay1(value)` nodes.

That mixed two responsibilities in one API:

- pre-allocating the canonical delay line,
- looking up the chosen `DelayLineInfo` during FIR emission.

It also meant standalone `Delay1` buffers were not part of the same up-front
strategy selection pass as `SIGDELAY`, even though `Delay1(x)` and `Delay(x, N)`
can share one underlying line.

### Fix

Extended `DelayManager::scan_signals` so the pre-scan also records standalone
non-recursive `Delay1(value)` nodes when `max_copy_delay >= 1`. This makes the
pre-pass responsible for all shareable delay-line geometry decisions.

Then replaced lowering-side opportunistic allocation with strict lookup:

- `lower_fixed_delay` now reads the pre-allocated line metadata via
  `delay_line_info(value)`.
- `lower_shift_delay1` now also uses `delay_line_info(value)` and assumes the
  pre-scan already chose the canonical shared geometry.

### Why this boundary is cleaner

Delay-line strategy selection is now centralized in one pass before lowering,
while FIR emission only consumes that decision. This removes the previous
"allocate-or-lookup" ambiguity from the lowering path and makes missing delay
resources fail explicitly as an internal fast-lane error.

Recursive `Delay1(Proj(...))` remains outside this mechanism because that path
can reuse recursion arrays instead of `fVec`/`iVec` delay lines.
