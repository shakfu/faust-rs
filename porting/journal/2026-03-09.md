### `propagate` / `tests` — Add `FlatBoxId` boundary validation for the post-`eval/a2sb` subset

Commit date: 2026-03-09

Source commit: pending

#### Context

The `FlatBoxId` plan established the intended propagation boundary, but the
crate still accepted raw `BoxId` everywhere. The first implementation slice was
to make that boundary executable without yet changing arity or lowering logic.

The goal of this step was therefore:

- introduce a typed handle rooted in `TreeId`,
- validate the post-eval flat-box subset structurally,
- reject evaluator-only syntax early,
- preserve the existing propagation behavior unchanged for now.

#### Changes

- `crates/propagate/src/lib.rs`
  - added `FlatBoxId(TreeId)` as the typed handle for the post-`eval/a2sb`
    propagation subset
  - added `FlatBoxBuildError::UnexpectedPostEvalBox { node, kind }`
  - added `try_build_flat_box(...)`
  - added internal `FlatNodeKind` decoding/validation over the admissible
    families
  - documented the new boundary in RustDoc, including the C++
    `evalprocess(...) -> a2sb(...) -> realPropagate(...)` provenance

- `crates/propagate/tests/core_api.rs`
  - added tests asserting that flat-box validation accepts valid post-eval
    families:
    - `symbolic`
    - `metadata`
    - `route`
    - `ondemand`
    - `upsampling`
    - `downsampling`
    - `soundfile`
    - `environment`
  - added tests asserting that flat-box validation rejects evaluator-only
    families:
    - `case`
    - `pattern_var`
    - `abstr`
    - `modulation`
    - local-definition shells
    - `component`
    - `library`
    - applicative/iterator forms
    - `ffunction`
  - added a nested regression test to ensure a non-flat subtree inside a larger
    flat-looking composition is still rejected

#### Validation

- `cargo test -p propagate --all-targets`

#### Outcome

This step makes the boundary check explicit without changing runtime lowering
yet:

- the crate now has a typed post-eval handle,
- the contract is enforced structurally,
- the `TreeArena` sharing model is preserved,
- later steps can move `box_arity(...)` and `propagate(...)` onto this boundary
  instead of matching against the full `BoxMatch` universe.

### `porting` / `propagate` — Specify a typed `FlatBoxId` boundary for post-eval propagation

Commit date: 2026-03-09

Source commit: pending

#### Context

The current Rust `propagate(...)` API accepts a generic `BoxId`, even though the
effective C++ production route is `evalprocess(...) -> a2sb(...) -> propagate`.
That means propagation conceptually does not consume arbitrary box syntax: it
consumes a stricter post-eval box language where lambda-calculus and pattern
matching have already disappeared.

This mismatch left the Rust contract too implicit:

- `propagate` still carried `UnsupportedBox` arms for evaluator-only forms that
  should never survive the eval boundary,
- valid C++ propagation families such as `route`, `ondemand`, `upsampling`,
  `downsampling`, and `soundfile` were mixed together with dead-by-contract
  syntax in the same unsupported bucket,
- the original plan draft proposed an owned structural `PropagateBox` IR, which
  would have weakened the existing `TreeArena` hash-consing and structural
  sharing model.

#### Changes

- Added [`porting/flat-box-input-plan-2026-03-09-en.md`](../flat-box-input-plan-2026-03-09-en.md)
  as the specification plan for a typed propagation boundary.

- The plan now defines the contract in terms of:
  - `FlatBoxId(TreeId)` as the typed post-eval propagation handle,
  - `FlatNodeKind` as the restricted internal decode view,
  - `FlatBoxBuildError` as the boundary error type.

- The representation choice was explicitly revised away from an owned copied IR
  toward a `TreeId`-based handle so the design keeps:
  - TreeArena hash-consing,
  - DAG structural sharing,
  - node identity parity with the rest of the compiler.

- The plan records the C++-derived family split:
  - valid after `eval/a2sb`: literals, slots/symbolic, primitives, widgets,
    groups, composition algebra, `Environment`, `Route`, `Inputs/Outputs`,
    `Ondemand`, `Upsampling`, `Downsampling`, `Soundfile`, waveform, foreign
    forms used by propagation,
  - forbidden after `eval/a2sb`: `Ident`, `Appl`, `Access`, `IPar`, `ISeq`,
    `ISum`, `IProd`, local-definition syntax, `Component`, `Library`, `Case`,
    `PatternVar`, `Abstr`, `Modulation`.

- The plan also keeps `Environment` in the flat contract to match the C++
  `realPropagate(...)` input language, while documenting it as operationally
  trivial (`(0,0)` arity and empty signal result).

#### Outcome

The resulting design is now aligned with the repo architecture:

- the typed boundary is explicit,
- the name `FlatBoxId` communicates that residual lambda/pattern syntax is gone,
- the representation remains rooted in `TreeId` instead of duplicating the box
  DAG,
- future propagation work can separate:
  - boundary violations from `eval/a2sb`,
  - true unimplemented propagation families.

#### Next steps

- implement `FlatBoxId`
- add `try_build_flat_box(...)`
- move dead-by-contract `UnsupportedBox` cases into the flat-box builder
- migrate `box_arity(...)` and `propagate(...)` internals toward `FlatBoxId`
  plus restricted decode views
