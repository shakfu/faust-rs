# 2026-04-16

## Plan recursive alias preservation extension

Commit date: 2026-04-16
Source commit: 6dfb612

Added `porting/eval-recursive-alias-preservation-plan-2026-04-16-en.md`, a new
plan for a targeted Rust-side compiler extension that would preserve selected
recursive aliases in `eval` instead of eagerly forcing them into a loop.

The plan makes two things explicit:

- the `auto_pan3.dsp` limitation is rooted in recursive alias forcing, not in
  forward AD alone,
- the same extension would be useful beyond `fad(exp, x)`, notably for clearer
  recursive source code, helper abstractions around state, and adaptive/online
  update patterns.

## Document explicit-seed FAD support boundary

Commit date: 2026-04-16
Source commit: 4427186

Clarified the current `fad(exp, x)` contract in the porting docs after checking
the `auto_pan3.dsp` family against both `faust-rs` and the upstream C++
compiler.

- Updated `porting/fad-explicit-diff-variable-plan-2026-04-15-en.md` to state
  that explicit-seed FAD is already usable for feed-forward gradients and for
  structured recursive uses covered by the existing `fad_recursive*.dsp`
  fixtures.
- Documented the narrower unsupported case: a seed reached only through a
  recursive alias currently being formed, e.g.
  `state = next ~ _; prev = state; grad = fad(loss(prev), prev);`.
- Recorded that this limit is not presently a Rust-only regression: the same
  recursive-alias motif also triggers an endless evaluation cycle in the
  upstream C++ Faust compiler.
- Updated `porting/faust-rs-supported-faust-subset-en.md` so the supported
  subset description now matches the explicit-seed implementation instead of
  the superseded multi-control `fad(expr)` design.
