### `eval`: memoize `a2sb` lowering to preserve residual sharing

Commit date: 2026-03-18

Source commit: `pending (recorded with this journal update)`

Source: `crates/eval/src/lib.rs`, `crates/eval/tests/core_eval.rs`, `porting/MEMOIZATION.md`

#### Context

`virtual_analog_oscillators.dsp` failed during evaluation even though Faust C++
compiled it. The failure reduced to reusing one residual case-derived value
through a local abstraction body, for example `foo(poly(6))(3)` with a body
that references the bound value multiple times.

The remaining semantic gap with C++ was that Rust lowered residual closures and
pattern matchers through `a2sb(...)` without memoizing the result by node
identity. Repeated occurrences of the same residual node therefore allocated
fresh symbolic slots, destroying sharing and changing arity/behavior. Faust C++
avoids that with `gSymbolicBoxProperty`.

#### Changes

1. Added `LoopDetector.symbolic_box_cache` as a per-evaluation
   `TreeId -> TreeId` memo for `a2sb(...)`.
2. Made `a2sb(...)` consult and populate that cache so one residual node lowers
   to one stable symbolic box shape across the session.
3. Kept the earlier C++-parity cleanup in closure application:
   abstraction arguments are rebound as evaluated values, and non-closure
   arity probing goes through `a2sb(...)`.
4. Added an eval regression test covering reuse of a residual case argument
   inside a local abstraction.
5. Documented the new memoization site in `porting/MEMOIZATION.md`.

#### Validation

- `cargo test -p eval eval_process_reuses_residual_case_argument_inside_local_abstraction -- --nocapture`
- `cargo fmt --all`
- `cargo run -p compiler -- /tmp/reuse_case_twice.dsp`
- `cargo run -p compiler -- /Users/letz/faust/tests/impulse-tests/dsp/virtual_analog_oscillators.dsp`

### `eval`: count residual closure arity correctly in non-closure application

Commit date: 2026-03-18

Source commit: `pending (recorded with this journal update)`

Source: `crates/eval/src/lib.rs`, `crates/eval/tests/core_eval.rs`

#### Context

`gate_compressor.dsp` failed in propagation even though Faust C++ compiled it
successfully. The failure reduced to `displaygain(compressor_stereo(...))`,
where the evaluator treated the partially applied `compressor_stereo` closure
as a single-output argument during non-closure application.

That incorrect fallback inserted an extra implicit wire before `displaygain`,
turning a valid `2 -> 2` residual function argument into an invalid `3 -> 3`
composition that later failed with a sequential bus-width mismatch.

#### Changes

1. Replaced the local argument-output probe in `apply_list(...)` with
   `list_outputs_for_apply(...)`, which can inspect evaluator-side residual
   forms.
2. Added `infer_box_arity_for_apply(...)` so `boxClosure(...)` and
   `boxPatternMatcher(...)` arguments are first lowered through `a2sb(...)`
   before inferring their actual arity.
3. Kept the existing permissive single-output fallback for cases that still
   cannot be typed locally after symbolic lowering.
4. Added an eval regression test covering
   `displaygain(compressor_stereo(5,-30,0.01,0.1))`, asserting that the
   resulting process remains well-typed as `2 -> 2`.

#### Validation

- `cargo fmt --all`
- `cargo test -p eval eval_box_non_closure_application_counts_residual_closure_outputs -- --nocapture`
- `cargo run -p compiler -- /tmp/displaygain_comp.dsp`
- `cargo run -p compiler -- /Users/letz/faust/tests/impulse-tests/dsp/gate_compressor.dsp`

### `signal_fir`: add FIR/backend support for integer shift binops

Commit date: 2026-03-18

Source commit: `a0ec54a`

Source: `crates/fir/src/lib.rs`, `crates/transform/src/signal_fir/module.rs`, `crates/transform/src/signal_fir/mod.rs`, `crates/codegen/src/backends/{c,cpp,interp,cranelift}/`

#### Context

`priority1.dsp` failed in the fast-lane with:

`[FRS-SFIR-0005] unsupported SIGBINOP operator 'lsh' in Step 2A`

The signal layer and interpreter runtime already had integer shift semantics,
but FIR and the fast-lane lowering still rejected shift binops entirely.

#### Changes

1. Extended `FirBinOp` with explicit integer shift operators:
   - `Lsh`
   - `ARsh`
   - `LRsh`
2. Updated FIR opcode encode/decode so the new operators round-trip through
   the FIR storage model.
3. Taught fast-lane `map_binop(...)` to lower signal `Lsh` / `ARsh` / `LRsh`
   to `Int32` FIR binops.
4. Added backend support:
   - C / C++ textual emitters now render shift expressions,
   - `LRsh` is emitted with an explicit `uint32_t` cast so it stays a logical
     right shift instead of inheriting signed `>>` semantics,
   - interp bytecode lowering maps to `LshInt` / `ARshInt` / `LRshInt`,
   - Cranelift lowering maps to `ishl` / `sshr` / `ushr`.
5. Added fast-lane structural tests for left and right shift lowering.

#### Validation

- `cargo fmt --all`
- `cargo test -p transform left_shift_binop_lowers_to_int32_fir_shift -- --nocapture`
- `cargo test -p transform right_shift_binops_lower_to_int32_fir_shifts -- --nocapture`
- `cargo run -p compiler -- /Users/letz/faust/tests/impulse-tests/dsp/priority1.dsp`

### `propagate`: memoize recursive De Bruijn helpers and document compiler memoization sites

Commit date: 2026-03-18

Source commit: `f9310e2`

Source: `crates/propagate/src/lib.rs`, `porting/MEMOIZATION.md`

#### Context

A macOS sampling report on `faust-rs` showed `propagate::liftn` dominating the
hot path during recursive propagation, with repeated visits of the same shared
De Bruijn subtrees. At the same time, the repository did not yet have one
operational document listing which memoization sites already exist and which
ones remain to be added progressively.

#### Changes

1. Added a local `PropagateMemo` for recursive propagation helpers:
   - `liftn`: memoized by `(TreeId, threshold)`
   - `aperture`: memoized by `TreeId`
2. Introduced `PropagateContext` to group the internal propagation state
   (`ArityCache`, `ControlIds`, `SlotEnv`, `PropagateMemo`, `clock_env`)
   without merging analysis and lowering caches into one type.
3. Updated the recursive propagation and clocked-wrapper lowering paths to use
   the shared context and memoized helper calls.
4. Added `porting/MEMOIZATION.md` as an operational roadmap describing:
   - memoization sites already implemented across `parser`, `eval`,
     `propagate`, `normalize`, `transform`, and `tlib`,
   - candidate memoization sites to add progressively next.

#### Validation

- `cargo fmt --all`
- `cargo test -p propagate --lib --no-run`
- `cargo test -p propagate liftn_and_aperture_memoize_shared_debruijn_subtrees -- --nocapture`

### `compiler`: default interpreter runtime helpers to the fast lane

Commit date: 2026-03-18

Source commit: `2ec2cfe`

Source: `crates/compiler/src/lib.rs`, `crates/compiler/tests/signal_fir_lane.rs`

#### Context

The CLI runtime paths for `-lang interp` and the `xtask` trace commands already
defaulted to the executable transform fast lane, but the public `Compiler`
helpers `compile_*_to_interp()` still defaulted to the legacy bridge.

That API mismatch made the library surface misleading: the same backend could
run either the real fast-lane FIR lowering or the current legacy placeholder
bridge depending on which entry point was used.

#### Changes

1. Switched `compile_source_to_interp`, `compile_file_to_interp`, and
   `compile_file_default_to_interp` to default to
   `SignalFirLane::TransformFastLane`.
2. Updated the surrounding Rustdoc comments so the default interpreter API
   documents the executable fast-lane behavior instead of the legacy bridge.
3. Added a non-regression test that checks the default interpreter helper emits
   the same `.fbc` bytecode as an explicit `TransformFastLane` request.

#### Validation

- `cargo fmt --all`
- `cargo test -p compiler default_interp_api_uses_fastlane_runtime_lowering -- --nocapture`

### `signal_fir`: fix table lowering parity and typed generator expansion

Commit date: 2026-03-18

Source commit: `9c2ef69`

Source: `crates/transform/src/signal_fir/module.rs`, `crates/transform/src/signal_fir/mod.rs`

#### Context

The impulse-test table corpus exposed two parity failures in the fast-lane FIR:

1. `rwtable` lowering treated mutable tables as `Static`, which produced
   incorrect generated C++/FIR for `table1.dsp` and `table2.dsp`.
2. Compile-time generator evaluation executed integer arithmetic as raw `f64`,
   which broke the noise-table construction in `table.dsp` and yielded sample
   traces that diverged from the reference Faust C++ compiler.

#### Changes

1. Added explicit table access tracking so readonly degenerate `rdtable`
   generators stay `AccessType::Static`, while mutable `rwtable` storage is
   declared as `AccessType::Struct` and initialized per instance.
2. Emitted `StoreTable` updates for mutable tables in the sample body, instead
   of treating them as static declarations.
3. Typed generator-expanded table literals with the declared FIR element type,
   avoiding `Float64` literals in `Int32`/`Float32` tables.
4. Prepared generator signals with `prepare_signals_for_fir` before evaluation
   and taught the generator interpreter to apply integer binops with wrapping
   semantics when the prepared signal type is `SimpleSigType::Int`.
5. Added structural non-regression tests covering mutable `rwtable` lowering
   and integer generator typing.

#### Validation

- `cargo fmt --all`
- `cargo test -p transform wrtbl_runtime_write_emits_store_table_update -- --nocapture`
- `cargo test -p transform wrtbl_readonly_generator_constant_lowers_to_declared_table -- --nocapture`
- `cargo test -p transform wrtbl_generator_expansion_preserves_integer_table_types -- --nocapture`
- `cargo test -p compiler legacy_and_fastlane_both_compile_table_fixtures -- --nocapture`
- `cargo test -p compiler legacy_and_fastlane_both_compile_c_table_fixtures_without_shims -- --nocapture`
- `cargo run -p xtask -- interp-trace-dump --case /Users/letz/faust/tests/impulse-tests/dsp/table.dsp --lane fast --sample-rate 48000 --block-size 16 --num-blocks 2`

#### Notes

The `interp` legacy lane still returns zeros for `table.dsp`, but this is a
separate limitation of the current legacy bridge in `crates/compiler/src/lib.rs`:
its legacy FIR dump is intentionally skeletal (`Label(...)` placeholders, no
real executable table lowering), so it is not a runtime parity oracle for
non-trivial DSPs.
