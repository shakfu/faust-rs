# 2026-03-23

## Refactor: dead-code sweep — remove compatibility wrappers and unused predicates

### Motivation

Several crates had accumulated dead code: public compatibility wrappers that
duplicated the typed API, private C++ parity predicates that were never wired
up, and orphaned utility functions left over from earlier porting phases.

### Changes

**`propagate`** — removed four public `BoxId` compatibility wrappers:
- `box_arity(arena, id, cache)` → callers now call `try_build_flat_box` +
  `box_arity_typed` explicitly.
- `propagate(arena, id, inputs, cache)` → callers use `try_build_flat_box` +
  `propagate_typed`.
- `propagate_with_ui(...)` → same pattern with `propagate_typed_with_ui`.
- `box_arity_flat(...)` — unused `#[doc(hidden)]` alias.
- Two internal calls to the now-removed `box_arity` inside `Inputs`/`Outputs`
  lowering were inlined as `try_build_flat_box` + `box_arity_typed`.
- Tests migrated to the typed API; `propagate_typed_uses_flat_boundary_and_matches_wrapper`
  renamed to `propagate_typed_enforces_flat_boundary` and simplified.

**`boxes`** — deleted 47 private C++ parity predicates (`is_node_wire`,
`is_node_cut`, `is_node_seq`, … `is_node_soundfile`, plus `node_ident_name`).
These were ports of C++ `isBox*` functions that were never needed in Rust
because `BoxMatch` pattern matching covers the same need idiomatically.
`box-ffi` implements the `CisBox*` C API surface via `match_unary_out!` /
`match_binary_out!` macros that use `BoxMatch` directly, so the private
predicates were truly unreachable.

**`cranelift-ffi`** — removed `encode_legacy_source_backed_bitcode` (V2
source-backed format, superseded by `FAUST_CLIF_V1`) and its helper
`esc_bitcode_field` which became orphaned by that removal.

**`codegen/interp`** — removed `parse_string_token` (C++ parity,
`parseStringToken()`, never called). Kept `unquote1` — it is called in the
`test_quote_unquote` unit test.

**`interval`** — removed `union_s` (signed interval union, C++ port artifact,
never called).

### Net result

`-563 / +151` lines across 9 files; zero warnings, all tests pass.

## Fix: `hi_or2` mask-rule off-by-one → exponential recursion in type annotator

### Symptom

macOS `sample -pid <dynamic-jack-gtk> 3s` (provided as *Échantillon de
dynamic-jack-gtk.txt*) showed **100% of CPU time (1904 / 1904 samples)** inside
`interval::bitwise::hi_or2` calling itself:

```
1904 interval::bitwise::hi_or2   (offset +456)
  1904 interval::bitwise::hi_or2 (offset +432)
    1904 interval::bitwise::hi_or2 (offset +456)
      …
```

Call path:

```
TypeAnnotator::infer_binop
  → interval::ops::logic::and
    → interval::bitwise::bitwise_signed_or
      → interval::bitwise::bitwise_unsigned_or
        → hi_or2   ← infinite recursion
```

### Root cause — `crates/interval/src/bitwise.rs`

`hi_or2` implements the Hacker's Delight OR-of-intervals algorithm.  It has a
mask short-circuit rule that fires when an interval already covers a complete
power-of-2 range `[lo, 2m-1]` (all bits below m's position are set).  The check
should be:

```rust
a.hi == 2 * m - 1
```

But the code computed:

```rust
a.hi == 2 * ma.wrapping_sub(1)   // = 2*(m-1) = 2m-2  ← wrong
```

The off-by-one meant the short-circuit never fired for intervals of the form
`[0, 2^n - 1]`.  Without the short-circuit, `hi_or2` makes **3 recursive
sub-calls** at each level (the `max(…, …, …)` branch for `ma == mb` with non-empty
`a0` and `b0`), giving **O(3^32) ≈ 10^15** calls for 32-bit ranges.

`bitwise_signed_and` implements AND via De Morgan:
`AND(a, b) = NOT(OR(NOT(a), NOT(b)))`.
`NOT` of any wide signed interval (e.g. `[INT_MIN, INT_MAX]`) produces exactly
the kind of full-range unsigned intervals that triggered the explosion.  The type
annotator calls `&` for every bitwise-AND signal, making it the natural trigger.

### Fix — `crates/interval/src/bitwise.rs` line 221

```rust
// Before (wrong: 2*(m-1) = 2m-2)
if (a.hi == 2 * ma.wrapping_sub(1)) || (b.hi == 2 * mb.wrapping_sub(1)) {

// After (correct: 2m-1, wrapping-safe for m = 0x80000000)
if (a.hi == ma.wrapping_add(ma).wrapping_sub(1))
    || (b.hi == mb.wrapping_add(mb).wrapping_sub(1)) {
```

`ma.wrapping_add(ma).wrapping_sub(1)` handles the `m = 0x80000000` edge case
correctly: `0x80000000 + 0x80000000 = 0x00000000` (wrapping), then `-1 =
0xFFFFFFFF`, which is the expected `2 * 0x80000000 - 1` in u32.

### Regression test added

`hi_or2_mask_rule_terminates_on_full_power_of_two_range` in
`crates/interval/src/bitwise.rs` tests:

- `[0, 0x7FFFFFFF] OR [0, 0x7FFFFFFF]` (lower half)
- `[0x80000000, 0xFFFFFFFF] OR [0x80000000, 0xFFFFFFFF]` (upper half)
- `[0, 0xFFFFFFFF] OR [0, 0xFFFFFFFF]` (full u32)
- `bitwise_signed_and([INT_MIN, INT_MAX], [INT_MIN, INT_MAX])` (De Morgan path)

### Result

| DSP                      | Before   | After  | Reference faust |
|--------------------------|----------|--------|-----------------|
| `guitarEffectChain.dsp`  | ∞ / 9.7 s | 2.0 s | — |
| `minimoog-novation.dsp`  | crash¹ / long | 4.7 s | 8.7 s |

¹ The Cranelift crash was fixed separately; the hang was the parser/type-annotator.

All 65 interval unit tests pass.

## fix(eval): integer div folding in seq numeric — `4/2` must produce `Int(2)`, not `Real(2.0)`

### Symptom

`zita_rev1.dsp` (and any DSP using `butterfly`/`hadamard`/`selector` from
`math.lib`) failed with:

```
sequential composition mismatch: left outputs (0) != right inputs (16)
```

### Root cause — `crates/eval/src/lib.rs` (`try_fold_seq_numeric`)

Commit `80b4701` changed `fold_binop` so that `Int(4) / Int(2)` produces
`Real(2.0)` instead of `Int(2)`.  This is correct at signal level (Faust
division is always real-valued), but breaks eval-time constant folding where
the result feeds into pattern matching.

Faust's `selector(i,n)` uses `with { S(i,i) = _; S(i,j) = !; }`.  Pattern
matching compares tree nodes by identity: `Real(2.0) != Int(2)`.  When
`selector(4/2, 4)` folds `4/2` to `Real(2.0)` instead of `Int(2)`, pattern
`S(i,i)` never matches, all outputs become cuts (`!`), and `butterfly(2)`
produces 0 outputs instead of 2.

### Fix — `crates/eval/src/lib.rs`

1. Added `all_inputs_are_int` helper: walks a `Par`/`Int`/`Real` tree and
   returns `true` only if every leaf is `Int`.

2. In `try_fold_seq_numeric`: when all inputs are `Int` and the signal result
   is `Real(x)` where `x.fract() == 0.0` and `x` fits `i32`, convert back
   to `Int(x as i32)`.

This preserves the "div always real" semantics at signal level while restoring
integer identity for eval-time pattern matching.

### Result

`zita_rev1.dsp` compiles successfully.  All existing tests pass.

5. LLVM/WASM backends scaffold only (P2)

## fix(test): update import_loading_cpp_gap — gap is now closed

### Symptom

After the docs commit, `cargo test` failed:

```
thread 'operator_test_is_still_the_known_cpp_only_import_loading_reproducer' has overflowed its stack
fatal runtime error: stack overflow, aborting
```

### Root cause

Two separate issues:

1. **The gap was already closed** by commit `c5ffe67` (zero-arity last-import-wins
   fix).  `dx7_tests.dsp` imports both `stdfaust.lib` and `demos.lib`, which both
   define `ba = library("basics.lib")` (and 18 other aliases).  Before `c5ffe67`
   this produced 19 parse errors and bailed early — the test saw "undefined symbol
   `ba`" as expected.  After `c5ffe67` all 19 aliases resolve correctly and
   evaluation completes successfully, making `expect_err` panic.

2. **Stack overflow at default 2 MB thread stack**.  `dx7_tests.dsp` / `operator_test`
   requires more than 2 MB of stack for deep recursive evaluation.  With
   `RUST_MIN_STACK=8388608` the compilation completes successfully, confirming the
   overflow is a resource limit, not an infinite recursion.

### Fix — `crates/compiler/tests/import_loading_cpp_gap.rs`

- Added `run_with_large_stack` helper (same pattern as `zita_pipeline.rs`):
  `thread::Builder::new().stack_size(64 * 1024 * 1024)`.
- Renamed test from `operator_test_is_still_the_known_cpp_only_import_loading_reproducer`
  to `operator_test_import_loading_parity_is_closed`.
- Converted from gap-tracker (expects `Err("undefined symbol ba")`) to
  parity-success (expects `Ok`).
- Updated module doc comment to reflect closed status.
