## feat(cranelift): support registered foreign globals and functions in backend and FFI

Commit date: 2026-03-27
Source commit: working tree after `f62c13d`

Completed the first explicit foreign-symbol registration model for the
Cranelift path so `fvariable(...)` and non-built-in `ffunction(...)` calls can
be compiled through a name + pointer contract instead of the earlier hardcoded
bring-up subset.

### Scope

- Updated
  [crates/codegen/src/backends/cranelift/mod.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/backends/cranelift/mod.rs)
  to:
  - add `CraneliftOptions::extern_data_symbols` for FIR `AccessType::Global`
    scalar bindings (`fvariable`)
  - add `CraneliftOptions::extern_function_symbols` for non-built-in FIR
    `FunCall` imports (`ffunction`)
  - register both data and function symbols in the Cranelift JIT builder
  - lower `LoadVar { access: Global }` through imported data symbols instead of
    rejecting globals as illegal `dsp*` fields
  - lower custom foreign function calls through explicit imported function
    signatures when the symbol is registered
  - keep subset-gap detection aligned with the new option maps so missing
    bindings fall back cleanly to the compute stub in non-strict mode
- Added focused backend regressions in
  [crates/codegen/src/backends/cranelift/tests.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/backends/cranelift/tests.rs)
  covering:
  - global scalar loads with and without `extern_data_symbols`
  - custom foreign function calls with and without `extern_function_symbols`
- Updated
  [crates/cranelift-ffi/src/factory.rs](/Users/letz/Developpements/RUST/faust-rs/crates/cranelift-ffi/src/factory.rs)
  to:
  - expose `registerCCraneliftForeignFunction(const char* name, void* fn_ptr)`
  - maintain a process-global foreign-function registry in the FFI layer
  - snapshot that registry into `CraneliftOptions::extern_function_symbols`
    during factory compilation
  - include the foreign-function registry fingerprint in factory
    `compile_options` and SHA/cache identity so cached factories do not silently
    reuse stale bindings
  - add an integration-style factory test proving a registered foreign function
    is honored by the public Cranelift FFI build path

### Validation

- `cargo fmt --all --check`
- `cargo test -p codegen cranelift -- --nocapture`
- `cargo test -p cranelift-ffi factory -- --nocapture`

## fix(wasm): restore soundfile runtime parity for faustwasm

Commit date: 2026-03-27
Source commit: working tree after `d2770a4`

successfully but produce silence inside `faustwasm`.

  [crates/codegen/src/json.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/json.rs)
  to the historical C++ form expected by `faustwasm`:
  - `{'foo'; 'bar'}` now renders as `{-foo-;-bar-}`
  - documented the exact runtime tokenization rule and failure mode in Rustdoc
- Added a JSON non-regression in
  [crates/codegen/src/json/tests.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/json/tests.rs)
  to lock the `soundfile.url` contract
- Updated the WASM linear-memory layout in
  [crates/codegen/src/backends/wasm/layout.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/backends/wasm/layout.rs)
  so `Soundfile*` struct fields are moved first, matching the C++ WASM backend
- Extended the WASM compute lowerer in
  [crates/codegen/src/backends/wasm/mod.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/backends/wasm/mod.rs)
  to support:
  - `LoadSoundfileLength`
  - `LoadSoundfileRate`
  - `LoadSoundfileBuffer`
- Aligned the WASM memory contract for soundfile DSPs:
  - soundfile modules now force imported/external memory
  - the backend now rejects unsupported `compute` bodies up front instead of
    silently emitting an empty `compute` stub
- Added focused WASM regressions in
  [crates/codegen/src/backends/wasm/tests.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/backends/wasm/tests.rs)

The Rust `faustwasm` compiler-module now regenerates `soundfile.dsp` artifacts

- the DSP field offset exposed as `ui[*].index` matches the installed
  `Soundfile*` pointer slot,
- the emitted WASM module uses imported memory and a real `compute` body rather
  than silently compiling to no-op audio.
