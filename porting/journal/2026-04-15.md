# 2026-04-15

## docs(propagate): enrich forward_ad.rs and lib.rs with full FAD rule table

Commit date: 2026-04-15
Source commit: pending (recorded by the step commit that includes this journal entry)

### Context

`crates/propagate/src/forward_ad.rs` existed with working code but minimal
documentation: the module-level doc listed scope, integration contract, and
ordering invariant, but did not explain the dual-number algebra or enumerate
the differentiation rules. Reviewers had to reconstruct the mathematical
intent from the signal-builder calls.

`crates/propagate/src/lib.rs` had no dedicated FAD section; the `fad`-related
behavior was scattered across `box_arity_typed`, `propagate_typed_with_ui`,
and inline comments in `propagate_inner`.

### What was done

#### `forward_ad.rs` — module doc rewritten

Added a `# Dual-number algebra` section explaining the `u + ε·u'` carrier
and the single-control seed convention, followed by a comprehensive
`# Differentiation rules by node kind` reference with one table per category:

- **Constants / audio inputs**: tangent 0
- **UI controls**: seed 1 for the selected control, 0 for all others;
  `button`/`checkbox` always 0
- **Arithmetic BinOp**: full table covering `Add`, `Sub`, product rule,
  quotient rule, `Rem`; shifts/bitwise/comparisons → 0
- **Unary transcendentals**: chain-rule table for all 11 functions
  (`sin`, `cos`, `tan`, `exp`, `log`, `log10`, `sqrt`, `abs`,
  `acos`, `asin`, `atan`) with explicit `f'(x)` and emitted expression
- **`pow(x,y)`**: general power rule `x^y · (y'·ln(x) + y·x'/x)`
- **`min`/`max`**: piecewise sub-gradient via `select2`
- **`delay1`**: linear shift, tangent is `delay1(x')`
- **`delay(x,d)`**: discrete-time Leibniz decomposition (content + time
  derivative via backward finite difference)
- **Casts**: `float_cast` linear; `int_cast` → 0; `bit_cast` → 0
- **`select2`, `prefix`**: transparent
- **`sigRec`/`sigRef`/`sigProj`**: parallel tangent group with `FAD_<var>`
  naming and cycle-breaking placeholder
- **Pass-through helpers** (`attach`, `enable`, `control`): left tangent
  forwarded only
- **Bargraphs**: zero tangent
- **Fallback**: zero tangent, primal unchanged

Added inline rule comments on every `transform_uncached` match arm and
a rule table in the `transform_binop` doc comment.

Enriched `unary_chain` doc to describe the two closure parameters precisely.

#### `lib.rs` — new `# Forward-mode automatic differentiation (FAD)` section

Added after `# Integer convention`, documenting:

- output arity expansion formula and its relation to `BoxADControlsCounter`
- the `[autodiff:false]` opt-out and why the box-level count is an upper bound
- the two-phase suppress/expand protocol for the `Rec` combinator
- reverse-mode AD status (`UnsupportedBox`)

#### Variable-delay simplification

Removed the unnecessary `zero` and `term2` intermediates in the
`SigMatch::Delay` arm:

```rust
// Before:
let zero = b.real(0.0);
let scaled_delay = b.mul(dual_d.tangent, delayed_time_gradient);
let term2 = b.sub(zero, scaled_delay);
Dual { primal, tangent: b.add(term1, term2) }

// After:
let scaled_delay = b.mul(dual_d.tangent, delayed_time_gradient);
Dual { primal, tangent: b.sub(term1, scaled_delay) }
```

`add(term1, sub(0, scaled))` and `sub(term1, scaled)` are identical;
the rewrite removes one intermediate signal node from the graph.

#### `porting/faust-rs-supported-faust-subset-en.md` updated

- Date bumped to 2026-04-15.
- Section 3: evidence block added for 2026-04-15 FAD documentation.
- Section 5.1: `fad(expr)` bullet added to the source-level subset list.
- Section 5.3: two new exclusion entries:
  - `rad(expr)` — completely unsupported (`UnsupportedBox`)
  - `fad(expr)` — propagation supported, end-to-end conditional;
    `[autodiff:false]` arity upper-bound caveat explained
- Section 6.1: FAD bullet added to "where Rust is close to C++".
- Section 6.2: `rad(expr)` added as the first item under "where C++ is
  still broader".
- Section 7.16 (new): full historical entry covering the rule table, the
  suppress/expand protocol, the `[autodiff:false]` opt-out, the 22 corpus
  entries with descriptions, validation status, and `rad` out-of-scope note.
- Section 8: `fad`/`rad` practical reading rule added.

### Decision

- The differentiation rules are now the canonical reference inside the
  codebase; the port plan and this journal entry provide historical context.
- The variable-delay simplification is a pure code-quality change with no
  semantic effect.

### Validation

- `cargo check -p propagate`
- `cargo doc -p propagate --no-deps` (no new warnings)

## docs(porting): plan `fad(exp, x)` explicit differentiation variable

Commit date: 2026-04-15
Source commit: pending (recorded by the step commit that includes this journal entry)

### Context

The current `fad(exp)` primitive differentiates `exp` against every
differentiable UI control it can reach, filtered by `[autodiff:false]`. The
implementation in `crates/propagate/src/forward_ad.rs` carries an
`ADControlCollector` that walks the signal DAG, consults the
`ControlSpec` registry, sorts controls by label, and runs one
`ForwardADTransform` instance per control. Output arity of `fad(exp)` is
`outputs * (1 + n_controls)`, which also forces the `FlatNodeKind::Rec` arm
of `propagate_inner` to count controls in both branches to predict the
post-expansion arity.

This design has several shortcomings:

- Coupling to UI metadata: `fad` cannot target an audio input, a derived
  signal, or any non-UI leaf.
- Unstable output layout: the label-based sort is the only thing keeping
  tangent order deterministic across runs.
- Combinatorial growth inside `~`: each extra control multiplies the
  tangent bundle.
- Divergence from the standard forward-mode AD contract (`jvp`-style), where
  the caller picks a single tangent direction.

### What was done

A full port plan `porting/fad-explicit-diff-variable-plan-2026-04-15-en.md`
was written to replace `fad(exp)` by the two-argument form `fad(exp, x)`.
The key points:

- **Surface**: `x` is a block whose **output arity is 1**. No constraint on
  its input arity — `x` may be a pure producer (slider, constant) or any
  `k → 1` transformer (`_`, `abs`, `sin`, `*(k)`, …). The outer `fad` exposes
  `max(exp.inputs, x.inputs)` inputs; `x` and `exp` share the same upstream
  bus, so `Input(i)` nodes hash-cons-share `SigId` between the two
  subgraphs.
- **Semantics**: directional derivative along the single propagated signal
  produced by `x`. The seeding rule collapses to a pointer comparison
  `sig == diff_seed` at the top of `transform_uncached`; every other leaf
  (sliders, inputs, constants, bargraphs, buttons) returns `tangent = 0`
  unconditionally. No enumeration, no per-control loop.
- **Output layout**: `[primal_1, tangent_1, primal_2, tangent_2, …]`, arity
  `outputs * 2`, independent of how many controls `exp` contains.
- **Box / parser / eval**: `BoxMatch::ForwardAD(exp, seed)` (two children),
  grammar rule `FAUTODIFF LPAR Expression COMMA Expression RPAR`,
  evaluator re-wraps both children, `infer_box_arity` and
  `box_arity_typed` check `seed.outputs == 1` and raise
  `EvalError::FadSeedShape` / `PropagateError::FadSeedArity` otherwise.
- **Propagate**: `FlatNodeKind::ForwardAD { body, seed }`; `propagate_inner`
  routes `inputs[..seed_arity.inputs]` to the seed, propagates it to get one
  `SigId`, then either calls `generate_fad_signals(arena, &body_sigs, seed_sig)`
  directly or defers through the existing `suppress_fad` mechanism when
  inside `~`. A new `ctx.pending_fad_seeds` list carries the seed across the
  `Rec` boundary so the deferred expansion retains the correct target.
- **`Rec` extra-outputs formula** collapses from
  `core_outputs * (1 + n_controls)` to
  `core_outputs + Σ count_ad_tangents(branch)`; `count_ad_controls` is dead
  and deleted.
- **`forward_ad.rs` simplification**: `ADControlCollector`,
  `is_autodiff_enabled`, `sort_controls_by_label`, and the dependency on
  `ControlSpec` / `ctx.ui_controls` are removed. The transform becomes
  single-pass per `fad` node.
- **Migration**: hard cutover. No alias, no feature flag. Existing corpus
  files are rewritten in the same PR. Plans
  `autodiff-forward-ad-port-plan-2026-04-13-en.md` and
  `fad-ddsp-library-plan-2026-04-14-en.md` will receive a banner pointing at
  the new plan. The C++ structural parity test for `fad` is gated behind a
  skip flag until upstream Faust adopts the same form.

### Test matrix documented in the plan

- Seed = slider used inside body (`fad(x*x, x)` → primal `x*x`, tangent `2x`).
- Seed = derived expression (`fad(sin(a*b), a*b)` → tangent `cos(a*b)`).
- Seed absent from body → every tangent is 0.
- Seed with 1 input (`fad(_ : sin, _)` → tangent `cos(input_0)`).
- Seed as transformer (`fad(_ : abs : sin, _ : abs)` → tangent `cos(abs(input_0))`).
- Seed = literal constant → tangent collapses to 0.
- Multiple `fad` inside `~` with `suppress_fad` active.
- Nested `fad(fad(x*x, x), x)` → four signals (primal, tangent, tangent of
  primal, tangent of tangent).

### Decision

- Land the rewrite as a single atomic PR covering boxes, parser, eval,
  propagate, forward_ad, corpus, and documentation banners.
- Treat this change as the authoritative form going forward; the
  single-argument `fad(exp)` form is removed without a deprecation window
  because the feature has not shipped in a tagged release yet.
- `rad` stays parse-only and single-argument (stub). It can follow the same
  two-argument redesign later if reverse-mode lands.

### Validation

- Plan written under `porting/` per the standard porting-plan workflow.
- No code changes in this step; implementation will follow as a sequence of
  phase-A through phase-H commits described in §10 of the plan.

## feat: implement fad(exp, x) — phases A–G

Commit date: 2026-04-15

### Context

Implementation of the plan from
`porting/fad-explicit-diff-variable-plan-2026-04-15-en.md`. Full hard cutover
from single-argument `fad(exp)` to two-argument `fad(exp, x)` where `x` is a
block with exactly one output.

### Changes landed

- **Phase A (boxes)**: `ForwardAD(BoxId)` → `ForwardAD(BoxId, BoxId)` (exp,
  seed). Tag moves from 1-child to 2-child arm in `matcher.rs`. Builder
  takes two args.
- **Phase B (parser)**: Grammar rule `FAUTODIFF LPAR Argument PAR
  Argument RPAR`. Old single-arg form rejected. C++ structural parity
  test updated.
- **Phase C (eval)**: Both children evaluated under the same lexical env.
  `infer_box_arity` returns `(max(exp_ins, seed_ins), exp_outs * 2)`.
- **Phase D (propagate)**: `FlatNodeKind::ForwardAD { body, seed }`.
  New error `FadSeedArity`. `count_ad_controls` removed, `count_fad_nodes`
  added. `PropagateContext` gains `pending_fad_seeds: Vec<SigId>`.
  `collect_ui_nodes` walks seed subtree. `ui_controls` field removed from
  `PropagateContext`.
- **Phase E+F (forward_ad)**: `ADControlCollector` deleted. `diff_control:
  ControlId` → `diff_seed: SigId`. Seed rule: pointer equality
  `sig == diff_seed` before the match. Sliders always return tangent 0.
  `generate_fad_signals(arena, outputs, diff_seed)`.
  `generate_fad_signals_multi(arena, outputs, seeds)` for Rec.
- **Phase G (corpus)**: All `fad_*.dsp` rewritten. `fad_autodiff_false.dsp`
  removed.

### Validation

`cargo test -p boxes -p parser -p eval -p propagate`
