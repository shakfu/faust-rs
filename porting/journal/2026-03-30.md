## docs(porting): broaden wasm ffunction plan into unified foreign-symbol support plan

Commit date: 2026-03-30
Source commit: working tree after `13e15e5`

Expanded the WebAssembly foreign-import planning note from a function-only
scope to a unified foreign-symbol plan covering `ffunction(...)`,
`fconst(...)`, and `fvar(...)`.

### Scope

- rewrote
  [porting/wasm-ffunction-support-plan-2026-03-30-en.md](/Users/letz/Developpements/RUST/faust-rs/porting/wasm-ffunction-support-plan-2026-03-30-en.md)
  into a broader plan centered on one wasm host ABI for foreign symbols
- documented the current Rust pipeline/backend split:
  - generic `ffunction(...)` still blocked in the wasm backend except for the
    historical helper whitelist
  - `fSamplingFreq` / `fSamplingRate` remain special lowered runtime constants
  - `count` remains a special lowered runtime variable while other `fvar(...)`
    accesses are still rejected in current wasm mode
- added a staged rollout:
  - preserve foreign-symbol provenance into wasm planning
  - generalize `ffunction(...)` imports first
  - add getter-based `fconst(...)` support next
  - add getter-based read-only `fvar(...)` support after that
- recorded the recommended ABI policy:
  - keep `fSamplingRate` and `count` internal
  - prefer getter-style imports for portable `fconst(...)` / `fvar(...)`
    support before considering imported globals
