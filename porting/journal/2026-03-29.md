## feat(parser,eval,compiler): expand parsed import nodes structurally like C++

Commit date: 2026-03-29
Source commit: working tree after `4ce6bf6`

Ported the next parser-parity step by moving file-backed import handling onto a
structural expansion pass over parsed `importFile(...)` nodes instead of
flattened source text, then added eval/compiler regressions for the reduced
inline `environment { import("..."); }` case that originally exposed the
miscompile.

### Scope

- Added a documented `StructuralImportExpander` in
  [crates/parser/src/lib.rs](/Users/letz/Developpements/RUST/faust-rs/crates/parser/src/lib.rs)
  that mirrors the C++ `SourceReader::getList(...)` ->
  `SourceReader::expandList(...)` boundary:
  - parses each file as its own source unit
  - expands `ImportFile` nodes structurally from parsed definition lists
  - rewrites nested local-definition containers so inline environment imports
    expand correctly inside `environment { ... }.process`
  - preserves deterministic `used_files` visitation order and shared metadata
    accumulation across imported files
- Narrowed
  [crates/parser/src/source_reader.rs](/Users/letz/Developpements/RUST/faust-rs/crates/parser/src/source_reader.rs)
  to provide raw source-unit resolution/loading helpers for the structural
  parser path instead of extending the old text-level import flattener
- Added parser/eval/compiler parity coverage:
  - [crates/parser/tests/structural_cpp_differential.rs](/Users/letz/Developpements/RUST/faust-rs/crates/parser/tests/structural_cpp_differential.rs)
    now asserts file-backed parsing expands inline environment imports and
    removes residual `ImportFile` nodes from the local definition list
  - [crates/eval/tests/core_eval.rs](/Users/letz/Developpements/RUST/faust-rs/crates/eval/tests/core_eval.rs)
    now checks `component("child.dsp")` accepts a loaded file whose local
    environment contains an inline import
  - [crates/compiler/tests/inline_environment_import_parity.rs](/Users/letz/Developpements/RUST/faust-rs/crates/compiler/tests/inline_environment_import_parity.rs)
    now checks the Rust compiler accepts the same reduced shape as C++, and
    differentially confirms the adjacent C++ compiler when `FAUST_CPP_BIN` is
    available
- Updated parity-facing comments:
  - [crates/eval/src/lib.rs](/Users/letz/Developpements/RUST/faust-rs/crates/eval/src/lib.rs)
    now documents that loaded files are parsed and expanded structurally like
    C++ before cloning definitions into the eval arena
  - [crates/propagate/src/lib.rs](/Users/letz/Developpements/RUST/faust-rs/crates/propagate/src/lib.rs)
    now explicitly rejects leftover `ImportFile` nodes with a comment pointing

## feat(parser,boxes): preserve `importFile` nodes through parse and definition normalization

Commit date: 2026-03-29
Source commit: working tree after `af15cd3`

Started the structural C++ import-parity execution by adding an explicit
`importFile` box node to the Rust parser/boxes layer and stopping
`import("...")` from being erased to `nil` during parsing.

### Scope

- Added a dedicated parser/import node in `crates/boxes`:
  - new tag constant and internal constructor for C++ `importFile(Tree filename)`
  - `BoxBuilder::import_file(...)`
  - `BoxMatch::ImportFile(...)` with Rustdoc documenting the C++
    `isImportFile(...)` role and the delayed structural-expansion boundary
- Updated `crates/parser/src/lib.rs` so:
  - `import_statement(...)` now returns a structural import node while still
    recording parser-context import metadata
  - `format_definitions(...)` preserves import nodes instead of trying to group
    them as ordinary definitions
  - Rustdoc now states that import-file nodes survive normalization and are
    intended for later C++-style `expandList(...)` handling
- Added focused parser regressions in
  [crates/parser/tests/structural_cpp_differential.rs](/Users/letz/Developpements/RUST/faust-rs/crates/parser/tests/structural_cpp_differential.rs):
  - top-level `import("ops.lib");` survives parse as `ImportFile`
  - inline `environment { import("child.dsp"); }.process` preserves the local
    import node structurally before any file-backed expansion path runs

### Validation

- `cargo fmt --all`
- `cargo test -p parser --test structural_cpp_differential --test api_bridge`
