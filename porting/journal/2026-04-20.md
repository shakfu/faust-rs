## Fix eval stack overflow on recursive FAD + local projection (follow-up to 2026-04-17)

Commit date: 2026-04-20
Source commit: pending

Running `faust-rs auto_wah_origin.dsp` aborted the process with
`thread 'main' has overflowed its stack / fatal runtime error: stack overflow`.
The reference Faust C++ compiler rejects the same pattern gracefully with
`ERROR : stack overflow in eval`.

### Root cause

`wahFreq = step ~ _` whose body references `dloss_df = fad(lossExpr, wahFreq)`
creates a self-referential cell where `wahFreq`, appearing as the second
argument of `fad`, is re-evaluated outside its enclosing `~`. The evaluator
enters `a2sb` / `a2sb_value` repeatedly, and each iteration allocates a fresh
`boxSlot` in `lower_abstraction_to_symbolic_value`. Because the slot is unique,
`LoopDetector::symbolic_box_cache` never hits and the identity-keyed
`LoopDetector::call_stack` guard in `enter_tree` / `enter_symbol_env` also
never fires — those guards only run from `eval_ident_value`, not from the
structural descent of `a2sb`.

The OS stack overflowed before any Rust error could propagate, so the process
aborted instead of returning a diagnostic.

### Fix

`crates/eval/src/loop_detector.rs`: added a `structural_depth: usize` counter
plus `enter_structural()` / `leave_structural()`. The counter is capped at
`min(max_depth, 256)` — a tighter bound than the 1024 symbol-lookup budget
because every structural hop also drags `eval_value` and
`apply_value_list_value` frames onto the OS stack, so each iteration consumes
an order of magnitude more bytes than a symbol-resolution loop.

`crates/eval/src/lib.rs`: `a2sb` and `a2sb_value` now wrap their bodies with
the new guard. `a2sb` was factored into an outer function (cache + guard) and
an inner `a2sb_match` that performs the actual match, so the guard entry/exit
runs exactly once per logical `a2sb` call and the cache still memoises
successful lowerings.

### Result

`faust-rs auto_wah_origin.dsp` now returns:

```
error [FRS-EVAL-0099] evaluation recursion depth exceeded (256)
```

No process abort, diagnostic shape matches the rest of the evaluator's error
surface. Validation: `cargo test -p eval` (72 passed), `cargo test -p compiler
--test signal_pipeline` (67 passed, including the recursive-FAD families).

The DSP itself remains semantically divergent and is still not compilable;
the fix only guarantees a clean error instead of a stack-overflow abort.

## Extend `fad(expr, seed)` to multi-output seeds

Commit date: 2026-04-20
Source commit: pending

Until now `fad(expr, seed)` required `seed` to produce exactly one output
signal. That forces the DSP author to instantiate one `fad` node per
differentiation variable even when several independent variables share the
same primal and identical propagation structure.

### Change

- `crates/propagate/src/lib.rs`: `box_arity_typed` now accepts any
  `seed_outputs >= 1`; the node advertises
  `outputs = body_outputs * (1 + seed_outputs)` instead of the hard-coded
  `* 2`. Only `seed_outputs == 0` is still rejected with `FadSeedArity`.
- `crates/propagate/src/lib.rs`: the `ForwardAD` propagation arm collects all
  seed signals and routes them through `forward_ad::generate_fad_signals_multi`
  (or pushes all of them into `pending_fad_seeds` under `suppress_fad`).
  The single-seed path is gone — `M = 1` is now just the degenerate case of
  the multi-seed pipeline.
- `crates/propagate/src/forward_ad.rs`: removed the now-unused
  `generate_fad_signals` helper and updated the module docstring to point at
  `generate_fad_signals_multi` as the canonical driver. The per-seed tangent
  layout is unchanged: `[p, t_s0, t_s1, …, t_s{M-1}]` per primal output.
- Error messages / diagnostic notes updated: "seed must produce at least 1
  output" instead of "exactly 1 output".

### Tests

- `crates/propagate/tests/core_api.rs`: two new propagation tests — a 2-seed
  case on `mul(f, g)` asserting the expanded arity (`1 * (1 + 2) = 3`) and a
  zero-output seed case (via `environment()`) asserting the dedicated
  `FadSeedArity { outputs: 0 }` error.
- `tests/corpus/fad_multi_seed.dsp`: corpus fixture
  `fad(f*g : sin, (f, g))` exercising a `par`-built two-output seed.
- `crates/compiler/tests/signal_pipeline.rs`: matching end-to-end test
  asserting 3 process outputs and both seed controls in the UI.

### Validation

`cargo test -p propagate` (42 passed), `cargo test -p compiler --test
signal_pipeline fad` (33 passed including the new `fad_multi_seed` test).
The release binary compiles the new corpus fixture to three C++ output
lanes: `sin(f*g)`, `g·cos(f*g)`, `f·cos(f*g)` — matching the forward-mode
dual-number derivation.

The user-facing subset document (`porting/faust-rs-supported-faust-subset-en.md`)
is updated to document the multi-output seed layout.
