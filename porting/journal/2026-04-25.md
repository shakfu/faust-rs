## perf(eval): reduce environment lookup overhead

Commit date: 2026-04-25

Source commit: pending

### Context

`faust-rs -time` showed that `clarinetMIDI.dsp` spends most of its time in the
Rust evaluation phase. Sampling the release binary showed deep recursive
`eval_value` stacks around identifier lookup, `map_children`, `eval_seq_value`,
and applications. Two attempted memoization probes were not retained: a
one-argument beta-reduction cache had very few hits on this DSP, and caching
numeric `seq` folding did not improve the measured time.

### Change

- Replaced the evaluator environment store from `Arc<Mutex<EnvStore>>` to
  `Rc<RefCell<EnvStore>>`, since one evaluation session is single-threaded and
  paid the mutex cost on every environment access.
- Added a per-layer `binding_index` so symbol lookups and local redefinition
  checks use direct symbol-to-binding lookup instead of reverse linear scans.

### Validation

- `cargo fmt --all`
- `cargo build --release -p compiler`
- `target/release/faust-rs -time /Users/letz/Developpements/Recherche/WAC/WAC\ 2017/Faust/clarinetMIDI.dsp -o /tmp/faust_rs_env_final.cpp`
- `cargo clippy -p eval -p compiler --all-targets -- -D warnings`
- `cargo test -p eval -p compiler --lib`

### Notes

On `clarinetMIDI.dsp`, release evaluation time moved from roughly `0.57s` in
the baseline run to roughly `0.53s` after the environment-store change. This is
a modest improvement; the remaining gap is not explained by a single missing
memoization cache. The next promising areas are repeated recursive
`eval_seq_value`/`map_children` work and eval-time parsing of `library(...)`
dependencies such as physical-model libraries.

## cli: expose internal compilation timings

Commit date: 2026-04-25

Source commit: pending

### Context

The C++ compiler exposes `-time`, which prints internal phase timings such as
parser, evaluation, propagation, signal preparation, and code generation. The
Rust CLI already accepted `-time`/`--compilation-time`, but it only reported a
coarse outer CLI phase, which was not enough to understand why some WAC 2017
programs compile slower in `faust-rs`.

### Change

- Added a compiler facade timing sink so internal stages can report durations
  without hard-wiring CLI output into the library.
- Wired parser, evaluation, box flattening, arity, propagation, signal type
  validation, signal-to-FIR, FIR verification, and backend codegen timing.
- Updated the CLI `-time` path to print Faust-like `end phase (duration : N)`
  lines and keep only the total outer timing.

### Validation

- `cargo fmt --all`
- `cargo check -p compiler`
- `cargo build --release -p compiler`
- `target/release/faust-rs -time /Users/letz/Developpements/Recherche/WAC/WAC\ 2017/Faust/clarinetMIDI.dsp -o /tmp/faust_rs_time_release.cpp`
- `faust -time /Users/letz/Developpements/Recherche/WAC/WAC\ 2017/Faust/clarinetMIDI.dsp -o /tmp/faust_cpp_time.cpp`

### Notes

On `clarinetMIDI.dsp`, release `faust-rs` reported about 0.66s in evaluation,
0.042s in signal-to-FIR, and 0.042s in C++ codegen, for about 0.771s total.
The C++ compiler reported about 0.069s in evaluation, 0.002s in propagation,
and 0.031s in `generateCode`. This confirms evaluation is currently the first
place to inspect for the Rust/C++ gap on this example.
