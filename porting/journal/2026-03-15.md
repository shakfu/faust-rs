### Cooperative cancellation for `--timeout` (library-safe timeout model)

Commit date: 2026-03-15

Source: `crates/eval/src/lib.rs`, `crates/compiler/src/lib.rs`, `crates/compiler/src/main.rs`

#### Context

The previous `--timeout` implementation spawned a watchdog thread that called
`process::exit(1)` after the configured duration. This is unsafe for libfaust
usage where the compiler runs inside a host process — killing the entire
process on timeout is unacceptable.

#### Changes

1. **Cooperative cancellation in the evaluator** — `LoopDetector` carries an
   `Arc<AtomicBool>` cancel flag. `eval_value` calls `check_cancel()` at the
   top of every recursive call and returns `EvalError::Cancelled` if the flag
   is set. New `LoopDetector::with_cancel(flag)` constructor accepts an
   externally-provided flag.

2. **New eval entry point** — `eval_entrypoint_with_source_context_and_cancel()`
   accepts an `Arc<AtomicBool>` and wires it into the `LoopDetector`. Existing
   entry points delegate to a shared `eval_entrypoint_full()` with `cancel: None`.

3. **`Compiler::with_cancel()`** — Builder method on the `Compiler` facade.
   When set, `pipeline_to_signals()` calls the cancel-aware eval entry point.

4. **CLI two-pronged timeout** — The watchdog thread now:
   - First sets the cooperative cancel flag (library-safe, covers eval hangs)
   - After a 2s grace period, falls back to `process::exit(1)` (CLI-only,
     covers hangs in propagation and other non-eval phases)

5. **Test cleanup** — Replaced stale `is_box_numeric` tests with
   `simplify_pattern` equivalents.

#### Files changed

| File | Change |
|------|--------|
| `crates/eval/src/lib.rs` | `LoopDetector::with_cancel`, `cancel_flag()`, `check_cancel()`, `EvalError::Cancelled`, `eval_entrypoint_with_source_context_and_cancel`, `eval_entrypoint_full`, test updates |
| `crates/compiler/src/lib.rs` | `Compiler.cancel` field, `with_cancel()` builder, cancel-aware eval call in `pipeline_to_signals` |
| `crates/compiler/src/main.rs` | Two-pronged watchdog (cooperative flag + fallback exit), `compiler_from_cli` accepts cancel flag |

#### Testing

- `cargo test -p eval -p compiler -p boxes -p propagate`: all tests pass, 0 failures.
- `--timeout 1` with `cubic_distortion.dsp` (propagation hang): aborts after ~3s.
- `--timeout 1` with `carre_volterra.dsp` (eval error): returns immediately with proper error.
- Normal compilation unaffected.

### Pattern matcher: `match_num` simplification, `boxPatternMatcher` side-table, Real→Int coercion

Commit date: 2026-03-15

Source: `crates/eval/src/lib.rs`, `crates/eval/src/pattern_matcher.rs`, `crates/boxes/src/lib.rs`

#### Context

9 out of 90 impulse-test DSP files failed with FRS-EVAL-0099 "no case rule matches
arguments". Root cause: recursive case expressions like `poly(max(1,min(6,4)))` pass
computed numeric arguments that the pattern matcher compared by tree identity against
literal constants. Since `max(1,min(6,4))` ≠ `int(4)` as tree nodes, no match occurred.

C++ solves this with `simplifyPattern(X)` in `apply_pattern_matcher_internal`: when a
state has numeric constant transitions (`match_num` flag), the argument is first reduced
to a numeric literal via `boxPropagateSig + simplify`.

#### Changes

1. **`simplify_pattern`** (eval/src/lib.rs) — New function that reduces a box expression
   to a numeric literal when possible. Uses `propagate_box_and_simplify` to get a signal,
   extracts `SigMatch::Int` or `SigMatch::Real`. When a `Real(x)` is an exact integer
   (e.g. `2.0`), returns `boxInt(x as i32)` for tree-identity parity with integer pattern
   constants. Mirrors C++ `simplifyPattern()` in `eval.cpp`.

2. **`apply_pattern_matcher` binding fix** (pattern_matcher.rs) — Critical bug fix:
   `match_num` simplification was only applied inside `apply_pattern_matcher_internal`
   (for constant comparison) but NOT in `apply_pattern_matcher` where variable bindings
   are extracted via `subtree(arena, x, ...)`. This meant variables like `i` in
   `factorial(i) = i * factorial(i-1)` were bound to the unsimplified expression
   (e.g. `sub(max(1,min(2,4)),1)`) instead of the literal `1`. Recursive case rules
   then produced ever-growing expressions → infinite recursion / stack overflow.
   Fix: simplify `x` at the top of `apply_pattern_matcher` so both matching AND binding
   use the simplified value.

3. **`eval_non_negative_count` Real acceptance** (eval/src/lib.rs) — After `match_num`
   fix, iteration counts like `ipar(i, max(1,min(2,4)), ...)` evaluate to `boxReal(2.0)`.
   Added `BoxMatch::Real(x)` arm that accepts whole-number reals as valid iteration counts.

4. **`boxPatternMatcher` tree node** (boxes/src/lib.rs, eval/src/lib.rs) — New tagged
   node kind `BOXPATMATCHER` with a single child (PM store key). Partially-applied
   pattern matchers are stored in a `Vec<PatternMatcherValue>` side-table inside
   `LoopDetector`. `force_value_to_box` returns `boxPatternMatcher(key)` instead of
   calling `lower_pattern_matcher_to_symbolic` (which caused stack overflow via
   `a2sb → eval → force_value_to_box` cycle). Handled in `eval_box` (returns as-is),
   `apply_list` (retrieves PM, delegates to `apply_pattern_matcher_value`), and `a2sb`
   (retrieves PM, lowers via `a2sb_value`).

5. **Supporting infrastructure** — `NumericLit` enum, `propagate_box_and_simplify`,
   `is_box_numeric` promoted from `#[allow(dead_code)]`. `apply_pattern_matcher_internal`
   signature changed from `&TreeArena` to `&mut TreeArena` (needed by `simplify_pattern`).

#### Files changed

| File | Change |
|------|--------|
| `crates/eval/src/lib.rs` | `simplify_pattern`, `force_value_to_box` PM side-table, `eval_non_negative_count` Real arm, `boxPatternMatcher` handling in eval_box/apply_list/a2sb, `EvalError::InternalError` variant |
| `crates/eval/src/pattern_matcher.rs` | `match_num` simplification in `apply_pattern_matcher` (binding fix) and `apply_pattern_matcher_internal`, `&mut TreeArena` signature |
| `crates/boxes/src/lib.rs` | `BOX_PATTERN_MATCHER_TAG`, `BoxMatch::PatternMatcher(BoxId)`, `BoxBuilder::pattern_matcher`, `node_pattern_matcher`, arity-1 `match_box` arm |
| `crates/propagate/src/lib.rs` | `BoxMatch::PatternMatcher` → `flat_box_unexpected` arm |
| `crates/eval/tests/core_eval.rs` | Updated partial-application test to expect `BoxMatch::PatternMatcher` instead of original case_expr |

#### Testing

- `cargo test -p eval -p boxes -p propagate`: 113 passed, 0 failed.
- `karplus.dsp`, `osc.dsp`, `APF.dsp`: OK.
- `os.sawN(2, 440)`, `os.square(440)`: OK (via stdfaust.lib).
- `factorial(5)`, `factorial(3-1)`: OK (recursive case with computed args).
- `carre_volterra.dsp`: no more stack overflow; residual match failure is a separate
  library import issue.

#### Plans

### CLI: add `--compilation-time` / `--timeout` flags

Commit date: 2026-03-15

Source: `crates/compiler/src/main.rs`

#### Context

The C++ Faust compiler provides `-time` / `--compilation-time` to display phase
timing information.  faust-rs had no equivalent, and no way to abort runaway
compilations.

#### Changes

1. **`CompilationTimer` struct** — tracks `Instant` per phase, prints
   `[phase] Xms (total Xms)` to stderr when `--compilation-time` is active,
   and aborts with exit code 1 when the global timeout is exceeded.

2. **CLI flags** (clap):
   - `--compilation-time` (`-time` legacy) — display phase timing on stderr.
   - `--timeout N` (`-timeout N` legacy) — maximum compilation time in seconds
     (default: 120).

3. **Legacy normalization** — `-time` → `--compilation-time`,
   `-timeout` → `--timeout` in `normalize_legacy_args()`.

4. **All dispatch blocks instrumented** — `--parse`, `--dump-box`, `--dump-sig`,
   `--dump-fir`, `--dump-fir-verify`, `--dump-interp`, `--dump-cranelift`,
   `--dump-cpp`, `--dump-c` each create a `CompilationTimer` and call
   `timer.phase()` after each major step.

### `signal_fir`: support multi-output recursion groups (SIGPROJ index > 0)

Commit date: 2026-03-15

Source: `crates/transform/src/signal_fir/module.rs`

#### Context

`dsp/freeverb.dsp` failed with FRS-SFIR-0004:

```
SIGPROJ index 1 unsupported in Step 2C.2 (only 0)
```

Freeverb's `allpass` filter uses a multi-output recursion group via `~ _`
that produces 2 outputs.  The fast-lane FIR lowerer only handled single-output
groups (SIGPROJ index 0).

#### Root cause

Four functions in `signal_fir/module.rs` assumed single-output recursion:

- `lower_proj`: hard-rejected `index != 0`
- `decode_symbolic_group`: extracted only `hd` (first element) of body list
- `active_recursion_info`: returned one `RecArrayInfo` per group (no index)
- `recursion_feedback_info`: rejected `index != 0`

The recursion stack (`Vec<RecArrayInfo>`) stored one array per group instead
of one per output.

#### Fix

1. Changed `recursion_stack` from `Vec<RecArrayInfo>` to `Vec<Vec<RecArrayInfo>>`
   — each entry is now a group of arrays, one per output body.
2. `active_recursion_info(group, proj_index)` — added projection index parameter.
3. `recursion_feedback_info` — removed the index-0 restriction.
4. Replaced `decode_symbolic_group` with `decode_symbolic_group_bodies` which
   uses `list_to_vec` to extract ALL body signals from the cons-list.
5. Rewrote `lower_proj` to:
   - Allocate recursion arrays for all bodies on first encounter
   - Lower all bodies in the same recursion context (shared stack push)
   - Use the group SigId as dedup key in `scheduled_state_updates`
   - Return the result for the requested projection index

#### Files changed

| File | Change |
|------|--------|
| `crates/transform/src/signal_fir/module.rs` | Multi-output recursion in `lower_proj` and supporting functions |

#### Testing

- Full test suite green (zero failures).
- `freeverb.dsp` compiles and produces valid C++ output.
- `karplus.dsp` and `delays.dsp` still compile correctly.

### `cranelift`: add i8 (bool) → float coercion paths

Commit date: 2026-03-15

Source: `crates/codegen/src/backends/cranelift/mod.rs`

#### Context

`dsp/karplus.dsp` failed at Cranelift lowering with FRS-CGEN-CLIF-0002:

```
unsupported Cranelift coercion i8 -> f32 for FIR target Float32
```

The Karplus-Strong model uses `>(0.0)` comparisons whose boolean result
(FIR `Bool` → Cranelift `i8`) is then multiplied with a float signal.

#### Root cause

`coerce_value_to_fir_type()` had explicit conversions for `I8→I32`,
`I8→I64`, `I32→F32`, `I32→F64`, etc., but **no path from `I8` directly
to `F32` or `F64`**.  When the FIR lowerer needed to cast a boolean to
float for arithmetic, it hit the catch-all error.

#### Fix

Added two new match arms: `(I8, F32)` and `(I8, F64)`.  Each widens the
`i8` to `i32` via `uextend`, then converts to float via `fcvt_from_sint`
— the same two-step sequence the existing arms would perform if chained.

#### Files changed

| File | Change |
|------|--------|
| `crates/codegen/src/backends/cranelift/mod.rs` | Add `I8→F32` and `I8→F64` coercion arms |

#### Testing

- Full test suite green (zero failures).
- `karplus.dsp` compiles and produces valid C++ output.

### `sigtype`: type annotator now recurses into cons-list recursion bodies

Commit date: 2026-03-15

Source: `crates/sigtype/src/rules.rs`

#### Context

`dsp/delays.dsp` (from the impulse-test suite) failed with FRS-SFIR-0004:

```
SIGDELAY requires a constant integer amount or a signal with a bounded
non-negative interval (expr=SIGDELAY(SIGINPUT(int(0)),
SIGINTCAST(SIGNUMENTRY(int(0)))))
```

The delay amount `nentry("d1", 200, 0, 1000, 1)` has explicit bounds
`[0, 1000]`, yet the FIR lowering could not find a bounded interval for it.

#### Root cause

`TypeAnnotator::infer_inner` did not handle **cons list** nodes.  Symbolic
recursion bodies (`SYMREC`) are stored in the arena as
`cons(signal, … cons(signal, nil))`.  When `match_sig` returns
`SigMatch::Unknown` for a cons node, the annotator fell through to the
conservative `make_maximal()` fallback — an unbounded `[-∞, +∞]` type —
**without recursing into the list elements**.  None of the inner signal
nodes (SIGDELAY, SIGINTCAST, SIGNUMENTRY, SIGBINOP, …) were ever typed.

Downstream, `scan_delay_lines` looked up the delay amount's interval in the
`sig_types` map and found no entry → error.

#### Fix

Added a `SigMatch::Unknown if self.arena.is_list(sig)` guard before the
generic `Unknown` catch-all.  When the node is a cons list, the annotator
now iterates with `hd`/`tl`, recursively infers each element's type, and
builds a `TupletType` via `make_tuplet` — matching the C++ `T(sig, env)`
list handling.

#### Files changed

| File | Change |
|------|--------|
| `crates/sigtype/src/rules.rs` | Add cons-list → `TupletType` handling in `infer_inner` |

#### Testing
