### CLI: add `--compilation-time` / `--timeout` flags

Commit date: 2026-03-15

Source: `crates/compiler/src/main.rs`

#### Context

The C++ Faust compiler provides `-time` / `--compilation-time` to display phase
timing information.  faust-rs had no equivalent, and no way to abort runaway
compilations.

#### Changes

1. **`CompilationTimer` struct** — tracks `Instant` per phase, prints
   `[phase] Xms (total Xms)` to stderr when `--compilation-time` is active,
   and aborts with exit code 1 when the global timeout is exceeded.

2. **CLI flags** (clap):
   - `--compilation-time` (`-time` legacy) — display phase timing on stderr.
   - `--timeout N` (`-timeout N` legacy) — maximum compilation time in seconds
     (default: 120).

3. **Legacy normalization** — `-time` → `--compilation-time`,
   `-timeout` → `--timeout` in `normalize_legacy_args()`.

4. **All dispatch blocks instrumented** — `--parse`, `--dump-box`, `--dump-sig`,
   `--dump-fir`, `--dump-fir-verify`, `--dump-interp`, `--dump-cranelift`,
   `--dump-cpp`, `--dump-c` each create a `CompilationTimer` and call
   `timer.phase()` after each major step.

### `signal_fir`: support multi-output recursion groups (SIGPROJ index > 0)

Commit date: 2026-03-15

Source: `crates/transform/src/signal_fir/module.rs`

#### Context

`dsp/freeverb.dsp` failed with FRS-SFIR-0004:

```
SIGPROJ index 1 unsupported in Step 2C.2 (only 0)
```

Freeverb's `allpass` filter uses a multi-output recursion group via `~ _`
that produces 2 outputs.  The fast-lane FIR lowerer only handled single-output
groups (SIGPROJ index 0).

#### Root cause

Four functions in `signal_fir/module.rs` assumed single-output recursion:

- `lower_proj`: hard-rejected `index != 0`
- `decode_symbolic_group`: extracted only `hd` (first element) of body list
- `active_recursion_info`: returned one `RecArrayInfo` per group (no index)
- `recursion_feedback_info`: rejected `index != 0`

The recursion stack (`Vec<RecArrayInfo>`) stored one array per group instead
of one per output.

#### Fix

1. Changed `recursion_stack` from `Vec<RecArrayInfo>` to `Vec<Vec<RecArrayInfo>>`
   — each entry is now a group of arrays, one per output body.
2. `active_recursion_info(group, proj_index)` — added projection index parameter.
3. `recursion_feedback_info` — removed the index-0 restriction.
4. Replaced `decode_symbolic_group` with `decode_symbolic_group_bodies` which
   uses `list_to_vec` to extract ALL body signals from the cons-list.
5. Rewrote `lower_proj` to:
   - Allocate recursion arrays for all bodies on first encounter
   - Lower all bodies in the same recursion context (shared stack push)
   - Use the group SigId as dedup key in `scheduled_state_updates`
   - Return the result for the requested projection index

#### Files changed

| File | Change |
|------|--------|
| `crates/transform/src/signal_fir/module.rs` | Multi-output recursion in `lower_proj` and supporting functions |

#### Testing

- Full test suite green (zero failures).
- `freeverb.dsp` compiles and produces valid C++ output.
- `karplus.dsp` and `delays.dsp` still compile correctly.

### `cranelift`: add i8 (bool) → float coercion paths

Commit date: 2026-03-15

Source: `crates/codegen/src/backends/cranelift/mod.rs`

#### Context

`dsp/karplus.dsp` failed at Cranelift lowering with FRS-CGEN-CLIF-0002:

```
unsupported Cranelift coercion i8 -> f32 for FIR target Float32
```

The Karplus-Strong model uses `>(0.0)` comparisons whose boolean result
(FIR `Bool` → Cranelift `i8`) is then multiplied with a float signal.

#### Root cause

`coerce_value_to_fir_type()` had explicit conversions for `I8→I32`,
`I8→I64`, `I32→F32`, `I32→F64`, etc., but **no path from `I8` directly
to `F32` or `F64`**.  When the FIR lowerer needed to cast a boolean to
float for arithmetic, it hit the catch-all error.

#### Fix

Added two new match arms: `(I8, F32)` and `(I8, F64)`.  Each widens the
`i8` to `i32` via `uextend`, then converts to float via `fcvt_from_sint`
— the same two-step sequence the existing arms would perform if chained.

#### Files changed

| File | Change |
|------|--------|
| `crates/codegen/src/backends/cranelift/mod.rs` | Add `I8→F32` and `I8→F64` coercion arms |

#### Testing

- Full test suite green (zero failures).
- `karplus.dsp` compiles and produces valid C++ output.

### `sigtype`: type annotator now recurses into cons-list recursion bodies

Commit date: 2026-03-15

Source: `crates/sigtype/src/rules.rs`

#### Context

`dsp/delays.dsp` (from the impulse-test suite) failed with FRS-SFIR-0004:

```
SIGDELAY requires a constant integer amount or a signal with a bounded
non-negative interval (expr=SIGDELAY(SIGINPUT(int(0)),
SIGINTCAST(SIGNUMENTRY(int(0)))))
```

The delay amount `nentry("d1", 200, 0, 1000, 1)` has explicit bounds
`[0, 1000]`, yet the FIR lowering could not find a bounded interval for it.

#### Root cause

`TypeAnnotator::infer_inner` did not handle **cons list** nodes.  Symbolic
recursion bodies (`SYMREC`) are stored in the arena as
`cons(signal, … cons(signal, nil))`.  When `match_sig` returns
`SigMatch::Unknown` for a cons node, the annotator fell through to the
conservative `make_maximal()` fallback — an unbounded `[-∞, +∞]` type —
**without recursing into the list elements**.  None of the inner signal
nodes (SIGDELAY, SIGINTCAST, SIGNUMENTRY, SIGBINOP, …) were ever typed.

Downstream, `scan_delay_lines` looked up the delay amount's interval in the
`sig_types` map and found no entry → error.

#### Fix

Added a `SigMatch::Unknown if self.arena.is_list(sig)` guard before the
generic `Unknown` catch-all.  When the node is a cons list, the annotator
now iterates with `hd`/`tl`, recursively infers each element's type, and
builds a `TupletType` via `make_tuplet` — matching the C++ `T(sig, env)`
list handling.

#### Files changed

| File | Change |
|------|--------|
| `crates/sigtype/src/rules.rs` | Add cons-list → `TupletType` handling in `infer_inner` |

#### Testing
