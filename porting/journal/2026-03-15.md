### `cranelift`: add i8 (bool) â†’ float coercion paths

Commit date: 2026-03-15

Source: `crates/codegen/src/backends/cranelift/mod.rs`

#### Context

`dsp/karplus.dsp` failed at Cranelift lowering with FRS-CGEN-CLIF-0002:

```
unsupported Cranelift coercion i8 -> f32 for FIR target Float32
```

The Karplus-Strong model uses `>(0.0)` comparisons whose boolean result
(FIR `Bool` â†’ Cranelift `i8`) is then multiplied with a float signal.

#### Root cause

`coerce_value_to_fir_type()` had explicit conversions for `I8â†’I32`,
`I8â†’I64`, `I32â†’F32`, `I32â†’F64`, etc., but **no path from `I8` directly
to `F32` or `F64`**.  When the FIR lowerer needed to cast a boolean to
float for arithmetic, it hit the catch-all error.

#### Fix

Added two new match arms: `(I8, F32)` and `(I8, F64)`.  Each widens the
`i8` to `i32` via `uextend`, then converts to float via `fcvt_from_sint`
â€” the same two-step sequence the existing arms would perform if chained.

### `sigtype`: type annotator now recurses into cons-list recursion bodies

Commit date: 2026-03-15

Source: `crates/sigtype/src/rules.rs`

#### Context

`dsp/delays.dsp` (from the impulse-test suite) failed with FRS-SFIR-0004:

```
SIGDELAY requires a constant integer amount or a signal with a bounded
non-negative interval (expr=SIGDELAY(SIGINPUT(int(0)),
SIGINTCAST(SIGNUMENTRY(int(0)))))
```

The delay amount `nentry("d1", 200, 0, 1000, 1)` has explicit bounds
`[0, 1000]`, yet the FIR lowering could not find a bounded interval for it.

#### Root cause

`TypeAnnotator::infer_inner` did not handle **cons list** nodes.  Symbolic
recursion bodies (`SYMREC`) are stored in the arena as
`cons(signal, â€¦ cons(signal, nil))`.  When `match_sig` returns
`SigMatch::Unknown` for a cons node, the annotator fell through to the
conservative `make_maximal()` fallback â€” an unbounded `[-âˆž, +âˆž]` type â€”
**without recursing into the list elements**.  None of the inner signal
nodes (SIGDELAY, SIGINTCAST, SIGNUMENTRY, SIGBINOP, â€¦) were ever typed.

Downstream, `scan_delay_lines` looked up the delay amount's interval in the
`sig_types` map and found no entry â†’ error.

#### Fix

Added a `SigMatch::Unknown if self.arena.is_list(sig)` guard before the
generic `Unknown` catch-all.  When the node is a cons list, the annotator
now iterates with `hd`/`tl`, recursively infers each element's type, and
builds a `TupletType` via `make_tuplet` â€” matching the C++ `T(sig, env)`
list handling.
