## Continue P3: run the causality gate for every prepared forest

Commit date: 2026-07-13
Source commit: this commit

Continuation of the structural-half commit below, closing P3's remaining
low-risk bullet: *"make signal dependency graph construction available for
every prepared forest, not only when clock domains exist."*

[`crates/transform/src/signal_fir/mod.rs`](../../crates/transform/src/signal_fir/mod.rs)'s
`compile_fastlane_inner` now runs the `clk_env::annotate` /
`hgraph::build_hgraph` / `hgraph::schedule` causality gate for non-clocked
programs too, using an empty `ClockDomainTable` (every reachable signal
degenerates to the `Top` environment). Immediately validated against the
full golden corpus (190 cases) before proceeding further — no regressions.

**Caught by the full test suite, not by golden-check:** one integration
test, `ondemand_signal_fir_rejects_with_clocked_not_lowered`, started
failing — not a false positive. It compiles a genuinely clocked program
through `compile_signals_to_fir_fastlane_with_ui`, the *plain*, clock-unaware
entry point (no `ClockDomainTable` is ever supplied there), and expects the
specific, structured `FRS-SFIR-0007` ("clocked node reached without a
domain table") rejection. The new unconditional gate ran first and threw
its own, less specific `ClockedViolation`-family error instead, because
`clk_env::annotate` cannot correctly resolve a *real* wrapper's clock
relationship from an *empty* domain table — it does not gracefully degrade
for that case, it produces a confusing error.

Fix: added `hgraph::contains_wrapper` (a pure structural scan, needs no
domain data — reuses `get_signal_dependencies`/`needs_subgraph`) and gated
the new unconditional causality check on it returning `false`. A program
reaching the non-clocked branch with an actual OD/US/DS wrapper node now
skips the new gate entirely and falls through to
`module::build_module`'s existing, unchanged `FRS-SFIR-0007` rejection —
exactly the pre-P3 behavior for that specific case. Ordinary wrapper-free
programs are unaffected. Added two `contains_wrapper` unit tests (a flat
program is `false`, an `ondemand` program is `true`).

Re-verified after the fix: full workspace test suite (78 suites, 0
failures, including the previously-failing test) and golden-check (190/190
`[OK]`).

**Still not done — same reasons as before:** wiring `Hsched`'s order into
actual FIR emission, and running CSE after routing. Both remain the
plan's "shadow mode, then make authoritative" activation steps, requiring
a dedicated design pass through `module::build.rs`/`core_lowering.rs`
(6,600+ lines, demand-driven via `SignalToFirLower::lower_output_signal`
with a plain, unordered `HashMap<SigId, FirId>` cache) before any attempt
to make them schedule-driven without risking silent FAD/RAD/BRA/clocked/
delay-ordering regressions.

Gates: `cargo fmt --all` clean, `cargo clippy --workspace --all-targets --

## Execute P3, structural half: Control graph + generalized Hgraph scheduling

Commit date: 2026-07-13
Source commit: this commit

Executed the safe, structural prerequisites of
[`vector-mode-signal-level-analysis-cpp-port-plan-2026-07-10-en.md`](../vector-mode-signal-level-analysis-cpp-port-plan-2026-07-10-en.md),
"P3 - Scalar scheduling activation", following the plan's own staging
("Run `Hsched` in shadow mode... before making it authoritative") and
`AGENTS.md`'s collaboration requirement to stop and report before a
parity-sensitive decision. **Deliberately stopped short of activation** —
see "Not done" below.

Verified before touching anything: `crate::hgraph::schedule`'s `Hsched`
result is, today, consumed only by the one production call site
(`signal_fir::compile_fastlane_inner`) as a **causality gate**
(`.map_err(...)?`) — its actual order is never read by `module::build_module`
or any FIR emission. This is what makes the structural changes below
provably behavior-preserving (confirmed by an unchanged `golden-check`), and
precisely matches the gap the module's own docs already named as pending:
*"C++ `controls`... is not populated in this slice... until backend
emission (P3) unifies the two."*

Changes to
[`crates/transform/src/hgraph/mod.rs`](../../crates/transform/src/hgraph/mod.rs):

- **`GraphKey::Control`** (plan §4.6): a global lifecycle graph for
  top-level `Konst`/`Block`-variability signals, redirected there from
  `Top` by a new `Builder::effective_key`. Scope deliberately narrower than
  full C++ parity: only *top-level* Konst/Block signals redirect; signals
  inside a wrapper's own inner-domain traversal keep their existing
  per-wrapper placement unchanged — nested per-domain control sections are
  left an open question rather than guessed at. Created **lazily** (on
  first redirect), not eagerly like `Top`, specifically so that
  `Hgraph::graphs().len()` does not silently grow by one for every program
  that redirects nothing (this was caught by an existing integration test
  asserting an exact graph count — see "Caught along the way").
- A reference from `Top` to a `Control` value is **not** recorded as a
  same-graph ordering edge (`Control` is an unconditional precondition, not
  schedule-dependent, per plan §4.2) — implemented as an explicit skip in
  the same-domain edge branch of `Builder::visit`.
- `audit_control_variability`: a new, independent check that `Control`
  never owns a `Samp`-variability signal (new `HgraphError::ControlVariabilityViolated`).
- `build_hgraph` gained a `sig_types: &HashMap<SigId, SigType>` parameter
  (an empty map is safe: every signal conservatively stays `Samp`,
  preserving pre-P3 routing exactly).
- `schedule` now takes a `SchedulingStrategy` and runs the shared P1
  generic scheduler (`crate::schedule::schedule`) through the already-
  existing `Digraph` → `ScheduleDag` adapter, per graph, **replacing** the
  module's hand-rolled depth-first walk. `ScheduleError::SelfEdge`/`Cycle`
  fold into the existing `HgraphError::InstantaneousCycle` shape (a
  self-edge is a length-1 cycle; a longer cycle reports its first
  stable-sorted member) rather than widening a public error type whose
  exact field shape other code may already match on.

The one production call site now passes `prepared.sig_types_map()` and
`SchedulingStrategy::DepthFirst` — the same strategy the old literal DFS
implemented, so the causality-gate behavior is unchanged even though the
exact (currently unobserved) node ordering may differ internally.

Caught along the way: `crates/compiler/tests/ondemand_pipeline.rs`'s
`clock_inference_and_hgraph_run_on_prepared_ondemand_program` asserted
`hgraph.graphs().len() == 2` and indexed `hgraph.graphs()[1]` positionally
to find the wrapper subgraph. With real `sig_types` data this DSP's gate
button *does* have a top-level Konst/Block signal, so `Control` is
genuinely populated (3 graphs) — this is the feature working correctly,
not a bug. Updated the assertion to check `Control`'s content explicitly
(plus `audit_control_variability`) and replaced the positional index with
an explicit `find` by `GraphKey::Wrapper(_)`, which was fragile even before
this change.

New tests in `hgraph/tests.rs` (8 total, 3 new): a Konst signal is
redirected to `Control` and its Top-side precondition edge is confirmed
absent, all four `-ss` strategies schedule the resulting three-graph
`Hgraph` validly; a signal absent from `sig_types` conservatively stays out
of `Control`; `audit_control_variability` rejects a hand-built malformed
`Hgraph` (the plan §8 negative-mutation policy).

**Not done — explicitly deferred, not silently skipped:**

- "Make scalar lowering visit the selected controls and top schedule" —
  wiring `Hsched`'s order into `module::build_module`'s actual FIR
  statement emission. This is the step the plan calls "shadow mode...
  before making it authoritative," and the one genuinely capable of
  changing golden output (different statement order, naming, CSE
  boundaries).
- "Run CSE only after statements have been routed to their lifecycle/domain
  regions" — depends on the above.
- Extending signal dependency graph construction to run unconditionally
  "for every prepared forest, not only when clock domains exist" — today's
  gate still only runs when `clock_domains` is non-empty; turning it on for
  every (including non-clocked) program is a new validation surface for
  currently-unvalidated DSPs and was not attempted here.

These require an explicit go-ahead: they are the actual "shadow mode
comparison, then flip" activation the plan describes, and unlike everything
above are not provably behavior-preserving without that comparison.

Also added `interval` as a new `transform` dev-dependency (needed to
hand-construct a `SigType` fixture in the new tests; no production use).

Gates: `cargo fmt --all` clean, `cargo clippy --workspace --all-targets --
-D warnings` clean, `cargo test --workspace --all-targets` 0 failures (78
test result lines), `cargo run -p xtask -- golden-check` all `[OK]` —
confirming, as claimed above, zero observable compiler behavior change.

## Implement R1: schedule certificate at L2

Commit date: 2026-07-13
Source commit: this commit

Next certified-plan phase after PV
([`lean-rust-certified-porting-plan-2026-07-11-en.md`](../lean-rust-certified-porting-plan-2026-07-11-en.md),
"R1 - Schedule certificate at L2"): the canonical, hashable artifact layer
around P1's already-existing `schedule`/`verify_schedule`.

New
[`crates/transform/src/schedule/certificate.rs`](../../crates/transform/src/schedule/certificate.rs)
(~450 lines, additive, not wired to production), field shapes mirroring the
`graphSnapshot`/`scheduleCertificate` definitions of
`porting/schemas/vector-verification-certificate-v1.schema.json`:

- `GraphSnapshot` (canonical `u64` nodes/edges) + `graph_hash()`: SHA-256 of
  a **hand-written** canonical byte encoding (`{"edges":[...],"nodes":[...]}`,
  sorted keys, no whitespace) — not `serde_json::to_string`, per the
  certified plan's explicit warning that the latter is not assumed
  canonical. Full RFC 8785 cross-language/multi-OS conformance stays R2
  scope, as the plan directs ("RV must already be green before this work
  becomes a phase blocker" — it now is).
- `ScheduleCertificate` (producer/program/graph/graph_hash/scope/strategy/
  node_count/ordered_nodes) and `SchedulingStrategy::canonical_name()`
  (the `-ss`-integer-independent string form: `-ss 3` and `-ss 42` both
  canonicalize to `"reverse_breadth_first"`).
- `certify_schedule` (producer): calls the existing `schedule()`, projects
  any `ScheduleDag` into a canonical `GraphSnapshot` via caller-supplied
  `node_id`/`edge_kind` mappings.
- `verify_schedule_certificate` (checker): never calls `schedule()` or any
  strategy. Checks, in order: schema version, `uint53` bounds, canonical
  node/edge ordering (noncanonical-but-equivalent sets are rejected, per
  plan §4.3), edge endpoints ⊆ nodes, `node_count` agreement with both graph
  and order, `ordered_nodes` validity (reusing `verify_schedule` itself —
  chaining an existing checker is not the same as calling the producer),
  and `graph_hash` recomputation.

Added `sha2 = "0.10"` as a new `transform` dependency (no existing SHA-256 in
the workspace).

Tests: round-trip for all four strategies, `-ss`-independence of the graph
and its hash (same graph/hash for `DepthFirst` vs `Special`, different
`ordered_nodes`), cyclic/self-edge rejection for every strategy, and one
negative-mutation test per `CertificateError` variant (wrong schema version,
out-of-range node, reordered/duplicated nodes, reordered edges, dangling
edge endpoint, both `node_count` mismatches, inverted schedule order,
altered `graph_hash`) — the plan §8 policy ("a checker without a
demonstrated rejecting mutation is not complete enough to serve as a trust
boundary").

Gates: `cargo fmt --all` clean, `cargo clippy --workspace --all-targets --
-D warnings` clean, `cargo test --workspace --all-targets` 0 failures (78
test result lines), `cargo run -p xtask -- golden-check` all `[OK]`.

## Fix the silent-passthrough bug in the shared table_state_delay fixture

Commit date: 2026-07-13
Source commit: this commit

Follow-up to the PV session below (`task_02414270`): fixed
[`crates/codegen/src/fixtures.rs::build_table_state_delay_test_module`](../../crates/codegen/src/fixtures.rs),
the only place this specific read-before-write ordering bug was found.

The fixture's delay read (`load_table("fDelay", ...)`) was referenced only
from `write_out`, the *second* statement in the loop body, after
`write_delay` (the overwriting store) — the *first* statement. Since FIR
expressions evaluate where they are textually embedded rather than at their
Rust-builder construction point, the read re-evaluated against the
already-overwritten ring slot, making the fixture a silent passthrough
(`output == input`) instead of a delay-by-4.

Fix: capture the read into an explicit `declare_var` temp-statement
(`tmpRead`) *before* the overwriting store, then reference the captured
value afterward — the same pattern already used in
`transform::signal_fir::pv_slice` (yesterday's PV work, which is where this
bug was first discovered and reproduced standalone).

Confirmed only one consumer before changing anything:
`crates/codegen/src/backends/wasm/tests.rs::wasm_compute_lowers_struct_tables_and_select2`,
which only asserts WASM opcode presence, never numeric output, and stays
green unchanged. `backend_test_fixtures()`'s other reference
(`crates/compiler/src/cli/runner.rs`, for the `--fir-fixture` listing/dispatch
CLI utility) is not a test and has no behavioral assertion either.

Added a regression guard next to the fixture,
`fixtures::tests::interp_delay_by_4_semantics_hold`: executes the fixture
through `codegen::backends::interp` with ten distinct input values and
asserts genuine delay-by-4 output (`[0,0,0,0,1,2,3,4,5,6]` for input
`[1..10]`), not the passthrough this bug produced. This is the first test
that checks this fixture's actual numeric behavior rather than structural
codegen properties.

Gates: `cargo fmt --all` clean, `cargo clippy --workspace --all-targets -- -D
warnings` clean, `cargo test --workspace --all-targets` 0 failures (78 test
result lines), `cargo run -p xtask -- golden-check` all `[OK]`.

## Execute PV: bit-exact signal-level vector split through the interpreter

Commit date: 2026-07-13
Source commit: this commit

Executed the certified plan's "RV - Early executable vertical slice" /
port plan's "PV - Early vertical vector execution slice" gate, in-session
(the Sonnet 5 delegation path was blocked by the account's monthly spend
limit mid-P0, so PV — the largest remaining phase — was implemented inline
rather than deferred).

**DSP shape.** `x = input(0) * 0.5`; `y = x @ 20` (persistent state, delay
strictly larger than every tested block size so no delay read ever needs a
same-block transported value — that combination is `P6` scope, not `PV`);
`z = x + 1.0` (a pure tail deliberately routed through a transport instead of
trivially inlining, to exercise cross-loop routing). `x` classifies as
`SeparateVectorizable` under the existing, unmodified
`loop_graph::needs_separate_loop` precedence (delay forces separation before
sharing is even considered) — the same function P4 will later generalize.

**New code**, both additive, neither wired into any production path:

- [`crates/transform/src/signal_fir/pv_slice.rs`](../../crates/transform/src/signal_fir/pv_slice.rs)
  (~400 lines): a genuine (if narrow) `SigId`-fact walk via the existing
  `signal_value_children` (occurrence counts, `max_delay`, not hardcoded);
  `PvPlan` (two loop nodes, one typed transport) implementing
  `schedule::ScheduleDag`; `pv_schedule` running the shared P1
  `SchedulingStrategy::DepthFirst` scheduler plus the independent
  `verify_schedule` postcondition checker; `route_pv_vector_fir` emitting FIR
  for both loop variants via `FirBuilder` directly (backend-neutral, no new
  vector types).
- [`crates/compiler/tests/pv_vector_slice.rs`](../../crates/compiler/tests/pv_vector_slice.rs):
  drives the scalar reference (existing, untouched
  `signal_prepare::prepare_signals_for_fir` +
  `signal_fir::compile_signals_to_fir_fastlane_with_ui` pipeline) and both new
  vector variants through `codegen::backends::interp`, across 7 blocks of
  varying, non-dividing sizes (`[7,8,8,5,3,11,6]`, total 48 samples > the
  20-sample delay), asserting bit-for-bit agreement on both outputs plus an
  explicit topology assertion (exactly two loops, dependency order, transport
  type/length) that would fail if the vector path silently serialized or
  inlined the whole case.

**Result: PASS.** `-vec -lv 0` (fixed-chunk-plus-remainder) and `-vec -lv 1`
(single variable-size loop) are bit-exact against the scalar reference, and
against each other, across all 48 samples and both outputs.

**Two FIR-execution-model facts learned the hard way, now documented in
`pv_slice.rs`** (neither was written down anywhere before this):

1. **Expressions evaluate where embedded, not at construction time.** A
   `load_table` value referenced later in a block re-reads at that later
   point in program order — reusing a raw load expression after an
   intervening write to the same table returns the *post-write* value.
   Discovered by probing the **existing, shared**
   `crates/codegen/src/fixtures.rs::build_table_state_delay_test_module`
   fixture through the interpreter: it is a **silent passthrough** (no delay
   effect at all), because its read-before-write pattern reads `read` inside
   a *later* statement (`write_out`) than the overwriting `write_delay`. No
   existing test caught this — the only test touching that fixture checks
   WASM opcode presence, never numeric output. The correct pattern (used
   throughout `pv_slice.rs`) captures the read into an explicit
   `declare_var` temp-statement *before* the overwrite. Flagged as a
   follow-up task (`task_02414270`, not fixed here — out of PV scope, shared
   fixture, its own blast radius).
2. **The general `for_loop` is a do-while, not a for.** Its own doc comment
   says so (`var = init; do { body; var += step } while (cond)`), unlike
   `simple_for_loop`. `Lv0`'s chunk-plus-remainder shape has two loops whose
   range can legitimately be empty (main loop when `count < 4`; remainder
   when `count` is an exact multiple of 4) — an unguarded `for_loop` executed
   one bogus iteration reading/writing one sample past `count` in exactly
   those cases (`io_oob` at `sample == count`, reproduced standalone for
   every `n` that is a multiple of the chunk width). Fixed by wrapping each
   with an explicit `if_` guard on the range being non-empty.
   Also: the general `for_loop`'s `init` must be an explicit `DeclareVar`
   (unlike `simple_for_loop`, which synthesizes its own); a bare constant
   left the loop variable unregistered (`undeclared variable: i0`).
   And: mid-function `DeclareTable` with `AccessType::Stack` is accepted by
   the FIR checker/some backends but **not implemented by the interp
   backend** (`unsupported FIR node`); the transport buffer was moved to a
   struct-level table instead, which is arguably more faithful to real
   vectorizing C++ backends anyway (`Vector*`/`Zec*` chunk buffers are class
   members, not stack locals).

Gates: `cargo fmt --all` clean, `cargo clippy --workspace --all-targets -- -D
warnings` clean, `cargo test --workspace --all-targets` 0 failures (78 test
result lines, all `ok`), `cargo run -p xtask -- golden-check` all `[OK]` —
