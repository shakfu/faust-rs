## Add canonical execution conditions and conservative signal effects

Commit date: 2026-07-13
Source commit: this commit

`analyze_vector_signals` entry point builds C++-compatible execution conditions
and then computes occurrence/effect facts from the same verified prepared
forest. The condition domain is deterministic positive DNF with absorption;

or backend output consumes the new facts. Final loop-level effect-edge

## Finalize P3 up to activation: implement shadow mode and record its finding

Commit date: 2026-07-13
Source commit: this commit

Implemented the P3 bullet that gates activation: *"Run `Hsched` in shadow
mode against demand-driven lowering first. Record statement-order... 
differences for `-ss 0` before making it authoritative."* Observation-only â€”
`golden-check` stays byte-identical (verified).

Mechanism (all additive):

- `SignalToFirLower` records `emission_order`: every distinct materialized
  `SigId` in first-lowering order, pushed at the module's **single** cache
  insertion site (`core_lowering.rs`, `lower_signal`). One `Vec::push` per
  signal; changes no emitted FIR.
- Exported on `SignalFirOutput::{emission_order, shadow_report}`.
- New `signal_fir::shadow` module: `compare_emission_order(hgraph, hsched,
  output) -> ShadowReport` computes, per graph, whether the demand-driven
  order respects every immediate edge (`respects_all_immediate_edges` â€” the
  activation-safety verdict) and whether it already equals the selected
  schedule on the comparable intersection (`matches_schedule_everywhere` â€”
  the no-churn verdict). Delayed (state) edges are correctly excluded;
  recursive-projection nodes that never enter the cache are counted as
  `uncovered`, not as inversions.
- `compile_fastlane_inner` now captures the causality gate's `(Hgraph,
  Hsched)` (previously discarded) and computes the report after lowering, so
  both inputs share the one prepared arena.

**The finding** (`crates/compiler/tests/p3_shadow_mode.rs`, nine plain
forward-time scalar corpus shapes â€” passthrough, gain+bias, stereo mix,
delay echo, one-pole and two-pole recursions, comb, fork/join, shared
expression):

- **Activation is dependency-safe.** Every sampled program has **zero
  immediate-edge inversions**: the demand-driven order already respects every
  same-tick `Hgraph` dependency. An `Hsched`-driven order (which respects them
  by construction) could therefore only reorder *independent* nodes.
- **Activation is not free.** Only **5/9** programs' demand-driven order
  already matches the `-ss 0` (DFS) schedule on the comparable intersection;
  the other four (delay echo, comb, shared expression, two-pole) genuinely
  differ. So flipping the schedule authoritative *would* change statement
  order â€” and thus goldens â€” for a real fraction of even trivial programs,
  confirming the plan's warning that activation "may legitimately change
  textual FIR" and needs per-case golden audit.

**Deliberately still not done** (the actual authoritative flip): wiring
`Hsched`'s order into `module::build_module`'s FIR emission, and
CSE-after-routing. Per this plan's exit criterion and `AGENTS.md`, that is a
parity-sensitive change requiring individually audited golden refreshes and
an explicit go-ahead â€” not something to merge silently. The shadow finding
above is precisely the go/no-go input for that decision; it is now recorded
rather than assumed. Port plan Â§P3 annotated with this status.

Gates: `cargo fmt --all` clean, `cargo clippy --workspace --all-targets --
-D warnings` clean, `cargo test --workspace --all-targets` 0 failures (79
suites â€” one new), `cargo run -p xtask -- golden-check` all `[OK]`.

## Fix the silent-passthrough bug in the shared table_state_delay fixture

Commit date: 2026-07-13
Source commit: this commit

Follow-up to the PV session below (`task_02414270`): fixed
[`crates/codegen/src/fixtures.rs::build_table_state_delay_test_module`](../../crates/codegen/src/fixtures.rs),
the only place this specific read-before-write ordering bug was found.

The fixture's delay read (`load_table("fDelay", ...)`) was referenced only
from `write_out`, the *second* statement in the loop body, after
`write_delay` (the overwriting store) â€” the *first* statement. Since FIR
expressions evaluate where they are textually embedded rather than at their
Rust-builder construction point, the read re-evaluated against the
already-overwritten ring slot, making the fixture a silent passthrough
(`output == input`) instead of a delay-by-4.

Fix: capture the read into an explicit `declare_var` temp-statement
(`tmpRead`) *before* the overwriting store, then reference the captured
value afterward â€” the same pattern already used in
`transform::signal_fir::pv_slice` (yesterday's PV work, which is where this
bug was first discovered and reproduced standalone).

Confirmed only one consumer before changing anything:
`crates/codegen/src/backends/wasm/tests.rs::wasm_compute_lowers_struct_tables_and_select2`,
which only asserts WASM opcode presence, never numeric output, and stays
green unchanged. `backend_test_fixtures()`'s other reference
(`crates/compiler/src/cli/runner.rs`, for the `--fir-fixture` listing/dispatch

**Two FIR-execution-model facts learned the hard way, now documented in
`pv_slice.rs`** (neither was written down anywhere before this):

1. **Expressions evaluate where embedded, not at construction time.** A
   `load_table` value referenced later in a block re-reads at that later
   point in program order â€” reusing a raw load expression after an
   intervening write to the same table returns the *post-write* value.
   Discovered by probing the **existing, shared**
   `crates/codegen/src/fixtures.rs::build_table_state_delay_test_module`
   fixture through the interpreter: it is a **silent passthrough** (no delay
   effect at all), because its read-before-write pattern reads `read` inside
   a *later* statement (`write_out`) than the overwriting `write_delay`. No
   existing test caught this â€” the only test touching that fixture checks
   WASM opcode presence, never numeric output. The correct pattern (used
   throughout `pv_slice.rs`) captures the read into an explicit
   `declare_var` temp-statement *before* the overwrite. Flagged as a
   follow-up task (`task_02414270`, not fixed here â€” out of PV scope, shared
   fixture, its own blast radius).
2. **The general `for_loop` is a do-while, not a for.** Its own doc comment
   says so (`var = init; do { body; var += step } while (cond)`), unlike
   `simple_for_loop`. `Lv0`'s chunk-plus-remainder shape has two loops whose
   range can legitimately be empty (main loop when `count < 4`; remainder
   when `count` is an exact multiple of 4) â€” an unguarded `for_loop` executed
   one bogus iteration reading/writing one sample past `count` in exactly
   those cases (`io_oob` at `sample == count`, reproduced standalone for
   every `n` that is a multiple of the chunk width). Fixed by wrapping each
   with an explicit `if_` guard on the range being non-empty.
   Also: the general `for_loop`'s `init` must be an explicit `DeclareVar`
   (unlike `simple_for_loop`, which synthesizes its own); a bare constant
   left the loop variable unregistered (`undeclared variable: i0`).
   And: mid-function `DeclareTable` with `AccessType::Stack` is accepted by
   the FIR checker/some backends but **not implemented by the interp
   backend** (`unsupported FIR node`); the transport buffer was moved to a
   struct-level table instead, which is arguably more faithful to real
   vectorizing C++ backends anyway (`Vector*`/`Zec*` chunk buffers are class
   members, not stack locals).
