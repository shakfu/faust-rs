## Fix the silent-passthrough bug in the shared table_state_delay fixture

Commit date: 2026-07-13
Source commit: this commit

Follow-up to the PV session below (`task_02414270`): fixed
[`crates/codegen/src/fixtures.rs::build_table_state_delay_test_module`](../../crates/codegen/src/fixtures.rs),
the only place this specific read-before-write ordering bug was found.

The fixture's delay read (`load_table("fDelay", ...)`) was referenced only
from `write_out`, the *second* statement in the loop body, after
`write_delay` (the overwriting store) — the *first* statement. Since FIR
expressions evaluate where they are textually embedded rather than at their
Rust-builder construction point, the read re-evaluated against the
already-overwritten ring slot, making the fixture a silent passthrough
(`output == input`) instead of a delay-by-4.

Fix: capture the read into an explicit `declare_var` temp-statement
(`tmpRead`) *before* the overwriting store, then reference the captured
value afterward — the same pattern already used in
`transform::signal_fir::pv_slice` (yesterday's PV work, which is where this
bug was first discovered and reproduced standalone).

Confirmed only one consumer before changing anything:
`crates/codegen/src/backends/wasm/tests.rs::wasm_compute_lowers_struct_tables_and_select2`,
which only asserts WASM opcode presence, never numeric output, and stays
green unchanged. `backend_test_fixtures()`'s other reference
(`crates/compiler/src/cli/runner.rs`, for the `--fir-fixture` listing/dispatch
CLI utility) is not a test and has no behavioral assertion either.

Added a regression guard next to the fixture,
`fixtures::tests::interp_delay_by_4_semantics_hold`: executes the fixture
through `codegen::backends::interp` with ten distinct input values and
asserts genuine delay-by-4 output (`[0,0,0,0,1,2,3,4,5,6]` for input
`[1..10]`), not the passthrough this bug produced. This is the first test
that checks this fixture's actual numeric behavior rather than structural
codegen properties.

Gates: `cargo fmt --all` clean, `cargo clippy --workspace --all-targets -- -D
warnings` clean, `cargo test --workspace --all-targets` 0 failures (78 test
result lines), `cargo run -p xtask -- golden-check` all `[OK]`.

## Execute PV: bit-exact signal-level vector split through the interpreter

Commit date: 2026-07-13
Source commit: this commit

Executed the certified plan's "RV - Early executable vertical slice" /
port plan's "PV - Early vertical vector execution slice" gate, in-session
(the Sonnet 5 delegation path was blocked by the account's monthly spend
limit mid-P0, so PV — the largest remaining phase — was implemented inline
rather than deferred).

**DSP shape.** `x = input(0) * 0.5`; `y = x @ 20` (persistent state, delay
strictly larger than every tested block size so no delay read ever needs a
same-block transported value — that combination is `P6` scope, not `PV`);
`z = x + 1.0` (a pure tail deliberately routed through a transport instead of
trivially inlining, to exercise cross-loop routing). `x` classifies as
`SeparateVectorizable` under the existing, unmodified
`loop_graph::needs_separate_loop` precedence (delay forces separation before
sharing is even considered) — the same function P4 will later generalize.

**New code**, both additive, neither wired into any production path:

- [`crates/transform/src/signal_fir/pv_slice.rs`](../../crates/transform/src/signal_fir/pv_slice.rs)
  (~400 lines): a genuine (if narrow) `SigId`-fact walk via the existing
  `signal_value_children` (occurrence counts, `max_delay`, not hardcoded);
  `PvPlan` (two loop nodes, one typed transport) implementing
  `schedule::ScheduleDag`; `pv_schedule` running the shared P1
  `SchedulingStrategy::DepthFirst` scheduler plus the independent
  `verify_schedule` postcondition checker; `route_pv_vector_fir` emitting FIR
  for both loop variants via `FirBuilder` directly (backend-neutral, no new
  vector types).
- [`crates/compiler/tests/pv_vector_slice.rs`](../../crates/compiler/tests/pv_vector_slice.rs):
  drives the scalar reference (existing, untouched
  `signal_prepare::prepare_signals_for_fir` +
  `signal_fir::compile_signals_to_fir_fastlane_with_ui` pipeline) and both new
  vector variants through `codegen::backends::interp`, across 7 blocks of
  varying, non-dividing sizes (`[7,8,8,5,3,11,6]`, total 48 samples > the
  20-sample delay), asserting bit-for-bit agreement on both outputs plus an
  explicit topology assertion (exactly two loops, dependency order, transport
  type/length) that would fail if the vector path silently serialized or
  inlined the whole case.

**Result: PASS.** `-vec -lv 0` (fixed-chunk-plus-remainder) and `-vec -lv 1`
(single variable-size loop) are bit-exact against the scalar reference, and
against each other, across all 48 samples and both outputs.

**Two FIR-execution-model facts learned the hard way, now documented in
`pv_slice.rs`** (neither was written down anywhere before this):

1. **Expressions evaluate where embedded, not at construction time.** A
   `load_table` value referenced later in a block re-reads at that later
   point in program order — reusing a raw load expression after an
   intervening write to the same table returns the *post-write* value.
   Discovered by probing the **existing, shared**
   `crates/codegen/src/fixtures.rs::build_table_state_delay_test_module`
   fixture through the interpreter: it is a **silent passthrough** (no delay
   effect at all), because its read-before-write pattern reads `read` inside
   a *later* statement (`write_out`) than the overwriting `write_delay`. No
   existing test caught this — the only test touching that fixture checks
   WASM opcode presence, never numeric output. The correct pattern (used
   throughout `pv_slice.rs`) captures the read into an explicit
   `declare_var` temp-statement *before* the overwrite. Flagged as a
   follow-up task (`task_02414270`, not fixed here — out of PV scope, shared
   fixture, its own blast radius).
2. **The general `for_loop` is a do-while, not a for.** Its own doc comment
   says so (`var = init; do { body; var += step } while (cond)`), unlike
   `simple_for_loop`. `Lv0`'s chunk-plus-remainder shape has two loops whose
   range can legitimately be empty (main loop when `count < 4`; remainder
   when `count` is an exact multiple of 4) — an unguarded `for_loop` executed
   one bogus iteration reading/writing one sample past `count` in exactly
   those cases (`io_oob` at `sample == count`, reproduced standalone for
   every `n` that is a multiple of the chunk width). Fixed by wrapping each
   with an explicit `if_` guard on the range being non-empty.
   Also: the general `for_loop`'s `init` must be an explicit `DeclareVar`
   (unlike `simple_for_loop`, which synthesizes its own); a bare constant
   left the loop variable unregistered (`undeclared variable: i0`).
   And: mid-function `DeclareTable` with `AccessType::Stack` is accepted by
   the FIR checker/some backends but **not implemented by the interp
   backend** (`unsupported FIR node`); the transport buffer was moved to a
   struct-level table instead, which is arguably more faithful to real
   vectorizing C++ backends anyway (`Vector*`/`Zec*` chunk buffers are class
   members, not stack locals).

Gates: `cargo fmt --all` clean, `cargo clippy --workspace --all-targets -- -D
warnings` clean, `cargo test --workspace --all-targets` 0 failures (78 test
result lines, all `ok`), `cargo run -p xtask -- golden-check` all `[OK]` —
compiler output unchanged, confirming PV stayed purely additive.
