# 2026-03-21

## Fix: slot_env de Bruijn lifting in FlatNodeKind::Rec

### Bug

`faust-rs -pn spectralCentroid_test tests/analyzers_tests.dsp` produced samples
different from the reference C++ `faust` output. The spectral centroid computation
converged to a wrong constant instead of tracking the signal spectrum.

### Root cause

In `crates/propagate/src/lib.rs`, the `FlatNodeKind::Rec` handler correctly lifts
external `inputs` by 1 de Bruijn level with `lift_signals(inputs)` when entering the
new DEBRUIJN scope. However, values captured in `ctx.slot_env` (from box abstraction
application, e.g. `lowpass(cf=centroid_feedback, x)`) were **not** lifted.

The `spectralCentroid` function has the structure:

```faust
spectralCentroid(...) = loop ~ _
    with {
        loop(fb) = centroid
            with {
                lowpass(cf, x) = y letrec { ... };
                xLRMS = an.rms_envelope_tau(t, lowpass(fb, x));
                ...
            };
    };
```

When `lowpass(cf=fb, x)` is applied and its inner `letrec` Rec handler runs, `cf`
holds `seed0 = delay1(proj(0, DEBRUIJNREF(1)))` — a reference to the outer centroid
loop. Without lifting `cf` before entering the inner DEBRUIJN scope, the inner binder
captures `DEBRUIJNREF(1)` as its own feedback (level 1 from the top), not the outer
centroid's level 1. The outer centroid reference is silently lost, making `lowpass`
filter at a constant cutoff regardless of the centroid feedback.

### Fix

Before propagating the Rec body (`left`), lift all slot_env values by 1 using
`liftn(arena, v, 1, memo)`, then restore the original slot_env after:

```rust
let lifted_slot_env: SlotEnv = ctx.slot_env
    .iter()
    .map(|(k, v)| (*k, liftn(arena, *v, 1, ctx.memo)))
    .collect();
let saved_slot_env = std::mem::replace(ctx.slot_env, lifted_slot_env);
let l2 = propagate_in_slot_env(arena, left, &rec_inputs, ctx)?;
*ctx.slot_env = saved_slot_env;
```

This mirrors `lift(slotenv)` at line 495 of
`compiler/propagate/propagate.cpp` in the C++ Faust compiler. In C++, `slotenv`
is a `Tree` cons-list so `lift` walks it structurally; in Rust we apply `liftn`
to each value individually for the same effect.

### Verification

All 796 workspace tests pass. The normalized `compute()` body of the generated C++
now matches the reference: the lowpass filter tan argument correctly reads
`fRecX[((fIOTA - 1) & 1)]` (previous centroid value) instead of a constant.

Also removed leftover `[DEBUG lower_proj]` eprintln calls from
`crates/transform/src/signal_fir/module.rs`.

## Fix: GeneratorInterpreter eval_delay1 does not advance recursive state

### Bug

`faust-rs -pn tabulate_test tests/basics_tests.dsp` produced `8.175 Hz` for every
sample instead of `261.625 Hz` (MIDI key 60, middle C).

The table generated by `ba.tabulate(1, ba.midikey2hz, 128, 0, 127, 60).lin` was
filled entirely with `midikey2hz(0) = 8.175` — every entry the same constant.

### Root cause

`GeneratorInterpreter::eval_delay1` in
`crates/transform/src/signal_fir/module.rs` handles `Delay1(Proj(idx, group))`
where `group` is a `SYMREC` or `Rec` recursion group by reading `prev[idx]` and
returning immediately, **without evaluating the recursion group body**.

The tabulate write function uses `ba.time = +(1)~_ : mem`, which after
`prepare_signals_for_fir` becomes:

```
Delay1(Proj(0, SYMREC(var, [Add(SYMREF(var), 1)])))
```

`eval_delay1` read `prev[0]` (= 0, initial) and returned without ever calling
`eval_rec_and_project`. Since `cur[0]` was never updated, `advance()` propagated
`prev ← cur = 0` at every step. `ba.time` evaluated to 0 for all 128 table
entries, so every entry became `midikey2hz(0)`.

### Fix

In `eval_delay1`, for the `Proj(idx, SYMREC/Rec(body))` path:
1. Save `prev_val = read_rec_prev(var, idx)` (the return value).
2. Call `eval_rec_and_project(var, Some(body), idx)` to compute and store the
   **current** step value in `cur[idx]`.
3. Return `prev_val`.

`advance()` then correctly propagates `cur → prev`, so the counter increments:
step 0 → 0, step 1 → 1, …, step 127 → 127, filling the table with
`midikey2hz(0), midikey2hz(1), …, midikey2hz(127)`.

After the fix:
- `tabulate_test`: faust-rs gives `261.6255798` vs ref `261.6255493` (Δ ≈ 3e-5,
  relative 1e-7 — float precision from precomputed vs runtime table).
- `tabulate_chebychev_test`: same quality match.
- All workspace tests pass, no regressions.
