## P0 — clean machine channel: JSON diagnostics on stdout, `--check`, frozen code table

Commit date: 2026-07-21
Source commit: this commit

Implements phase P0 of `porting/mcp-server-analysis-and-plan-2026-07-21-en.md`.
The motivation is independent of the MCP work (plan §6.3, C2): until now no
automated consumer — CI, IDE, the xtask harnesses — could read faust-rs
diagnostics without string-scraping.

**D1 — JSON on stdout, alone.** `crates/compiler/src/cli/diagnostics.rs:49`
(`print_structured_diagnostics`) writes the JSON branch to stdout instead of
stderr. `crates/compiler/src/cli/runner.rs:550` adds
`report_pipeline_failure(prefix, err, cli)`, which emits the human prefix line
only under `--error-format human`; the 17 per-backend
`eprintln! + print_structured_diagnostics + exit(1)` sites collapse onto it.
`--error-format human` output is unchanged, guarded by a regression test.

Two design points settled during implementation:

- *stdout conflict rule* — on a successful dump-mode compile, generated output
  stays on stdout and no diagnostics payload is added. The CLI never emitted
  one on success outside `--check`, so there is nothing to interleave. Rule
  documented in the module docs and in `docs/user-diagnostics-guide-en.md`.
- *no-bundle fallback* — `CompilerError` variants carrying no
  `DiagnosticBundle` (backend codegen, import failures; reached in practice by
  an unresolved `import(...)`) now emit a `code: null` envelope of the same
  shape via `format_fallback_diagnostics_json` (`diagnostics.rs:87`) rather
  than nothing, so the schema is uniform across every failure path.

**D2 — `--check`.** New flag (`cli/args.rs:113`), wired into the mode-exclusion
array and the `--architecture` / `--fir-fixture` / `--no-fir-verify`
validation blocks. Dispatch at `runner.rs:1440` runs
`compile_file_to_fir_with_lane` (parse → eval → propagate → type → FIR verify,
no codegen — the same call `--dump-fir-verify` uses). Success emits an empty
`diagnostics` array and exit 0, so success and failure share one schema and a
consumer needs one parser, not two.

**D3 — frozen code table.** `docs/diagnostics-codes-en.md` documents every
code with stage, meaning and raise site. `cli/tests.rs:1298`
(`frozen_frs_code_table_matches_source`) walks `crates/` natively and diffs
against the documented set, so adding an undocumented code or renumbering an
existing one fails CI.

**Validation.** New `crates/compiler/tests/cli_diagnostics_channel.rs`: 11
subprocess tests spawning the real binary, covering PARSE/EVAL/PROP/COMP/SFIR
from corpus cases plus FIR and SRC from purpose-written fixtures under
`crates/compiler/tests/fixtures/cli_diagnostics/`. LEX has no CLI-reachable
fixture (see below) and is covered by a synthetic-diagnostic unit test at
`cli/tests.rs:1166` instead — recorded rather than skipped silently.

Rejecting mutations, each applied, observed, and reverted: (a) prefix
reintroduced onto the JSON stream → 9/11 checker tests fail; (b) payload routed
back to stderr → 7/11 fail (the 4 survivors exercise the success and SRC
fallback paths, which that mutation does not touch); (c) `FRS-EVAL-0003`
renumbered → the frozen-table test fails with an explicit diff. Mutation (b)
was independently re-applied and re-verified outside the implementing agent.

Gate met: **218/218** corpus files emit a parseable JSON document on stdout
under `--check --error-format json`. `cargo fmt --check`, `cargo clippy
--workspace --all-targets -- -D warnings` and `cargo test --workspace` clean.
(`normalize::tests::normalize_add_term_debug_rejects_incoherent_de_bruijn_input`
fails under `--release` only: it is `#[should_panic]` on a `debug_assert!`
compiled out in release — pre-existing, unrelated, passes in debug.)

**Two corrections to the plan document**, both found by doing the work:

- §1.4.3 claimed `tests/corpus/ondemand_fft_roundtrip_id_016.dsp` exceeded
  120 s. It compiles in ~0.2 s. The original timing was taken while that file
  carried an uncommitted working-tree modification and while the measuring loop
  invoked a non-existent `timeout` binary. No slow-compile outlier is
  demonstrated; risk R4 is downgraded from observed to speculative. The
  `timeout(1)`-unavailable finding stands.
- §1.4.5's "34 diagnostic codes" is the output of a textual grep and
  overcounts. `FRS-SRC-0001..0003` and `FRS-COMP-0001..0003` are declared in
  `crates/errors/src/codes.rs` but never constructed; `FRS-EVAL-0100` is a
  literal inside the unit test `bundle_counts_error_severity_only`
  (`crates/errors/src/lib.rs:310`), not an emitted code; `FRS-LEX-0001` has a
  live call site but is unreachable from the CLI because the lexer's catch-all
  rule in `faustlexer.l` matches every byte, so lexical failures surface as
  `FRS-PARSE-0001` or via the D1 fallback. Real surface: **27 constructible,
  26 CLI-reachable**. The frozen table still pins all 34 (self-consistent with
  the extraction), but pinning dead and test-only codes into a public contract
  is a mistake — recorded as open question O4, to settle before any exposure.

Also noted, out of scope: a DSP with constant-zero division (`_ / (0 : *(0))`)
triggers a caught-but-noisy panic in `crates/normalize/src/mterm.rs:400` during
FIR-verifier-warning cases.

Implemented by a Sonnet 5 subagent; D1/D2 behavior, the 218-file gate and
mutation (b) were re-verified independently before commit.


## Analyze exposing faust-rs as an MCP server, and whether it helps the compiler

Commit date: 2026-07-21
Source commit: this commit

New plan document `porting/mcp-server-analysis-and-plan-2026-07-21-en.md`,
answering two questions: should `faust-rs` be exposed as an MCP server and with
which API (Parts 1–5), and does that kind of tooling help improve the compiler
itself (Part 6).

**Positioning.** Five Faust MCP servers are already connected in this
environment (`faustcode`, `faustnode`, `faustplugins`, `faustremote`,
`faust-mcp-app`). The compile → wasm → run → spectrum → MIDI space is
saturated by two independent implementations, so the proposed server is
strictly *static*: diagnostics, compilation metadata, IR explanation, option
comparison, automatic differentiation, block diagrams. Six tools, deliberately
not thirty — the closed option set and the absence of a flag-passthrough
escape hatch are design constraints, not omissions.

**Measured constraints** (release build at `6f56dfbe`):

- Token budget is the dominant constraint. `auto_chorus_stereo_fad_host.dsp`
  (62 lines) produces 1.2 MB of signal IR and 5.0 MB of FIR IR — roughly 1.2 M
  tokens. No tool may return a raw dump; the default must be a structured
  summary, and building that summarizer is the actual engineering work.
- The machine channel is not usable as-is: `--error-format json` writes the
  payload to stderr behind a plain-text `C++ pipeline failed:` prefix, so
  stderr does not parse as JSON and stdout is empty on failure. Exit codes are
  0/1. Fixing this is P0 and belongs in the CLI, not in a wrapper.
- Some corpus files exceed the 120 s `--timeout` default, and this platform has
  no `timeout(1)`; the server must enforce its own wall clock (15–20 s).
- 34 `FRS-*` diagnostic codes across 8 stage namespaces become a public
  contract on exposure and must be frozen first.

**Part 6 — effect on compiler development.** The answer splits. As a tool
surface for maintainers: no. MCP earns its overhead when a host boundary must
be crossed, and in-repo agents already have `Bash`, the corpus and `xtask`;
the xtask command surface is phase-driven and must stay fluid, so a published
schema would create version drag; and the real bottleneck is knowledge (the
validation-gate traps), which no tool call transmits.

As a forcing function: yes. Three capabilities it requires stand on their own
merits.

- **C1** — the structural IR summarizer serves maintainers at least as much as
  users, since compiler work *is* IR reading and the megabyte dumps above are
  a maintainer's problem too. Consequence: P2 is re-scoped as a `crates/`-level
  library capability callable from `xtask`, tests and the CLI, with a thin MCP
  adapter — not as MCP-server work.
- **C2** — the clean JSON channel (P0) is a CI/harness win independent of MCP;
  today every automated consumer must string-scrape.
- **C3** — query-shaped differential access, new and independent of the server.

**Pre-existing defects surfaced by the audit** (unrelated to MCP; visible only
because the question forced an audit of what the maintainer-facing interface
actually returns):

- `porting/phases/phase-4-corpus-status-diff-report-en.md` is dated 2026-06-10
  and reports 190 cases against a corpus that now holds 218 `.dsp` files, with
  no staleness signal in the document.
- The same report lists 78 `ERR/OK` entries, predominantly
  `undefined symbol : fad|rad` — expected divergence, since the reference
  compiler lacks those primitives — classified identically to a genuine parity
  break. Read cold, the report is misleading.

These make C3 (a staleness-aware, expected-divergence-classifying, per-case
parity query mode) the recommended first step, ahead of P1: smallest of the
three, no dependency on the MCP work, and it fixes a currently misleading
report. Recorded as open question O3.

**Dogfooding as a diagnostics instrument (§6.4).** Diagnostic quality is
normally unmeasured. The session that motivated the document is the example:
on `repro_fad_error.dsp`, the compiler's message was *correct* —
`too many arguments: expected at most 2, got 3`, with `expr=min` and both
spans — yet the file's own header comment attributed the failure to a
`fad`/`fft` interaction above `N > 8`. A correct message that readers
routinely misread is a diagnostic defect, and the missing element is
identifiable: the message never states *why* three arguments were parsed, i.e.
that `,` re-associated inside the argument list. The P5 evaluation set is
therefore given a second purpose beyond the go/no-go gate: it is the
instrument that turns such cases into measurements.

Documentation only; no code, no behavior change.
