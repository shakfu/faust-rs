## Stop SFIR diagnostics from repeating their own code

Commit date: 2026-07-21
Source commit: this commit

`signal_fir_diagnostic` built the `Diagnostic` message from
`SignalFirError`'s `Display`, which renders as `[<code>] <message>`. Since the
`Diagnostic` already carries the code, the human renderer printed it twice:

```
error [FRS-SFIR-0004] [FRS-SFIR-0004] signal preparation failed: ...
```

Noticed while wiring the division-by-zero diagnostic and confirmed on an
unrelated corpus case before concluding it was pre-existing rather than
introduced there. `SignalFirError` already exposes `message()` — the bare text
without the bracketed prefix — so the fix is to use it and let the diagnostic
own the code. One line.

The regression guard lives in the existing
`check_json_sfir_family_failure_is_clean`: the message must not contain the
code the diagnostic already reports. Rejecting mutation (restore
`error.to_string()`) confirmed to fail it.

**The extraction trap caught me again.** Explaining the fix in a comment, I
wrote the code pattern as `FRS-SFIR-000N`, and the frozen-table checker — which
scans `.rs` source textually — extracted `FRS-SFIR-000` as a new undocumented
code, failing the workspace suite. This is the same trap documented one commit
earlier for the retired `COMP` numbers, hit again within the day. Anything
code-shaped in a `.rs` file becomes a code, comments included; the comment now
says `[<the SFIR code>]`. Worth keeping in mind for any future diagnostics work.

Workspace suite 0 failures, clippy `-D warnings` clean.


## Wire up `FRS-CODEGEN-*`: no compiler error reaches the user without a code

Commit date: 2026-07-21
Source commit: this commit

Closes the last `code: null` paths. Five backend-emission variants (`Codegen`,
`CodegenC`, `CodegenJulia`, `CodegenInterp`, `CodegenWasm`) plus `MissingRoot`
returned `None` from `CompilerError::diagnostics()`, so a backend failure was as
unstructured as an unresolved import was before this morning. Note the list is
five, not the four originally identified: `CodegenInterp` was never named
because it fell into the `_ => None` catch-all rather than an explicit arm.

**Design: one code, not five.** The intended shape was one `FRS-CODEGEN-*` per
backend, until an inventory showed the emitters already own a **separate
taxonomy** — 27 stable codes shaped `FRS-CGEN-<LANG>-NNNN` across ASC, C, CLIF,
CPP, INTERP, JULIA, RUST and WASM — that simply never reached a
`DiagnosticBundle`. Minting five more codes on top would have created two
competing schemes for identical events. Instead this mirrors the pattern already
established for FIR (`FRS-FIR-0002` + `fir_code=...`): a single
`FRS-CODEGEN-0001`, with the discriminating detail as notes:

```
error [FRS-CODEGEN-0001] wasm backend code generation failed: unsupported
  compute local collector node in WASM subset: SimpleForLoop { … }
  = note: backend: wasm
  = note: codegen_code=FRS-CGEN-WASM-0003
  = help: this is a backend limitation or a compiler bug, not a DSP syntax error
  = help: try another `-lang` backend to confirm the front-end result is sound
```

The failure class is identical across backends and the backend is a parameter,
so it belongs in a note rather than in the code. Verified on a real corpus case
(`err_rad_delay_temporal_unsupported.dsp -lang wasm`), not only on a fixture.

`FRS-CGEN-*` codes cannot leak into the frozen set: the extra `-<LANG>` segment
means the extraction regex `FRS-[A-Z]+-[0-9]+` never matches them. Checked
explicitly and documented, given how the retirement comment resurrected codes in
the previous commit.

**`MissingRoot` → `FRS-COMP-0005`**, documented as an internal invariant guard:
a parse reporting no errors always exposes a root, and an empty file fails later
with `FRS-EVAL-0001`, so reaching it means a compiler bug. The diagnostic says
so and asks for a report rather than implying a DSP mistake.

**Supporting changes.** Backend error types gained the accessors needed to split
code from message without parsing rendered text: `message()` on the C++, C,
Julia and WASM error structs, and `as_str` made public on the interpreter's code
enum (it alone was private). Each codegen variant now carries its bundle, built
by constructors (`CompilerError::codegen_wasm`, `::missing_root`, and the shared
`codegen_diagnostics`) so the bundle and the error can never disagree — the same
shape used for `Import`.

**The compiler found the end state.** Once the last variant was wired, `rustc`
reported the `_ => None` arm of `diagnostics()` as unreachable: no path returns
`None` any more. The arm is removed and the match left deliberately exhaustive,
so a future variant that forgets its bundle fails to compile. That catch-all is
exactly how the gap persisted for imports and backends in the first place. The
method keeps its `Option` return for source compatibility, with the invariant
documented.

Net: 32 codes, 9 namespaces, and every `CompilerError` carries structured
diagnostics.

**Validation.** 13/13 CLI diagnostics-channel tests including the new
`codegen_backend_failure_is_structured`; workspace suite 0 failures; clippy
`-D warnings` clean; P0 gate still 218/218; 32 codes in source = 32 documented,
registry and frozen table in agreement. Rejecting mutations, applied, observed
and reverted: dropping the `codegen_code` note → the test fails listing the
remaining notes; unwiring the WASM bundle → the test fails on `code: Null`.


## Curate the frozen code table: wire up `FRS-SRC-*`, retire three `FRS-COMP-*`

Commit date: 2026-07-21
Source commit: this commit

Settles open question O4 from `porting/mcp-server-analysis-and-plan-2026-07-21-en.md`:
the table frozen by P0's D3 contained codes that were never emitted, and
publishing those as a public contract would promise diagnostics that can never
arrive. Auditing them split the set three ways rather than the expected two.

**`FRS-SRC-0001..0003` — missing diagnostics, now written.**
`parser::source_reader::SourceReaderError` has exactly three variants, mapping
one-to-one onto the three codes, and all three fire today (unreadable path,
unresolved import, import cycle). They were not dead reservations: the taxonomy
had anticipated these failures precisely and nothing had converted them.
`SourceReaderError::to_diagnostics` now builds a real bundle for each, and
`CompilerError::import` attaches it — the variant carries the bundle so the two
can never disagree. An unresolved import previously reached the user through
the `code: null` no-bundle envelope with no span and no notes, for what is the
single most common newcomer failure; it now carries `FRS-SRC-0002`, a span on
the `import(...)` directive, and the ordered list of searched directories.
Reference C++ reports only `ERROR : unable to open file <name>`, so this is
deliberately more informative than parity rather than a port of it.

Two obstacles were worth the detour:

- *Two import-expansion paths exist.* The `source_reader` one scans lines and
  knows the position; the box-level one works from an already-parsed tree whose
  nodes carry no location, and that is the path the CLI takes. Rather than
  threading locations through the box tree, `ImportSite::locate_in` re-scans the
  file to recover the directive — on the error path only, never during a
  successful compile. The site stays `Option`, so a failed relocation yields no
  label rather than a wrong one.
- *A pre-existing comment-scanner bug surfaced by failing the new test.*
  `advance_block_comment_state` searched for `/*` without stopping at `//`, so
  an ordinary line comment containing a glob (`// see tests/corpus/*.dsp`) read
  as opening a block comment and hid every following `import` from the source
  reader. Scope was checked before concluding: the box-level path still resolves
  those imports, so there is no miscompile — the observable symptom was the lost
  span. One-line fix; the fixture now deliberately carries such a glob as a
  regression guard.

**`FRS-COMP-0001..0003` — retired.** These meant "parse / eval / propagate stage
failed", but each of those stages already has its own family (`FRS-PARSE-*`,
`FRS-EVAL-*`, `FRS-PROP-*`, with `0099` generic fallbacks) carrying strictly
more information. `FRS-COMP-0004` stays because the type stage has no family of
its own — there is no `FRS-TYPE-*` — which is why it alone was ever wired up. It
is deliberately **not** renumbered into the gap: renumbering a live code is the
one operation the freeze rule forbids, and a numbering gap is the correct end
state. A new "Retired codes" table in `docs/diagnostics-codes-en.md` burns the
numbers permanently: deleting a never-emitted code is safe, reusing its number
later is the same silent break the freeze rule prevents, merely delayed.

**`FRS-EVAL-0100` — removed.** Never a code: a literal inside the unit test
`bundle_counts_error_severity_only`, captured only because the extraction is
textual. The test now uses a registered code (`EVAL_GENERIC_FAILURE`).

**`FRS-LEX-0001` — kept, deliberately.** Unlike the above it is not an unused
constant but one arm of an exhaustive `match` over `lrpar::LexParseError`, a
third-party enum. Deleting it would remove no code path, only force that arm to
report a less accurate code. Unreachable today solely because the lexer's
catch-all rule matches every byte.

**`FRS-EVAL-0006` — registry gap fixed.** `EVAL_SLIDER_INIT_OUT_OF_RANGE` was
declared and genuinely emitted from `crates/eval/src/error.rs:692`, yet was
absent from `codes::all_codes()`, so a consumer enumerating codes through the
registry would never have seen it. Nothing compared the registry with the frozen
table, which is why it went unnoticed; the new
`code_registry_matches_frozen_table` now checks both directions.

**A property of the frozen-table checker, learned the hard way.** Writing the
retired numbers in full inside a `codes.rs` comment resurrected them: the
checker extracts codes *textually* from `.rs` source, so anything shaped like a
code becomes one, comment or not. The retirement note therefore spells them
without the `FRS-` prefix, with the reason stated on the spot, and the
authoritative record lives in the doc.

Net effect: 30 codes, all emitted except `FRS-LEX-0001` and `FRS-FIR-0001`,
whose unreachability is documented with its cause.

**Validation.** 12/12 CLI diagnostics-channel tests including the rewritten
`check_json_src_family_unresolved_import_is_structured` (code, span, searched
directories); workspace suite 0 failures; clippy `-D warnings` clean; P0 gate
still 218/218. Rejecting mutations, each applied, observed and reverted:
unwiring the `Import` bundle → the SRC test fails; reverting the comment-scanner
fix → the label assertion fails; dropping `EVAL_SLIDER_INIT_OUT_OF_RANGE` from
the registry again → the new registry check fails naming the code.

**Next gap, not addressed here:** `Codegen`, `CodegenC`, `CodegenJulia`,
`CodegenWasm` and `MissingRoot` still surface as `code: null`. A backend failure
today is as unstructured as an unresolved import was this morning; that argues
for an `FRS-CODEGEN-*` family.


## Reject constant division by zero as a diagnostic instead of swallowing a panic

Commit date: 2026-07-21
Source commit: this commit

`crates/normalize/src/mterm.rs` guarded division by a constant zero with a bare
`assert!`. The resulting panic was caught by `simplify_signals_fastlane`, whose
`Err(_)` arm cleared the cache and returned the *original, unsimplified*
signal. Net effect on `process = _ / (0 : *(0));`: raw panic text on stderr,
then `Check OK: 0 diagnostics` and **exit 0** — a false green on a program the
reference compiler rejects. The panic noise was the visible symptom; the
swallowed rejection was the actual defect.

**C++ reference behavior, established before choosing a fix.** The homologous
site `mterm::operator/=(Tree)` (`compiler/normalize/mterm.cpp:172`) does not
warn — it throws `faustexception("ERROR : division by 0 in ...")`, as does
`operator/=(const mterm&)` (`mterm.cpp:292`). Confirmed against the reference
binary: `faust fir_constant_zero_division.dsp` prints
`ERROR : division by 0 in IN[0] / 0.0f` and exits 1, **identically with and
without `-wall`**.

The `-wall` warning machinery does exist, but governs a different site:
`sigPromotion.cpp:340` reports a *potential* division by zero found by interval
analysis (`j1.hasZero()`), gated by `-me`/`gMathExceptions` to compute it, with
`-wall`/`gAllWarning` only deciding whether accumulated warnings are printed at
the end (`main.cpp:51`). Two distinct gates, and neither applies to the
constant case. Porting that warning would be new work and depends on intervals
being attached to signals (interval crate steps 6–7, still open).

**Fix.** The panic now carries a typed `DivisionByZero` payload. The existing
`catch_unwind` in `simplify_signals_fastlane` downcasts it and returns
`NormalFormError::DivisionByZero`, which `signal_prepare` maps to a new
`SignalPrepareError::DivisionByZero`, surfacing as `FRS-SFIR-0004` with exit 1.
A panic hook installed once suppresses the default report **only** for this
payload and delegates every other payload to the hook it replaced, so genuine
panics keep their normal output. All four assertion sites are converted
(`div_nums` ×2, `div_sig`, `div_mterm`).

Two design points, documented in the code:

- *Unwind rather than `Result`.* `add_sig`/`sub_sig` are mutually recursive and
  reach every simplification rule, so threading a `Result` would convert the
  whole `Mterm`/`Aterm`/`simplify` call graph for a condition that is fatal
  anyway — and C++ uses an exception at the same point for the same reason. The
  typed payload keeps the existing mechanism while making an expected rejection
  distinguishable from a compiler bug, which the bare `assert!` was not.
- *Detection cannot move earlier.* In `_ / (0 : *(0))` the divisor only becomes
  the constant zero after constant folding, so a pre-normalization scan for
  `Div(_, 0)` would miss it. C++ detects it at the same point for the same
  reason.

No new `FRS-*` code was added: `FRS-SFIR-0004` ("signal preparation failed")
already covers the case, and adding one would enlarge a table already flagged
for curation (open question O4 in the MCP plan).

**Validation.** New `division_by_zero_is_a_clean_diagnostic_not_a_panic`
asserts exit 1, the `FRS-SFIR-0004` code, a message naming the division, and
the absence of panic text on stderr. Rejecting mutations, each applied,
observed and reverted: removing the quiet hook → the stderr assertion fails;
re-swallowing the payload → the exit-1 assertion fails. Full workspace suite
0 failures, clippy `-D warnings` clean, fmt applied. The P0 gate still holds at
218/218, and a corpus-wide scan confirms **0 files print panic text** under
`--check`.

**Regression fixed in passing, with a correction.** `check_json_fir_family_
failure_is_clean` broke, as expected — the DSP no longer reaches the FIR
verifier. Repairing it showed the test's original premise was wrong:
`--fir-verify-strict` fails on *any* DSP (FIR-M07, `getSampleRate` not
declared), so no dedicated fixture was ever needed to reach the FIR family. The
test now uses an ordinary corpus DSP; the fixture backs the new
division-by-zero test; both comments are corrected.

**Pre-existing defect noted, not fixed:** SFIR diagnostics duplicate their code
in the rendered message (`error [FRS-SFIR-0004] [FRS-SFIR-0004] signal
preparation failed: ...`), reproduced on an unrelated corpus case. Out of scope
here, but exactly the diagnostic-quality class §6.4 of the MCP plan targets.


## P0 — clean machine channel: JSON diagnostics on stdout, `--check`, frozen code table

Commit date: 2026-07-21
Source commit: this commit

Implements phase P0 of `porting/mcp-server-analysis-and-plan-2026-07-21-en.md`.
The motivation is independent of the MCP work (plan §6.3, C2): until now no
automated consumer — CI, IDE, the xtask harnesses — could read faust-rs
diagnostics without string-scraping.

**D1 — JSON on stdout, alone.** `crates/compiler/src/cli/diagnostics.rs:49`
(`print_structured_diagnostics`) writes the JSON branch to stdout instead of
stderr. `crates/compiler/src/cli/runner.rs:550` adds
`report_pipeline_failure(prefix, err, cli)`, which emits the human prefix line
only under `--error-format human`; the 17 per-backend
`eprintln! + print_structured_diagnostics + exit(1)` sites collapse onto it.
`--error-format human` output is unchanged, guarded by a regression test.

Two design points settled during implementation:

- *stdout conflict rule* — on a successful dump-mode compile, generated output
  stays on stdout and no diagnostics payload is added. The CLI never emitted
  one on success outside `--check`, so there is nothing to interleave. Rule
  documented in the module docs and in `docs/user-diagnostics-guide-en.md`.
- *no-bundle fallback* — `CompilerError` variants carrying no
  `DiagnosticBundle` (backend codegen, import failures; reached in practice by
  an unresolved `import(...)`) now emit a `code: null` envelope of the same
  shape via `format_fallback_diagnostics_json` (`diagnostics.rs:87`) rather
  than nothing, so the schema is uniform across every failure path.

**D2 — `--check`.** New flag (`cli/args.rs:113`), wired into the mode-exclusion
array and the `--architecture` / `--fir-fixture` / `--no-fir-verify`
validation blocks. Dispatch at `runner.rs:1440` runs
`compile_file_to_fir_with_lane` (parse → eval → propagate → type → FIR verify,
no codegen — the same call `--dump-fir-verify` uses). Success emits an empty
`diagnostics` array and exit 0, so success and failure share one schema and a
consumer needs one parser, not two.

**D3 — frozen code table.** `docs/diagnostics-codes-en.md` documents every
code with stage, meaning and raise site. `cli/tests.rs:1298`
(`frozen_frs_code_table_matches_source`) walks `crates/` natively and diffs
against the documented set, so adding an undocumented code or renumbering an
existing one fails CI.

**Validation.** New `crates/compiler/tests/cli_diagnostics_channel.rs`: 11
subprocess tests spawning the real binary, covering PARSE/EVAL/PROP/COMP/SFIR
from corpus cases plus FIR and SRC from purpose-written fixtures under
`crates/compiler/tests/fixtures/cli_diagnostics/`. LEX has no CLI-reachable
fixture (see below) and is covered by a synthetic-diagnostic unit test at
`cli/tests.rs:1166` instead — recorded rather than skipped silently.

Rejecting mutations, each applied, observed, and reverted: (a) prefix
reintroduced onto the JSON stream → 9/11 checker tests fail; (b) payload routed
back to stderr → 7/11 fail (the 4 survivors exercise the success and SRC
fallback paths, which that mutation does not touch); (c) `FRS-EVAL-0003`
renumbered → the frozen-table test fails with an explicit diff. Mutation (b)
was independently re-applied and re-verified outside the implementing agent.

Gate met: **218/218** corpus files emit a parseable JSON document on stdout
under `--check --error-format json`. `cargo fmt --check`, `cargo clippy
--workspace --all-targets -- -D warnings` and `cargo test --workspace` clean.
(`normalize::tests::normalize_add_term_debug_rejects_incoherent_de_bruijn_input`
fails under `--release` only: it is `#[should_panic]` on a `debug_assert!`
compiled out in release — pre-existing, unrelated, passes in debug.)

**Two corrections to the plan document**, both found by doing the work:

- §1.4.3 claimed `tests/corpus/ondemand_fft_roundtrip_id_016.dsp` exceeded
  120 s. It compiles in ~0.2 s. The original timing was taken while that file
  carried an uncommitted working-tree modification and while the measuring loop
  invoked a non-existent `timeout` binary. No slow-compile outlier is
  demonstrated; risk R4 is downgraded from observed to speculative. The
  `timeout(1)`-unavailable finding stands.
- §1.4.5's "34 diagnostic codes" is the output of a textual grep and
  overcounts. `FRS-SRC-0001..0003` and `FRS-COMP-0001..0003` are declared in
  `crates/errors/src/codes.rs` but never constructed; `FRS-EVAL-0100` is a
  literal inside the unit test `bundle_counts_error_severity_only`
  (`crates/errors/src/lib.rs:310`), not an emitted code; `FRS-LEX-0001` has a
  live call site but is unreachable from the CLI because the lexer's catch-all
  rule in `faustlexer.l` matches every byte, so lexical failures surface as
  `FRS-PARSE-0001` or via the D1 fallback. Real surface: **27 constructible,
  26 CLI-reachable**. The frozen table still pins all 34 (self-consistent with
  the extraction), but pinning dead and test-only codes into a public contract
  is a mistake — recorded as open question O4, to settle before any exposure.

Also noted, out of scope: a DSP with constant-zero division (`_ / (0 : *(0))`)
triggers a caught-but-noisy panic in `crates/normalize/src/mterm.rs:400` during
FIR-verifier-warning cases.

Implemented by a Sonnet 5 subagent; D1/D2 behavior, the 218-file gate and
mutation (b) were re-verified independently before commit.


## Analyze exposing faust-rs as an MCP server, and whether it helps the compiler

Commit date: 2026-07-21
Source commit: this commit

New plan document `porting/mcp-server-analysis-and-plan-2026-07-21-en.md`,
answering two questions: should `faust-rs` be exposed as an MCP server and with
which API (Parts 1–5), and does that kind of tooling help improve the compiler
itself (Part 6).

**Positioning.** Five Faust MCP servers are already connected in this
environment (`faustcode`, `faustnode`, `faustplugins`, `faustremote`,
`faust-mcp-app`). The compile → wasm → run → spectrum → MIDI space is
saturated by two independent implementations, so the proposed server is
strictly *static*: diagnostics, compilation metadata, IR explanation, option
comparison, automatic differentiation, block diagrams. Six tools, deliberately
not thirty — the closed option set and the absence of a flag-passthrough
escape hatch are design constraints, not omissions.

**Measured constraints** (release build at `6f56dfbe`):

- Token budget is the dominant constraint. `auto_chorus_stereo_fad_host.dsp`
  (62 lines) produces 1.2 MB of signal IR and 5.0 MB of FIR IR — roughly 1.2 M
  tokens. No tool may return a raw dump; the default must be a structured
  summary, and building that summarizer is the actual engineering work.
- The machine channel is not usable as-is: `--error-format json` writes the
  payload to stderr behind a plain-text `C++ pipeline failed:` prefix, so
  stderr does not parse as JSON and stdout is empty on failure. Exit codes are
  0/1. Fixing this is P0 and belongs in the CLI, not in a wrapper.
- Some corpus files exceed the 120 s `--timeout` default, and this platform has
  no `timeout(1)`; the server must enforce its own wall clock (15–20 s).
- 34 `FRS-*` diagnostic codes across 8 stage namespaces become a public
  contract on exposure and must be frozen first.

**Part 6 — effect on compiler development.** The answer splits. As a tool
surface for maintainers: no. MCP earns its overhead when a host boundary must
be crossed, and in-repo agents already have `Bash`, the corpus and `xtask`;
the xtask command surface is phase-driven and must stay fluid, so a published
schema would create version drag; and the real bottleneck is knowledge (the
validation-gate traps), which no tool call transmits.

As a forcing function: yes. Three capabilities it requires stand on their own
merits.

- **C1** — the structural IR summarizer serves maintainers at least as much as
  users, since compiler work *is* IR reading and the megabyte dumps above are
  a maintainer's problem too. Consequence: P2 is re-scoped as a `crates/`-level
  library capability callable from `xtask`, tests and the CLI, with a thin MCP
  adapter — not as MCP-server work.
- **C2** — the clean JSON channel (P0) is a CI/harness win independent of MCP;
  today every automated consumer must string-scrape.
- **C3** — query-shaped differential access, new and independent of the server.

**Pre-existing defects surfaced by the audit** (unrelated to MCP; visible only
because the question forced an audit of what the maintainer-facing interface
actually returns):

- `porting/phases/phase-4-corpus-status-diff-report-en.md` is dated 2026-06-10
  and reports 190 cases against a corpus that now holds 218 `.dsp` files, with
  no staleness signal in the document.
- The same report lists 78 `ERR/OK` entries, predominantly
  `undefined symbol : fad|rad` — expected divergence, since the reference
  compiler lacks those primitives — classified identically to a genuine parity
  break. Read cold, the report is misleading.

These make C3 (a staleness-aware, expected-divergence-classifying, per-case
parity query mode) the recommended first step, ahead of P1: smallest of the
three, no dependency on the MCP work, and it fixes a currently misleading
report. Recorded as open question O3.

**Dogfooding as a diagnostics instrument (§6.4).** Diagnostic quality is
normally unmeasured. The session that motivated the document is the example:
on `repro_fad_error.dsp`, the compiler's message was *correct* —
`too many arguments: expected at most 2, got 3`, with `expr=min` and both
spans — yet the file's own header comment attributed the failure to a
`fad`/`fft` interaction above `N > 8`. A correct message that readers
routinely misread is a diagnostic defect, and the missing element is
identifiable: the message never states *why* three arguments were parsed, i.e.
that `,` re-associated inside the argument list. The P5 evaluation set is
therefore given a second purpose beyond the go/no-go gate: it is the
instrument that turns such cases into measurements.

Documentation only; no code, no behavior change.
