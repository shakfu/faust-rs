## Clarify C++ DSP instance destruction in the root README

Commit date: 2026-07-22
Source commit: this commit

Replaced the two `processor.reset()` calls in the Interpreter and Cranelift C++
examples with explicit scopes. The calls were technically
`std::unique_ptr<dsp>::reset()`, not members of the Faust `dsp` API, but the
spelling misleadingly suggested that `dsp` exposed a `reset` lifecycle method.
The scoped examples now express the actual ownership requirement directly: the
DSP instance is destroyed before the factory that created it.

warnings`, `cargo test --workspace --all-targets`, and `git diff --check`.

## Regenerate the vector coverage baseline for the new corpus DSP

Commit date: 2026-07-22
Source commit: this commit

CI failed on the Ubuntu `vector-coverage-check` job with `vector mode
precision=f32 -lv 0 -ss 0 does not cover the exact corpus`. The cause was a
stale baseline, not a compiler regression: commit `e42547d1` ("Fix vector
identity recursion aliases") added
`tests/impulse-tests/dsp/recursive_identity_passthrough.dsp` to the vector
corpus without regenerating `tests/vector-coverage/corpus-baseline.json`, whose
last refresh dated from `a180aa6b`. `validate_vector_coverage` requires every
mode report to list the corpus exactly, so 132 listed entries against 133 files
on disk rejected the first mode it validated.

Regenerated the sixteen `count_vector_corpus` reports (both precisions, `-lv
0/1`, `-ss 0..3`) and merged them with `vector-coverage-merge`. The new DSP is
certified in all sixteen modes, so the diff is limited to it: per-mode totals
move from 132 to 133, certified counts from 96 to 97, and
`certified-dspfiles.txt` gains a single line. No other corpus entry changed
status, and no fallback or error classification moved.

Validated with `cargo run -p xtask -- vector-coverage-check`, which reports OK
over 1552 retained certified mode/DSP pairs across 16 modes and 133 corpus
DSPs.

Retention note: any commit adding a `.dsp` under `tests/impulse-tests/dsp/`
must regenerate this baseline in the same commit, otherwise the coverage gate
fails closed on the very first mode.

## Refresh FAD/RAD user documentation and add the bilingual synthesis

Commit date: 2026-07-22
Source commit: this commit

Refreshed the FAD and RAD user notes against the current compiler contracts:
FAD output ordering is per primal output and seed lane; RAD returns primal
outputs followed by gradients of their implicit all-ones sum; temporal RAD uses
the current `compute(count)` block as its reverse horizon with zero terminal
adjoint state. The notes now distinguish exact symbolic rules from the
documented finite-difference table-index rule and zero-tangent or explicit
diagnostic boundaries.

Added the English `docs/fad-rad-synthesis-en.md` counterpart to the French
synthesis, cross-linked both documents, and replaced references to local,
unversioned sketches with tracked corpus coverage or explicit illustrative
labels. The main README now introduces clock-domain primitives (`ondemand`,
`upsampling`, `downsampling`) and FAD/RAD, with links to the user-facing notes.

The FAD note records FAD's clock-domain payload handling and RAD's current
clock-boundary rejection. The RAD notes also document block-local BRA behavior,
Signal-IR seed identity, unsupported-family behavior, and the current `abs(0)`
formula limitation. Removed stale FAD/synthesis PDF copies rather than leaving
them inconsistent with their Markdown sources.

Validated with `cargo test -p compiler --test fad_recursive_runtime --test
ondemand_pipeline --test rad_runtime` (96 passed; one pre-existing
`abs(0)`-related RAD test remains ignored), `cargo check -p compiler --examples`,
