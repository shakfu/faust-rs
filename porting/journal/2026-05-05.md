## docs(rad): fix E1 scalar-seed gradient convention

Commit date: 2026-05-05
Source commit: `4c1786c`

Recorded the phase-E1 decision that scalar seed/parameter gradients are exposed
as per-sample contributions, not as an implicit block-reduced scalar.

### Decision

- `rad(...)` emits `grad_p[n]`, the contribution of frame `n` to `dJ/dp`.
- The block gradient remains an explicit reduction:

  ```text
  dJ/dp = Σ_{n=0..count-1} grad_p[n]
  ```

- `rad(...)` does not repeat the block scalar on every frame and does not hide
  it on a sentinel frame.
- DSP or host code that wants a block update can reduce explicitly over the
  current block size; Faust `maths.lib` exposes that size as `ma.BS`.

### Impact

- Phase E1 can keep the output surface signal-like and sample-aligned.
- Optimizer examples must make accumulation explicit instead of assuming the
  seed-gradient lane is already block-reduced.

## fix(ui): trim newlines from widget labels and sort group children by metadata index

Comparison of `clarinetMIDI.dsp.json` (C++ Faust) against the faust-rs output
revealed two bugs in `UiProgramBuilder` / `split_label_metadata`.

### Bug 1 — trailing `\n` in `label`, `shortname`, `address`

`trim_space_tab` only stripped spaces and tabs, leaving any `\n` or `\r`
characters present in the raw DSP label string.  Fixed by adding `'\n' | '\r'`
to the `trim_matches` pattern in `crates/ui/src/lib.rs`.

### Bug 2 — wrong item order within groups

The C++ compiler uses the bare-integer metadata keys (`[0:]`, `[1:]`, `[2:]`, …)
to sort items within each UI group.  faust-rs inserted items in DAG traversal
order, ignoring these keys, so the JSON widget order differed from C++.

**Fix:**

- Added `pub fn ordering_key_from_metadata(metadata: &UiMetadata) -> i64` in
  `crates/ui/src/lib.rs`: scans metadata for the first key that parses as `i64`,
  returns `i64::MAX` when none is found.
- Changed `UiDraftNode::Group.children` from `Vec<usize>` to `Vec<(i64, usize)>`
  (order key + node id).
- Replaced `children_mut()` with `insert_child_sorted()`, which uses
  `partition_point(|(k,_)| *k <= order_key)` — stable ascending sort that
  preserves insertion order for equal keys.
- `find_or_create_group()` now extracts the ordering key from the group spec's
  metadata and calls `insert_child_sorted`.
- The three public methods `insert_input_control`, `insert_output_control`,
  `insert_soundfile` now take an explicit `order_key: i64` parameter.
- In `crates/propagate/src/lib.rs` the three insertion sites call
  `ui::ordering_key_from_metadata(&metadata)` before moving `metadata` into
  `register_control`, and forward the key to the builder.

Result: the `clarinetMIDI` JSON UI section now matches the C++ reference exactly
(order and clean labels).  All existing tests pass without modification to their
assertions.

### Validation

- `cargo test -p ui -p propagate`
- `cargo test --workspace --all-targets`

## plan(normalize): De Bruijn coherence check during factorisation

Wrote `porting/normalize-debruijn-coherence-check-plan-2026-05-05-en.md`.

### Motivation

The FAD cache-poisoning bug (`61f1cb2`) produced a signal tree that was
**structurally closed** (aperture ≤ 0 at the root — existing gate in
`de_bruijn_to_sym` would not have caught it) but **semantically wrong**: a
cached tangent computed inside a `DEBRUIJNREC` body escaped into the outer
scope because its `SigId` hash-consed identically to the FAD seed signal.

The plan adds a **strictly stronger invariant**: at every local traversal depth
`d` (number of enclosing `DEBRUIJNREC` binders), any `DEBRUIJNREF(k)` must
satisfy `k < d`.  This catches the class of structural incoherence that the
global aperture check misses.

### Distinction from `de_bruijn_aperture`

`de_bruijn_aperture(root) <= 0` checks that the *root-level aperture* is
non-positive — references that are closed by a nested binder contribute
negatively to the root aperture and hide violations.  The new check traverses
with an explicit depth counter and rejects any reference that escapes its
**local** enclosing binders, even if a farther outer binder would absorb it.

### Plan structure (5 steps)

1. **`tlib`** — `check_de_bruijn_coherence(arena, root) -> Result<(), RecursionError>` +
   new `RecursionError::IncoherentDeBruijnReference { node, level, depth }`.
   Memo key is `(TreeId, depth)` — same hash-consed node at different depths is
   a distinct check.  Five unit tests, including the critical
   `coherence_vs_aperture_distinction` case that shows a tree passing
   `is_de_bruijn_closed` while failing the new check.

2. **`propagate`** — Post-transform gate in `generate_fad_signals_multi` and
   `generate_rad_signals`.  New `PropagateError::DeBruijnCoherence { pass, detail }`.
   First line of defense, closest to the bug site.

3. **`normalize`** — Pre-`de_bruijn_to_sym` gate in `prepare_signals_multi`.
   Catches incoherent trees before the symbolic conversion attempt.
   Propagates via the existing `From<RecursionError>` into `NormalFormError::Recursion`.

4. **`normalize`** — `#[cfg(debug_assertions)]` panic guard in
   `normalize_add_term`.  Zero cost in release; documents the invariant
   (symbolic form only) and catches pipeline ordering bugs in debug/test.

5. **Compiler integration test** — disables the cache-restore fix temporarily,
   verifies the FAD gate returns `DeBruijnCoherence` rather than silently
   emitting wrong output.

No new `CompilerError` variant required; existing plumbing (`PropagateError` →
`CompilerError::Propagate`) covers the primary detection site.
