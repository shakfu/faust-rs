### `eval` / `tests` — Port captured-environment rewrites for `boxModifLocalDef`

Commit date: 2026-03-06

Source commit: pending (pre-commit worktree for the next closure-port step in this session)

#### Context

After restoring the parser and IR node for `boxModifLocalDef`, the main
remaining semantic gap was still the evaluator contract behind
`copyEnvReplaceDefs(...)` / `updateClosures(...)`. Without that rewrite step,
Rust could parse `expr [ defs ]` but still had no way to:

- copy one captured lexical layer,
- retarget closures that originally captured that layer,
- bind replacement definitions in the rewrite call-site environment,
- re-evaluate the closure value through the rewritten environment.

That behavior is the clearest place where the true closure model materially
differs from eager environment threading.

#### Changes

- `crates/eval/src/lib.rs`
  - added `Environment::spawn_child_with_parent(...)` and `layer_snapshot(...)`
    so the evaluator can clone one captured layer while reusing the same
    parent stack
  - added `EvalError::ExpectedClosureValue` for operators that require a
    captured closure value
  - added `eval_modif_local_def_value(...)` and wired
    `BoxMatch::ModifLocalDef` into `eval_value(...)`
  - ported the C++ environment rewrite core with:
    - `rewrite_captured_env(...)`
    - `copy_env_replace_defs(...)`
  - documented the direct provenance from C++
    `compiler/evaluate/environment.cpp` and corrected Rustdoc examples to use
    the postfix bracket syntax `expr [ defs ]`

- `crates/eval/tests/core_eval.rs`
  - added regression coverage asserting that:
    - rewriting an environment updates already-captured sibling closures
      (`b = a` follows rewritten `a`)
    - rewriting an abstraction closure changes its captured free variable view
    - replacement definitions capture the *current* rewrite scope, not the
      original captured environment

### `boxes` / `parser` / `propagate` / `tests` — Reintroduce `boxModifLocalDef` syntax and IR shape

Commit date: 2026-03-06

Source commit: pending (pre-commit worktree for the next closure-port step in this session)

#### Context

The true closure model cannot reach C++ parity for captured-environment
rewrites if the Rust frontend still lacks the corresponding IR node.
The C++ compiler exposes that feature through `boxModifLocalDef` and the
postfix bracket syntax `expr [ defs ]`. Rust still lowered environment literals
through `with_local_def`, but it had no generic representation for the rewrite
operator itself.

This step restores the parser and box-layer surface so the evaluator can port
`copyEnvReplaceDefs(...)` on top of a real node instead of another ad hoc form.

#### Changes

- `crates/boxes/src/lib.rs`
  - added `BOXMODIFLOCALDEF` support to the canonical box layer:
    - `BoxBuilder::modif_local_def(...)`
    - `BoxMatch::ModifLocalDef(BoxId, BoxId)`
    - `node_modif_local_def(...)` / matcher support
  - documented the direct C++ provenance from `boxes.cpp`

- `crates/parser/src/grammar/faustparser.y`
  - added the postfix grammar rule:
    - `InfixExp LCROC DefList RCROC`
  - declared postfix precedence for `LCROC` so the Rust parser generator keeps
    the same intended reading as the C++ grammar

- `crates/parser/src/lib.rs`
  - added `ParseState::modif_local_def_box(...)` as the boxed equivalent of
    C++ `boxModifLocalDef`

- `crates/parser/tests/structural_cpp_differential.rs`
  - added a differential structural case showing that:
    - `(e [ a = 1; ]).a`

- `crates/propagate/src/lib.rs`
  - marked `ModifLocalDef` as unsupported at propagation time, matching the

### `eval` / `tests` — Apply abstractions and access through captured closures

Commit date: 2026-03-06

Source commit: pending (pre-commit worktree for the next closure-port step in this session)

#### Context

Captured parser definitions restored lexical forcing for plain identifiers, but
the evaluator still normalized abstractions eagerly and still approximated
`access` structurally. That remained short of the C++ closure contract for:

- applying lambda values in their captured lexical environment,
- carrying symbolic arguments through nested closure applications,
- resolving `environment` access through the captured environment instead of
  the caller scope.

This step moves the evaluator one level deeper into the true closure model by
introducing closure-aware intermediate evaluation for abstractions and access.

#### Changes

- `crates/eval/src/lib.rs`
  - split the evaluator into:
    - `eval_value(...) -> EvalValue`
    - `force_value_to_box(...) -> TreeId`
  - changed `eval_box(...)` into a thin wrapper that evaluates to `EvalValue`
    first, then lowers to box IR only at the boundary
  - made `BoxMatch::Abstr` and `BoxMatch::Environment` evaluate to captured
    `EvalValue::Closure` values instead of eagerly normalizing them
  - added closure-aware application with `apply_value_list(...)`, so:
    - abstraction bodies are evaluated in the closure's captured environment,
    - nested symbolic arguments retain the caller environment through closure
      rebinding,
    - environment values reject extra arguments explicitly
  - rewrote `access` evaluation through `eval_access_value(...)`, so
    `expr.ident` resolves `ident` in the captured environment when `expr`
    evaluates to a closure/environment value
  - updated Rustdoc to mark abstraction/environment closure capture as
    implemented while environment-rewrite and pattern-matcher closure parity
    remain pending

- `crates/eval/tests/core_eval.rs`
  - added regression coverage asserting that:
    - `f(x) = y; process = f(0) with { y = 2; };` still resolves `y` from the
      closure's captured environment,
    - `e = environment with { a = 1; }; process = e.a with { a = 2; };`
      resolves `a` from the captured environment

### `eval` / `tests` — Store parser definitions as captured closures and force them in their binding environment

Commit date: 2026-03-06

Source commit: pending (pre-commit worktree for the next closure-port step in this session)

#### Context

After introducing stable `EnvId` identities, the next closure-port step was to
stop treating Faust definitions as raw `TreeId` aliases. The previous Rust
evaluator still resolved `f = x;` by re-evaluating `x` in the caller
environment, which is not the C++ contract once shadowing enters the picture.

This step makes captured definition closures explicit in the evaluator while
keeping abstraction/environment values on the adapted path for now.

#### Changes

- `crates/eval/src/lib.rs`
  - introduced internal evaluator values:
    - `EvalValue::Box(TreeId)`
    - `EvalValue::Closure(ClosureValue { expr, env })`
  - changed `Environment` layers to store `EvalValue` bindings instead of bare
    `TreeId`s
  - kept the public `Environment::bind(...)` API for box values, but added
    internal closure-aware binding/lookup helpers for evaluator use
  - changed `bind_definitions(...)` to store parser definitions as captured
    closures bound to the current `Environment`
  - changed `eval_box(...)` on `Ident` to:
    - resolve the binding layer id,
    - force `EvalValue::Closure` in the captured environment,
    - keep raw `EvalValue::Box` forcing for non-definition bindings
  - changed loop detection from raw-tree-only frames to:
    - `LoopFrame::Tree(TreeId)`
    - `LoopFrame::SymbolEnv { sym, env_id }`
    so definition recursion can now be keyed by `(symbol, env)` instead of only
    by expression shape
  - changed `eval_process_with_stats(...)` to enter evaluation through the
    `process` identifier again, matching the closure-based forcing path
  - updated Rustdoc to mark definition capture as implemented while
    abstraction/environment closures remain pending

- `crates/eval/tests/core_eval.rs`
  - added a regression asserting that:
    - `x = 1; f = x; process = f with { x = 2; };`

### `eval` / `tests` — Introduce stable `EnvId` layer identity as closure-port groundwork

Commit date: 2026-03-06

Source commit: pending (pre-commit worktree for the next closure-port step in this session)

#### Context

The current Rust evaluator still stores raw `TreeId` bindings, but the next
closure-port stages need a stable notion of environment identity comparable to
the C++ `ENV_LAYER` node identity. Without that, later steps cannot key
recursion tracking and captured closures on `(symbol, env)` the same way the
C++ evaluator does.

This first implementation step keeps the external evaluator semantics unchanged
while replacing the cloned parent-chain representation with a shared arena of
environment layers addressed by `EnvId`.

#### Changes

- `crates/eval/src/lib.rs`
  - introduced public `EnvId` as the stable identity of one evaluator
    environment layer
  - replaced `Environment { bindings, parent: Box<Environment>, barrier }` with
    a shared arena-backed representation:
    - `Environment { store: Arc<Mutex<EnvStore>>, current: EnvId }`
    - `EnvStore { layers: Vec<EnvLayer> }`
    - `EnvLayer { bindings, parent: Option<EnvId>, barrier }`
  - rewrote:
    - `lookup`
    - `lookup_local`
    - `lookup_until_barrier`
    - `push_scope`
    - `push_barrier_scope`
    - diagnostic name collection helpers
  - added `Environment::id()` for explicit stable layer identity
  - updated Rustdoc to reflect the new O(1) scope-push cost and the fact that
    Rust now already uses stable environment ids as preparation for the true
    closure model

- `crates/eval/tests/core_eval.rs`
  - added regression coverage asserting that:
    - child and barrier scopes allocate fresh stable ids,
    - parent bindings remain visible through normal lookup,
    - barrier lookup still stops before the parent binding

### `eval` / `pattern_matcher` / `tests` — Evaluate `case` patterns before matching and restore barrier semantics

Commit date: 2026-03-06

#### Context

After restoring parser-side definition normalization, two `case`-specific
semantic mismatches remained active:

- rule automata were still built from raw parser patterns instead of evaluated
  and simplified patterns,
- repeated pattern-variable checks still consulted the outer lexical
  environment, unlike the C++ `pushEnvBarrier` / `searchIdDef` behavior.

These two issues had to be fixed together because they both live in the
`evalCase` path and share the same rule-application semantics.

#### Changes

- `crates/eval/src/lib.rs`
  - added Rustdoc provenance for the C++ `evalRuleList` / `evalPattern` /
    `patternSimplification` path
  - introduced:
    - `eval_rule_list`
    - `eval_pattern_list`
    - `eval_pattern`
    - `pattern_simplification`
  - folded constant-only numeric pattern subgraphs before automaton
    construction, covering the minimal `(1+1)` mismatch
  - changed `apply_case_rules` to:
    - evaluate/simplify rule LHS patterns before building the automaton,
    - cache automata by evaluated rule-list `TreeId`,
    - initialise one barrier child environment per rule instead of a plain
      scope child

- `crates/eval/src/lib.rs` — `Environment`
  - added explicit barrier semantics to the environment model:
    - `barrier: bool`
    - `push_barrier_scope()`
    - `lookup_until_barrier()`
  - documented the direct C++ mapping to `pushEnvBarrier` / `searchIdDef`

- `crates/eval/src/pattern_matcher.rs`
  - updated automaton-cache documentation to reflect evaluated-rule keys
  - switched repeated variable checks from `lookup(...)` to
    `lookup_until_barrier(...)`

- `crates/eval/tests/core_eval.rs`
  - added regression tests for:
    - numeric pattern folding before matching,
    - pattern-variable lookup stopping at the barrier scope

- `tests/cpp_parity_known_gaps/README.md`
