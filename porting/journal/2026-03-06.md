### `eval` / `pattern_matcher` / `tests` — Evaluate `case` patterns before matching and restore barrier semantics

Commit date: 2026-03-06

#### Context

After restoring parser-side definition normalization, two `case`-specific
semantic mismatches remained active:

- rule automata were still built from raw parser patterns instead of evaluated
  and simplified patterns,
- repeated pattern-variable checks still consulted the outer lexical
  environment, unlike the C++ `pushEnvBarrier` / `searchIdDef` behavior.

These two issues had to be fixed together because they both live in the
`evalCase` path and share the same rule-application semantics.

#### Changes

- `crates/eval/src/lib.rs`
  - added Rustdoc provenance for the C++ `evalRuleList` / `evalPattern` /
    `patternSimplification` path
  - introduced:
    - `eval_rule_list`
    - `eval_pattern_list`
    - `eval_pattern`
    - `pattern_simplification`
  - folded constant-only numeric pattern subgraphs before automaton
    construction, covering the minimal `(1+1)` mismatch
  - changed `apply_case_rules` to:
    - evaluate/simplify rule LHS patterns before building the automaton,
    - cache automata by evaluated rule-list `TreeId`,
    - initialise one barrier child environment per rule instead of a plain
      scope child

- `crates/eval/src/lib.rs` — `Environment`
  - added explicit barrier semantics to the environment model:
    - `barrier: bool`
    - `push_barrier_scope()`
    - `lookup_until_barrier()`
  - documented the direct C++ mapping to `pushEnvBarrier` / `searchIdDef`

- `crates/eval/src/pattern_matcher.rs`
  - updated automaton-cache documentation to reflect evaluated-rule keys
  - switched repeated variable checks from `lookup(...)` to
    `lookup_until_barrier(...)`

- `crates/eval/tests/core_eval.rs`
  - added regression tests for:
    - numeric pattern folding before matching,
    - pattern-variable lookup stopping at the barrier scope

- `tests/cpp_parity_known_gaps/README.md`
  - updated `gap_03` and `gap_04` to mark them as fixed in the eval phase and
    ready for promotion into regular parity guardrails

#### Validation

- `cargo test -p eval --all-targets`
- `cargo test -p parser --all-targets`
- `cargo run -p compiler -- --dump-sig tests/cpp_parity_known_gaps/gap_03_case_pattern_constant_folding.dsp`
- `cargo run -p compiler -- --dump-sig tests/cpp_parity_known_gaps/gap_04_case_pattern_scope_barrier.dsp`

### `parser` / `eval` / `tests` — Restore patterned and multi-clause definition normalization

Commit date: 2026-03-06

#### Context

The first correction step from the parser/pattern/eval parity plan was to stop
treating Faust function definitions as identifier-only parameter lists. The
Rust parser still emitted raw repeated definitions and the evaluator then
rejected valid C++ programs as plain symbol redefinitions.

#### Changes

- `crates/parser/src/grammar/faustparser.y`
  - switched function-definition parsing from `ParamList` to full `ArgList`
  - kept `ParamList` only for lambda syntax, matching the C++ parser split

- `crates/parser/src/lib.rs`
  - ported the core `formatDefinitions` flow from C++:
    - group same-name raw definitions,
    - detect standard identifier arglists,
    - lower single standard definitions to nested `abstr`,
    - lower patterned or multi-clause definitions to `case`
  - documented the C++ provenance (`standardArgList`, `makeDefinition`,
    `formatDefinitions`)

- `crates/eval/src/lib.rs`
  - clarified that parser-originated definitions are expected to be
    pre-normalized before binding
  - retained the raw-args fallback only for direct test construction

- `crates/parser/tests/structural_cpp_differential.rs`
  - added parser differential coverage for:
    - `foo(0) = _; foo(x) = x; process = foo;`
    - `foo(x) = x; foo(y) = y; process = foo;`
  - asserted C++ acceptance, Rust parse success, single grouped `foo`
    definition, and `BOXCASE` normalization

- `tests/cpp_parity_known_gaps/README.md`
  - updated `gap_01` and `gap_02` status notes to reflect the new remaining
    blocker: `case` nodes now survive until `propagate`

#### Validation

- `cargo test -p parser --all-targets`
- `cargo test -p eval --all-targets`
- `cargo run -p compiler -- --dump-sig tests/cpp_parity_known_gaps/gap_01_pattern_def_constant_clause.dsp`
- `cargo run -p compiler -- --dump-sig tests/cpp_parity_known_gaps/gap_02_pattern_def_clause_grouping.dsp`

### `parser` / `eval` / `compiler` / `tests` / `porting` — Add minimal DSP corpus for current C++ parity gaps

Commit date: 2026-03-06

#### Context

Parser/pattern-matching/evaluation analysis identified a set of small Faust
programs that are accepted by the C++ reference compiler but still diverge in
the Rust port. Adding them directly under `tests/corpus/rep_*.dsp` would have
been wrong at this stage because:

- `crates/parser/tests/production_corpus.rs` currently requires every
  `rep_*.dsp` fixture to parse successfully in Rust,
- `xtask` corpus discovery scans `tests/corpus/*.dsp`,
- snapshotting the current Rust behavior would freeze known bugs instead of
  tracking them as parity work items.

#### Changes

- Added a dedicated excluded corpus under `tests/cpp_parity_known_gaps/` with
  four minimal reproducers:
  - `gap_01_pattern_def_constant_clause.dsp`
  - `gap_02_pattern_def_clause_grouping.dsp`
  - `gap_03_case_pattern_constant_folding.dsp`
  - `gap_04_case_pattern_scope_barrier.dsp`

- Documented the purpose, current C++/Rust status, and manual repro commands in:
  - `tests/cpp_parity_known_gaps/README.md`

- Recorded the semantic analysis and closure order in:
  - `porting/parser-pattern-eval-cpp-parity-gap-analysis-2026-03-06-en.md`
