### `porting` / `eval` — Reframe evaluator parity work around the true captured-closure model

Commit date: 2026-03-06

Source commit: pending (pre-commit planning update in this session)

#### Context

The first March 6 parity fixes restored several concrete mismatches in parser
definitions, `case`, residual `a2sb` lowering, and modulation. That left a
clear next architectural question for `eval`: keep extending the eager
environment-threading adaptation, or port the C++ closure model itself.

The C++ reference still relies on captured closures stored in environment
layers, plus `copyEnvReplaceDefs(...)` / `updateClosures(...)` for
environment-rewrite semantics. The Rust evaluator does not yet implement that
contract and still documents the current representation too strongly as if it
were already closure-equivalent.

#### Changes

- added [porting/eval-true-closure-model-port-plan-2026-03-06-en.md](../eval-true-closure-model-port-plan-2026-03-06-en.md)
  as the new architecture note and staged plan for porting:
  - explicit evaluator values,
  - stable `EnvId` environment identity,
  - captured closure forcing,
  - `copyEnvReplaceDefs(...)` parity,
  - closure-aware pattern-matcher capture,
  - cleanup of the current adapted `a2sb` path
- updated
  [porting/parser-pattern-eval-cpp-parity-gap-analysis-2026-03-06-en.md](../parser-pattern-eval-cpp-parity-gap-analysis-2026-03-06-en.md)
  with a status note pointing to the closure-model follow-up
- revised Rustdoc in `crates/eval/src/lib.rs` so the current evaluator is
  described as an `adapted` model rather than a full closure-parity endpoint

#### Validation

- no code-generation or behavior change intended
- no tests rerun for this documentation-only planning update

### `boxes` / `parser` / `eval` / `propagate` / `compiler` / `tests` — Lower residual closures through `a2sb`, restore modulation semantics, and prepare grouped-definition patterns

Commit date: 2026-03-06

Source commit: pending (pre-commit worktree for the next parity commit in this session)

#### Context

After restoring grouped definitions and `case` evaluation semantics, the Rust
pipeline still diverged from the C++ compiler in three places that mattered in
practice:

- residual `abstr` / `case` nodes were reaching `propagate`,
- `modulation` forms still failed in propagation instead of being rewritten
  during evaluation,
- grouped patterned definitions still built raw `case` rules without
  `prepare_pattern`, so variable clauses were not wrapped as `BOXPATVAR`.

Those gaps blocked the remaining Phase-4 parity fixtures (`rep_13`, `rep_16`,
`rep_24`, `rep_27`, `rep_32`, `rep_33`) and kept the minimal C++ parity DSPs in
`tests/cpp_parity_known_gaps/` out of the regular guardrails.

#### Changes

- `crates/boxes/src/lib.rs`
  - added `boxSlot` / `boxSymbolic` equivalents to `BoxBuilder`
  - extended `BoxMatch` and `match_box(...)` with:
    - `Slot(i32)`
    - `Symbolic(BoxId, BoxId)`
  - documented the direct C++ provenance in Rustdoc

- `crates/eval/src/lib.rs`
  - added an adapted Rust `a2sb(...)` pass, with provenance comments pointing
    to C++ `a2sb` / `real_a2sb`
  - threaded a local slot-id counter through `LoopDetector`, replacing the C++
    `gBoxSlotNumber` global
  - changed `eval_process_with_stats(...)` to run `a2sb` after evaluation so
    `propagate` no longer receives raw `abstr` or `case` nodes
  - ported `modulation` evaluation semantics from C++:
    - evaluate the target label and modulation circuit,
    - check arity (`<= 2` inputs, exactly `1` output),
    - evaluate and `a2sb`-lower the body,
    - implant the modulation circuit into matching widgets/groups,
    - wrap the rewritten body in `symbolic(slot, ...)` only when needed
  - added local Rustdoc for the adapted modulation rewrite path and the label
    matching subset currently supported
  - extended local arity inference with `Slot` / `Symbolic`

- `crates/propagate/src/lib.rs`
  - added `Slot` / `Symbolic` arity handling
  - introduced a local slot environment in propagation so symbolic boxes bind
    their first input like the C++ `slotenv`
  - kept the public `propagate(...)` API stable while documenting the adapted
    internal representation in Rustdoc

- `crates/parser/src/lib.rs`
  - fixed grouped patterned definitions to use `node_case_checked(...)` instead
    of building raw `BOXCASE` rules directly
  - this restores the C++ `prepareRule(s)` behavior for multi-clause definition
    families, so identifier clauses become `BOXPATVAR`

- `crates/eval/tests/core_eval.rs`
  - added regression coverage for:
    - residual abstraction lowering to symbolic boxes,
    - residual case lowering to symbolic boxes,
    - modulation with no matching widget,
    - modulation with a matching slider and default `mul` implantation

- `crates/propagate/tests/core_api.rs`
  - added explicit propagation coverage for raw slots and nested symbolic boxes

- `crates/parser/tests/structural_cpp_differential.rs`
  - tightened grouped-definition checks to assert that variable clauses are
    wrapped as `BOXPATVAR`

- `crates/compiler/tests/signal_pipeline.rs`
  - added integration coverage for `rep_32_modulation_single.dsp` and
    `rep_33_modulation_chain.dsp`

- `crates/compiler/tests/diagnostic_errors.rs`
  - updated former propagate-failure fixtures for compound / under-applied case
    paths because they now lower successfully to signals

- `crates/compiler/src/main.rs`
  - updated CLI snapshot tests that had been asserting the old propagate-stage
    failure for the compound case fixture

- `tests/cpp_parity_known_gaps/README.md`
  - updated `gap_01` and `gap_02` to mark them fixed
  - the known-gap folder now contains only fixed reproducers waiting for
    promotion into standard parity guardrails

### `eval` / `pattern_matcher` / `tests` — Evaluate `case` patterns before matching and restore barrier semantics

Commit date: 2026-03-06

#### Context

After restoring parser-side definition normalization, two `case`-specific
semantic mismatches remained active:

- rule automata were still built from raw parser patterns instead of evaluated
  and simplified patterns,
- repeated pattern-variable checks still consulted the outer lexical
  environment, unlike the C++ `pushEnvBarrier` / `searchIdDef` behavior.

These two issues had to be fixed together because they both live in the
`evalCase` path and share the same rule-application semantics.

#### Changes

- `crates/eval/src/lib.rs`
  - added Rustdoc provenance for the C++ `evalRuleList` / `evalPattern` /
    `patternSimplification` path
  - introduced:
    - `eval_rule_list`
    - `eval_pattern_list`
    - `eval_pattern`
    - `pattern_simplification`
  - folded constant-only numeric pattern subgraphs before automaton
    construction, covering the minimal `(1+1)` mismatch
  - changed `apply_case_rules` to:
    - evaluate/simplify rule LHS patterns before building the automaton,
    - cache automata by evaluated rule-list `TreeId`,
    - initialise one barrier child environment per rule instead of a plain
      scope child

- `crates/eval/src/lib.rs` — `Environment`
  - added explicit barrier semantics to the environment model:
    - `barrier: bool`
    - `push_barrier_scope()`
    - `lookup_until_barrier()`
  - documented the direct C++ mapping to `pushEnvBarrier` / `searchIdDef`

- `crates/eval/src/pattern_matcher.rs`
  - updated automaton-cache documentation to reflect evaluated-rule keys
  - switched repeated variable checks from `lookup(...)` to
    `lookup_until_barrier(...)`

- `crates/eval/tests/core_eval.rs`
  - added regression tests for:
    - numeric pattern folding before matching,
    - pattern-variable lookup stopping at the barrier scope

- `tests/cpp_parity_known_gaps/README.md`
  - updated `gap_03` and `gap_04` to mark them as fixed in the eval phase and
    ready for promotion into regular parity guardrails

### `parser` / `eval` / `tests` — Restore patterned and multi-clause definition normalization

Commit date: 2026-03-06

#### Context

The first correction step from the parser/pattern/eval parity plan was to stop
treating Faust function definitions as identifier-only parameter lists. The
Rust parser still emitted raw repeated definitions and the evaluator then
rejected valid C++ programs as plain symbol redefinitions.

#### Changes

- `crates/parser/src/grammar/faustparser.y`
  - switched function-definition parsing from `ParamList` to full `ArgList`
  - kept `ParamList` only for lambda syntax, matching the C++ parser split

- `crates/parser/src/lib.rs`
  - ported the core `formatDefinitions` flow from C++:
    - group same-name raw definitions,
    - detect standard identifier arglists,
    - lower single standard definitions to nested `abstr`,
    - lower patterned or multi-clause definitions to `case`
  - documented the C++ provenance (`standardArgList`, `makeDefinition`,
    `formatDefinitions`)

- `crates/eval/src/lib.rs`
  - clarified that parser-originated definitions are expected to be
    pre-normalized before binding
  - retained the raw-args fallback only for direct test construction

- `crates/parser/tests/structural_cpp_differential.rs`
  - added parser differential coverage for:
    - `foo(0) = _; foo(x) = x; process = foo;`
    - `foo(x) = x; foo(y) = y; process = foo;`
  - asserted C++ acceptance, Rust parse success, single grouped `foo`
    definition, and `BOXCASE` normalization

- `tests/cpp_parity_known_gaps/README.md`
  - updated `gap_01` and `gap_02` status notes to reflect the new remaining
    blocker: `case` nodes now survive until `propagate`
