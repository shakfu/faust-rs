# 2026-06-14

## Port the C++ impulse-response test harness into faust-rs

Commit date: 2026-06-14
Source commit: this commit

Ported the C++ Faust `tests/impulse-tests` mechanism into faust-rs as a
Makefile-driven, backend-comparison test suite under `tests/impulse-tests/`,
plus a new `crates/impulse-runner` binary. Design and full results are in
`porting/impulse-tests-harness-port-plan-2026-06-14-en.md`.

Design (confirmed with the maintainer):

- **Reference oracle = genuine native C++, full 4-pass protocol** (impulse +
  random-split + polyphonic-4 + polyphonic-1 = 60000 frames at 44100 / block 64
  / `-double`), produced by the reference C++ Faust compiler wrapped in the
  original `impulsearch.cpp` / `controlTools.h`. `Make.ref`-generated files are
  byte-identical to the committed C++ `reference/`. The oracle and the native
  C/C++ paths legitimately depend on a C++ Faust checkout (architecture headers
  + impulse arch); the fragile headers are referenced in place via overridable
  variables in `common.mk` rather than vendored.
- **C / C++ backends** are wrapped in the *same* arch. faust-rs's generated
  class implements the standard Faust `dsp` interface (`clone()`,
  `buildUserInterface()`, `compute()`…), so they run the full polyphonic
  protocol and are compared to the reference exactly (no `-part`). The C backend
  uses `impulsearch2.cpp`; faust-rs `-i` is dropped so `c++ -I$(FAUST_ARCH)`
  resolves the faust headers.
- **Interpreter backend** has no polyphonic / MIDI / soundfile runtime, so
  `impulse-runner` reproduces only the scalar impulse pass (15000 frames) and is
  compared with `filesCompare -part` — the prefix approach the C++ suite already
  uses for its Rust target. The runner compiles via the faust-rs library to
  interp bytecode (`generate_interp_module::<f64>`), drives `button`/`checkbox`
  zones (discovered through `ui_instructions()`, written with `set_real_zone`)
  high during block 0 then low, and emits the exact reference text format with
  the same `normalize()` zero-clamp.

Curation (`known.mk` + `KNOWN_FAILURES.md`): every mismatch was classified by
its *max* delta over the whole run. Genuinely bounded rounding bands get a
per-DSP `PRECISION_<dsp>` tolerance; real divergences are excluded with a
documented cause via `KNOWN_FAIL_<backend>`. The makefiles `filter-out` the
excluded names and apply the per-DSP tolerance, so the aggregate targets are
green gates. `tools/filesCompare.cpp` was patched (the upstream tool ignored a
tolerance when combined with `-part`, only honoring `argv[3]` at `argc==4`).

Baseline over the 93 DSPs (`-double`, default tolerance 2e-6): references 93/93;
raw match cpp 92, c 87, interp 74; one shared compile-fail (`subcontainer1`,
sub-container codegen gap). Because the C++ backend reproduces the full
60000-frame reference exactly on 92/93, the remaining mismatches are
backend-specific divergences the suite pinpoints (interp delay lines in
`comb_delay*`, a few math/UI primitives, recursive-filter drift; C `grain3`).
Curated green gates: `make cpp` 92, `make c` 91, `make interp` 78.

Generated `reference/`, `ir/`, `build/` and the compiled `filesCompare` are
gitignored. Next phases (in the plan): cranelift and wasm/wast runners, an
`xtask` entry point for toolchain-free CI, and a pure-Rust comparator.
