### `codegen` / `compiler` / `box-ffi` / `cranelift-ffi` / `xtask` — Propagate `--double` to Interp serialization; Cranelift already correct

Commit date: 2026-03-07

Source commit: pending

#### Context

After adding the `--double` flag and threading `RealType` through the
transform fast lane, the interpreter backend still emitted
`interpreter_dsp_factory float` even when `--double` was requested.

  - `serialize_factory` is now generic:
    `fn serialize_factory<R: FbcReal>(factory: &FbcDspFactory<R>) -> Result<String, String>`
  - Both `lower_signals_to_interp_legacy_bridge` and
    `lower_signals_to_interp_transform_fastlane` now accept `real_type: RealType`
    and dispatch: `Float32` → `FbcDspFactory<f32>`, `Float64` → `FbcDspFactory<f64>`
  - `lower_signals_to_interp_legacy_bridge` is now also passed `real_type`

  - `--dump-fir-verify` branch gains `.with_real_type(selected_real_type(&cli))`

### `eval` / `transform` / `compiler` / `box-ffi` — Introduce parametric `real_type` for internal DSP arithmetic; separate from `FAUSTFLOAT` external interface

Commit date: 2026-03-07

Source commit: pending

#### Context

The transform fast lane previously used `FirType::FaustFloat` for every
floating-point value — both internal arithmetic nodes (accumulators, math
calls, tables, delay-state slots) and the external DSP interface (audio
buffers, UI zones).  This conflated two orthogonal concerns and made it
impossible to run internal computation in `double` while keeping the external
interface at `FAUSTFLOAT`.

The reference Faust compiler's `-double` option performs exactly this
separation: internal DSP nodes use `double`, but `compute(...)` argument
arrays and UI zone pointers remain at the declared `FAUSTFLOAT` width.

#### Changes

  - added `SamplePrecision` public enum (`Float32` / `Float64`, default `Float32`)
  - added `pub sample_precision: SamplePrecision` field to `EvalSourceContext`

    it in `for_loaded_file`

- `crates/transform/src/signal_fir/mod.rs`
  - added `RealType` public enum (`Float32` / `Float64`, default `Float32`)
    with `as_fir_type() -> FirType` converter
  - added `pub real_type: RealType` field to `SignalFirOptions`
  - updated `compile_signals_to_fir_fastlane` to pass `options.real_type.as_fir_type()`
    to `build_module`
  - added `unwrap_output_cast` test helper to strip the mandatory
    `Cast(FaustFloat, ...)` wrapper on output-buffer stores before asserting
    on the inner computation node

- `crates/transform/src/signal_fir/module.rs` (core refactor)
  - `build_module` now accepts `real_ty: FirType` and stores it in

  - **Internal nodes → `real_ty`**: float constants (`float_const`), cast

    - output stores: `Cast(FaustFloat, <computation in real_ty>)`

### `parser` / `tests` — Fix scientific notation float parsing; validate precision qualifiers end-to-end

Commit date: 2026-03-07

Source commit: pending

#### Context

All scientific-notation float forms (`1.5e-3`, `1.5e+3`, `1.5e3`, `.75e-2`, `1.192092896e-07`)
failed to parse because the lrlex `FLOAT` rule listed shorter non-exponent alternatives
**before** the exponent alternatives.  lrlex uses leftmost-first alternation (not
longest-match), so `[0-9]+\.[0-9]*f?` consumed `1.192092896` and left `e-07` to trigger
a parse error.

This was the actual blocker for `singleprecision`/`doubleprecision` precision qualifiers:
all precision-qualified infrastructure (tokens, grammar rules, `accept_definition()`,
`prepend_statement_with_variant()`) was already in place; the lexer was just failing on
the floating-point constants that appear in every standard library.

#### Changes

- `crates/parser/src/grammar/faustlexer.l`
  - reordered `FLOAT` rule alternatives so all exponent forms come first:
    `[0-9]+\.[0-9]*e[-+]?[0-9]+f?` | `[0-9]+e[-+]?[0-9]+f?` | `\.[0-9]+e[-+]?[0-9]+f?`
    then non-exponent forms:
    `[0-9]+\.[0-9]*f?` | `\.[0-9]+f?` | `[0-9]+f`

- `tests/corpus/rep_54_scientific_notation_float.dsp`
  - new corpus fixture covering: negative/positive/bare exponents, dot-leading floats,
    and all four precision qualifier keywords

- `tests/golden/rust/rep_54_scientific_notation_float/compiler_stdout.txt`
  - generated golden snapshot via `cargo run -p xtask -- golden-gen-rust`

#### Validation

### `parser` — Fix `-I` vs `local_dir` ordering in `resolve_import_from` to match C++

Commit date: 2026-03-07

Source commit: pending

#### Context

In C++, `-I` paths are inserted at the **front** of `gImportDirList` via
`insert(begin())`, so they always take precedence over the master directory
and any directories added dynamically by `fopenSearch`. The importing file's
own directory is only appended to `gImportDirList` at the **back**, after the
file is successfully opened.

The Rust `resolve_import_from` placed `local_dir` (the importing file's
parent) **before** `search_paths`, reversing this contract: a `-I` override
containing a file with the same name as a sibling of a library would be
silently ignored for nested imports.

#### Changes

- `crates/parser/src/source_reader.rs`
  - `resolve_import_from`: moved `search_paths` candidates before `local_dir`
    candidate (with deduplication), matching C++ `gImportDirList` ordering
  - added Rustdoc comment explaining the C++ ordering rationale
  - added unit test `search_paths_take_precedence_over_local_dir_matching_cpp_import_order`
    that creates two real on-disk directories and asserts the override wins

#### Validation

- `cargo test --workspace --all-targets` — 0 failures

### `compiler` / `interp-ffi` / `cranelift-ffi` — Port C++ library import search path model to Rust

Commit date: 2026-03-07

Source commit: pending

#### Context

The Rust compiler previously resolved imports using only the parent directory
of the source file (`default_search_base`), matching none of the C++ library
discovery logic. As a result, any `.dsp` that `import`s a standard library
(e.g. `stdfaust.lib`) failed to compile in Rust even though the libraries
were present on the system.

The C++ frontend searches libraries in a well-defined order:

1. Caller-supplied `-I` / `--import-dir` paths
2. The source file's parent directory
3. `FAUST_LIB_PATH` environment variable
4. Executable-relative `share/faust/`
5. `/usr/local/share/faust`
6. `/usr/share/faust`

#### Changes

- `crates/compiler/src/lib.rs`
  - added `build_import_search_paths(path, extra, faust_lib_path, exe)` — core
    builder that assembles the ordered, deduplicated search list matching the
    C++ model
  - added `default_import_search_paths(path) -> Vec<PathBuf>` — public helper
    that reads `FAUST_LIB_PATH` and `current_exe()` from the environment
  - added `merge_import_search_paths(path, extra)` — used internally by all
    `compile_file*` entry points to prepend caller paths before the defaults
  - removed `default_search_base(path) -> PathBuf` (single-path fallback)
  - all `compile_file_default*` and equivalent fast-lane entry points now pass
    `&[]` as extra paths and let `merge_import_search_paths` build the full list
  - added 5 unit tests covering ordering, deduplication, bare-filename fallback,
    and the public helper
  - updated rustdoc to document the C++ parity intent

- `crates/interp-ffi/src/factory.rs`
  - updated import and call site to use `default_import_search_paths`

- `crates/cranelift-ffi/src/factory.rs`
  - same update as `interp-ffi`

#### Validation

- `cargo build` — clean build, no errors
