## fix(cranelift): guard oversized AArch64 compute bodies

Commit date: 2026-04-30

Source commit: this commit

Reproduced the `fad_tracking3.dsp` Cranelift backend crash through the FFI
runtime path:

- `cargo run -q -p cranelift-ffi --example compute_bench -- fad_tracking3.dsp`
  exited through a native `EXC_BAD_INSTRUCTION` before this fix.
- `lldb` showed the crash inside the finalized JIT `compute` body called from
  `computeCCraneliftDSPInstance`, at a Cranelift-emitted `udf` padding/data
  area rather than in the Rust/C FFI wrapper.

Added an AArch64-only guard for oversized monolithic `compute` bodies. When
the first lowering succeeds but the generated `compute` CLIF exceeds the
current conservative safety threshold, `generate_cranelift_module` drops that
JIT module and retries with the existing no-op stub fallback. This keeps hosts
from aborting while the Cranelift backend still lacks function splitting or a
proven-safe lowering strategy for these large FAD/delay-heavy functions.

`fad_tracking3.dsp` now reports `compute_body_lowered=false` and the FFI
`compute_bench` reproducer completes without crashing. This is a controlled
fallback, not full Cranelift runtime parity for that DSP.

## feat(eval): make evaluator depth budgets configurable

Commit date: 2026-04-30

Source commit: this commit

Added process environment overrides for the evaluator recursion budgets:

- `FAUST_RS_DEFAULT_EVAL_MAX_DEPTH` overrides the default
  identity-tracked evaluator recursion budget used by `LoopDetector::new`
  and `LoopDetector::with_cancel`.
- `FAUST_RS_STRUCTURAL_HARD_MAX_DEPTH` overrides the structural lowering
  cap used by `a2sb` / `a2sb_value`.

Both variables are read when a `LoopDetector` is constructed so one
evaluation pass keeps stable recursion semantics. Missing, invalid, or
zero values fall back to the compiled defaults (`1_024` and `4_096`).
`LoopDetector::with_max_depth(...)` remains an explicit caller override for
the identity-tracked evaluator budget, while structural lowering is still
clamped by the configured structural cap.

Raising `FAUST_RS_STRUCTURAL_HARD_MAX_DEPTH` above the default is an
explicit opt-in: previous validation showed that `16_384` can still overflow
the real Rust/OS stack before the logical `RecursionDepthExceeded` guard on
the recursive `case` diagnostic fixture.

- `cargo clippy -p eval --all-targets -- -D warnings`

- `cargo test`

## docs(eval): clarify evaluator depth caps

Commit date: 2026-04-30

Source commit: this commit

Renamed the evaluator recursion constants to separate the two guarded
depth notions:

- `DEFAULT_EVAL_MAX_DEPTH` is the default budget for identity-tracked
  evaluator recursion. It applies to `LoopDetector::call_stack`, where
  frames carry stable tree/environment identities and can therefore
  detect direct cycles as well as excessive acyclic recursion.
- `STRUCTURAL_HARD_MAX_DEPTH` is the hard cap for structural lowering
  recursion in `a2sb` / `a2sb_value`. That path cannot rely on identity
  cycle detection because fresh symbolic slots may be allocated while
  descending, so it keeps a separate counter and clamps even explicit
  `with_max_depth(...)` requests.

The default evaluator budget is `1_024`. The structural hard cap is
`4_096`. They remain distinct so a future caller can raise the
identity-tracked evaluator budget for a known deep acyclic program
without implicitly raising the structural-lowering stack risk. The CLI
stack comment now documents that the spawned 64 MiB compiler thread is
the stack contract for these guarded budgets; library embedders need
comparable stack headroom or a lower evaluator depth budget.

Added unit tests that enter structural lowering exactly
the default budget and assert that the next entry returns
`RecursionDepthExceeded`. A second test raises the general evaluator
budget above `STRUCTURAL_HARD_MAX_DEPTH` and verifies that structural
lowering still clamps at the structural hard cap.

`cargo fmt --all` also formatted existing Rust files touched by recent RAD
work; those formatting-only diffs are included in this commit.

### Validation

- `cargo fmt --all`
- `cargo test -p eval structural_depth_`
- `cargo test -p compiler --test diagnostic_errors diverging_recursive_case_reports_eval_error_instead_of_aborting`
- `cargo clippy -p eval -p compiler --all-targets -- -D warnings`
- `cargo build -p compiler --release`
- `cargo test`

The attempted `16_384` evaluator budget is not safe for the recursive
`case` diagnostic fixture: in debug tests, the process reaches a real OS
stack overflow before `RecursionDepthExceeded`. Restoring the default
identity-tracked evaluator budget to `1_024` makes the fixture return the
controlled `stack overflow in eval` diagnostic again.
