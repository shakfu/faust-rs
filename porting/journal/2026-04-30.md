# 2026-04-30

## docs(eval): clarify evaluator depth caps

Commit date: 2026-04-30

Source commit: this commit

Renamed the evaluator recursion constants to separate the two guarded
depth notions:

- `DEFAULT_EVAL_MAX_DEPTH` is the default budget for identity-tracked
  evaluator recursion. It applies to `LoopDetector::call_stack`, where
  frames carry stable tree/environment identities and can therefore
  detect direct cycles as well as excessive acyclic recursion.
- `STRUCTURAL_HARD_MAX_DEPTH` is the hard cap for structural lowering
  recursion in `a2sb` / `a2sb_value`. That path cannot rely on identity
  cycle detection because fresh symbolic slots may be allocated while
  descending, so it keeps a separate counter and clamps even explicit
  `with_max_depth(...)` requests.

The default evaluator budget is `1_024`. The structural hard cap is
`4_096`. They remain distinct so a future caller can raise the
identity-tracked evaluator budget for a known deep acyclic program
without implicitly raising the structural-lowering stack risk. The CLI
stack comment now documents that the spawned 64 MiB compiler thread is
the stack contract for these guarded budgets; library embedders need
comparable stack headroom or a lower evaluator depth budget.

Added unit tests that enter structural lowering exactly
the default budget and assert that the next entry returns
`RecursionDepthExceeded`. A second test raises the general evaluator
budget above `STRUCTURAL_HARD_MAX_DEPTH` and verifies that structural
lowering still clamps at the structural hard cap.

`cargo fmt --all` also formatted existing Rust files touched by recent RAD
work; those formatting-only diffs are included in this commit.

### Validation

- `cargo fmt --all`
- `cargo test -p eval structural_depth_`
- `cargo test -p compiler --test diagnostic_errors diverging_recursive_case_reports_eval_error_instead_of_aborting`
- `cargo clippy -p eval -p compiler --all-targets -- -D warnings`
- `cargo build -p compiler --release`
- `cargo test`

The attempted `16_384` evaluator budget is not safe for the recursive
`case` diagnostic fixture: in debug tests, the process reaches a real OS
stack overflow before `RecursionDepthExceeded`. Restoring the default
identity-tracked evaluator budget to `1_024` makes the fixture return the
controlled `stack overflow in eval` diagnostic again.
