## docs(eval): clarify evaluator depth caps

Commit date: 2026-04-30

Source commit: this commit

Renamed the evaluator recursion constants to separate the two guarded
depth notions:

- `DEFAULT_EVAL_MAX_DEPTH` is the default budget for identity-tracked
  evaluator recursion. It applies to `LoopDetector::call_stack`, where
  frames carry stable tree/environment identities and can therefore
  detect direct cycles as well as excessive acyclic recursion.
- `STRUCTURAL_HARD_MAX_DEPTH` is the hard cap for structural lowering
  recursion in `a2sb` / `a2sb_value`. That path cannot rely on identity
  cycle detection because fresh symbolic slots may be allocated while
  descending, so it keeps a separate counter and clamps even explicit
  `with_max_depth(...)` requests.

Both caps are currently set to `16_384`, but they remain distinct so a
future caller can raise the identity-tracked evaluator budget without
implicitly raising the structural-lowering stack risk. The CLI stack
comment now documents that the spawned 64 MiB compiler thread is the
stack contract for this structural cap; library embedders need comparable
stack headroom or a lower evaluator depth budget.

Added unit tests that enter structural lowering exactly
`STRUCTURAL_HARD_MAX_DEPTH` times and assert that the next entry returns
`RecursionDepthExceeded`, including the case where `with_max_depth(...)`
requests a higher general evaluator budget.

`cargo fmt --all` also formatted existing Rust files touched by recent RAD
work; those formatting-only diffs are included in this commit.
