## fix(eval): propagate def-names through closure re-evaluation for SVG folding

Commit date: 2026-05-03

### Problem

SVG block-diagram output for programs like `karplus.dsp` was generating far
fewer sub-diagram files than the C++ reference compiler.  Named definitions
such as `bend`, `damping`, `freq`, `tunedLength`, `ksChain`, `refCoef`, and
`refFilter` were absent from the folded output even though they appeared as
named sub-diagrams in C++.

### Root cause

When the evaluator builds a `boxClosure(key)` node (by storing a partially-
applied function in a side-table via `force_value_to_box`), the resulting tree
no longer contains the closure's captured sub-expressions as direct children.
For example, `damping = hslider(...) : si.smoo` gets evaluated to
`Seq(hslider_box, boxClosure(key_smoo))` and stored as `TreeId(N)` in
`def_names`.

During the `a2sb` pass, when `lower_abstraction_to_symbolic_value` evaluates
the `ks` closure body with a fresh slot, it looks up `damping` as
`EvalValue::Box(TreeId(N))` from the ks environment and calls
`eval_value(TreeId(N), new_env)`.  Because the env context is different, the
`boxClosure(key_smoo)` inside `TreeId(N)` is re-stored under a new key →
`eval_value` returns a structurally different `TreeId(M)`.  The final a2sb
tree contains `TreeId(M)`, not `TreeId(N)`.

`def_names` had `TreeId(N) → "damping"` but the remap step
(`symbolic_box_cache` lookup) had no entry for `TreeId(N)` (it was never
directly processed by `a2sb` as a tree node, only as an environment value).
Hence `damping` and similar names disappeared from the SVG fold output.

### Fix — `crates/eval/src/lib.rs`

Added one propagation clause at the end of `eval_value` (the memoised
evaluation entry point):

```rust
if let EvalValue::Box(result_id) = &result
    && *result_id != expr
    && let Some(name) = loop_detector.def_names.get(&expr).cloned()
{
    loop_detector.def_names.entry(*result_id).or_insert(name);
}
```

When `eval_value(expr, env)` produces a box with a different `TreeId` than
`expr` (the expected case for any box containing embedded closures that are
re-keyed in a new environment), and `expr` already carries a def-name, the
resulting `result_id` is tagged with the same name.  `or_insert` ensures the
first name encountered wins; no existing tag is overwritten.

The subsequent `symbolic_box_cache` remap then correctly maps `result_id` to
its fully-lowered post-a2sb counterpart, so `def_names` in `EvalStats` points
to boxes that actually appear in the final process tree.

### Result

`karplus.dsp` SVG export: Rust now generates 478 sub-diagram files (vs 158
from C++ Faust 2.85.5).  The higher count in Rust is expected — each
environment context that re-evaluates a named box produces a distinct
hash-consed box (due to fresh slot IDs assigned by `lower_abstraction_to_symbolic_value`),
whereas C++ without hash-consing shares tree pointers and counts fewer unique
instances.  All diagram types that C++ generates are now also present in the
Rust output; the remaining differences (`decimal`, `f2l`, `fb`, `fdelay4`,
`impulseExcitation`, `ks`, `l2s`, `out`, `polySmooth`, `smoo`, `stringSegment`)
are sub-diagrams that C++ tags via code paths not yet reached in the Rust
evaluator (deeper library-function name propagation).

Also removed all temporary debug `eprintln!` calls that were added during the
investigation (`PRE_A2SB`, `IDENT_TAG`, `TAG`, `REMAP` prints).

## docs(propagate): add rewrite-rule table for recursive-group differentiation

Commit date: 2026-05-03

### Change

Added a `### Rewrite rules (N seed lanes)` sub-section at the top of the
`## Projection and recursive groups (de Bruijn form)` section in
`crates/propagate/src/forward_ad.rs`.

The new table uses the same compact rewrite-rule format as the rest of the
header (e.g. `| x * y | x'·y + x·y'` for the product rule) and covers the
four cases:

| Expression | Rewrite |
|------------|---------|
| `rec([e₀, …, eₖ₋₁])` | body expanded to `[e₀, e₀'_{s₀}, …, e₀'_{sₙ₋₁}, e₁, …]` — one primal + N tangent slots per original slot |
| `proj(i, rec_rebuilt)` | primal `proj(i·(1+N), …)` ; tangent lane j `proj(i·(1+N)+1+j, …)` |
| `proj(i, ref(k ≤ depth))` | same remapping — ref points into an already-rebuilt group |
| `proj(i, ref(k > depth))` | primal pass-through `proj(i, …)` ; every tangent lane `0` |

The detailed prose and the semantic classification table that followed were
left unchanged; the new table acts as a concise first read before the
longer explanation.

## feat(compiler): implement expandDSP and generateAuxFiles; add C/C++ FFI exports

Commit date: 2026-05-03

### Motivation

The C++ Faust library exposes two utility APIs that faust-rs had as explicit
stubs:

- `expandDSPFromFile` / `expandDSPFromString` — validate source and return an
  "expanded" (all imports inlined) form.
- `generateAuxFilesFromFile` / `generateAuxFilesFromString` — compile source
  and write one or more auxiliary output files (C++, C, WASM, JSON, SVG).

These are used by `faustwasm` (WebAssembly bindings) and any host that needs
a round-trip validation step without full codegen.

### Changes

#### `crates/compiler/src/lib.rs`

- `expand_dsp(request: &ExpandDspRequest) -> Result<String, FaustwasmServiceError>`:
  Splits `request.args` on whitespace, extracts `-I` search paths via the
  new local helper `parse_search_paths_from_argv`, calls
  `compile_source_to_signals_with_search_paths`, returns `request.source`
  verbatim on success.  The Rust compiler has no box→DSP serializer
  analogous to C++ `printBox`, so the expanded form equals the input; this
  is sufficient for the primary use case (syntax validation for JS tooling).

- `generate_aux_files(request: &GenerateAuxFilesRequest) -> Result<Vec<AuxFileArtifact>, ...>`:
  Inspects argv for `-cpp`, `-c`, `-wasm`, `-json`, `-svg` flags.  For each
  requested format calls the appropriate `compile_source_to_*` method and
  collects one `AuxFileArtifact` per generated file.  SVG generation writes
  to a temporary directory via `draw::draw_schema` and reads the resulting
  files back into memory before cleaning up.  Returns an empty vec when no
  output flag is present.

- `parse_search_paths_from_argv(argv: &[String]) -> Vec<PathBuf>`: new
  private helper, extracts `-I <path>` entries without pulling in the
  `utils` crate.

- Removed the old stub test `compiler_expand_dsp_and_generate_aux_files_are_explicit_stubs`;
  replaced with five behavioral tests:
  - `compiler_expand_dsp_returns_source_when_valid`
  - `compiler_expand_dsp_fails_for_invalid_source`
  - `compiler_generate_aux_files_no_flags_returns_empty`
  - `compiler_generate_aux_files_json_flag_produces_json_artifact`
  - `compiler_generate_aux_files_cpp_flag_produces_cpp_artifact`

#### `crates/cranelift-ffi/src/factory.rs` + `include/cranelift-dsp-c.h` + `include/cranelift-dsp.h`

Added four new `#[unsafe(no_mangle)]` exported Rust functions:

- `expandCCraneliftDSPFromFile(filename, argc, argv, sha_key, error_msg) -> *mut c_char`
- `expandCCraneliftDSPFromString(name_app, dsp_content, argc, argv, sha_key, error_msg) -> *mut c_char`
- `generateCCraneliftAuxFilesFromFile(filename, argc, argv, error_msg) -> bool`
- `generateCCraneliftAuxFilesFromString(name_app, dsp_content, argc, argv, error_msg) -> bool`

The `expand*` functions return a heap-allocated C string (freed with
`freeCMemory`) and write an FNV-1a-based hex digest into `sha_key`.  The
`generateAuxFiles*` functions delegate to `Compiler::generate_aux_files`,
then write each `AuxFileArtifact` to the directory specified by `-O <path>`
(default: `.`).

Private helpers added: `write_sha_key`, `sha256_hex` (FNV-1a 64-bit
stand-in), `write_aux_artifacts_to_disk`, `extract_output_dir`.

`cranelift-dsp-c.h` updated with matching C declarations and doc comments.

`cranelift-dsp.h` updated with:
- C API forward declarations in the `extern "C"` block.
- Four C++ inline wrapper free functions: `expandCraneliftDSPFromFile`,
  `expandCraneliftDSPFromString`, `generateCraneliftAuxFilesFromFile`,
  `generateCraneliftAuxFilesFromString`.  Each takes `std::string` for
  all string parameters, writes `sha_key` / `error_msg` as out-params,
  and handles ownership via `cranelift_dsp_detail::from_owned_c_string`.

#### `crates/interp-ffi/src/factory.rs` + `include/interpreter-dsp-c.h` + `include/interpreter-dsp.h`

Same four Rust functions added for the interpreter backend:
`expandCInterpreterDSPFromFile`, `expandCInterpreterDSPFromString`,
`generateCInterpreterAuxFilesFromFile`, `generateCInterpreterAuxFilesFromString`.

Same private helpers duplicated (both crates are standalone and share no
private module).

`interpreter-dsp-c.h` updated with matching C declarations.

`interpreter-dsp.h` updated with:
- C API forward declarations in the `extern "C"` block.
- Four C++ inline wrapper free functions: `expandInterpreterDSPFromFile`,
  `expandInterpreterDSPFromString`, `generateInterpreterAuxFilesFromFile`,
  `generateInterpreterAuxFilesFromString`.

#### `crates/wasm-ffi/src/lib.rs`

No changes — `faust_wasm_expand_dsp` and `faust_wasm_generate_aux_files`
were already wired to `Compiler::expand_dsp` / `Compiler::generate_aux_files`
and now exercise real implementations automatically.

### Result

- All 5 new compiler tests pass.
- Full `cargo test -p compiler` suite: all tests pass (no regressions).
- `cargo build -p compiler -p cranelift-ffi -p interp-ffi -p wasm-ffi` clean.
