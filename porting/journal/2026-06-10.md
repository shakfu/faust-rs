## Analyze ondemand/upsampling/downsampling clock domains and write port plan

Commit date: 2026-06-10
Source commit: this commit

Wrote a full analysis of the C++ on-demand computation concept
(`ondemand` / `upsampling` / `downsampling` primitives with a clock signal
as first input) and a porting plan for faust-rs:
`porting/ondemand-clock-domains-analysis-port-plan-2026-06-10-en.md`.

C++ reference: `RUST/faust`, branch `master-dev-ocpp-od-fir-2-FIR19`,
commit `8eebea429`.

Analysis content:

- semantics: clock-guarded blocks (`if`/`for`/modulo-counter), temp-var
  inputs, perm-var sample-and-hold outputs, per-domain local time (IOTA),
  `ma.SR` adaptation under US/DS; generated-code examples captured from the
  branch binary;
- stage-by-stage C++ implementation walkthrough: propagation with threaded
  `clockenv`, the new signal IR nodes (`SIGOD/US/DS`, `SIGCLOCKED`,
  `SIGTEMPVAR/PERMVAR/ZEROPAD/SEQ`), typing rules, clock environment
  inference (`clkEnvInference.cpp`), hierarchical dependency graph
  (`Hgraph`/`Hsched`), block-structured code generation;
- algorithm properties: the inference is a monomorphic clock calculus over a
  finite tree of strictly nested domains, solved by a Jacobi-style Kleene
  iteration (monotone, terminating, deterministic, least fixed point places
  recursive groups in the shallowest consistent domain); graph construction
  is a single O(V+E) DFS partitioning signals by domain;
- critical assessment: algorithms are sound and architecturally right;
  encodings are prototype-grade (cons-tuple clock envs with a
  function-pointer cast, pseudo-bottom type hack, tree-property state);
- Rust recommendation: port the algorithms 1:1, modernize representations
  (`ClockDomain` arena + side maps, explicit block type, structured
  diagnostics); do not simplify the double-clocking of inputs nor the
  separate bottom-up inference phase;
- faust-rs state audit: front half (parser â†’ propagation â†’ typing) already
  at parity; missing back half (inference, hierarchical scheduling,
  FIR structured blocks, backends); two concrete bugs identified
  (`signal_prepare` traverses clock envs as signals â†’ FRS-SFIR-0004;
  `make_clock_env` omits the slotenv/path uniqueness token);
- 8-step port plan with the FIR block extension as the architectural
  centerpiece, plus differential validation against the branch binary.

  simplified to constants, their `f64` values are extracted via `match_box`

  `checkRange` call in `eval.cpp`.
