### `parser` / `signals` — Unblock more `impulse-tests` corpus cases with import-comment support and non-recursive signal dumps

Commit date: 2026-03-12

Source commit: pending

#### Context

After fixing repeated transitive import expansion, the next external
`impulse-tests` blockers were infrastructure issues rather than clear semantic
parity failures:

- `filter.lib` imported `music.lib` as `import("music.lib"); // ...`, which the
  Rust source reader failed to recognize as an import line
- very large DSPs such as `modulations.dsp` stressed parser source-location
  bookkeeping
- large signal trees produced during `--dump-sig` needed a non-recursive dump
  path to avoid tool-side stack exhaustion in diagnostic workflows

The immediate goal was to remove these front-end/tooling blockers before
continuing deeper evaluator/propgation debugging on the remaining corpus
failures.

#### Changes

- [source_reader.rs](../../crates/parser/src/source_reader.rs)
  - taught `parse_import_line(...)` to accept `import("...");` followed by a
    trailing line comment
  - added regression coverage for the commented import form

- [lib.rs](../../crates/parser/src/lib.rs)
  - precomputed line-start offsets for parser inputs
  - replaced repeated `lexer.line_col(...)` cursor updates with binary-search
    lookup against the cached line-start table
  - kept source-origin remapping behavior unchanged for expanded imported files

- [lib.rs](../../crates/signals/src/lib.rs)
  - rewrote `dump_sig(...)` and `dump_sig_readable(...)` to use an explicit task
    stack instead of recursive descent

- [core_api.rs](../../crates/signals/tests/core_api.rs)
  - added a deep left-spine dump regression to ensure large signal trees can be
    rendered without recursive stack growth

#### Validation

- `cargo fmt --all`
- `cargo test -p parser source_reader -- --nocapture`
- `cargo test -p signals -- --nocapture`
- targeted check: `library("filter.lib").if` now resolves successfully through
  the transitive `filter.lib -> music.lib -> math.lib` chain
- targeted rescan of `impulse-tests` signal dumps after the fix set
  - result: `73` OK, `19` fail, `1` timeout
  - `cubic_distortion.dsp` no longer fails on `undefined symbol if`; it now
    reaches a later `stack overflow` blocker

#### Outcome

`faust-rs` now handles commented import lines like the legacy Faust library
files expect, performs cheaper parser cursor mapping on very large sources, and
uses a non-recursive signal dumper suitable for large signal trees. The
remaining `impulse-tests` blockers are now more clearly concentrated in deeper
evaluation/propagation stack-overflow paths rather than import/dump plumbing.

### `parser` — Deduplicate transitive imports during source expansion to match C++ `SourceReader`

Commit date: 2026-03-12

Source commit: pending

#### Context

Scanning the external Faust `impulse-tests` DSP corpus exposed four parser
failures that were not true syntax problems:

- `bandfilter.dsp`
- `dbmeter.dsp`
- `lfboost.dsp`
- `vumeter.dsp`

All four files import both `math.lib` and `music.lib`, while `music.lib`
already re-imports `math.lib`. The Rust parser source expander was inlining the
same imported file multiple times, which duplicated zero-argument top-level
definitions such as `SR`, `BS`, and `PI` before `format_definitions(...)`
grouping. That later surfaced as the misleading parser diagnostic
`multiple definitions of a zero-argument symbol are not allowed`.

The C++ reference `SourceReader::expandRec(...)` guards import expansion with a
`visited` set and therefore expands each imported file at most once.

#### Changes

- [source_reader.rs](../../crates/parser/src/source_reader.rs)
  - added `expanded_files` tracking to the Rust `SourceReader`
  - cleared that tracking at each top-level `read_file*` entry point
  - skipped re-injecting already-expanded imports while preserving cycle
    detection through the existing `visiting` set
  - added a regression test covering a direct import plus a transitive
    re-import of the same file

#### Validation

- `cargo fmt --all`
- `cargo test -p parser source_reader -- --nocapture`
- `cargo run -p compiler -- --parse -I /Users/letz/Developpements/RUST/faust/tests/impulse-tests/dsp -I /Users/letz/Developpements/RUST/faust/tests/impulse-tests /Users/letz/Developpements/RUST/faust/tests/impulse-tests/dsp/bandfilter.dsp`
- `cargo run -p compiler -- --parse -I /Users/letz/Developpements/RUST/faust/tests/impulse-tests/dsp -I /Users/letz/Developpements/RUST/faust/tests/impulse-tests /Users/letz/Developpements/RUST/faust/tests/impulse-tests/dsp/vumeter.dsp`
- targeted parse rescan of `/Users/letz/Developpements/RUST/faust/tests/impulse-tests/dsp`
  - result after fix: `92` parse OK, `0` parse errors, `1` timeout (`modulations.dsp`)

#### Outcome

`faust-rs` now matches the C++ import-expansion behavior for repeated transitive
imports, which removes the false parser failures in the `impulse-tests` DSP
corpus caused by duplicated library definitions.

### `compiler` / `eval` / `transform` — Add `-pn` / `--process-name` support for custom DSP entry points

Commit date: 2026-03-12

Source commit: pending

#### Context

Testing wider external DSP corpora such as `faustlibraries/tests` requires the
Rust compiler to accept the same custom top-level entry-point override as the
C++ compiler (`-pn <name>` / `--process-name <name>`).

Before this change, the Rust pipeline hard-coded `process`:

- evaluator lookup,
- missing-entry diagnostics,
- compiler facade orchestration,
- some alias-trace/source-label fallbacks in diagnostics,
- CLI entrypoint selection.

That made parse/eval/propgate scans impossible on DSPs whose exported entry is
not named `process`.

#### Changes

- [lib.rs](../../crates/eval/src/lib.rs)
  - added entry-point-parameterized evaluator entry points on top of the
    existing `eval_process*` wrappers
  - generalized `MissingProcessDefinition` so diagnostics mention the requested
    entry-point name instead of always `process`

- [lib.rs](../../crates/compiler/src/lib.rs)
  - added `Compiler::with_process_name(...)`
  - wired the configured entry-point through `pipeline_to_signals(...)`
  - updated alias/source-label fallback helpers so diagnostic traces can start
    from the configured entry-point instead of always `process`

- [main.rs](../../crates/compiler/src/main.rs)
  - added `--process-name <name>` CLI support
  - added legacy compatibility normalization for `-pn <name>`
  - wired the configured entry-point through the CLI-built `Compiler`

- [core_eval.rs](../../crates/eval/tests/core_eval.rs)
  - added regression coverage for custom-entrypoint missing-definition
    diagnostics

- [lib.rs](../../crates/compiler/src/lib.rs)
  - added regression coverage proving `compile_source_to_signals(...)` works
    with a custom top-level entry-point

- [main.rs](../../crates/compiler/src/main.rs)
  - added CLI regressions for `-pn` normalization and `--process-name` parsing

#### Validation

- `cargo test -p eval eval_error_converts_to_structured_diagnostic_codes -- --exact`
- `cargo test -p compiler process_name -- --nocapture`
- `cargo test -p compiler custom_entrypoint -- --nocapture`
- `cargo run -p compiler -- --dump-sig --process-name dsp /tmp/faust_rs_custom_entry.dsp`

#### Outcome

`faust-rs` can now target a custom top-level DSP entry-point name through both
its public compiler facade and the CLI, which removes the `process` hard-stop
for parse/eval/signal scans over external DSP suites.
