### `eval` / `tests` — Preserve partial `case` matcher state during structural lowering

Commit date: 2026-03-12

Source commit: pending

#### Context

After fixing parsed abstraction reconstruction, the next `impulse-tests`
investigation moved to the `no case rule matches` family, starting from
`carre_volterra.dsp`.

The first reduced reproducer was:

```faust
poly(1,x)=x;
poly(6,x)=x*x*(7.0 + x*x*(-5.0 + x*x));
saw1l = _;
process = saw1l : poly(6);
```

This should behave like a partially-applied multi-rule function returning a DSP
of one signal input, but Rust still failed with `no case rule matches
arguments`. The root cause was not pattern evaluation itself; it was structural
rebuilding. `map_children(...)` re-evaluated every child through `eval_box(...)`,
which forced a partially-applied `PatternMatcher` back to raw `case` syntax and
discarded the already-matched argument state.

#### Changes

- [lib.rs](../../crates/eval/src/lib.rs)
  - changed `map_children(...)` to lower child evaluator values through
    `a2sb_value(...)` instead of `eval_box(...)`
  - this preserves residual partial `case` matcher state as symbolic first-order
    boxes instead of collapsing it back to the original raw `case`

- [core_eval.rs](../../crates/eval/tests/core_eval.rs)
  - updated the under-applied `case` regression to assert a residual symbolic
    function shape rather than the weaker previous behavior of returning the
    original raw `case` syntax

#### Validation

- `cargo test -p eval eval_process_case_supports_incremental_partial_application --test core_eval -- --exact --nocapture`
- targeted mini-repro:
  - `poly(1,x)=x; poly(6,x)=...; saw1l = _; process = saw1l : poly(6);`
  - result after fix: `--dump-sig` succeeds

#### Outcome

Rust now preserves partial `case` application state across structural lowering
boundaries, which fixes one real evaluator bug exposed by the `carre_volterra`
reduction work.

The original `carre_volterra.dsp` case still does not fully pass: a second,
distinct blocker remains when the partially-applied `case` argument is itself a
constant expression such as `max(1,min(2,4))` that has not yet been simplified
to an integer before matching. That constant-simplification issue was left out
of this commit on purpose.

### `boxes` / `compiler` — Fix parsed abstraction argument order and unblock higher-order corpus application

Source commit: pending

`reverb_designer.dsp` and `reverb_tester.dsp` first suggested a higher-order
evaluation bug: passing a named function as an argument and reapplying it later
failed in Rust with `too many arguments`.

The minimal reproducer was:

```faust
gui_group(x) = hgroup("top", x);
apply(f, x) = f(x);
```

### `parser` — Deduplicate transitive imports during source expansion to match C++ `SourceReader`

Commit date: 2026-03-12

Source commit: pending

#### Context

Scanning the external Faust `impulse-tests` DSP corpus exposed four parser
failures that were not true syntax problems:

- `bandfilter.dsp`
- `dbmeter.dsp`
- `lfboost.dsp`
- `vumeter.dsp`

All four files import both `math.lib` and `music.lib`, while `music.lib`
already re-imports `math.lib`. The Rust parser source expander was inlining the
same imported file multiple times, which duplicated zero-argument top-level
definitions such as `SR`, `BS`, and `PI` before `format_definitions(...)`
grouping. That later surfaced as the misleading parser diagnostic
`multiple definitions of a zero-argument symbol are not allowed`.

The C++ reference `SourceReader::expandRec(...)` guards import expansion with a
`visited` set and therefore expands each imported file at most once.

#### Changes

- [source_reader.rs](../../crates/parser/src/source_reader.rs)
  - added `expanded_files` tracking to the Rust `SourceReader`
  - cleared that tracking at each top-level `read_file*` entry point
  - skipped re-injecting already-expanded imports while preserving cycle
    detection through the existing `visiting` set
  - added a regression test covering a direct import plus a transitive
    re-import of the same file

#### Validation

- `cargo fmt --all`
- `cargo test -p parser source_reader -- --nocapture`
