### `porting` / `compiler` / `transform` — Add living snapshot of the Faust subset currently supported by `faust-rs`

Commit date: 2026-03-12

Source commit: pending

#### Context

The repository had many detailed parity plans and differential reports, but no
single document describing the currently supported Faust language subset in a
way that distinguishes:

- front-end acceptance (`parse -> eval -> propagate -> signals`),
- end-to-end backend support on the active fast-lane route,
- and the specific reasons why some valid Faust programs still fail in Rust
  while they compile in the C++ compiler.

This made it hard to answer basic scope questions such as “what compiles
today?”, especially for cases like variable delays where the front-end and the
backend have different support envelopes.

#### Changes

- [faust-rs-supported-faust-subset-en.md](../../porting/faust-rs-supported-faust-subset-en.md)
  - added a living status document for the currently supported Faust subset
  - separated front-end corpus parity from end-to-end backend support
  - recorded the current snapshot:
    - front-end corpus parity at `75/75` status-matched cases
    - backend parity at `59/60` valid corpus cases, with one remaining valid
      backend gap in non-trivial stream wrappers
  - documented the major current backend exclusions, notably:
    - variable delays
    - non-trivial stream wrappers
    - interval-dependent backend behaviors
    - complex `SIGGEN` table initializers outside the active slice
  - added a short historical progression from the fast-lane bootstrap through
    parser/pattern/eval closure and fixed-delay support

- [phase-4-corpus-status-diff-report-en.md](../../porting/phases/phase-4-corpus-status-diff-report-en.md)
  - refreshed the front-end corpus status snapshot used by the new document
  - current summary is now:
    - `75` total cases
    - `60` valid `OK/OK`
    - `15` invalid `ERR/ERR`
    - no remaining `OK/ERR` or `ERR/OK` mismatches

- [phase-6-backend-full-corpus-diff-report-en.md](../../porting/phases/phase-6-backend-full-corpus-diff-report-en.md)
  - refreshed the end-to-end fast-lane backend snapshot used by the new document
  - current summary is now:
    - `75` total cases
    - `59` end-to-end supported cases
    - `16` unsupported cases
    - one remaining valid backend gap in `rep_18_stream_wrappers.dsp`

- [2026-03-12.md](../../porting/journal/2026-03-12.md)
  - added this journal entry

#### Validation

- `cargo run -p xtask -- corpus-status-report`
- `cargo run -p xtask -- backend-full-corpus-diff-report`
- manual source cross-check against:
  - `crates/transform/src/signal_prepare.rs`
  - `crates/transform/src/signal_fir/mod.rs`
  - `crates/transform/src/signal_fir/module.rs`
  - parity/history documents under `porting/`

#### Outcome

The repository now has one stable document that answers, in a way that can be
updated over time, which Faust programs compile in Rust today and where the
current end-to-end boundary still sits relative to the C++ compiler.

### `porting` / `interval` — Add full C++ interval-library port plan

Commit date: 2026-03-12

Source commit: pending

#### Context

The Rust workspace still treats `crates/interval` as a scaffold, while several
parity-sensitive compiler paths now depend on interval facts to progress,
especially bounded-variable delay support in the fast-lane.

The immediate need was to stop framing interval work as a local `SIGDELAY`
patch and instead define a proper subsystem port for the full C++
`compiler/interval/` surface.

#### Changes

- [interval-port-plan-2026-03-12-en.md](../../porting/interval-port-plan-2026-03-12-en.md)
  - added a dedicated porting plan for the full C++ interval subsystem
  - scoped the work to the complete `compiler/interval/` library, including
    core interval data, operator algebra, placeholders, tests, and downstream
    integration
  - documented the current Rust gap, the target crate architecture, the
    implementation phases, and the validation matrix

- [2026-03-12.md](../../porting/journal/2026-03-12.md)
  - added this journal entry

- [JOURNAL.md](../../JOURNAL.md)
  - added the `2026-03-12` daily journal file to the top-level index

- [README.md](../../porting/journal/README.md)
  - added the `2026-03-12` day entry to the journal day index

#### Validation

- manual review of the new plan against:
  - current `crates/interval` scaffold status
  - current `signal_prepare` reduced typing model
  - current fast-lane `SIGDELAY` limitation
  - C++ interval and signal-typing reference entry points

#### Outcome

The repository now has an explicit parity-first plan for porting the full C++
interval library into Rust, instead of leaving interval work implicit inside
transform-level follow-up tasks.
