# 2026-05-04

## fix(draw): correct AD output arity in draw + RecSchema bounds guard

`RecSchema::draw` and `RecSchema::collect_traits` iterated `0..self.s2.inputs()`
and called `self.s1.output_point(i)` for each index, assuming the Faust type
invariant `s1.outputs() >= s2.inputs()`.  Any DSP that uses `fad(…)` /
`ForwardAD` or `rad(…)` / `ReverseAD` can break this invariant at the draw level
and trigger a `panic_bounds_check` in `BlockSchema::output_point`.

### Root cause — AD output-expansion rules were missing from the draw

Both AD wrappers expand the output count of their inner expression at compile
time.  The draw module was rendering them as plain decorators around the
**unmodified inner expression**, inheriting its schema arity without the
expansion rules:

```rust
// Before — wrong arity for both AD variants
BoxMatch::ForwardAD(expr, _seed) =>
    make_decorate(generate_inside(arena, expr, config), 10.0, "fad"),
BoxMatch::ReverseAD(expr, _seeds) =>
    make_decorate(generate_inside(arena, expr, config), 10.0, "rad"),
```

The compile-time rules (from `propagate/src/lib.rs`) are:

| Wrapper | `outputs` formula |
|---|---|
| `fad(expr, seed)` | `expr.outputs × (1 + seed.outputs)` — primal + one tangent lane per seed output |
| `rad(expr, seeds)` | `expr.outputs + seeds.outputs` — primal lanes + one gradient lane per seed output |

Any `SeqSchema`, `SplitSchema`, `RecSchema`, or other composition built on top of
an AD box therefore received a draw-level arity that disagreed with what the
type-checker proved at compile time.

In `fad_pendulum_cello4.dsp`, the chain was:
- `train ~ si.bus(2)` → `RecSchema(s1=train_body, s2=si.bus(2))`
- `train_body` contains `ad.grad(…)` = `fad(expr, p) : !, _`
- The `ForwardAD` draw schema reported fewer outputs than the compiled form
- Downstream `SeqSchema` cable balancing compensated locally but shifted the
  mismatch further up the tree
- A nested `RecSchema` (inside `cello_string` / `de.fdelay` / `fi.lowpass`)
  ended up with `s2.inputs() = 4` but `s1.outputs() = 1`, triggering the panic

The same panic would occur for any DSP that uses `rad(…)` in a recursive context.

### Fix 1 — expand AD output count with derivative-lane stubs

New shared helper `ad_augment_extra_outputs(inner, n_extra, label)` in
`translate.rs`, with thin wrappers for each AD variant:

```rust
fn ad_augment_extra_outputs(inner: Box<dyn Schema>, n_extra: usize, label: &str)
    -> Box<dyn Schema>
{
    if n_extra == 0 { return inner; }
    make_par(inner, make_block(0, n_extra, label, COLOR_SLOT, ""))
}

fn fad_augment_tangents(inner: Box<dyn Schema>, seed_outs: usize) -> Box<dyn Schema> {
    let n_tangent = inner.outputs() * seed_outs;     // primal × seed outputs
    ad_augment_extra_outputs(inner, n_tangent, "∂")
}

fn rad_augment_gradients(inner: Box<dyn Schema>, seeds_outs: usize) -> Box<dyn Schema> {
    ad_augment_extra_outputs(inner, seeds_outs, "∇") // one grad lane per seed output
}
```

The stub block (`0 → n_extra`, colored with `COLOR_SLOT`) is placed in parallel
with the primal inner schema so the decorator's port count matches the compiled
arity while preserving the visual representation of the inner expression.

Both `generate_inside` and `generate_inside_folded` now call these helpers:

```rust
BoxMatch::ForwardAD(expr, seed) => {
    let inner = generate_inside(arena, expr, config);
    let seed_outs = generate_inside(arena, seed, config).outputs().max(1);
    make_decorate(fad_augment_tangents(inner, seed_outs), 10.0, "fad")
}
BoxMatch::ReverseAD(expr, seeds) => {
    let inner = generate_inside(arena, expr, config);
    let seeds_outs = generate_inside(arena, seeds, config).outputs().max(1);
    make_decorate(rad_augment_gradients(inner, seeds_outs), 10.0, "rad")
}
```

### Fix 2 — defensive `min` clamp in RecSchema (retained as safety net)

`feedback_ports = s1.outputs().min(s2.inputs())` is kept as the loop bound in
both `RecSchema::draw` and `RecSchema::collect_traits` as a belt-and-suspenders
guard against any future arity-skew cases.  The non-recursive outputs loop starts
from `feedback_ports` accordingly.

### Test added

`rec_schema_tolerates_more_feedback_inputs_than_left_outputs` — constructs a
`RecSchema` with `s1.outputs()=1` and `s2.inputs()=2`, places it, calls
`draw()` and `collect_traits()` and verifies no panic.

All 25 draw unit tests pass.  `fad_pendulum_cello4.dsp`, `fad_filter1.dsp`, and
`fad_biquad1.dsp` all generate SVG without errors.

## feat(wasm-ffi): inject user virtual sources via --virtual-source name=base64

Two commits enabling user library files (e.g. `ad.lib`) to reach the Rust
compiler in the faustide Rust backend.

### feat(wasm-ffi): --virtual-source flag parsed in compile path

New helper `virtual_sources_from_argv(argv)` parses `--virtual-source name=base64`
flags out of the compiler argument vector and merges the decoded sources on top
of the embedded standard library bundle.  `parse_compile_request` now uses this
helper so that any user library registered on the TypeScript side arrives at the
Rust compiler without filesystem access.

New test: `parse_compile_request_injects_virtual_source_from_flag`.

### fix(wasm-ffi): parse --virtual-source flags in generate_aux_files paths

`faust_wasm_generate_aux_files` and `faust_wasm_generate_aux_files_json` were
previously hardcoding `embedded_standard_library_sources()`, ignoring any
`--virtual-source` flags.  Both functions now call `virtual_sources_from_argv`
on their parsed argv, so user library files are available during SVG/aux-file
generation as well.  This fixed the "library could not resolve source file
`ad.lib`" error that appeared in SVG generation even when compilation succeeded.

## docs(porting): document SVG hierarchy and browser navigation model in faustwasm plan

Enriched `porting/faustwasm-svg-aux-files-plan-2026-05-03-en.md` with the
missing detail about the hierarchical structure of Faust SVG block diagrams.

### SVG hierarchy

The Faust compiler produces a *tree* of SVG files for any non-trivial program:
`process.svg` is the entry point and contains standard SVG `<a href="...">` links
(relative paths) pointing to child sub-diagrams, which may themselves link
deeper.  This tree must be returned in its entirety by
`faust_wasm_generate_aux_files_json(...)`: the JSON artifact array now
explicitly requires all `.svg` files, with `process.svg` first and relative
`path` keys that match the `href` attributes embedded in the SVG source.

### Browser navigation model

The classic Faust IDE interaction — click a dark-blue block to drill down, click
outside to go up — was analysed against three candidate strategies:

- **(a) `data:` URLs** — fails at depth > 1; relative `href` cannot resolve from
  a `data:` URL.
- **(b) `URL.createObjectURL`** — same problem; `blob:` URLs are opaque to
  relative-path resolution.
- **(c) application-managed navigation stack** — selected. The host maintains a
  stack of relative SVG paths, renders each from the in-memory
  `Record<string, string>` map, intercepts `<a>` clicks to push the target and
  background clicks to pop. Matches the existing Emscripten-based Faust IDE
  behaviour exactly.

The `faustwasm` adapter is responsible for returning a complete map; navigation
state is the caller's responsibility.

## feat(wasm-ffi): implement faust_wasm_generate_aux_files_json

Full Rust-side implementation of the SVG auxiliary-files JSON ABI described in
`porting/faustwasm-svg-aux-files-plan-2026-05-03-en.md`.

Four commits:

### fix(compiler): sort SVG artifacts so process.svg is always first

`generate_aux_files` previously collected SVG files via `read_dir`, which does
not guarantee ordering.  The artifact list is now sorted: `process.svg` first,
then remaining files in stable alphabetical order.  Incidentally fixed the
`draw_schema` call to pass `"process"` as the diagram name instead of the DSP
stem, so the root file is always named `process.svg` (matching the C++ compiler
`--process-name` default and the `faustwasm` expectation).

### feat(wasm-ffi): add WasmAuxFileArtifact DTO and faust_wasm_generate_aux_files_json export

New raw WASM export:
```
faust_wasm_generate_aux_files_json(
    name_ptr, name_len,
    source_ptr, source_len,
    args_ptr, args_len
) -> text_result_handle
```

Returns a UTF-8 JSON array of all generated auxiliary files, each encoded as
base64 (`content_base64`).  `process.svg` is always first.  On failure the
handle is an error text result with the compiler diagnostic.

Supporting helpers kept inline (no new crate dependencies):
- `base64_encode`: RFC 4648 standard alphabet, pure Rust.
- `json_escape`: minimal escaper for path values.
- `artifacts_to_json`: serialises `Vec<WasmAuxFileArtifact>` to JSON array.

The existing `faust_wasm_generate_aux_files` boolean export is unchanged.

### test(wasm-ffi): 7 new tests (20 total)

- `base64_encode`: 5 cases (empty, 1-byte, 2-byte, 3-byte, RFC 4648 vector).
- `artifacts_to_json`: empty array, single round-trip, JSON escaping.
- `generate_aux_files_json`: SVG artifact presence, base64→SVG decode, invalid
  source error, backward-compatible boolean wrapper.

### feat(xtask): add faust_wasm_generate_aux_files_json to required exports

`required_wasm_ffi_exports()` and the WAT fixture now include the new export,
so `cargo run -p xtask -- build-faustwasm-compiler-module` will fail if the
ABI regresses.

## fix(compiler): inject embedded stdlib in generate_aux_files via virtual_sources

Fixed "malformed definition node 8" error when `generate_aux_files` is called
via the WASM backend on DSPs that use `import("stdfaust.lib")`.

### Root cause

`generate_aux_files` called `compile_source_to_signals_with_search_paths` which
always passed `&VirtualSourceMap::default()` — an empty map.  The embedded
standard library bundle (populated at build time from `/usr/local/share/faust/`)
was only injected on the `faust_wasm_compile_dsp` path via `parse_compile_request`,
not on the auxiliary-files path.

### Fix

`GenerateAuxFilesRequest` gains a `pub virtual_sources: VirtualSourceMap` field
(derives `Default`) and `generate_aux_files` now calls
`compile_source_to_signals_with_import_context` passing `&request.virtual_sources`.

The two `wasm-ffi` call sites (`faust_wasm_generate_aux_files` and
`faust_wasm_generate_aux_files_json`) now pass
`virtual_sources: embedded_standard_library_sources()`.  The `cranelift-ffi`,
`interp-ffi`, and compiler-test sites use `..Default::default()` (empty map —
those targets use filesystem search paths, not the embedded bundle).

### Verification

```
node scripts/faust2svg.js test/mono.dsp /tmp/out-rust \
    --rust --rust-wasm .../faust_wasm_ffi.wasm
# Generated 8 files.
# process.svg  SR-…  hardreset-…  incraux-…  period-…  phasorimp-…
# phasortable-…  sinwaveform-…
```

All 20 `wasm-ffi` unit tests pass.

## fix(propagate/fad): discard per-REC-body cache entries on DEBRUIJNREC exit

Incorrect gradient produced by `fad(loss, prev_gain)` when `prev_gain` is the
feedback signal of an outer recursive loop and the loop body contains an inner
`DEBRUIJNREC` whose back-edge is structurally identical to `prev_gain`.

### Root cause — FAD transform cache not scoped to DEBRUIJNREC bodies

The `ForwardADTransform` kept a single flat `SigId → Dual` cache across the
entire traversal.  Inside a `DEBRUIJNREC` body the seeds are lifted by one De
Bruijn level (`DEBRUIJNREF(1)` → `DEBRUIJNREF(2)`) so that equality checks fire
correctly within the body.  Any signal node computed during body traversal was
cached under the same `SigId` it has at the outer scope — but with the
body-scope (lifted-seed) tangent.

Concrete chain in `fad_gain1.dsp`:

```
outer loop:  loop(prev_gain, in) = next_gain
  seed = delay1(proj(0, DEBRUIJNREF(1)))   ← prev_gain
  inner rec (no.noise):
    DEBRUIJNREC([1103515245 * delay1(proj(0, DEBRUIJNREF(1))) + 12345])
```

The inner noise body contains `delay1(proj(0, DEBRUIJNREF(1)))`, which hash-conses
to the **same `SigId`** as `prev_gain` (both are `delay1(proj(0, DEBRUIJNREF(1)))`).
When the transform entered the noise body:

1. The lifted seed became `delay1(proj(0, DEBRUIJNREF(2)))` — no match.
2. `delay1(proj(0, DEBRUIJNREF(1)))` was processed via the `Delay1` rule and
   cached as `Dual { tangent: delay1(proj(1, DEBRUIJNREF(1))) }`.
3. After exiting the body, the outer-scope seed check for `prev_gain` found
   the poisoned cache entry and returned the wrong tangent instead of `1.0`.

Effect on the generated C++ (before fix):

```cpp
// fad_gain1.dsp — gradient was:
// -constant * fRec395 * fTemp0^2 * (fRec395 - fSlow0)   ← WRONG (extra fRec395)
// correct gradient:
// +constant * fTemp0^2 * (fRec395 - fSlow0)
```

The extra `fRec395 = prev_gain` factor and wrong sign caused the gradient-descent
loop to diverge instead of converging toward `target_gain`.

### Fix — snapshot + restore cache on DEBRUIJNREC entry/exit

Before entering a `DEBRUIJNREC` body, the set of `SigId`s already in the cache
is saved.  After the body is traversed (seeds and depth restored), every cache
entry whose key was not in the pre-entry snapshot is discarded.  The final
`Dual` for the `DEBRUIJNREC` node itself is then inserted with the outer-scope
value.

```rust
let outer_cache_keys: AHashSet<SigId> = self.cache.keys().copied().collect();
// ... process body with lifted seeds ...
self.cache.retain(|k, _| outer_cache_keys.contains(k));
// insert final dual for sig
```

### Regression test

`fad_seed_not_poisoned_by_inner_rec_back_edge` in `forward_ad::tests`:

```rust
inner_rec = DEBRUIJNREC([k * delay1(proj(0, ref(1)))])
seed      = delay1(proj(0, DEBRUIJNREF(1)))   // same SigId as the rec back-edge
expr      = proj(0, inner_rec) + seed
// tangent must be Add(proj(1, fad_rec), real(1.0))
// rhs = real(1.0) proves tangent(seed) = 1.0
```

19 forward-AD unit tests pass; 54 propagate integration tests pass.
`fad_gain1.dsp`, `fad_pendulum_cello4.dsp`, `fad_filter1.dsp` all compile cleanly.
