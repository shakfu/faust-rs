# 2026-06-22

## Legibility experiment: signal_prepare.rs simplification analysis (EN)

A design/analysis doc (no code change) applying the same method as the
2026-06-21 `signal_fir/delay` experiment to the staging phase
`crates/transform/src/signal_prepare.rs` (1475 lines). Written as a single
English file in `porting/`:

- `signal-prepare-simplification-experiment-2026-06-22-en.md`

Contents:

- **Context** — `signal_prepare` is stage 2 (staging) of `transform`, between
  propagation and `signal_fir`. Entry `prepare_signals_for_fir_verified`
  (called from `signal_fir/mod.rs:214`); the heart is a fixed linear pipeline of
  ~16 full-forest traversals, **5 of them `infer_full_types`**, threading 5
  hand-named `sig_types_*` snapshots. Boundary contract: de-Bruijn recursion /
  mixed numerics / non-canonical delays in → symbolic `SYMREC`, casts inserted,
  canonical `Delay1`, reduced `SimpleSigType` + full `sig_types` out, in a
  private arena.
- **Five reading frictions** (F1–F5), each anchored to `file:line`: the five
  hand-threaded type snapshots (F1), the "re-type after a structural change"
  rule as convention not structure (F2), the inline W4 inter-pass `debug_assert`
  clutter (F3), producer and ~700-line verifier sharing one file (F4), and the
  absence of a uniform pass contract (F5).
- **Three independent, orthogonal restructurings:** A — a typed staging pipeline
  (the pass sequence as data over a `Staging` value, the driver owning the
  re-type schedule); B — split the verification boundary into
  `signal_prepare/verify.rs` (producer vs. checker, a pure move mirroring the
  delay split); C — extract the two canonicalization rewrites into
  `signal_prepare/rewrites.rs` as testable units. Comparison table + recommended
  order **B → C → A**.
- **Safe/testable migration** anchored on a FIR-output (or structural prepared)
  snapshot plus the existing **32 `signal_prepare` tests**, the 103 transform
  tests, the impulse oracle, and the built-in `PreparedSignals::verify`
  postcondition checker; per-proposal recipe with differential testing for A.
  Performance argument: A reproduces the exact 5-infer schedule (same traversal
  count); B/C are pure moves. Making the schedule explicit later *enables*
  auditing redundant re-types (out of scope for "same perf").
- **Worked before/after** of the pipeline body: the five hand-threaded
  `sig_types_*` snapshots vs. a flat `PIPELINE` list of named `Pass` entries.

Validation: documentation only — no code changed, no tests affected. Companion
doc links verified to resolve; `file:line` references checked against current
`signal_prepare.rs`.
