## Expose boxFad/boxRad in the Box C/C++ API

Commit date: 2026-06-09
Source commit: this commit

Added forward/reverse automatic differentiation constructors to the libfaust
Box API surface, which previously had no FFI entry point even though the
`boxes` builder already supported them.

Implementation updates:

- added `CboxFad(exp, seed)` and `CboxRad(exp, seeds)` to `box-ffi` via the
  existing `prim2!` macro, dispatching to `BoxBuilder::forward_ad` /
  `reverse_ad`;
- declared `Box CboxFad(...)` / `Box CboxRad(...)` in `libfaust-box-c.h` and
  added the C++ inline wrappers `boxFad` / `boxRad` in `libfaust-box.h`;
- factored the builder's inlined `intern_tag(BOX_FORWARD_AD_TAG, ...)` /
  `intern_tag(BOX_REVERSE_AD_TAG, ...)` into new `node_forward_ad` /
  `node_reverse_ad` helpers in `internals.rs`, matching the node-helper
  convention used by `node_inputs`, `node_ondemand`, etc. (mirroring the C++
  `boxForwardAD` / `boxReverseAD` names); removed the now-unused tag import from
  `builder.rs`.

Scope notes:

- kept the builder method names `forward_ad` / `reverse_ad`, consistent with the
  descriptive `*_ad` vocabulary used across `propagate` (`mod forward_ad`,
  `contains_forward_ad`, `block_reverse_ad`) and the ~30 call sites in `eval`
  and tests; the short `fad` / `rad` names stay at the language/FFI boundary
  (tags, `CboxFad` / `boxFad`);
- used a `//` line comment (not `/* */`) for the new C header decl so the
  `libfaust-export-check` header parser does not misread the comment text as a
  symbol.

Validation:

- `cargo fmt --all`
- `cargo clippy -p boxes -p box-ffi --all-targets -- -D warnings`
- `cargo test -p boxes -p box-ffi`
- `cargo run -p xtask -- libfaust-export-check`
