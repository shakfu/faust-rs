## Factorization Part A — split signal_fir/module.rs into a module/ directory

Commit date: 2026-05-25
Source commits: `0d1291c`, `17cc40e`, `964b4a0`, `caf1939`, `c494754`,
`96a7804`, `e340594`

Executed Part A of `porting/factorization-god-files-plan-2026-05-25-en.md`. The
5020-line `crates/transform/src/signal_fir/module.rs` is now a `module/`
directory of cohesive, single-concern files:

| File | LOC | Concern |
|---|---|---|
| `mod.rs` | 508 | struct + fields, shared constants/free fns, residual helpers, submodule wiring |
| `build.rs` | 584 | `build_module` FIR-assembly entry point (re-exported `pub(crate)`) |
| `setup.rs` | 375 | constructor + shared lowering utilities |
| `core_lowering.rs` | 794 | `lower_signal` dispatch + leaf lowering |
| `state.rs` | 348 | delay/recursion state + named-struct/register-init helpers |
| `bra.rs` | 908 | BlockReverseAD lowering (backward sweep, adjoint, tapes, carries) |
| `ui_lowering.rs` | 531 | UI control + metadata + buildUserInterface emission |
| `tables.rs` | 423 | waveform/rdtbl/wrtbl + table sizing/index helpers |
| `arithmetic.rs` | 522 | binop/math/minmax/abs/cast/select2/proj + map_binop |
| `rad_formula_builder.rs` | 124 | FIR adapter for backend-neutral RAD formulas |

Mechanics and findings:

- Pure code move, no logic edits. The single `impl SignalToFirLower` was already
  partitioned into 5 `impl` blocks aligned to section dividers.
- Cross-module methods were promoted to `pub(super)`; struct fields stayed private
  (descendant modules can access an ancestor struct's private fields, but a method
  defined in a child-module `impl` needs `pub(super)` to be callable from siblings
  or the parent). `lower_int32_const` and `unsupported_node` stayed private in
  `mod.rs`.
- Two child-module names had to deviate to avoid clashes: `ui_lowering` (the `ui`
  crate is imported) and `core_lowering` (the std `core` crate).
- Sub-range extractions had to use **doc-comment-aware boundaries**: cutting at a
  `fn` line orphans that method's doc comment and grabs the next method's doc.
- `build_module` was lifted out (beyond the plan's original mapping) so `mod.rs`
  lands at ~510 lines, under the plan's ≤900 target.

Validation:

- `cargo fmt`, `cargo clippy -p transform --all-targets -- (clean)`,
  `cargo test -p transform` (97 ok) after every extraction.
- End-to-end gate: `cargo test -p compiler` (50/34/91/2 ok) and
  `cargo build --workspace` green at Part A completion. Behavior frozen.
- Note: `xtask golden-check` has a pre-existing unrelated failure (missing golden
  for `rad_delay1_block_fallback`), so per-crate test suites were used as the
  behavior gate.

## Factorization plan for the three largest source files

Commit date: 2026-05-25
Source commit: this commit (`Add code-quality assessment and god-file factorization plan`)

Added `porting/factorization-god-files-plan-2026-05-25-en.md`, a concrete,
behavior-preserving plan to split the three largest "god files" identified in the
quality assessment:

- `transform/src/signal_fir/module.rs` (5020 LOC) — the 127 methods of the
  module-private `SignalToFirLower` struct are already grouped into 5 `impl`
  blocks aligned to section dividers, so the file converts to a `module/`
  directory (`mod.rs` + `core`/`state`/`bra`/`ui`/`tables`/`arithmetic`/
  `rad_formula_builder`) with no visibility changes, relying on the Rust rule
  that child modules may access private fields of an ancestor's struct. Single
  entry point `build_module` (`pub(super)`) and no external references confirm
  the split is internal to the `signal_fir::module` subtree.
- `eval/src/lib.rs` (4946 LOC) — extract free-function clusters
  (`ui_widgets`, `modulation`, `label`, `definitions`, `apply`, `iteration`,
  plus case/pattern consolidation) via `pub(crate)` moves, keeping the evaluator
  dispatch core in `lib.rs`.
- `compiler/src/lib.rs` (4599 LOC) — extract helper concerns
  (`diagnostics`, `signal_lowering`, `json_options`, `box_preview`, `paths`,
  `definition_graph`, `golden`, `names`) as `pub(crate)` (never `pub`, since five
  FFI crates depend on `compiler`), then thin `impl Compiler` to stage
  orchestration as a final, separately-reviewed pass.

The plan specifies per-file target tables (new file → methods/functions → source
line ranges), two reuse mechanics (impl-across-child-modules vs `pub(crate)`
free-function moves), a mandatory per-extraction gate
(`fmt`/`clippy -D`/`test`/`golden-check`, where any golden diff signals a
forbidden behavior change), four milestones (M1–M4) ordered by ascending risk,
suggested commit messages, a Definition of Done, and three open questions left to
the author (BRA module naming, case/pattern placement, façade depth).

  `utils` "placeholder" header, 220 untracked scratch files at the repo root).

  re-architecture (linearize-once + transpose), already planned.
