## Add AssemblyScript impulse-test runner and gate

Commit date: 2026-06-15
Source commit: this commit

Added an AssemblyScript impulse-test target under `tests/impulse-tests`.
The new `tools/impulseasc.js` runner compiles each DSP with
`faust-rs -lang asc`, wraps the generated class with exported buffer/control
entry points, compiles it with `asc`, executes the resulting WebAssembly module
through Node, and emits the scalar impulse `.ir` format.

Backend fixes needed for the runner:

- `compute` now uses `Array<StaticArray<f32>>` channel buffers, matching the
  generated method body's `StaticArray` loads.
- the AssemblyScript backend maps FaustFloat math helpers to `Mathf`/`f32`
  intrinsics consistently with its current `f32` ABI.
- generated modules include small helpers for FIR/libm functions that `asc`
  does not expose directly with C names: `fmod`, `remainder`, `rint`, `exp10`,
  `isnan`, `isinf`, and `copysign`.

The measured gate is **66/93**: the shared `subcontainer1` compile gap remains
excluded globally and 26 AssemblyScript-specific cases are listed in
`KNOWN_FAIL_assemblyscript`. Most exclusions are f32-vs-double numeric
divergences still needing tolerance classification; `sound` is a real soundfile
FIR lowering gap, and `table`/`table2` still show table/indexing divergences.

## Add WASM impulse-test runner and gate

Commit date: 2026-06-15
Source commit: this commit

Added an initial WASM impulse-test target under `tests/impulse-tests`.
The new `tools/impulsewasm.js` runner compiles each DSP with `faust-rs
-lang wasm -double`, instantiates the generated module with Node's native
WebAssembly runtime, drives the scalar impulse pass, and emits the same `.ir`
text format used by the existing runners.

The runner follows the current WASM ABI:

- DSP base pointer is `0`.
- JSON `size` gives the start of the host-managed I/O zone.
- input/output pointer tables and sample buffers are placed after that zone.
- `init(dsp, sample_rate)` is called before block processing.
- `compute(dsp, count, inputs, outputs)` is called block by block.
- `button` UI items from the JSON are driven for the first block only, matching
  the scalar impulse harness behavior.

`make wasm` now uses `Make.wasm` and is a green known-failure gate. The initial
classification is **70/93** passing: `subcontainer1` remains the shared compile
gap and 22 WASM-specific cases are excluded in `KNOWN_FAIL_wasm` for follow-up
classification.

- `make -C tests/impulse-tests -f Make.wasm ir/wasm/LPF.ir`
- `make -k -C tests/impulse-tests wasm` to classify the initial failures

## Document backend lifecycle contract for future backends

Commit date: 2026-06-15
Source commit: this commit

The lifecycle fix in `9ac18a9485e0cd56a3defd29fcada4c22f0b35b4` established
backend invariants that must apply to every future backend, not only to the
interpreter and Cranelift implementations corrected in that commit.

Documented a stable contract in
`porting/backend-lifecycle-contract-en.md`:

- `init(sample_rate)` must call `classInit(sample_rate)` before
  `instanceInit(sample_rate)`.
- `instanceInit(sample_rate)` must call `instanceConstants(sample_rate)`,
  `instanceResetUserInterface()`, and `instanceClear()` in that order.
- `instanceInit(sample_rate)` must not call `classInit(sample_rate)`.
- backend runtimes must not duplicate `instanceClear` with ad-hoc field
  clearing.
- compiled `instanceConstants` bodies are authoritative when present.

Added a short contributor rule in `AGENTS.md` requiring a backend-specific
lifecycle conformance test before any new backend is added to impulse, golden,
or parity gates.

Validation:

- documentation-only change; inspected diff locally

## Fix Cranelift impulse UI, soundfile, and mixed select2 parity

Commit date: 2026-06-15
Source commit: this commit

The Cranelift impulse gate still excluded eight DSPs:
`bells`, `karplus`, `karplus32`, `UITester`, `reverb_designer`,
`reverb_tester`, `sound`, and `grain3`.

Root causes:

- `impulse_cranelift` did not replay `buildUserInterface`, so it never captured
  button zones and never installed soundfile pointers. This diverged from the
  C++ impulse harness, which drives `FUI::setButtons` and installs the
  `TestMemoryReader` soundfile fixture through `SoundUI`.
- Under `-double`, UI zone callback pointers still arrive through the C ABI
  shape, but their addresses point to `f64` fields in the JIT state. The runner
  must therefore treat zone pointers as raw addresses and write at the compiled
  sample/control width.
- `grain3` exposed a Cranelift verifier failure in `select2`: the lowering
  emitted a CLIF `select` with an `i8` bool branch and an `i32` state branch.
  Both branches must be coerced to the FIR result type before emitting `select`.

Fixes:

- `impulse_cranelift` now replays UI callbacks, captures only `addButton` zones
  (matching `FUI::setButtons`, not checkboxes), and drives them for the first
  scalar impulse block.
- `impulse_cranelift` now installs an in-memory `Soundfile*` fixture equivalent
  to the C++ `TestMemoryReader`, including modulo channel sharing.
- Cranelift `Select2` lowering now coerces both branch values to the declared
  FIR result type before emitting CLIF.
- Added a Cranelift unit test for mixed bool/int `select2` lowering.

## Fix C backend grain3: preserve double literal precision

Commit date: 2026-06-15
Source commit: this commit

The C backend's last impulse-test exclusion was `grain3`, with a localized
drift around frame 14016 (`~2.6e-3`) and the same drift repeated on the second
impulse pass.

Root cause: the C emitter formatted `Float64` literals with `"{value:.15}"`,
which truncated sensitive constants such as `1.0 / 2147483647.0` to
`0.000000000465661`. The C++ backend emitted enough precision
(`0.0000000004656612875245797`). In `grain3`, that constant drives the
pseudo-random sample-and-hold delay path, so the truncation changed the delayed
sample interpolation at a few frames.

Fix: align the C backend double-literal formatting with the C++ backend by using
Rust's round-trip `f64` formatting and only appending `.0` for integer-looking
finite values. Added a unit test for the `grain3` PRNG scale literal.

Validation:

- `cargo test -p codegen --lib backends::c`
- `cargo build --release -p compiler --bin faust-rs`
- `make -C tests/impulse-tests -f Make.gcc outdir=c lang=c ir/c/grain3.ir`
- `make -B -C tests/impulse-tests -f Make.gcc outdir=c lang=c`

Result: `grain3` is removed from `KNOWN_FAIL_c`; the C backend gate is now
**92/93** (only the shared `subcontainer1` compile gap remains).

## Align backend lifecycle with the Faust C++ backend contract

Commit date: 2026-06-15
Source commit: this commit

The Cranelift `table2` impulse divergence exposed a lifecycle parity issue
rather than a DSP-specific rwtable problem. The external Faust C++ backend was
checked with `/Users/letz/Developpements/RUST/faust/build/bin/faust` on
`tests/corpus/rep_10_two_in_two_out_ui.dsp`; it emits:

- `init(sample_rate)`: `classInit(sample_rate)` then
  `instanceInit(sample_rate)`;
- `instanceInit(sample_rate)`: `instanceConstants(sample_rate)`,
  `instanceResetUserInterface()`, then `instanceClear()`.

Fixes:

- interpreter runtime now keeps `class_init` in `init`, not `instance_init`;
- interpreter FBC-to-C++ emission now follows the same split;
- Cranelift FFI now keeps `class_init_instance` in `init`, not
  `instanceInit`;
- Cranelift `instanceClear` no longer has a descriptor-side field clear policy;
  the compiled FIR `instanceClear` body is the authority;
- Cranelift descriptor constant replay is used only when no compiled
  `instanceConstants` body exists.

Guardrails added:

- generated C++/C lifecycle tests check both `init` and `instanceInit`;
- interpreter runtime executes order-sensitive lifecycle bytecode;
- interpreter FBC-to-C++ checks the emitted lifecycle scaffold;
- Cranelift FFI has a structural lifecycle test and rejects runtime-side clear
  policy reintroduction.

Documented the policy and validation plan in
`porting/backend-lifecycle-cpp-parity-plan-2026-06-15-en.md`.

## Fix Cranelift prefix/phasor: run the JIT instanceClear at init

Commit date: 2026-06-15
Source commit: this commit

The impulse harness surfaced that Cranelift's `prefix(init, x)` returned 0 at
frame 0 instead of `init` (breaking `prefix` and `phasor`, which builds on it).

Root cause: `prefix` fills its 1-sample delay buffer (`fRec7`) with the init
value inside `instanceClear` (`fRec7[i] = 1.0`). The Cranelift backend only
JIT-compiled `compute` and `instanceConstants`, so the runtime's
`instanceClearCCraneliftDSPInstance` zeroed state via the runtime descriptor but
never ran the compiled `instanceClear` body — `fRec7` stayed 0 and the first
prefix sample read 0. The interpreter and C++/C backends run `instanceClear`, so
they were correct.

Fix: api.rs now JIT-compiles `instanceClear` alongside `instanceConstants`;
`JitDspModule` carries `instance_clear_entry_addr`; and
`instanceClearCCraneliftDSPInstance` runs it (after the descriptor-based clear),
mirroring how `instanceConstants` is wired.

Result: `prefix` and `phasor` now match the reference exactly; the curated
Cranelift gate goes 81 -> **83/93**. No regression (cranelift-ffi 31 + codegen
251 tests pass).

## Fix interpreter short-delay silence (general ForLoop ignored is_reverse)

Commit date: 2026-06-15
Source commit: this commit

The impulse harness surfaced that the interpreter backend produced **silence**
for short fixed delays `@(N)` with `3 <= N <= mcd` (the shift-array strategy);
`@(1)`/`@(2)` (direct mem) and `@(N > mcd)` (ring buffer) were fine.

Root cause: `FirToFbcCompiler::compile_for_loop` (general `ForLoop`) took
`_is_reverse` and ignored it — it compiled the `step`/`end` nodes as bare
expressions and never updated the loop variable or built a real exit condition.
The shift loop is a reverse `ForLoop` (`var = N; var > 0; var += -1`), so it ran
zero useful iterations and the delay line was never shifted -> output stayed 0.

Fix: rewrite `compile_for_loop` to the canonical contract already implemented by
the Cranelift backend (`lower_for_loop`): `init` is a `DeclareVar(kLoop)` that
seeds the variable; `step`/`end` are the signed increment and exclusive bound;
the loop runs `do { body; var += step } while (is_reverse ? var > end : var <
end)` (`GTInt`/`LTInt`). Updated the `test_roundtrip_for_loop` unit test, which
had encoded a different, non-canonical convention (step = update statement, end
= condition expression).

Result: `comb_delay1`, `comb_delay2`, `math_simp`, `norm3` now match the
reference; the curated interp gate goes 78 -> **82/93**. No regression (codegen
251 tests pass). The same `is_reverse` handling already worked in C++/C (native)
and Cranelift, so only the interpreter was affected.

## Make the Cranelift backend work in 64-bit (double) and wire it into the impulse tests

Commit date: 2026-06-15
Source commit: this commit

The Cranelift JIT backend was 32-bit only (`FirType::FaustFloat` hard-mapped to
`F32`), so its impulse responses could not be compared against the `-double`
reference impulse-test oracle. Extended it to 64-bit and added it to the
`tests/impulse-tests` harness.

Backend changes (`crates/codegen/src/backends/cranelift`):

- New `CraneliftOptions::double_precision` flag, threaded through the type map
  (`fir_type_to_clif_type`), struct-layout sizing (`fir_type_layout_scalar`,
  `build_struct_layout_for_module`), static-table data emission
  (`define_static_tables_in_jit`), and the `compute`/`instanceConstants`
  lowering (`declare_jit_function` -> `FunctionBodyLoweringContext` ->
  `ComputeLowering`). Under `double`, `FaustFloat` resolves to `F64` / 8-byte.
- Op- and const-selection sites that lumped `FaustFloat` with `Float32` are now
  routed through a `canon_real` helper, so a `FaustFloat` value selects f64
  ops/consts under `-double` (binop, math `FunCall`, default-value emission).
  Load/store widths and buffer strides follow automatically because they go
  through `fir_type_to_clif`/`coerce_value_to_fir_type`.

FFI changes (`crates/cranelift-ffi`):

- `createCCraneliftDSPFactoryFromFile`/`FromString`/box/signal paths parse
  `-double` (via `parse_ffi_compile_args`) and build the FIR with
  `RealType::Float64` and the Cranelift module with `double_precision = true`.
- `apply_control_defaults` (UI-zone init writer) now writes a `FaustFloat`
  control zone at the field's actual width. Previously it always wrote f32,
  which under `-double` stored 4 bytes into an 8-byte zone and corrupted slider
  inits — the bug that made `freeverb` diverge from sample 0 (0.93) until fixed.
- New `src/bin/impulse_cranelift.rs`: JIT-compiles a DSP and runs the scalar
  impulse pass, emitting the reference `.ir` text. Allocates f64 I/O buffers
  under `-double` (the C-API only forwards the pointers).

Harness: `Make.cranelift` + `KNOWN_FAIL_cranelift` (`known.mk`) +
`make cranelift` target. The Cranelift gate is green at **81/93** against the
`-double` reference (was f32-only; better than interp's 74). Remaining gaps are
curated: `prefix`/`phasor` (prefix primitive, upstream bug #1071), `table2`
(rwtable), `bells`, `karplus`/`karplus32`, `UITester` (button zones not driven
yet), `reverb_designer`/`reverb_tester` (shared drift), `sound` (soundfile JIT
crash), `grain3`.

No f32 regression: `cranelift-ffi` (31) and `codegen` (251) tests still pass.
