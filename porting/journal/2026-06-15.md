## Fix Cranelift prefix/phasor: run the JIT instanceClear at init

Commit date: 2026-06-15
Source commit: this commit

The impulse harness surfaced that Cranelift's `prefix(init, x)` returned 0 at
frame 0 instead of `init` (breaking `prefix` and `phasor`, which builds on it).

Root cause: `prefix` fills its 1-sample delay buffer (`fRec7`) with the init
value inside `instanceClear` (`fRec7[i] = 1.0`). The Cranelift backend only
JIT-compiled `compute` and `instanceConstants`, so the runtime's
`instanceClearCCraneliftDSPInstance` zeroed state via the runtime descriptor but
never ran the compiled `instanceClear` body — `fRec7` stayed 0 and the first
prefix sample read 0. The interpreter and C++/C backends run `instanceClear`, so
they were correct.

Fix: api.rs now JIT-compiles `instanceClear` alongside `instanceConstants`;
`JitDspModule` carries `instance_clear_entry_addr`; and
`instanceClearCCraneliftDSPInstance` runs it (after the descriptor-based clear),
mirroring how `instanceConstants` is wired.

Result: `prefix` and `phasor` now match the reference exactly; the curated
Cranelift gate goes 81 -> **83/93**. No regression (cranelift-ffi 31 + codegen
251 tests pass).

## Fix interpreter short-delay silence (general ForLoop ignored is_reverse)

Commit date: 2026-06-15
Source commit: this commit

The impulse harness surfaced that the interpreter backend produced **silence**
for short fixed delays `@(N)` with `3 <= N <= mcd` (the shift-array strategy);
`@(1)`/`@(2)` (direct mem) and `@(N > mcd)` (ring buffer) were fine.

Root cause: `FirToFbcCompiler::compile_for_loop` (general `ForLoop`) took
`_is_reverse` and ignored it — it compiled the `step`/`end` nodes as bare
expressions and never updated the loop variable or built a real exit condition.
The shift loop is a reverse `ForLoop` (`var = N; var > 0; var += -1`), so it ran
zero useful iterations and the delay line was never shifted -> output stayed 0.

Fix: rewrite `compile_for_loop` to the canonical contract already implemented by
the Cranelift backend (`lower_for_loop`): `init` is a `DeclareVar(kLoop)` that
seeds the variable; `step`/`end` are the signed increment and exclusive bound;
the loop runs `do { body; var += step } while (is_reverse ? var > end : var <
end)` (`GTInt`/`LTInt`). Updated the `test_roundtrip_for_loop` unit test, which
had encoded a different, non-canonical convention (step = update statement, end
= condition expression).

Result: `comb_delay1`, `comb_delay2`, `math_simp`, `norm3` now match the
reference; the curated interp gate goes 78 -> **82/93**. No regression (codegen
251 tests pass). The same `is_reverse` handling already worked in C++/C (native)
and Cranelift, so only the interpreter was affected.

## Make the Cranelift backend work in 64-bit (double) and wire it into the impulse tests

Commit date: 2026-06-15
Source commit: this commit

The Cranelift JIT backend was 32-bit only (`FirType::FaustFloat` hard-mapped to
`F32`), so its impulse responses could not be compared against the `-double`
reference impulse-test oracle. Extended it to 64-bit and added it to the
`tests/impulse-tests` harness.

Backend changes (`crates/codegen/src/backends/cranelift`):

- New `CraneliftOptions::double_precision` flag, threaded through the type map
  (`fir_type_to_clif_type`), struct-layout sizing (`fir_type_layout_scalar`,
  `build_struct_layout_for_module`), static-table data emission
  (`define_static_tables_in_jit`), and the `compute`/`instanceConstants`
  lowering (`declare_jit_function` -> `FunctionBodyLoweringContext` ->
  `ComputeLowering`). Under `double`, `FaustFloat` resolves to `F64` / 8-byte.
- Op- and const-selection sites that lumped `FaustFloat` with `Float32` are now
  routed through a `canon_real` helper, so a `FaustFloat` value selects f64
  ops/consts under `-double` (binop, math `FunCall`, default-value emission).
  Load/store widths and buffer strides follow automatically because they go
  through `fir_type_to_clif`/`coerce_value_to_fir_type`.

FFI changes (`crates/cranelift-ffi`):

- `createCCraneliftDSPFactoryFromFile`/`FromString`/box/signal paths parse
  `-double` (via `parse_ffi_compile_args`) and build the FIR with
  `RealType::Float64` and the Cranelift module with `double_precision = true`.
- `apply_control_defaults` (UI-zone init writer) now writes a `FaustFloat`
  control zone at the field's actual width. Previously it always wrote f32,
  which under `-double` stored 4 bytes into an 8-byte zone and corrupted slider
  inits — the bug that made `freeverb` diverge from sample 0 (0.93) until fixed.
- New `src/bin/impulse_cranelift.rs`: JIT-compiles a DSP and runs the scalar
  impulse pass, emitting the reference `.ir` text. Allocates f64 I/O buffers
  under `-double` (the C-API only forwards the pointers).

Harness: `Make.cranelift` + `KNOWN_FAIL_cranelift` (`known.mk`) +
`make cranelift` target. The Cranelift gate is green at **81/93** against the
`-double` reference (was f32-only; better than interp's 74). Remaining gaps are
curated: `prefix`/`phasor` (prefix primitive, upstream bug #1071), `table2`
(rwtable), `bells`, `karplus`/`karplus32`, `UITester` (button zones not driven
yet), `reverb_designer`/`reverb_tester` (shared drift), `sound` (soundfile JIT
crash), `grain3`.

No f32 regression: `cranelift-ffi` (31) and `codegen` (251) tests still pass.
