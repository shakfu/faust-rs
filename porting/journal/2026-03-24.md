## Bug fix: float literal pattern matching (`t8.dsp`)

`t8.dsp` exposed a pattern-matching regression:

```faust
foo2(1.0) = 456;
process = foo2(1.0);   // → "no case rule matches arguments"
```

### Root cause — `simplify_pattern` in `crates/eval/src/lib.rs`

The automaton stores pattern constants as their original `TreeId`:
`foo2(1.0) = 456;` → `TransKind::Constant(float_bits(0x3ff0000000000000))`.
Matching checks `argument_tree_id == pattern_tree_id`.

`simplify_pattern` had two illegal coercions that converted integer-valued
`Real` values to `Int`:

```rust
// Fast-path literal branch (was):
BoxMatch::Real(x) => {
    let i = x as i32;
    if (i as f64) == x { return BoxBuilder::new(arena).int(i); }  // 1.0 → int(1)
    return box_id;
}

// Non-literal arithmetic branch (was):
SigMatch::Real(x) => {
    let i = x as i32;
    if (i as f64) == x { Some(NumericLit::Int(i)) }  // 2.0 → Int(2)
    else { Some(NumericLit::Real(x)) }
}
```

The argument `1.0` was coerced to `int(1)` before the `TreeId` comparison.
`int(1) ≠ float_bits(1.0)` → match failed.

The same bug affected `foo(4/2)` and `foo4(4.0/2.0)`: Faust `/` is always
real-valued division (even for integer operands), so both patterns store a
`float_bits(2.0)` constant, and the coercion produced a mismatched `int(2)`.

### Fix — align with C++ `simplifyPattern`

C++ `simplifyPattern` returns any `boxInt` / `boxReal` literal unchanged —
no Real→Int coercion at any point.

```rust
// Fast-path — return any literal as-is:
BoxMatch::Int(_) | BoxMatch::Real(_) => return box_id,

// Non-literal — preserve signal type:
SigMatch::Int(i)  => Some(NumericLit::Int(i)),
SigMatch::Real(x) => Some(NumericLit::Real(x)),   // no coercion
```

### Corpus entry and regression test

- `tests/corpus/rep_72_float_literal_pattern.dsp` — four cases:
  `foo1(1)`, `foo2(1.0)`, `foo(4/2)`, `foo4(4.0/2.0)`.
- `corpus_float_literal_pattern_matching_compiles_like_cpp` in
  `crates/compiler/tests/signal_pipeline.rs`.

All 62 eval + 35 compiler tests pass, zero new warnings.

## Analysis: `latch_demo_test` parity gap vs C++ reference

Investigated why `faust-rs -pn latch_demo_test tests/demos_tests.dsp` produces
different samples from the C++ reference (`faust 2.85.5`).

### Root cause

`ba.latch` uses `os.oscrs` which internally calls `fi.nlf2(f, 1)`.  The `nlf2`
definition computes `th = 2*pi*f/ma.SR` where `f = float(ma.SR)/1000.0`.

The C++ compiler reduces `float(SR)/1000.0/SR → 1/1000.0` at compile time via
its `simplify()` pass, then embeds precomputed float constants:
`sin(2π/1000) = 0.006283144f`, `cos(2π/1000) = 0.9999803f`.

faust-rs currently skips `simplify()` before FIR lowering, so the sin/cos
arguments `6.2831855f * SR/1000/SR` are evaluated in the hot loop every sample
using runtime float arithmetic.  After 200 iterations the oscillator phase
accumulates enough floating-point error to shift the latch trigger by 1 sample
(fires at sample 200 instead of 201), capturing a different value of `x`.

The fix is to apply `simplify()` between type annotation and FIR conversion —
exactly what the existing patch
`0001-feat-signal_prepare-wire-algebraic-simplification-be.patch` does.
**The patch is not applied yet** (deferred by decision on 2026-03-24).

### Confirmed by

- `cargo run -p xtask -- interp-trace-dump` (faust-rs): trigger at sample 200, latch = 0.951058
- Reference C++ harness at SR=48000: trigger at sample 201, latch = 0.952981

## Analysis: C++ `patternSimplification` / `isBoxNumeric` — porting gap

Compared the C++ `patternSimplification` (called from `evalPattern` during
automaton construction) against the Rust `pattern_simplification`.

### Key differences found

| Aspect | C++ | Rust (current) |
|--------|-----|----------------|
| Order | `isBoxNumeric` on whole expr **first**, then recurse | Recurse into children **first**, then weak fold |
| Recursion scope | PatternOp only (Par/Seq/Split/Merge/Rec) | Also recurses into HGroup/VGroup/TGroup/Route |
| Simplification power | Full propagation + `simplify()` via `isBoxNumeric` | Literal arithmetic only (`simplify_numeric_pattern`) |

The full-power path already exists in `simplify_pattern` (line 1841) and is
correctly wired into the pattern *matcher* (`match_num` guard).  It is not yet
used in `pattern_simplification` called from `eval_pattern`.

The three helpers `simplify_numeric_pattern`, `eval_numeric_pattern_value`,
`eval_numeric_binary_op` (~120 lines) are superseded by `simplify_pattern` and
can be deleted after the refactor.
