## refactor(fir,transform): move reusable FIR loop helpers into fir::helpers

Commit date: 2026-04-10

Source commit: pending

### Context

`crates/transform/src/signal_fir/loops.rs` no longer contained
`signal_fir`-specific policy. It only packaged low-level FIR loop builders used
by both delay-line and recursion-carrier lowering.

Keeping that code under `signal_fir` made the ownership misleading and blocked
reuse from other FIR-producing passes.

### Implementation

Updated:

- `crates/fir/helpers.rs`
- `crates/fir/src/lib.rs`
- `crates/fir/src/tests.rs`
- `crates/transform/src/signal_fir/delay.rs`
- `crates/transform/src/signal_fir/recursion.rs`
- `crates/transform/src/signal_fir/mod.rs`

Implemented:

- moved the shared loop helpers out of `signal_fir` into `crates/fir/helpers.rs`
- exposed them as `fir::helpers` through a `#[path = "../helpers.rs"]` module
  declaration in `crates/fir/src/lib.rs`
- renamed the destination module from the initial `loop_helpers` name to the
  more generic `helpers` path requested during the move
- rewired `signal_fir` delay and recursion lowering to import
  `emit_reverse_array_shift_loop` and `fresh_loop_var` from `fir::helpers`
- added a `fir` unit test locking the emitted reverse shift-loop shape
- removed the old local `signal_fir/loops.rs` module once all call sites were
  migrated

## fix(signal-fir): align recursive delay carrier lowering with Faust copy-loop thresholds

Commit date: 2026-04-10

Source commit: pending

### Context

The first recursion-carrier alignment pass switched small delayed recursion
groups away from the power-of-two circular buffer, but it still did not match
the C++ compiler on the exact lowering boundary used by `fi.iir(...)`-style
feedback chains.

Using `t13.dsp` as a differential probe showed the remaining mismatch:

- Faust C++ unrolls delay copies for carrier delays `1..2`
- Faust C++ emits a reverse copy loop for carrier delays `3..15`
- Faust C++ switches to a circular buffer at carrier delay `16`

`faust-rs` was still unrolling all exact-shift carriers up to and including the
`max_copy_delay` boundary.

### Implementation

Updated:

- `crates/transform/src/signal_fir/recursion.rs`
- `crates/transform/src/signal_fir/module.rs`
- `crates/transform/src/signal_fir/tests.rs`

Implemented:

- an exact recursion-carrier threshold check matching Faust's `delay <
  max_copy_delay` rule, so the boundary case now lowers to the circular path
- reverse-loop finalization for exact-shift recursion carriers once the delayed
  recursion depth exceeds two samples
- retention of direct unrolled copies for the small `delay <= 2` cases
- regression coverage proving the shift-loop path before the circular boundary
  and the circular fallback at and beyond the boundary

## perf(signal-fir): lower simple recursion carriers to one scalar state cell

Commit date: 2026-04-10

Source commit: pending

### Context

The recursion abstraction plan had already identified a narrower optimization
window for simple unary feedback loops such as:

```faust
process = + ~ _;
```

The fast-lane still lowered these cases through the generic two-slot carrier
path:

- one persistent `fRec*/iRec*[2]` array field
- one immediate current-slot write
- one post-output `slot1 = slot0` finalize copy

That was correct but too conservative for recursion shapes that only need one
previous-sample state cell.

### Implementation

Updated:

- `crates/transform/src/signal_fir/recursion.rs`
- `crates/transform/src/signal_fir/module.rs`
- `crates/transform/src/signal_fir/tests.rs`
- `crates/compiler/tests/signal_fir_lane.rs`
- `porting/signal-fir-recursion-abstraction-plan-2026-04-08-en.md`

Implemented:

- a `SingleScalar` recursion storage strategy selected for conservative unary
  recursion carriers whose delayed recursion usage does not exceed one sample
- scalar struct-backed recursion state (`fRec*` / `iRec*`) initialized directly
  in `instanceClear()` without array-clear loops
- stack-local current-sample bindings (`fRecCur*` / `iRecCur*`) inside the
  sample loop so top-level projections can observe the current sample without
  overwriting the previous-sample state too early
- post-output scalar state finalization via one `StoreVar(kStruct, fRec*)`
  instead of the old mandatory two-slot copy
- fallback retention of `TwoSlotShift` and `Circular` recursion carriers for
  multi-output groups and deeper delayed recursion chains
- regression coverage for:
  - simple unary recursion lowering to scalar state
  - unary `delay1` recursion reuse over scalar state
  - integer scalar recursion state
  - multi-output recursion fallback to array-backed carriers
  - C/C++ integration fixtures updated to the scalar-recursion contract

## docs(signal-fir): plan scalar recursion carrier fast path

Commit date: 2026-04-10

Source commit: pending

### Context

The existing Signal -> FIR recursion plan already documented the current
runtime carrier split between:

- `TwoSlotShift` for ordinary one-sample feedback
- `Circular` for widened recursion carriers serving delayed reads

That plan still did not describe the narrower optimization opportunity for
very simple feedback loops such as:

```faust
process = + ~ _;
```

where the recurrence only needs one previous-sample state cell instead of a
two-slot array.

### Documentation

Updated:

- `porting/signal-fir-recursion-abstraction-plan-2026-04-08-en.md`
- `porting/journal/README.md`

The recursion plan now documents:

- a planned `SingleScalar` storage strategy as an optimization-only extension
  of the current recursion lowering model
- why this remains a lowering decision rather than a change to the canonical
  `sigRec/sigProj` signal representation
- conservative eligibility rules for switching from `fRec*[2]`-style storage
  to one scalar state cell
- the intended implementation sequence, fallback policy, and validation matrix
  for simple one-sample feedback loops
