## fix(cranelift): JIT-compile and execute FIR `instanceConstants`

Commit date: 2026-04-04
Source commit: working tree after `56cb70b`

Reworked the Cranelift backend/runtime boundary so `instanceConstants` is no
longer replayed through a Rust-side subset interpreter. The backend now
compiles `instanceConstants` as a real JIT function and the `cranelift_dsp`
runtime invokes that entry directly during instance initialization, aligning
the lifecycle model with the C/C++ backends and covering the full FIR subset
that Cranelift already knows how to lower.

## fix(sigtype): preserve kSamp floor through recursive fixed-point iterations

Commit date: 2026-04-04

### Problem

`update_rec_types` in `crates/sigtype/src/rules.rs` replaced the full
approximation type for each recursive-group component with the freshly inferred
body type on every fixed-point iteration, merging only the **interval**.
Variability, computability, and other lattice dimensions were discarded.

Consequence: if a recursion body ignores its feedback input (e.g. `! : 1`),
the body infers as `Konst`. The `TREC` seed's `Samp` variability was thrown
away on the first iteration, allowing the group тАФ and its `Proj(i, SYMREC)`
nodes тАФ to converge to `Konst`. The variability-driven placement pass in
`signal_fir` would then hoist those projections into `instanceConstants`,
breaking the sample-by-sample feedback contract.

In C++ Faust, `updateRecTypes` calls `joinType` which joins **all** type
dimensions (including variability) with the previous approximation, so
`kKonst тИи kSamp = kSamp` preserves the floor. The Rust port was only doing
the equivalent of the interval join.

### Fix

After inferring the fresh body component, promote its variability and
computability by the old approximation's values before merging the interval:

```rust
component = component
    .promote_variability(old_comp.variability())
    .promote_computability(old_comp.computability());
```

This makes the `kSamp`/`kInit` floor from the `TREC` seed sticky across
every iteration, matching C++ `joinType` semantics.

### Revert of workaround

A prior commit had added `depends_on_recursive_projection` тАФ a transitive
graph walk in the codegen layer тАФ to guard against hoisting expressions derived
from `Proj` nodes. That guard was a symptom-level fix for the type system bug.
It is removed. `is_recursive_projection` is kept as a belt-and-suspenders
defensive check (the type system now guarantees `Proj` nodes are always `Samp`,
so the guard is redundant but harmless).

### Scope

- `crates/sigtype/src/rules.rs` тАФ `update_rec_types`: join variability and
  computability with old approximation before merging interval
- `crates/transform/src/signal_fir/module.rs` тАФ remove
  `depends_on_recursive_projection`, `depends_on_recursive_projection_inner`,
  and `recursive_dependency_cache`; restore `is_recursive_projection` guard
  with updated docstring; remove appended unit test

### Validation

- `cargo test -p sigtype -p transform` тАФ 103 tests pass

## feat(fir/checker): FIR-LC01 тАФ detect struct fields read in instanceConstants before instanceClear initializes them

Commit date: 2026-04-04

### Problem

`phasor.dsp` compiled without error after commit `7a11966`
(variability-driven placement Phase 1), but produced wrong output:

```
Line : 1 output : 2 sample1 : 0.1 different from sample2 : 0.8 delta : 0.7
```

The FIR dump revealed the root cause: the variability placement pass had
hoisted waveform index counter reads (`iWave48`, `iWave26`) into
`instanceConstants` as if they were compile-time constants:

```
instanceConstants(sr):
    fConst0 = fTbl48[iWave48]   тЖР iWave48 read here
    fConst2 = fTbl26[iWave26]   тЖР iWave26 read here

instanceClear():
    iWave48 = 0                 тЖР only initialized HERE, runs later
    iWave26 = 0
```

In C++ this happens to work because the allocator zero-initialises the
struct, and `0` is the correct starting index.  In our JIT/interpreter
path the DSP struct is also zeroed, so the *value* was accidentally
correct тАФ but the pattern is structurally wrong and masked a deeper
signal-typing bug (waveform index counters not carrying `Samp`
variability).

The FIR verifier had no check for this lifecycle ordering violation, so
the bad module sailed through to codegen silently.

### Fix

Three additions to `crates/fir/`:

**`src/lib.rs`** тАФ promoted `child_ids` from `fn` to `pub(crate) fn`
so the checker can drive its own iterative subtree walkers without
duplicating the traversal logic.

**`src/checker.rs`**

- Added `FIR-LC01 | E` to the Phase 2 diagnostics table.
- Added `check_lifecycle_order` method on `VerifyCtx`:
  - Locates `instanceConstants` and `instanceClear` bodies.
  - Computes `cleared_only` = fields stored in `instanceClear` but never
    stored in `instanceConstants`.
  - Walks `instanceConstants` for `LoadVar(kStruct, name)` where
    `name тИИ cleared_only` and emits a `Severity::Error` FIR-LC01 per hit,
    with `fir_function` and `fir_variable` context metadata.
- Added two iterative helpers using `child_ids`:
  - `collect_struct_stores(store, root) тЖТ HashSet<String>`
  - `collect_struct_loads(store, root) тЖТ Vec<(String, FirId)>`
- Called `self.check_lifecycle_order()` at the end of `check_phase2`.

**`src/checker/tests.rs`** тАФ two new tests:
- `lc01_instance_constants_reads_clear_only_field` тАФ reproduces the
  phasor pattern; asserts FIR-LC01 fires as `Error` with correct field
  name and function context.
- `lc01_no_warn_when_field_initialized_in_instance_constants` тАФ field is
  pre-written in `instanceConstants` before being read; asserts no
  FIR-LC01.

### Why Error, not Warning

`maybe_verify_fir_module` only promotes warnings to fatal when
`--fir-verify-strict` is passed.  With the default `strict=false` the
warning was silently dropped and compilation proceeded.  A struct field
read before it is ever written is a correctness bug тАФ not a style
concern тАФ so FIR-LC01 is `Severity::Error`, which triggers the default
`has_errors()` fatal path without any special flag.

  ```
  error [FRS-FIR-0001] struct field 'iWave48' is read in `instanceConstants`
    but is only initialized in `instanceClear` (which runs later)
  ```
