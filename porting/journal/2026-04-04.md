# 2026-04-04

## feat(transform): variability-driven statement placement (Phase 1)

Commit date: 2026-04-04
Plan: `porting/fir-cse-runtime-optimizations-plan-2026-04-03-en.md` §2

Implemented Phase 1 of the FIR runtime optimization plan: variability-driven
statement placement.  Non-trivial expressions whose signal-level variability
is slower than sample rate are now hoisted out of the per-sample loop into the
correct execution-tier bucket.

### What changed

**`crates/transform/src/signal_fir/module.rs`**

- Added `Bucket` enum (`Constants`, `Control`) and free function
  `is_trivial_fir()` to classify FIR value nodes that should never be
  materialized (literals, variable loads, null values).
- Added `const_counter: u32` and `slow_counter: u32` fields to
  `SignalToFirLower` for generating unique `fConst*` / `fSlow*` variable names.
- Added helper methods:
  - `variability_of(sig)` — reads `Variability` from the `sig_types` map.
  - `is_recursive_projection(sig)` — guards against hoisting recursive
    projections that carry sample-by-sample feedback state.
  - `materialize_in_bucket(value, bucket)` — emits a named variable
    (`DeclareVar` for stack / `StoreVar` for struct) in the appropriate
    lifecycle accumulator, returns a `LoadVar` reference.
- Modified `lower_signal()` to insert a variability check after the main
  match dispatch: `Konst` → `constants_statements`, `Block` →
  `control_statements`, `Samp` → stays inline.

### Naming convention (C++ parity)

| Bucket | Prefix | Storage | Lifecycle section |
|--------|--------|---------|-------------------|
| Constants | `fConst` | `AccessType::Struct` | `instanceConstants` |
| Control | `fSlow` | `AccessType::Stack` | `compute` preamble |
| Sample | (inline) | — | `compute` sample loop |

### Edge cases

- **Recursive projections**: guarded by `is_recursive_projection()`; stay in
  the sample loop even if typed `Konst`.
- **Trivial nodes**: literals and variable loads are never hoisted (free to
  duplicate).
- **Control bucket DeclareVar**: `fSlow*` uses `DeclareVar(Stack, init)` so
  that the WASM and Cranelift backends discover the local during their
  pre-scan pass.
