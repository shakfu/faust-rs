## fix(cranelift): JIT-compile and execute FIR `instanceConstants`

Commit date: 2026-04-04
Source commit: working tree after `56cb70b`

Reworked the Cranelift backend/runtime boundary so `instanceConstants` is no
longer replayed through a Rust-side subset interpreter. The backend now
compiles `instanceConstants` as a real JIT function and the `cranelift_dsp`
runtime invokes that entry directly during instance initialization, aligning
the lifecycle model with the C/C++ backends and covering the full FIR subset
that Cranelift already knows how to lower.

### Scope

- updated
  [crates/codegen/src/backends/cranelift/mod.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/backends/cranelift/mod.rs)
  to:
  - generalize the single-function JIT path so it can lower executable FIR
    helpers beyond `compute`
  - compile `instanceConstants` as `module::instanceConstants` when present
  - expose the finalized `instanceConstants` entry address in `JitDspModule`
  - add a backend regression asserting that a module with FIR
    `instanceConstants` gets a live JIT entry
- updated
  [crates/codegen/src/backends/cranelift/tests.rs](/Users/letz/Developpements/RUST/faust-rs/crates/codegen/src/backends/cranelift/tests.rs)
  with a dedicated fixture whose `compute` depends on a struct field written by
  `instanceConstants`
- updated
  [crates/cranelift-ffi/src/instance.rs](/Users/letz/Developpements/RUST/faust-rs/crates/cranelift-ffi/src/instance.rs)
  so `instanceConstantsCCraneliftDSPInstance` calls the JIT-compiled
  `instanceConstants` function instead of interpreting a runtime-local subset
- simplified
  [crates/cranelift-ffi/src/runtime.rs](/Users/letz/Developpements/RUST/faust-rs/crates/cranelift-ffi/src/runtime.rs)
  back to its intended role: runtime descriptor extraction for UI, metadata,
  reset/default state, and sample-rate field tracking, without duplicating FIR
  execution semantics
