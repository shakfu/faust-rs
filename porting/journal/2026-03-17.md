### `signal_fir`: fix `select2` branch inversion

Commit date: 2026-03-17

Source: `crates/transform/src/signal_fir/module.rs`

#### Context

Faust's `select2(sel, x, y)` returns `x` when `sel=0` and `y` when `sel=1`.
The fast-lane lowering destructured `SigMatch::Select2(cond, s1, s2)` as
`(cond, then_value, else_value)`, but in the signal tree `s1` is the `sel=0`
branch (else) and `s2` is the `sel=1` branch (then).  This produced an
inverted ternary (`sel ? x : y` instead of `sel ? y : x`), giving wrong
output for any DSP using `select2`.

#### Changes

Swapped the destructuring at the call site:
`SigMatch::Select2(cond, else_value, then_value)` so the ternary emitted
by `lower_select2` is `sel ? y : x`, matching the reference compiler.

### CLI: accept `-double` legacy flag

Commit date: 2026-03-17

Source: `crates/compiler/src/main.rs`

#### Context

The reference Faust compiler accepts `-double` (single dash) to enable
double-precision mode.  Our CLI only accepted `--double`, breaking
compatibility with existing scripts and workflows.

#### Changes

Added `-double` → `--double` normalization in `normalize_legacy_args`,
following the same pattern used by `-time`, `-lang`, `-cn`, etc.

### Default generated class/struct name to `mydsp`

Commit date: 2026-03-17

Source: `crates/compiler/src/lib.rs`

#### Context

The reference Faust compiler uses `mydsp` as the default C++/C class/struct
name when no `-cn` option is given.  Our compiler was instead deriving the
name from the source filename (e.g. `sine_phasor.dsp` → `sine_phasor`),
which broke parity with the reference output.

#### Changes

1. `resolve_module_name` now returns `"mydsp"` when no explicit class name
   is provided, instead of sanitizing the source filename stem.
2. The cranelift/FFI path (`lower_signals_to_fir`) continues to derive the
   FIR module name from the source filename, keeping bitcode save/restore
   consistent.

### `signal_fir`: use circular buffers with `fIOTA` for delay1 and recursion

Commit date: 2026-03-17

Source: `crates/transform/src/signal_fir/module.rs`

#### Context

The fast-lane `signal_fir` used two different delay strategies:
- **SIGDELAY** (multi-sample): circular buffers with `fIOTA` — correct
- **delay1/recursion**: scalar state + deferred shift in `compute_updates`

The deferred-shift approach caused ordering bugs: depth-first lowering pushed
inner dependency updates before outer ones, so signals like APF.dsp produced
incorrect sample output (e.g. sample[1] was `0.77` instead of the correct
`-0.23`).

The C++ reference `signalFIRCompiler` uses circular buffers with `fIOTA` for
**all** delays uniformly (`writeReadDelay`), which is inherently order-
independent.

#### Changes

1. **`ensure_state_slot`** — delay1 state changed from scalar to 2-element
   array, initialized via `register_clear_recursion_array`.

2. **`lower_delay_state`** — now uses circular buffer read/write:
   - Read previous: `state[(fIOTA - 1) & 1]`
   - Write current: `state[fIOTA & 1] = value` (immediate, in sample body)
   - No deferred shift in `compute_updates`

3. **`lower_proj`** — recursion projections use the same circular buffer
   pattern instead of fixed indices 0/1 with deferred shift.

4. **Removed topological sort** — `sort_compute_updates`, `collect_var_reads`,
   `written_var_name` deleted along with unused imports (`FirMatch`,
   `match_fir`).  Reverted `child_ids` in `crates/fir/src/lib.rs` to private.

5. **Updated tests** — `signal_fir_lane.rs` and `signal_fir/mod.rs` tests
   updated to expect circular buffer patterns (`fIOTA & 1` indexing, single
   store per recursion) instead of scalar shift patterns.

### `signal_fir`: fix delay-line sizing mismatch for multi-delay carried signals

Commit date: 2026-03-17

Source: `crates/transform/src/signal_fir/module.rs`

#### Context

`frenchBell.dsp` (and other DSPs with modal synthesis) failed with:

```
[FRS-SFIR-0004] internal fast-lane delay-line sizing mismatch for signal 3282:
existing size 2 < required 4
```

The error occurred because `prepare_delay_lines` allocated delay-line buffers
eagerly during a single-pass DAG walk.  When the same carried signal appeared in
multiple `SIGDELAY` nodes with different delay amounts, the first (smaller) delay
determined the buffer size; encountering a larger delay for the same signal then
triggered a fatal sizing-mismatch error.

#### Changes

1. **Two-pass scan** — `prepare_delay_lines` now separates collection from
   allocation.  The first pass (`scan_delay_lines`) walks the DAG and records
   `max(delay)` per carried signal into a `HashMap<SigId, i32>`.  The second
   pass iterates the map and calls `ensure_delay_line_decl` once per signal
   with the correct (maximum) size.

2. **Signature update** — `scan_delay_lines` and `scan_delay_child` now take
   a `&mut HashMap<SigId, i32>` parameter instead of calling
   `ensure_delay_line_decl` directly.

#### Note

After this fix, `frenchBell.dsp` hits a separate stack-overflow issue (infinite
recursion during lowering), which is unrelated to the delay-line sizing and
needs separate investigation.
