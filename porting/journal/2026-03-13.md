### `interval` — Port the full C++ interval library (Steps 1–5 of plan)

Commit date: 2026-03-13

Source commit: interval-port-2026-03-13

#### Context

The `crates/interval` crate was a scaffold with no real implementation.
Downstream transforms (`signal_prepare`, `signal_fir`) were blocked on
interval facts needed for variable `SIGDELAY` sizing and arithmetic safety
checks.

The porting plan (`porting/interval-port-plan-2026-03-12-en.md`) identified
the full C++ interval subsystem under `compiler/interval/` as the target.
This commit delivers plan Steps 1–5 (standalone library, all operators,
test matrix, stable public API).

#### C++ reference

- `compiler/interval/interval_def.hh` — `itv::interval` type and set
  operations.
- `compiler/interval/interval_algebra.hh` — algebra surface (all operator
  declarations).
- `compiler/interval/interval*.cpp` — one concrete implementation per
  operator family.
- `compiler/interval/bitwiseOperations.{hh,cpp}` — `SInterval`/`UInterval`
  structures and MSB-split OR algorithm.
- `compiler/interval/utils.hh` and `compiler/interval/precision_utils.hh` —
  `min4`/`max4`, `minValAbs`/`maxValAbs`, `exactPrecisionUnary`.
- `compiler/interval/check.hh` — `taylor_lsb` constant (false), test harness.

#### Design decisions

- `exact_precision_unary` takes `fn(f64) -> f64` (no closures with captures)
  to avoid allocations; calling code inlines captured-variable computations
  directly.
- `gen` is a reserved keyword in Rust 2024; exported as `r#gen`.
- `not` uses the analytic `[~hi, ~lo]` formula instead of the C++ brute-force
  loop (equivalent result, O(1) instead of O(n)).
- Placeholder semantics are preserved explicitly and documented per the plan
  requirement (risk 11.3).

### `sigtype` — Plan for porting the full C++ signal type system

Commit date: 2026-03-13

Source commit: sigtype-port-plan-2026-03-13

#### Context

Discussion of Steps 6–7 of the interval port plan revealed that attaching
intervals to prepared signals as a separate `HashMap<SigId, Interval>` is
an architectural deviation from the C++ compiler.

In C++, `AudioType` (the base of `SimpleType`, `TableType`, `TupletType`)
carries `fInterval` directly as a member field. Type and interval are a single
object; every `promote*` operation returns a new type preserving the interval,
and the type inference engine (`sigtyperules.cpp`) calls `getInterval()` on the
type of each sub-expression to propagate bounds through the operator algebra.

Separating them in Rust would require keeping two maps in sync through every
type promotion — a reliable source of divergence bugs.

The correct architecture is to port the full C++ type system and replace
`SimpleSigType { Int, Real, Sound }` with a proper `SigType` that embeds
`Interval`.

#### Plan

See `porting/sigtype-port-plan-2026-03-13-en.md` for the full plan.

Summary:
- New crate `crates/sigtype` depending on `signals` and `interval`.
- Enums: `Nature`, `Variability`, `Computability`, `Vectorability`, `Boolean`
  with bitwise-OR `join` semantics matching the C++ lattice.
- Struct `Res { valid: bool, index: i32 }` for fixed-point precision.
- Types: `SimpleType`, `TableType`, `TupletType` unified under `SigType` enum.
- Value semantics; `PartialEq` ignores `res` (matches C++, ensures fixpoint
  convergence).
- Operators: `union_types`, `product_types`, cast helpers (`int_cast`,
  `bit_cast`, `float_cast`, `samp_cast`, `bool_cast`, `num_cast`,
  `cast_interval`), merge functions, check assertions.
- Factories: `make_simple`, `make_table_type`, `make_tuplet` (no global
  memoization — Rust value equality replaces C++ pointer identity).
- `TypeAnnotator` (port of `sigtyperules.cpp`): bottom-up `infer` dispatch on
  `SigMatch` + recursive fixed-point loop for `SymRec`/`Proj` groups.
- Migration: `signal_prepare.rs` uses `SigType` from `sigtype`; `SimpleTyper`,
  `TypeSlot`, `SimpleSigType` are deleted.

#### Rationale for a new crate rather than extending `signals`

`sigtyperules.cpp` dispatches on all signal node kinds (`SigMatch`), so it must
depend on the `signals` crate. But `signals` must not depend on the type system
(circular). A dedicated `sigtype` crate sits between `signals` and `transform`
in the dependency graph.
