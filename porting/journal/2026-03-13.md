### `porting` — Freeze pathname parity analysis and the Rust relative-group extension plan in the UI IR contract

Commit date: 2026-03-13

Source commit: pending

#### Context

The grouped-UI rewrite had already frozen ownership around `UiProgram`, label
metadata extraction, and FIR/backend emission from explicit grouped UI.

One remaining area was still under-specified in the architecture note:

- Faust label-as-pathname semantics for widgets
- the current Rust parity gap on widget pathnames such as `../volume`
- whether Rust should stay strictly at current C++ behavior for groups or
  extend the model to support relative path navigation in explicit group labels

The requested direction was to document the C++ baseline accurately, then
freeze a coherent Rust plan that preserves widget parity first and allows a
limited, explicit Rust-only extension for relative group rebasing.

#### Changes

- [ui-ir-architecture-contract-2026-03-12-en.md](../../porting/ui-ir-architecture-contract-2026-03-12-en.md)
  - expanded the ownership contract so UI IR now explicitly owns:
    - pathname-normalized structural placement
    - terminal display labels
    - extracted metadata
  - documented the C++ pathname baseline with provenance to:
    - `compiler/propagate/labels.cpp`
    - `compiler/propagate/propagate.cpp`
    - `compiler/generator/uitree.cpp`
    - `compiler/generator/compile.cpp`
  - recorded the observed C++ rule that widget labels are normalized with
    `normalizePath(...)`, while explicit group labels are still treated as
    direct folder labels
  - documented the current Rust parity gap:
    - pathname grammar is not yet parsed in grouped UI construction
    - labels such as `../volume` are still emitted literally
    - the current recursive `collect_ui_nodes(...) -> Vec<UiId>` shape is not
      sufficient for rebasing insertions
  - froze the canonical correction ordering:
    - raw evaluated label
    - pathname normalization
    - metadata extraction
    - canonical `UiProgram`
  - added a concrete implementation plan by file and by slice, covering:
    - new pathname-focused DSP fixtures under `tests/corpus/`
    - failing tests
    - pathname parser/normalizer ownership
    - path-aware `UiProgram` insertion helpers
    - `propagate` refactor points
    - Rust-only group-label extension rollout
  - froze an `adapted` Rust-only extension plan for explicit group labels:
    - navigation-only prefixes (`./`, `../`, `/`)
    - no typed path segments inside group labels yet
    - no arbitrary intermediate group synthesis from `foo/bar`
    - clamp above-root navigation at the canonical root
  - extended API classification, validation requirements, and follow-up slices
    for pathname work

- [README.md](../../porting/journal/README.md)
  - added the `2026-03-13` journal day
  - corrected the visible `2026-03-12` entry count in the day index

#### Outcome

The grouped-UI architecture contract now distinguishes clearly between:

- C++ pathname parity that Rust must match for widgets
- current C++ non-support for relative group-label rebasing
- the deliberate Rust extension scope for navigation-only relative group
  pathnames

This gives the future pathname implementation a precise semantic target before
any code refactor begins.
