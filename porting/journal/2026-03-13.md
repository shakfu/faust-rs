### `ui/propagate` — Add the Rust-only relative group-label rebasing extension

Commit date: 2026-03-13

Source commit: pending

#### Context

Widget pathname parity was now closed, but the architecture contract also froze
one deliberate Rust extension on top of the C++ baseline:

- explicit group labels may use navigation-only pathname prefixes
  (`./`, `../`, `/`)
- rebasing changes the insertion site of the explicit group itself
- rebased explicit groups may leave an ancestor chain visible even when that
  ancestor becomes structurally empty after the rebase

The tricky part was preserving deterministic source order without accidentally
emitting all empty groups. `propagate` therefore needed explicit group
placeholders plus a conditional prune step, rather than returning to the old
purely recursive `Vec<UiId>` model.

#### Changes

- [ui/src/lib.rs](../../crates/ui/src/lib.rs)
  - extended `UiProgramBuilder` with:
    - `find_group_path(...)`
    - `group_has_children(...)`
    - `remove_group_if_empty(...)`
  - documented these helpers as the structural support needed for explicit
    group placeholders reserved during `propagate`

- [ui/tests/core_api.rs](../../crates/ui/tests/core_api.rs)
  - added a structural regression test proving empty group placeholders can be
    removed while populated paths remain intact

- [propagate/src/lib.rs](../../crates/propagate/src/lib.rs)
  - introduced `UiCollectSummary { has_ui, preserve_ancestor_chain }`
  - switched explicit group traversal to:
    - normalize the group label with `normalize_group_label_navigation(...)`
    - reserve the explicit group path in source order
    - recurse through descendants under the normalized parent path
    - keep or prune the placeholder depending on actual subtree emission and
      ancestor-preservation needs
  - documented the adapted behavior relative to C++: explicit group rebasing is
    now Rust-only, while widget pathname parity remains unchanged

- [propagate/tests/core_api.rs](../../crates/propagate/tests/core_api.rs)
  - added direct grouped-UI coverage for:
    - `vgroup("../Bar", ...)` rebasing to the parent
    - root-clamped rebasing with repeated `../../..`

- [tests/corpus/rep_63_ui_relative_group_rebase.dsp](../../tests/corpus/rep_63_ui_relative_group_rebase.dsp)
- [tests/corpus/rep_64_ui_relative_group_root_clamp.dsp](../../tests/corpus/rep_64_ui_relative_group_root_clamp.dsp)
  - added dedicated corpus fixtures for the Rust-only extension

- [compiler/tests/signal_pipeline.rs](../../crates/compiler/tests/signal_pipeline.rs)
  - added compiler-level assertions for both new extension fixtures, including
    synthesized-root labeling and preserved empty ancestor groups

- [tests/golden/rust/rep_63_ui_relative_group_rebase/compiler_stdout.txt](../../tests/golden/rust/rep_63_ui_relative_group_rebase/compiler_stdout.txt)
- [tests/golden/rust/rep_64_ui_relative_group_root_clamp/compiler_stdout.txt](../../tests/golden/rust/rep_64_ui_relative_group_root_clamp/compiler_stdout.txt)

### `propagate/ui` — Normalize widget pathnames into grouped UI construction

Commit date: 2026-03-13

Source commit: fa14139

#### Context

The previous UI slice added pathname parsing and a path-aware `UiProgramBuilder`
inside `crates/ui`, but grouped UI extraction in `propagate` still rebuilt UI
with the old recursive `Vec<UiId>` collector.

That older path could only preserve explicit group nesting literally. It could
not reproduce current Faust C++ widget-pathname behavior such as:

- rebasing `../volume` out of an intermediate group
- creating canonical typed ancestors from labels like `h:Oscillator/freq`
- extracting inline metadata only after pathname normalization

The next required parity step was therefore to move grouped UI construction in
`propagate` onto the new path-aware builder while keeping the scope limited to
current C++ widget behavior. Explicit relative group-label rebasing remains a
later Rust-only extension.

#### Changes

- [propagate/src/lib.rs](../../crates/propagate/src/lib.rs)
  - replaced the old `TreeArena`-local `UiCollector` node assembly with
    `UiProgramBuilder`
  - normalized widget, bargraph, and soundfile labels through
    `normalize_widget_label_path(...)` before metadata extraction
  - converted the explicit group traversal into path-context accumulation for
    descendants instead of eager group node materialization
  - documented the temporary parity rule that groups escaped through widget
    `../` rebasing are omitted, matching current C++
  - preserved root-group synthesis and empty-root renaming policy after the new
    builder materialization step

### `ui` — Add pathname normalization helpers and a path-aware grouped UI builder

Commit date: 2026-03-13

Source commit: pending

#### Context

Implementing pathname-aware grouped UI in `propagate` needed two missing
primitives first:

- a Rust-side pathname model that mirrors the C++ widget normalization rules
  closely enough to test in isolation
- a path-aware grouped UI builder capable of inserting groups and controls
  under normalized target paths, instead of only building direct recursive
  subtrees

Without these pieces, the existing `collect_ui_nodes(...) -> Vec<UiId>` flow in
`propagate` had no clean place to express rebasing such as `../volume` or the
planned Rust-only relative group-label extension.

#### Changes

- [ui/src/lib.rs](../../crates/ui/src/lib.rs)
  - added raw pathname carriers:
    - `UiGroupPathSegment`
    - `UiNormalizedWidgetPath`
    - `UiNormalizedGroupPath`
  - added canonical stored group carrier:
    - `UiGroupSpec`
  - added pathname normalization helpers with Rustdoc provenance to
    `labels.cpp`:
    - `normalize_widget_label_path(...)`
    - `normalize_group_label_navigation(...)`
    - `canonicalize_group_spec(...)`
  - implemented sequential pathname operation handling so mixed forms such as
    typed segments followed by `../` normalize correctly
  - added `UiProgramBuilder`, a path-aware grouped UI builder that:
    - merges repeated group paths deterministically
    - preserves leaf insertion order
    - materializes the resulting forest back into `TreeArena`/`UiId`

- [ui/tests/core_api.rs](../../crates/ui/tests/core_api.rs)
  - added isolated coverage for:
    - relative widget rebasing
    - typed widget pathname normalization
    - relative group-label rebasing and root clamp
    - metadata extraction after pathname normalization
    - deterministic `UiProgramBuilder` insertion/merge behavior

### `porting` — Freeze pathname parity analysis and the Rust relative-group extension plan in the UI IR contract

Commit date: 2026-03-13

Source commit: pending

#### Context

The grouped-UI rewrite had already frozen ownership around `UiProgram`, label
metadata extraction, and FIR/backend emission from explicit grouped UI.

One remaining area was still under-specified in the architecture note:

- Faust label-as-pathname semantics for widgets
- the current Rust parity gap on widget pathnames such as `../volume`
- whether Rust should stay strictly at current C++ behavior for groups or
  extend the model to support relative path navigation in explicit group labels

The requested direction was to document the C++ baseline accurately, then
freeze a coherent Rust plan that preserves widget parity first and allows a
limited, explicit Rust-only extension for relative group rebasing.

#### Changes
