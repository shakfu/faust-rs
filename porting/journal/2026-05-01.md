# 2026-05-01

## fix(cranelift): remove over-aggressive AArch64 CLIF size guard

Commit date: 2026-05-01

Source commit: this commit

Commit `7fa314a` introduced a post-lowering CLIF text size guard
(`should_fallback_large_aarch64_compute`) that discarded a successfully
JIT-compiled module and fell back to the no-op stub whenever the `compute`
function's CLIF text exceeded 32 KiB on AArch64. This silenced any
moderately complex DSP — `STunedBar.dsp` being the first reported victim —
because 32 KiB of CLIF text is a normal size for a non-trivial Faust
`compute` body and bears no reliable relationship to whether the generated
machine code will actually trigger the branch-range overflow.

The guard was also logically wrong: if `try_generate_cranelift_module`
returned `Ok(compiled)` the JIT had already finalized successfully, meaning
all branch veneer insertions had been resolved without a panic. Discarding
that result and replacing it with a no-op stub only produced silence.

Removed:
- `AARCH64_MAX_SAFE_COMPUTE_CLIF_BYTES` constant
- `compute_clif_exceeds_aarch64_safe_size` function
- `should_fallback_large_aarch64_compute` function
- the `Ok(Ok(compiled))` arm that invoked the guard
- the corresponding unit test `aarch64_large_compute_clif_guard_has_strict_threshold`

The `catch_unwind` path added in `7fa314a` for the actual panic case
(AArch64 branch-range overflow) is preserved and remains the sole protection
for `fad_tracking3.dsp`-class crashes.

### Validation

- `cargo test -p codegen cranelift`
- `cargo test -p codegen`
