## fix(cranelift): remove over-aggressive AArch64 CLIF size guard

Commit date: 2026-05-01

Source commit: af925ab

Commit `7fa314a` introduced a post-lowering CLIF text size guard
(`should_fallback_large_aarch64_compute`) that discarded a successfully
JIT-compiled module and fell back to the no-op stub whenever the `compute`
function's CLIF text exceeded 32 KiB on AArch64. This silenced any
moderately complex DSP — `STunedBar.dsp` being the first reported victim —
because 32 KiB of CLIF text is a normal size for a non-trivial Faust
`compute` body and bears no reliable relationship to whether the generated
machine code will actually trigger the branch-range overflow.

The guard was also logically wrong: if `try_generate_cranelift_module`
returned `Ok(compiled)` the JIT had already finalized successfully, meaning
all branch veneer insertions had been resolved without a panic. Discarding
that result and replacing it with a no-op stub only produced silence.

Removed:
- `AARCH64_MAX_SAFE_COMPUTE_CLIF_BYTES` constant
- `compute_clif_exceeds_aarch64_safe_size` function
- `should_fallback_large_aarch64_compute` function
- the `Ok(Ok(compiled))` arm that invoked the guard
- the corresponding unit test `aarch64_large_compute_clif_guard_has_strict_threshold`

The `catch_unwind` path added in `7fa314a` for the actual panic case
(AArch64 branch-range overflow) is preserved and remains the sole protection
for `fad_tracking3.dsp`-class crashes.

### Validation

- `cargo test -p codegen cranelift`
- `cargo test -p codegen`

## fix(normalize): add x - x → 0 simplification rule

Commit date: 2026-05-01

Source commit: this commit

Analysis of `fad_rec.dsp` (where both outputs should reduce to zero because
`process_fad` and `fad(process_original, (g))` are semantically equivalent)
revealed two independent causes preventing simplification:

**Cause 1 (structural — not fixed here):** `SIGREC(body)` nodes are interned
by content, but their bodies contain `SIGPROJ(i, group)` nodes that reference
the Rec's own SigId. Two separately-constructed recursive systems therefore
always get distinct SigIds even when semantically identical, so the `t1 == t2`
guard in the self-operation rule block can never fire across them. Fixing
this requires a recursive graph isomorphism pass upstream of simplification.

**Cause 2 (rule gap — fixed here):** Even when `t1 == t2` (same SigId on both
sides of a subtraction, as happens with plain splits), `BinOp::Sub` was absent
from the self-operation rule block in `simplification()`. All other symmetric
operations (`Rem`, `Xor`, `Gt`, `Lt`, …) already reduced `x op x` to a
constant; `Sub` was the only missing case.

Added `BinOp::Sub => return SigBuilder::new(arena).int(0)` to the block and a
covering unit test `simplify_sub_self_is_zero`.

### Validation

- `cargo test -p normalize simplify_sub_self_is_zero`
- `cargo test -p normalize`

## docs(porting): plan for isomorphic Rec group merging in normalize

Commit date: 2026-05-01

Source commit: 715c390

The `x - x → 0` rule fixed in the previous commit cannot fire on `fad_rec.dsp`
because the two recursive systems (`process_fad` and `fad(process_original, (g))`)
are constructed by independent pipeline stages and therefore always receive
distinct `SigId`s, even though they implement the same recurrence.  The root
cause is that `SIGREC(body)` is hash-consed by content, but `body` contains
`SIGPROJ(i, R)` back-edges pointing to `R` itself; two separately-built Rec
nodes that reference their own distinct identifiers cannot be unified by the
arena's hash-consing alone.

Wrote a design plan for a new pre-simplification pass in `crates/normalize/`:

- **Opening a Rec**: replace every `Proj(i, R)` in the body by a canonical
  sentinel `Hole(i)` to produce an acyclic DAG.  Two Rec nodes are isomorphic
  iff their opened DAGs hash-cons to the same `SigId`.
- **New module** `normalize/src/rec_merge.rs`, entry point
  `merge_isomorphic_rec_groups(arena, roots) -> Vec<SigId>`, called before
  `simplify` in the normalize pipeline.
- **Steps**: collect reachable Rec nodes → open each → group by opened
  signature → elect canonical representative → apply substitution over the
  full signal forest → run simplify (the Sub rule then fires).
- **Complexity**: O(N + R·D) — linear in graph size, negligible in practice.
- **6 planned unit tests** covering single-output, multi-output, non-isomorphic
  pairs, full round-trip to `Int(0)`, nested groups, and idempotence.

Plan file: `porting/normalize-rec-isomorphism-merge-plan-2026-05-01-en.md`.

## fix(propagate): silence clippy::too_many_arguments on input/output_control

Commit date: 2026-05-01

Source commit: 07a71e3

`cargo clippy --all-targets -- -D warnings` failed on `crates/propagate/src/lib.rs`
with two `clippy::too_many_arguments` errors on the private helper methods
`input_control` and `output_control` (8 parameters each, limit is 7).

Both methods forward UI control registration to `register_control` and
`UiBuilder`; introducing an intermediate struct would add abstraction without
benefit for private helpers of this kind.  Added
`#[allow(clippy::too_many_arguments)]` to each function.
