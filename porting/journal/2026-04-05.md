# 2026-04-05

## fix(sigtype): SIGWAVEFORM must carry `Samp` variability, not `Konst`

Commit date: 2026-04-05

### Problem

`SIGWAVEFORM` was typed with `Variability::Konst` via `make_table_type`, which
hardcodes `kKonst` matching C++ Faust's classification of the **static table**.
In the Rust signal graph after propagation, however, `SIGWAVEFORM` represents
the **cycling output** — each sample yields the next element (advancing an
implicit counter modulo the table length).  The output changes every sample,
so its variability must be `Samp`.

With `Konst` variability, the placement pass in `signal_fir/module.rs`
hoisted every expression that depends on the waveform into `instanceConstants`.
For `phasor.dsp` this included `clk = reset > 0` (a `SigBinOp` inheriting
`max(Konst, Konst) = Konst`), dragging `LoadTable(fTbl48, iWave48)` inside
`instanceConstants` as a sub-expression.  The FIR-LC01 verifier (added in the
previous commit) caught this as a lifecycle-order violation and aborted
compilation with an error.

### Root cause

`make_table_type` correctly classifies the **data** as compile-time constant;
the bug was applying that classification to the **signal output** as well.  The
node-level fix must be in sigtype: promote the result to `Samp` so that all
dependent expressions inherit the correct floor via the lattice join
`max(Samp, ·) = Samp`.

### Fix — `crates/sigtype/src/rules.rs`

```rust
// empty waveform
Variability::Samp   // was Variability::Konst

// non-empty waveform
Ok(make_table_type(t.promote_interval(itv)).promote_variability(Variability::Samp))
// was: Ok(make_table_type(t.promote_interval(itv)))
```

Both cases now carry `Samp`, so `clk`, `reset > 0`, and every other expression
derived from a waveform stays in the sample loop where it belongs.

### What was considered but rejected

A belt-and-suspenders guard `&& !matches!(match_sig(..., sig), SigMatch::Waveform(_))`
was briefly added to the placement `if`-condition in `module.rs` to protect the
`SigWaveform` node itself from hoisting.  It was reverted because:

1. **Dead code after the type fix.** With `Samp` propagating from sigtype, the
   placement match always falls through to `_ => lowered` for waveform nodes.
   The guard can never fire.
2. **Wrong in principle.** Placement logic is supposed to be driven entirely by
   the type system.  A structural bypass that says "even if variability says
   Konst, don't hoist this node kind" works around a type-system bug rather
   than fixing it.  Once the type system is correct the bypass becomes a lie
   about the architecture.
3. **It protects only the waveform node itself**, not parent expressions that
   inherit the wrong variability.  The root fix had to be in sigtype anyway.

### Scope

- `crates/sigtype/src/rules.rs` — `SigMatch::Waveform` arm: empty case
  `Variability::Konst → Variability::Samp`; non-empty case promotes result
  with `.promote_variability(Variability::Samp)`; extensive inline comment
  explaining the cycling-output vs. static-table distinction.

### Validation

- `cargo test --workspace` — all tests pass (0 failures)
- `faust-rs phasor.dsp` no longer emits FIR-LC01 errors
- `instanceConstants` FIR dump for `phasor.dsp` contains only SR-derived
  constants (`fConst0/1/2/3`); no `iWave*` or table reads appear
