## feat(transform): CSE materialization pass (Phase 2 of FIR runtime optimizations)

Commit date: 2026-04-05

### Overview

Implements Phase 2 of the FIR runtime optimization plan: intra-bucket Common
Subexpression Elimination (CSE).  After variability-driven placement (Phase 1)
has routed each expression to the correct execution tier, CSE detects
multi-referenced non-trivial value nodes within each bucket and materializes
them into named temporary variables (`DeclareVar` + `LoadVar`).

### How it works

1. **Reference counting** — for each bucket (`constants_statements`,
   `control_statements`, `sample_statements`), walk the statement trees and
   count how many times each `FirId` value node is referenced as a child.
   Hash-consing ensures structurally identical subtrees share the same `FirId`,
   so fan-out detection is automatic.

2. **Bottom-up rewrite** — process statements in order.  For each statement,
   recursively rewrite its value children.  When a non-trivial value node has
   `ref_count ≥ 2`, emit a `DeclareVar(prefix<N>)` and replace all subsequent
   references with `LoadVar(prefix<N>)`.

3. **Insertion at point of first use** — declarations are emitted immediately
   before the statement that first triggers them, preserving sequential data
   dependencies between statements.  (An earlier version prepended all
   declarations before all statements, which broke `constants_statements` where
   `fConst1` may depend on a struct field stored by `fConst0`'s init statement.)

### Naming conventions

| Bucket | Prefix | Counter start |
|--------|--------|---------------|
| `constants_statements` | `fConst` | from Phase 1's `const_counter` |
| `control_statements` | `fSlow` | from Phase 1's `slow_counter` |
| `sample_statements` | `fTemp` | 0 |

### Trivial-node filter

Literals (`Int32`, `Float32`, …), variable loads (`LoadVar`, `LoadVarAddress`),
and `NullValue` are never materialized — they are free to duplicate.

### Example effect

`rep_38_sine_phasor.dsp` — two phasor recursions share circular-buffer index
expressions `(fIOTA & 1)` and `((fIOTA - 1) & 1)`.  Before CSE, each was
expanded inline at 2 use sites (4 evaluations total).  After CSE:

```cpp
int fTemp0 = (fIOTA & 1);
int fTemp1 = ((fIOTA - 1) & 1);
fRec76[fTemp0] = (fRec76[fTemp1] + fSlow2);
output0[i0] = ...fRec76[fTemp0]...;
fRec83[fTemp0] = (fRec83[fTemp1] + fSlow4);
output1[i0] = ...fRec83[fTemp0]...;
```

### Scope

- **New**: `crates/transform/src/signal_fir/cse.rs` — `count_fir_value_uses()`,
  `materialize_shared_values()`, `rewrite_stmt()`, `rewrite_value()`,
  `rewrite_value_children()`, `value_children_of()`, `is_trivial_value()`.
- **Modified**: `crates/transform/src/signal_fir/mod.rs` — register `mod cse`.
- **Modified**: `crates/transform/src/signal_fir/module.rs` — call CSE on all
  three buckets after lowering, before FIR module assembly.
- **Modified**: `crates/compiler/tests/signal_fir_lane.rs` — relax sine phasor
  test assertion to accept both inline and CSE-materialized circular-buffer
  index patterns.

### Validation

- `cargo test --workspace` — all tests pass (0 failures)
- Impulse-test regression fixed (initial version broke APF due to declaration
  prepending; fixed by inserting at point of first use)

## fix(sigtype): SIGWAVEFORM must carry `Samp` variability, not `Konst`

Commit date: 2026-04-05

### Problem

`SIGWAVEFORM` was typed with `Variability::Konst` via `make_table_type`, which
hardcodes `kKonst` matching C++ Faust's classification of the **static table**.
In the Rust signal graph after propagation, however, `SIGWAVEFORM` represents
the **cycling output** — each sample yields the next element (advancing an
implicit counter modulo the table length).  The output changes every sample,
so its variability must be `Samp`.

With `Konst` variability, the placement pass in `signal_fir/module.rs`
hoisted every expression that depends on the waveform into `instanceConstants`.
For `phasor.dsp` this included `clk = reset > 0` (a `SigBinOp` inheriting
`max(Konst, Konst) = Konst`), dragging `LoadTable(fTbl48, iWave48)` inside
`instanceConstants` as a sub-expression.  The FIR-LC01 verifier (added in the
previous commit) caught this as a lifecycle-order violation and aborted
compilation with an error.

### Root cause

`make_table_type` correctly classifies the **data** as compile-time constant;
the bug was applying that classification to the **signal output** as well.  The
node-level fix must be in sigtype: promote the result to `Samp` so that all
dependent expressions inherit the correct floor via the lattice join
`max(Samp, ·) = Samp`.

### Fix — `crates/sigtype/src/rules.rs`

```rust
// empty waveform
Variability::Samp   // was Variability::Konst

// non-empty waveform
Ok(make_table_type(t.promote_interval(itv)).promote_variability(Variability::Samp))
// was: Ok(make_table_type(t.promote_interval(itv)))
```

Both cases now carry `Samp`, so `clk`, `reset > 0`, and every other expression
derived from a waveform stays in the sample loop where it belongs.

### What was considered but rejected

A belt-and-suspenders guard `&& !matches!(match_sig(..., sig), SigMatch::Waveform(_))`
was briefly added to the placement `if`-condition in `module.rs` to protect the
`SigWaveform` node itself from hoisting.  It was reverted because:

1. **Dead code after the type fix.** With `Samp` propagating from sigtype, the
   placement match always falls through to `_ => lowered` for waveform nodes.
   The guard can never fire.
2. **Wrong in principle.** Placement logic is supposed to be driven entirely by
   the type system.  A structural bypass that says "even if variability says
   Konst, don't hoist this node kind" works around a type-system bug rather
   than fixing it.  Once the type system is correct the bypass becomes a lie
   about the architecture.
3. **It protects only the waveform node itself**, not parent expressions that
   inherit the wrong variability.  The root fix had to be in sigtype anyway.

### Scope

- `crates/sigtype/src/rules.rs` — `SigMatch::Waveform` arm: empty case
  `Variability::Konst → Variability::Samp`; non-empty case promotes result
  with `.promote_variability(Variability::Samp)`; extensive inline comment
  explaining the cycling-output vs. static-table distinction.

### Validation

- `cargo test --workspace` — all tests pass (0 failures)
- `faust-rs phasor.dsp` no longer emits FIR-LC01 errors
- `instanceConstants` FIR dump for `phasor.dsp` contains only SR-derived
  constants (`fConst0/1/2/3`); no `iWave*` or table reads appear

## fix(transform): skip Phase 1 hoisting for SIGWRTBL nodes

Commit date: 2026-04-05

### Problem

`table1.dsp` failed with FIR-LC01: `iWave54` was read inside
`instanceConstants` before `instanceClear` had initialized it.

### Root cause

`SIGWRTBL` inherits `Konst` variability from `make_table_type`, which
correctly describes the **static table content**.  However, `lower_wrtbl`
returns the **write signal's value** — an expression that may reference
`iWave*` cycling counters (Samp-rate state allocated in `instanceClear`).

Phase 1 variability-driven placement saw `Konst` and hoisted the
expression into `instanceConstants`, placing a `LoadVar("iWave54")` before
the variable existed.

### Fix — `crates/transform/src/signal_fir/module.rs`

Added a guard to the Phase 1 placement check:

```rust
let lowered = if !is_trivial_fir(&self.store, lowered)
    && !self.is_recursive_projection(sig)
    && !matches!(match_sig(self.arena, sig), SigMatch::WrTbl(..))
{
```

`SIGWRTBL` nodes now always stay inline (in the sample loop), bypassing
the variability-based hoist.

### What was considered but deferred

The principled fix is in sigtype: `infer_write_table_type` should promote
the result variability to `max(tbl_var, ws_var)` so the write signal
inherits the cycling counter's `Samp` floor.  This was deferred because:
1. The guard is safe and narrowly scoped.
2. Changing sigtype rules requires verifying downstream impacts across
   all backends and the recursive fixed-point iteration.

### Scope

- `crates/transform/src/signal_fir/module.rs` — Phase 1 placement guard

### Validation

- `cargo test --workspace` — all tests pass (0 failures)
- `faust-rs dsp/table1.dsp` compiles without FIR-LC01 errors

## fix(sigtype): remove spurious `samp_cast` from pure math type inference

Commit date: 2026-04-05

### Problem

`sin(ma.SR)` was evaluated inside the sample loop (`std::sin(fConst2)`)
instead of being precomputed once in `instanceConstants`.

For `process = *(sin(ma.SR)*button("gate"))`, faust-rs generated:

```cpp
for (int i0 = 0; i0 < count; ++i0) {
    output0[i0] = (fSlow0 * (input0[i0] * std::sin(fConst2)));  // sin every sample!
}
```

While C++ Faust correctly hoists the `sin` call:

```cpp
void instanceConstants(int sample_rate) {
    fConst0 = std::sin(std::min<float>(1.92e+05f, std::max<float>(1.0f, float(fSampleRate))));
}
```

### Root cause — porting bug in `infer_unary_math`

`infer_unary_math` applied `samp_cast(tx)` unconditionally, promoting the
result variability to `Samp` regardless of the argument's variability.
This affected all transcendental and math operations: `sin`, `cos`, `tan`,
`asin`, `acos`, `atan`, `sqrt`, `exp`, `log`, `log10`, `floor`, `ceil`,
`rint`, `round`.  The same bug was present in binary math ops (`atan2`,
`fmod`, `remainder`).

The C++ Faust source (`sinprim.hh`, `cosprim.hh`, `sqrtprim.hh`, etc.)
uses `floatCast(t)` — **without** `sampCast` — for all pure math
primitives.  The `inferSigType` pattern is always:

```cpp
return castInterval(floatCast(t), gAlgebra.Sin(i));
```

The variability of the argument is preserved intact; only the nature is
promoted to Real.  The Rust port incorrectly added `samp_cast`.

### Fix — `crates/sigtype/src/rules.rs`

**`infer_unary_math`**: replaced `samp_cast(tx)` with plain `tx`:

```rust
Ok(float_cast(tx).promote_interval(itv))
// was: Ok(float_cast(samp_cast(tx)).promote_interval(itv))
```

**Binary math** (`Atan2`, `Fmod`, `Remainder`): removed `samp_cast` from
the `union_types` result, preserving argument-derived variability.

Pure math functions are referentially transparent — `sin(Konst) → Konst`,
`sin(Block) → Block`, `sin(Samp) → Samp`.

### Scope

- `crates/sigtype/src/rules.rs` — `infer_unary_math`, `Atan2`, `Fmod`,
  `Remainder` arms

### Validation

- `cargo test --workspace` — all tests pass (0 failures)
- `faust-rs t10.dsp`: `std::sin(fConst2)` now in `instanceConstants`,
  sample loop reduced to `fSlow1 * input0[i0]`

## perf(transform): signal-sharing-aware Phase 1 placement

Commit date: 2026-04-05

### Problem

Phase 1 variability-driven placement materialized **every** non-trivial
Block/Konst signal node into a named variable (`fSlow*`/`fConst*`),
regardless of whether the node was shared in the signal DAG.  For
`STunedBar.dsp`, this created 285 `fSlow` variables where C++ Faust only
needed 43.

Each intermediate sub-expression in a Block-rate chain (e.g.
`cos(fConst * slider)`) got its own variable: `fSlow9 = fConst10 * fSlow4`,
`fSlow10 = cos(fSlow9)`, `fSlow11 = fConst9 * fSlow10`.  C++ Faust
composes these as one expression: `fSlow4 = fConst6 * cos(fConst7 * fSlow2)`.

### Root cause

`materialize_in_bucket` was called unconditionally for every non-trivial
node with Block/Konst variability.  C++ Faust only creates named variables
for **shared** signal sub-trees (referenced ≥ 2 times in the DAG).
Single-use intermediate nodes are inlined into their parent's expression.

### Fix — `crates/transform/src/signal_fir/module.rs`

Added a pre-lowering signal DAG analysis pass (`analyze_signal_sharing`)
that computes:

1. **`sig_ref_counts`**: how many parent nodes reference each `SigId`.
2. **`sig_at_boundary`**: set of nodes where at least one parent has
   strictly higher variability (e.g. a Block node consumed by a Samp node).

Phase 1 placement now only materializes a node into a named variable when:
- `sig_shared` (ref_count ≥ 2): the node is used in multiple places,
  materialization prevents duplication; **OR**
- `at_boundary`: the node sits at a variability transition (e.g.
  Block→Samp) — it must be materialized to ensure it executes in the
  correct bucket, even if single-use.

Single-use nodes within a same-variability chain stay inline, producing
compact compound expressions like the C++ output.

### Results on `STunedBar.dsp`

| Metric | Faust C++ | faust-rs before | faust-rs after |
|--------|-----------|-----------------|----------------|
| `float fSlow` | 43 | 285 | 43 |
| `int fSlow` | 0 | 0 | 40 |
| Total `fSlow` | 43 | 285 | 83 |

The 43 `float fSlow` variables now match C++ exactly.  The 40 `int fSlow`
are delay-index computations that C++ inlines into the sample loop —
a separate optimization opportunity.

### Scope

- `crates/transform/src/signal_fir/module.rs` — `analyze_signal_sharing`,
  `analyze_sig_rec`, Phase 1 placement guard

### Validation

- `cargo test --workspace` — all tests pass (0 failures)
- `faust-rs STunedBar.dsp`: `float fSlow` count reduced from 285 to 43
- `faust-rs t10.dsp`: compound expression `fConst0 = sin(fmin(192000, ...))`
  in `instanceConstants`, `fSlow0 = fConst0 * float(fButton0)` in compute
