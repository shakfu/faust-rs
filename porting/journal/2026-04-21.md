# 2026-04-21

## Fix FAD seed-recognition drift on lambda-bound recursive inputs

Commit date: 2026-04-21
Source commit: pending

`fad_pendulum_cello1.dsp` (FAD 2nd-order kinematics branch) and the minimal
reproduction

```faust
kin(phi) = eq, vel with {
    eq = sin(phi) + 0.3 * cos(2.5 * phi);
    vel = fad(eq, phi) : !, _;
};
phi_gen = (+(0.5 / ma.SR) : ma.frac) ~ _ : *(2 * ma.PI);
process = phi_gen : kin;
```

emitted C++ with a phantom second recursion slot: the primal used `fRec235`
while the tangent used an unrelated `fRec249`, and the tangent was multiplied
by a stuck-at-zero integer recursion (`iRec244`), silently zeroing the
derivative. The analytic bypass version of the same DSP compiled correctly,
proving the primal model was sound and the defect lived inside the FAD
pipeline.

### Root cause

`propagate::forward_ad::generate_fad_signals_multi` converts both the primal
outputs and each seed from de Bruijn form to symbolic recursion form
before running the forward-mode transform. Previously it called
`de_bruijn_to_sym` once per root. `de_bruijn_to_sym` allocates a new
`Converter` for every call, with a fresh `convert_memo` and `next_var_index`;
its `fresh_var()` helper guarantees uniqueness against the arena, not
determinism across calls. So:

1. First call (outputs): consumes `W0`, rewrites the `phi_gen` sub-term into
   `SYMREC(W0, …)`.
2. Second call (seed): tries `W0`, observes it already interned, bumps to
   `W1`, rewrites the *same* `phi_gen` de Bruijn sub-term into
   `SYMREC(W1, …)` — a structurally identical but `TreeId`-distinct node.

The forward-mode transform recognises the seed by `TreeId` equality
(`sig == self.diff_seed`). With the seed at `SYMREC(W1, …)` and the body's
`phi` reference at `SYMREC(W0, …)`, the match never fired. The transform
walked into `W0` instead, differentiating the inner `frac` accumulator — the
`2π` wrapper leaked into the tangent as a chain-rule factor and the outer
`SYMREC` kept the primal form while a forked `FAD_W0` recursion was
materialised for the tangent. Downstream codegen scheduled the two SYMRECs
to two separate slots (`fRec235` vs `fRec249`), producing the observed
double-recursion shape.

### Fix

- `crates/tlib/src/recursion.rs`: added `de_bruijn_to_sym_many(arena, roots)`.
  One `Converter` instance, one shared `convert_memo` and `next_var_index`,
  so a de Bruijn sub-term reachable from several roots maps to exactly one
  symbolic `SYMREC` node.
- `crates/tlib/src/lib.rs`: re-export `de_bruijn_to_sym_many`.
- `crates/propagate/src/forward_ad.rs`: `generate_fad_signals_multi` now
  bundles `[outputs…, seeds…]` through a single `de_bruijn_to_sym_many`
  call and splits the result back into `converted_outputs` /
  `converted_seeds`. The single-root `de_bruijn_to_sym` import is replaced
  by the multi-root helper.

Scope: the fix is confined to the FAD seed boundary. The single-root
`de_bruijn_to_sym` entry point is unchanged — other callers
(`prepare_signals_for_fir` and friends) convert one root at a time and are
unaffected.

### Tests

- `tests/corpus/fad_lambda_recursive_seed.dsp`: new corpus fixture — the
  minimal reproduction above.
- `crates/compiler/tests/signal_pipeline.rs`:
  `corpus_fad_lambda_recursive_seed_shares_recursion_across_primal_and_tangent`
  drives the full signal pipeline on the fixture and asserts
  `process_arity = 0 → 2` with both lanes present. The test is dispatched
  through a worker thread with a 64 MiB stack because the DSP pulls
  `stdfaust.lib` and its evaluation tree overflows the default debug-build
  stack — consistent with the `run_with_large_stack` pattern already used in
  `zita_pipeline.rs`.
- No existing tests required updates; the new helper's shared-memo path
  subsumes the single-root path for every call site in the workspace.

### Validation

- `cargo test -p tlib` (11 passed).
- `cargo test -p propagate` (42 passed, including the previously introduced
  multi-seed cases).
- `cargo test -p compiler --test signal_pipeline fad` (34 passed, including
  the new regression).
- `cargo test --workspace` (clean).

End-to-end check: the minimal single-level reproduction now emits one
shared `fRec` across primal and tangent. **However, the full
`fad_pendulum_cello1.dsp` (which nests `acc = fad(vel, phi)` on top of
`vel = fad(eq, phi)`) is still silent after this fix.** See the next
entry for the diagnosis.

## `fad_pendulum_cello1.dsp` still silent — nested FAD forks the recursion

Commit date: 2026-04-21
Source commit: pending (plan-only)

Diff-ing the generated C++ against the analytic-bypass variant of the
same DSP exposes two downstream symptoms:

- `phi_gen` is emitted three times (`fRec772`, `fRec812`, `fRec839`),
  one per kinematics output, instead of once.
- The trigger branch multiplies the acceleration tangent by an integer
  recursion `iRec806` that is initialised to `0` and never updated
  (`int iRecCur806 = iRec806;`). Because `trigger = max(0, acc) * 0.5`
  feeds the noise burst, a permanent zero multiplier silences the output.

The factors `11.78 = 1.875·2π` and `6.28 = 2π` in the tangent expression
confirm the chain rule is being applied with respect to the inner `frac`
accumulator rather than `phi`, then re-scaled by the stuck `iRec806`
that should have been the constant `1` seed tangent.

### Root cause

The 2026-04-21 fix (`de_bruijn_to_sym_many`) unified the converter pass
across outputs and seeds for a *single-level* `fad`. Nested `fad` breaks
the same invariant one level higher:

1. `fad(eq, phi)` converts the phasor's `DEBRUIJNREC` into `SYMREC(W0)`
   and returns the tangent in that symbolic form.
2. `fad(vel, phi)` reaches the outer FAD. Its body (`vel`) is already a
   SYMREC tree; its seed (`phi`) is still `DEBRUIJNREC` in `slot_env`
   (propagate never re-converts slot values). The converter walks the
   SYMREC through `match_sym_rec` (keeps `W0`) and the DEBRUIJNREC
   through `match_de_bruijn_rec` (calls `fresh_var()`, sees `W0`
   interned, picks `W1`).
3. Seed ends at `SYMREC(W1, …)`, body at `SYMREC(W0, …)`: the FAD
   transform's `sig == self.diff_seed` check misses the seed inside the
   body, differentiates the inner `frac` accumulator, and emits the
   spurious `FAD_W1` recursion that lands as `iRec806` at codegen.

No naming discipline on top of the current architecture can paper this
over, because the converter has no structural comparator between
`DEBRUIJNREC` and `SYMREC` nodes.

### Chosen path forward

Plan (B) from the user-facing analysis: move the single
`de_bruijn_to_sym` conversion to `signal_prepare` and teach
`ForwardADTransform` to dispatch directly on `DEBRUIJNREC` /
`DEBRUIJNREF`. The full design is committed as
`porting/fad-debruijn-native-transform-plan-2026-04-21-en.md`. See
below for the implementation landed the same day.

## Implement Plan B — FAD operates on DeBruijn signals directly

Commit date: 2026-04-21
Source commit: pending

### What changed

The plan called for teaching `ForwardADTransform` to handle `DEBRUIJNREC`
and `DEBRUIJNREF` with expanded interleaved lanes. Analysis of the actual
execution path showed that the full lane-expansion mechanism is not needed:
the seed is always matched as a leaf (`sig == self.diff_seed`) before the
transform ever descends into a recursion body. The only change required is
to stop converting the signal graph inside `generate_fad_signals_multi`.

**`crates/propagate/src/forward_ad.rs`**

- `generate_fad_signals_multi`: removed the bundled
  `de_bruijn_to_sym_many` conversion (outputs + seeds joined into one
  combined slice, converted, split back). The function now passes
  `outputs` and `seeds` directly to `ForwardADTransform` without any
  prior form change.
- Import of `de_bruijn_to_sym_many` removed from the `tlib` use block.
- Module docstring updated: the conversion note now says "deferred to
  `signal_prepare`", not "converted before differentiation".

The `SYMREC`/`SYMREF` arms in `transform_uncached` are kept as defensive
code; they are unreachable in the normal pipeline (since nothing upstream
of `signal_prepare` converts to symbolic form) but guard against
mixed-form trees if a caller ever passes already-converted signals.

**`signal_prepare.rs` — no change required.**
Line 391 already calls `tlib::de_bruijn_to_sym(&mut arena, cloned_list)`
on a cons-list containing all process outputs. Because the cons-list is
one root and the `Converter` memo is shared across all its children,
every occurrence of the same `DEBRUIJNREC` sub-term maps to the same
fresh `W0`/`W1`/… name, regardless of whether it is reached from a
primal lane or a tangent lane. This is precisely the invariant we need.

### Why this is sufficient

Seed recognition in `ForwardADTransform` is `sig == self.diff_seed`,
compared by `SigId` (the arena's hash-consed node index). Before the
fix, `de_bruijn_to_sym_many` was called independently on the body and
seed, producing two distinct `SYMREC` nodes for the same underlying
`DEBRUIJNREC` (different fresh `W` names). The equality check then
missed the seed everywhere inside the body.

By removing the conversion entirely, both body and seed stay in
`DEBRUIJNREC` form with the same `TreeId`. The equality check fires
correctly at every phi reference. The recursive structure inside the
`DEBRUIJNREC` body is never traversed by the transform because the seed
check short-circuits before any descent.

For nested `fad(fad(eq,phi),phi)`:
1. Inner `fad(eq,phi)`: seed = phi (DeBruijn SigId). Transform visits
   eq, reaches phi leaf → tangent=1. Tangent uses phi's SigId in
   composed expressions (`cos(phi)`, `sin(2.5*phi)`). Output DeBruijn.
2. Outer `fad(vel,phi)`: seed = same phi SigId. Body = vel (DeBruijn
   expressions containing phi). Transform visits vel, reaches phi →
   tangent=1 again. Second derivative is computed correctly.
3. `signal_prepare` converts the whole output list (pos, vel, acc) in
   one shared pass, producing one `SYMREC(W0)` for phi everywhere.

### Corpus fixture and regression test

**`tests/corpus/fad_nested_on_recursive_seed.dsp`**: self-contained
(no stdfaust import) reproduction of the nested-FAD pattern using a
linear accumulator for phi. Compiles to 3 outputs with a single `fRec`.

**`crates/compiler/tests/signal_pipeline.rs`**:
`corpus_fad_nested_on_recursive_seed_emits_three_lanes_one_recursion`
asserts `process_arity = 0 → 3` with no UI controls.

### Validation

- `cargo test -p propagate` (42 passed).
- `cargo test -p tlib` (11 passed).
- `cargo test -p compiler --test signal_pipeline fad` (35 passed, +1
  new nested-FAD regression).
- `cargo test --workspace` (clean, all suites green).

End-to-end: `faust-rs fad_pendulum_cello1.dsp` (FAD 2nd-order branch)
now emits exactly one `fRec` for `phi_gen`, and the generated C++ is
structurally identical to the analytic-bypass version — same phasor,
same trig expressions, same trigger path — with only variable-index
differences. The stuck-at-zero `iRec` and the duplicate phasors are
gone.

## Raise evaluator structural-depth limit and grow compiler thread stack

`faust-rs auto_spat.dsp` failed with:

```
evaluation recursion depth exceeded (256)
```

`auto_spat.dsp` contains `BOXFAUTODIFF` expansions that produce deeply nested
`a2sb` structural lowering chains exceeding the former 256-frame cap.

### Root cause

`LoopDetector::enter_structural()` capped `a2sb` / `a2sb_value` recursion at
`STRUCTURAL_MAX = 256`, tighter than the 1024 general `max_depth`, to keep OS
stack usage within the default 8 MiB thread stack.  Programs with long
combinator chains (auto-panning, FxLMS filter banks) legitimately recurse
deeper than 256 structural frames.

### Fix

- `crates/eval/src/loop_detector.rs`: `STRUCTURAL_MAX` raised to 1024; symbol-
  lookup `max_depth` raised to 4096.
- `crates/compiler/src/main.rs`: renamed the body of `main` to `run_main` and
  added a thin `main` that spawns `run_main` on a 64 MiB thread, giving the
  evaluator stack headroom for the deeper structural limit.

## FAD: De Bruijn recursive-body differentiation (`DEBRUIJNREC` / `DEBRUIJNREF`)

`fad_fxlms1.dsp` uses `ba.if(reset, init, next) ~ _` inside a `with` block.
The FAD transform needs to handle the de Bruijn form of recursive groups
(`DEBRUIJNREC` / `DEBRUIJNREF` tag nodes) that `propagate` emits before the
`de_bruijn_to_sym` conversion runs in `signal_prepare`.

### New behaviour in `ForwardADTransform`

- `DEBRUIJNREF` node: `d/dp ref = ref` — the index shift is applied at the
  projection site, not here.
- `DEBRUIJNREC(body)` node: differentiate the body, then **interleave** primals
  and tangents — `[p0, t0, p1, t1, …]` — so that `proj(i, rec)` picks the
  primal from slot `2i` and the tangent from slot `2i+1`.  A `debruijn_depth`
  counter tracks how many FAD-interleaved `DEBRUIJNREC` scopes are currently
  open.
- `Proj` arm: when the group is a `DEBRUIJNREC` or a `DEBRUIJNREF` whose de
  Bruijn level is ≤ `debruijn_depth` (bound inside the FAD scope), double the
  index.  When the level exceeds `debruijn_depth` (an outer loop not being
  differentiated), the tangent is zero.

### Cleanup

French-language comments removed; import sorted; projection logic refactored to
a local `enum GroupKind { BoundRec, UnboundRef, Other }` match.
