# 2026-04-21

## Fix FAD seed-recognition drift on lambda-bound recursive inputs

Commit date: 2026-04-21
Source commit: pending

`fad_pendulum_cello1.dsp` (FAD 2nd-order kinematics branch) and the minimal
reproduction

```faust
kin(phi) = eq, vel with {
    eq = sin(phi) + 0.3 * cos(2.5 * phi);
    vel = fad(eq, phi) : !, _;
};
phi_gen = (+(0.5 / ma.SR) : ma.frac) ~ _ : *(2 * ma.PI);
process = phi_gen : kin;
```

emitted C++ with a phantom second recursion slot: the primal used `fRec235`
while the tangent used an unrelated `fRec249`, and the tangent was multiplied
by a stuck-at-zero integer recursion (`iRec244`), silently zeroing the
derivative. The analytic bypass version of the same DSP compiled correctly,
proving the primal model was sound and the defect lived inside the FAD
pipeline.

### Root cause

`propagate::forward_ad::generate_fad_signals_multi` converts both the primal
outputs and each seed from de Bruijn form to symbolic recursion form
before running the forward-mode transform. Previously it called
`de_bruijn_to_sym` once per root. `de_bruijn_to_sym` allocates a new
`Converter` for every call, with a fresh `convert_memo` and `next_var_index`;
its `fresh_var()` helper guarantees uniqueness against the arena, not
determinism across calls. So:

1. First call (outputs): consumes `W0`, rewrites the `phi_gen` sub-term into
   `SYMREC(W0, …)`.
2. Second call (seed): tries `W0`, observes it already interned, bumps to
   `W1`, rewrites the *same* `phi_gen` de Bruijn sub-term into
   `SYMREC(W1, …)` — a structurally identical but `TreeId`-distinct node.

The forward-mode transform recognises the seed by `TreeId` equality
(`sig == self.diff_seed`). With the seed at `SYMREC(W1, …)` and the body's
`phi` reference at `SYMREC(W0, …)`, the match never fired. The transform
walked into `W0` instead, differentiating the inner `frac` accumulator — the
`2π` wrapper leaked into the tangent as a chain-rule factor and the outer
`SYMREC` kept the primal form while a forked `FAD_W0` recursion was
materialised for the tangent. Downstream codegen scheduled the two SYMRECs
to two separate slots (`fRec235` vs `fRec249`), producing the observed
double-recursion shape.

### Fix

- `crates/tlib/src/recursion.rs`: added `de_bruijn_to_sym_many(arena, roots)`.
  One `Converter` instance, one shared `convert_memo` and `next_var_index`,
  so a de Bruijn sub-term reachable from several roots maps to exactly one
  symbolic `SYMREC` node.
- `crates/tlib/src/lib.rs`: re-export `de_bruijn_to_sym_many`.
- `crates/propagate/src/forward_ad.rs`: `generate_fad_signals_multi` now
  bundles `[outputs…, seeds…]` through a single `de_bruijn_to_sym_many`
  call and splits the result back into `converted_outputs` /
  `converted_seeds`. The single-root `de_bruijn_to_sym` import is replaced
  by the multi-root helper.

Scope: the fix is confined to the FAD seed boundary. The single-root
`de_bruijn_to_sym` entry point is unchanged — other callers
(`prepare_signals_for_fir` and friends) convert one root at a time and are
unaffected.

### Tests

- `tests/corpus/fad_lambda_recursive_seed.dsp`: new corpus fixture — the
  minimal reproduction above.
- `crates/compiler/tests/signal_pipeline.rs`:
  `corpus_fad_lambda_recursive_seed_shares_recursion_across_primal_and_tangent`
  drives the full signal pipeline on the fixture and asserts
  `process_arity = 0 → 2` with both lanes present. The test is dispatched
  through a worker thread with a 64 MiB stack because the DSP pulls
  `stdfaust.lib` and its evaluation tree overflows the default debug-build
  stack — consistent with the `run_with_large_stack` pattern already used in
  `zita_pipeline.rs`.
- No existing tests required updates; the new helper's shared-memo path
  subsumes the single-root path for every call site in the workspace.

### Validation

- `cargo test -p tlib` (11 passed).
- `cargo test -p propagate` (42 passed, including the previously introduced
  multi-seed cases).
- `cargo test -p compiler --test signal_pipeline fad` (34 passed, including
  the new regression).
- `cargo test --workspace` (clean).

End-to-end check: the minimal single-level reproduction now emits one
shared `fRec` across primal and tangent. **However, the full
`fad_pendulum_cello1.dsp` (which nests `acc = fad(vel, phi)` on top of
`vel = fad(eq, phi)`) is still silent after this fix.** See the next
entry for the diagnosis.

## `fad_pendulum_cello1.dsp` still silent — nested FAD forks the recursion

Commit date: 2026-04-21
Source commit: pending (plan-only)

Diff-ing the generated C++ against the analytic-bypass variant of the
same DSP exposes two downstream symptoms:

- `phi_gen` is emitted three times (`fRec772`, `fRec812`, `fRec839`),
  one per kinematics output, instead of once.
- The trigger branch multiplies the acceleration tangent by an integer
  recursion `iRec806` that is initialised to `0` and never updated
  (`int iRecCur806 = iRec806;`). Because `trigger = max(0, acc) * 0.5`
  feeds the noise burst, a permanent zero multiplier silences the output.

The factors `11.78 = 1.875·2π` and `6.28 = 2π` in the tangent expression
confirm the chain rule is being applied with respect to the inner `frac`
accumulator rather than `phi`, then re-scaled by the stuck `iRec806`
that should have been the constant `1` seed tangent.

### Root cause

The 2026-04-21 fix (`de_bruijn_to_sym_many`) unified the converter pass
across outputs and seeds for a *single-level* `fad`. Nested `fad` breaks
the same invariant one level higher:

1. `fad(eq, phi)` converts the phasor's `DEBRUIJNREC` into `SYMREC(W0)`
   and returns the tangent in that symbolic form.
2. `fad(vel, phi)` reaches the outer FAD. Its body (`vel`) is already a
   SYMREC tree; its seed (`phi`) is still `DEBRUIJNREC` in `slot_env`
   (propagate never re-converts slot values). The converter walks the
   SYMREC through `match_sym_rec` (keeps `W0`) and the DEBRUIJNREC
   through `match_de_bruijn_rec` (calls `fresh_var()`, sees `W0`
   interned, picks `W1`).
3. Seed ends at `SYMREC(W1, …)`, body at `SYMREC(W0, …)`: the FAD
   transform's `sig == self.diff_seed` check misses the seed inside the
   body, differentiates the inner `frac` accumulator, and emits the
   spurious `FAD_W1` recursion that lands as `iRec806` at codegen.

No naming discipline on top of the current architecture can paper this
over, because the converter has no structural comparator between
`DEBRUIJNREC` and `SYMREC` nodes.

### Chosen path forward

Plan (B) from the user-facing analysis: move the single
`de_bruijn_to_sym` conversion to `signal_prepare` and teach
`ForwardADTransform` to dispatch directly on `DEBRUIJNREC` /
`DEBRUIJNREF`. The full design is committed as
`porting/fad-debruijn-native-transform-plan-2026-04-21-en.md`. This
entry is the commit-log pointer; no code landed today.
