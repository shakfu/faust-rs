## fix(eval): preserve higher-order tuple children for C++-parity case matching

Fixed an eval parity bug exposed by `faust-rs -pn resistor_test tests/wdmodels_tests.dsp`
against `faustlibraries`. The failure path was `wd.buildtree` → `builddown`/`parres`, where
tupled local closures such as `(res_leaf, probe)` must remain structurally matchable by
`case` rules like `parres((Ap1, Ap2)) = ...`.

### Root cause

`map_children` in `crates/eval/src/lib.rs` rebuilt generic parent nodes by calling
`a2sb_value(...)` on each child. That was too eager for residual closures and pattern
matchers: tuple/`par` nodes carrying higher-order local definitions were being symbolically
forced before later case dispatch, so the matcher no longer saw the same callable tuple shape
that Faust C++ preserves.

### Fix

- `map_children` now uses `force_value_to_box(...)` instead of `a2sb_value(...)` when
  reconstructing generic child nodes.
- This keeps residual callable children intact as box nodes while still allowing later
  forcing when a phase genuinely needs symbolic lowering.

### Regression coverage

Added two eval tests:

- `eval_process_preserves_tupled_higher_order_locals_for_case_matching` — in-memory regression
  for tupled local closures capturing `wdmodels` library functions.
- `eval_process_tupled_higher_order_locals_match_cpp_reference` — self-contained file-backed
  differential that compiles the same fixture with the C++ reference compiler when
  `FAUST_CPP_BIN` (or `/usr/local/bin/faust`) is available, then checks the Rust evaluator on
  the same source.

### Validation

- `cargo fmt --all`
- `cargo test -p eval eval_case_pattern_var_lookup_stops_at_barrier_scope -- --nocapture`
- `cargo test -p eval eval_process_preserves_tupled_higher_order_locals_for_case_matching -- --nocapture`
- `cargo test -p eval eval_process_tupled_higher_order_locals_match_cpp_reference -- --nocapture`
- `cargo run -p compiler -- -pn resistor_test /Users/letz/Developpements/faustlibraries/tests/wdmodels_tests.dsp`

## Soundfile support: interpreter and Cranelift backends

## feat(cranelift): add soundfile support in Cranelift backend

Two complementary changes that bring the Cranelift JIT backend to full
soundfile parity with the C/C++ backends.

### FFI layer (`cranelift-ffi/src/instance.rs`)

Added `soundfile_zone_ptr()` alongside the existing `zone_ptr()`.  For
`FirType::Sound` fields in the `dsp*` struct (pointer-sized, 8 bytes on
LP64), it returns the address of that slot cast as `*mut *mut c_void` —
the `Soundfile**` expected by `SoundUI::addSoundfile`.

The `buildUserInterfaceCCraneliftDSPInstance` `Soundfile` arm now calls
`soundfile_zone_ptr` instead of falling back through `zone_ptr` (which
returned `None` for Sound fields, producing a null zone pointer and a
`SoundUI::addSoundfile` crash).  The host now writes the loaded
`Soundfile*` directly into the JIT struct field at the field's natural
offset.

### Codegen (`codegen/src/backends/cranelift/mod.rs`)

Three new `ComputeLowering` methods implement the three soundfile
expression nodes, using the C++ `Soundfile` packed-struct offsets:

| Field | Offset | Use |
|-------|--------|-----|
| `fBuffers` | 0 | `void*` — `float**` or `double**` |
| `fLength` | 8 | `int*` — length per part |
| `fSR` | 16 | `int*` — sample rate per part |
| `fOffset` | 24 | `int*` — frame offset per part |

- **`load_soundfile_ptr`** — loads `dsp->fSoundN` (ptr-sized struct field) → `Soundfile*`.
- **`lower_load_soundfile_length`** — `fSoundN->fLength[part]` (offset 8, int array stride 4).
- **`lower_load_soundfile_rate`** — `fSoundN->fSR[part]` (offset 16, int array stride 4).
- **`lower_load_soundfile_buffer`** — loads `fBuffers[chan]` (pointer array, pointer stride),
  reads `fOffset[part]` (offset 24), adds `idx`, loads sample with element-type stride.

`subset_expr_gap_reason` updated to return `None` for all three variants —
DSPs using `soundfile` primitives no longer fall back to the no-op stub.

## fix(interp-ffi): wire soundfile zones so loaded audio reaches executor

After `5facc6a` the crash was fixed (no more null zone pointer), but
`compute()` still played silence because the executor held
`default_silence()` stubs instead of the real audio data.

### Problem

After `buildUserInterface`, each `soundfile_zones[slot]` held a valid
`Soundfile*` written by `SoundUI::addSoundfile`, but the executor's
`soundfiles[slot]` was never updated.

### Fix — three files

**`ui.rs`** — `AddSoundfile` arm passes `&mut soundfile_zones[slot]` as
`void**` zone pointer instead of `null_mut()`.

**`types.rs`**:
- `dispatch_ui_glue` forwards the `soundfile_zones` slice to
  `dispatch_ui_f32` / `dispatch_ui_f64`.
- `FbcExecutorAny::new_for_factory` pre-populates `executor.soundfiles`
  with `default_silence()` stubs (bounds safety before first UI call).
- New `set_soundfile(slot, sf)` method replaces a soundfile slot.

**`instance.rs`**:
- `InterpreterDspInstance` gains `soundfile_zones: Vec<*mut c_void>`
  (one null sentinel per soundfile at creation time).
- After `dispatch_ui_glue` returns, `sync_soundfiles_from_zones` reads
  each non-null C++ `Soundfile*` via a `#[repr(C, packed)]` `CSoundfile`
  mirror struct, copies `fLength`/`fSR`/`fOffset` arrays and per-channel
  audio buffers (handling both `fIsDouble=false` f32 and `=true` f64),
  then calls `set_soundfile()` to replace the stubs.

The `Soundfile` C++ packed-struct layout mirrored:

```rust
#[repr(C, packed)]
struct CSoundfile {
    f_buffers:   *mut c_void,  // offset  0 — float** or double**
    f_length:    *mut i32,     // offset  8
    f_sr:        *mut i32,     // offset 16
    f_offset:    *mut i32,     // offset 24
    f_channels:  i32,          // offset 32
    f_parts:     i32,          // offset 36
    f_is_double: bool,         // offset 40
}
```

`fIsDouble=false` buffers are copied as-is (f32 → f64 widening);
`fIsDouble=true` buffers are copied directly as f64.

## feat(interp): implement soundfile support in interpreter backend

Full end-to-end soundfile pipeline for `faust-rs -lang interp`.

### Problem

`faust-rs -lang interp tp0.dsp` crashed with:
```
[FRS-CGEN-INTERP-0003] unsupported FIR node: LoadSoundfileBuffer { var: "fSound0", ... }
```

### New module: `codegen/src/backends/interp/soundfile.rs`

Defines the Rust-side `Soundfile` struct mirroring the C++ fields:
`num_channels`, `num_parts`, `lengths`, `sample_rates`, `offsets`,
`buffers: Vec<Vec<f64>>`.  `default_silence()` produces the same
1-part/1-channel/zero-length/44100 Hz fallback as C++ `defaultsound`.
`read_sample(chan, part, idx)` is bounds-safe (returns 0.0 on OOB).

### Compiler (`codegen/src/backends/interp/compiler.rs`)

- `soundfile_slots: HashMap<String, usize>` tracks `fSoundN → slot` index.
- `predeclare_var_storage` and `compile_declare_var`: Sound-typed vars use
  `alloc_soundfile_slot` (no heap scalar allocation).
- `compile_add_soundfile`: stores slot in `instr.offset`, URL in `instr.key`.
- Three new compile methods → FBC opcodes:
  - `compile_load_soundfile_length` → `LoadSoundFieldInt` (`int_value=0`)
  - `compile_load_soundfile_rate` → `LoadSoundFieldInt` (`int_value=1`)
  - `compile_load_soundfile_buffer` → `LoadSoundFieldReal` (chan/part/idx on int stack)

### Executor (`codegen/src/backends/interp/executor.rs`)

- `FbcExecutor<R>` gains `pub soundfiles: Vec<Box<Soundfile>>`.
- `LoadSoundFieldInt`: pops `part`, dispatches on `iv` (0 = fLength, 1 = fSR).
- `LoadSoundFieldReal`: pops `idx`, `part`, `chan`; calls `sf.read_sample`.

### Factory / Instance

- `FbcDspFactory::soundfile_count()` counts `AddSoundfile` opcodes in `ui_block`.
- `FbcDspInstance::new` pre-populates `soundfiles` with `default_silence()`.
- `FbcExecutorAny::new_for_factory` likewise pre-populates (FFI path).

`tp0.dsp` compiles and runs without error; silence stubs are replaced with
real audio after `buildUserInterface`.
