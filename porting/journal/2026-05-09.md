## Implement RAD Phase B2 — reference BPTT executor + FD oracle

Landed Phase B2 of
`porting/rad-block-reverse-ad-signal-ir-plan-2026-05-07-en.md`: a
self-contained, tape-based TBPTT(BS, BS) reference executor that operates
directly on Signal-IR nodes, independent of the FIR lowering pipeline.  This
executor is the correctness oracle for Phase B3 (backend lowering).

New file: `crates/compiler/tests/block_reverse_ad.rs`

Public API:

- `collect_postorder(arena, root, visited, order)` — DFS postorder traversal
  covering all signal-node kinds with children (BinOp, unary math, Delay1,
  Select2, etc.).  Shared `visited` set allows multi-output bodies to reuse a
  single traversal.

- `eval_sig(arena, sig, vals, inputs, ui_vals, delay1_state) -> f64` — forward
  evaluation of one node given already-computed child values.  Handles: `Real`,
  `Int`, `Input`, `HSlider`/`VSlider`/`NumEntry`/`Button`/`Checkbox` (via
  `ui_vals`), all 17 `BinOp` variants, full trig/exp/log/sqrt/abs/floor/ceil
  suite, `IntCast`, `FloatCast`, `Select2`, and `Delay1` (reads from
  `delay1_state`, keyed by the `Delay1` node's own `SigId`).

- `propagate_adj(arena, sig, y_bar, vals, adj, adj_carry)` — one backward
  adjoint step.  Differentiable rules: Add/Sub/Mul/Div product-rule, Pow,
  Min/Max (subgradient), full trig/exp/log/sqrt/abs chain rule, FloatCast
  identity, Fmod/Remainder pass-through.  `Delay1` is anti-causal: deposits
  `y_bar` into `adj_carry[x]` instead of `adj[x]` so the contribution arrives
  at the backward step for the previous sample.

- `block_bptt_eval(arena, body_sigs, seed_sigs, cotangents, input_block,
  ui_vals, block_size) -> BpttResult` — full TBPTT(BS, BS) loop.  Forward
  pass: builds per-sample `tape[sample][sig] = f64`, updates `delay1_state`
  after each sample.  Backward pass (reverse sample order): starts each step
  with carry from sample+1, injects cotangents into primal adjoints, runs
  reverse-postorder backprop, records per-seed adjoints.  Returns
  `BpttResult { primals, grads, grads_total }`.

- Finite-difference helpers:
  - `forward_block_sum` — forward-only block evaluation for FD oracle.
  - `fd_const_seed` — central-difference estimate for constant (UI) seeds.
  - `fd_audio_seed` — forward-difference estimate for per-sample audio seeds
    (exact because block sum is linear in any single input sample).

Tests (7, all green):

- `block_ref_const_seed_linear` — `body=[2*x]`, x=0.5 constant; per-sample
  grad=2, total=16, matches FD.
- `block_ref_const_seed_sin` — `body=[sin(x)]`, x=0.5; grad=cos(0.5) per
  sample, total=BS*cos(0.5), matches FD.
- `block_ref_const_seed_delay1` — `body=[delay1(x)]`, x=0.5; forward gives
  0 at sample 0 then 0.5; BPTT total=BS-1 via anti-causal carry; FD matches.
- `block_ref_const_seed_square` — `body=[x*x]`, x=2.0; grad=4 per sample,
  matches FD.
- `block_ref_audio_seed_linear` — `body=[2*input(0)]`; per-sample grad=2,
  confirmed per sample via FD forward-difference.
- `block_ref_audio_seed_delay1` — `body=[delay1(input(0))]`; grad[j]=1 for
  j<BS-1, grad[BS-1]=0 (last sample's carry exits the block); confirmed by FD.
- `block_ref_carrier_round_trip` — builds a full `SigBlockReverseAD` carrier
  via `SigBuilder::block_reverse_ad`, extracts body/seed/cotangent lists with
  `match_sig` + `list_to_vec`, feeds to executor; guards B0 encoding contract.

## Implement RAD Phase B4 — forward tape for non-trivially-re-evaluable operands

Landed Phase B4 of the RAD BlockReverseAD plan: the FIR backward sweep now
correctly handles body expressions that contain **non-trivially-re-evaluable**
signals (e.g. `Delay1(x)`) as operands in multiplication, division, or unary
math nodes.  The fix is a **per-sample forward tape**: forward values of such
operands are stored during the forward sample loop and loaded during the
reverse sweep instead of calling `lower_signal` (which would read incorrect
delay state in the reverse loop).

### Background

Phase B3 had a correctness limitation: `propagate_bra_adj` called
`lower_signal(rhs)` directly for the `Mul` backward rule.  When `rhs =
Delay1(x)`, the signal cache for the reverse loop does not hold the correct
forward value, so the gradient would be computed from the wrong delay state.
Phase B4 fixes this by introducing `load_bra_fwd_value`, which transparently
falls back to a tape load for non-trivial signals.

### Changes

- `crates/transform/src/signal_fir/block_reverse_ad.rs`:
  - Updated module-level doc to describe both B3 (tape-free) and B4
    (per-sample tape) scopes with a reference table for tape-needed signals.
  - Added `BinOp` import for the `Mul | Div` pattern in
    `collect_tape_needed_values`.
  - New function `collect_tape_needed_values(arena, postorder) ->
    HashSet<SigId>`: iterates the postorder and marks which signal values
    must be taped.  Rules:
    - `Mul(lhs, rhs)` / `Div(lhs, rhs)`: tape `lhs` if not trivial; tape
      `rhs` if not trivial.
    - `Sin(x)` / `Cos(x)` / `Log(x)`: tape `x` if not trivial.
    - `Exp(x)` / `Sqrt(x)`: tape the node itself (`sig`) if `x` is not
      trivial (the backward rule reuses `val[sig]` directly).
    - All other node kinds: no tape requirement.

- `crates/transform/src/signal_fir/module.rs`:
  - Added `const MAX_BRA_TAPE_BLOCK_SIZE: usize = 8192` with Rustdoc
    explaining the per-block tape capacity and host `compute()` constraint.
  - Updated `block_reverse_ad` import to also pull in
    `collect_tape_needed_values`.
  - New struct fields on `SignalToFirLower`:
    - `bra_tape_store_var: HashMap<SigId, String>` — maps a tape-needed
      signal to the name of its `fBraTapeN` struct array.
    - `bra_tape_stores_scheduled: HashSet<SigId>` — idempotency guard for
      `ensure_bra_tape_stores`.
  - New method `ensure_bra_tape_stores(group, body_sigs, …)`:
    - Builds the unified postorder over all body roots.
    - Calls `collect_tape_needed_values` to get the set of signals to tape.
    - For each tape-needed signal: allocates a `fBraTapeN: Array(real_ty,
      8192)` struct field, lowers the signal in the forward loop context,
      emits `store_table(fBraTapeN, Struct, i0, v_fir)` to
      `sample_phases.immediate` (not `sample_end` — see bug note below),
      and records the `SigId → name` mapping.
    - Guarded by `bra_tape_stores_scheduled` to prevent double-scheduling.
    - **Bug discovered and fixed during testing**: an initial version emitted
      tape stores to `sample_phases.sample_end`, which runs AFTER `post_output`
      updates delay/state variables.  For `Delay1(x)`, `sample_end` would
      re-read the updated register (x[n] instead of the delayed value x[n-1]),
      causing `tape[0] = 2.0` instead of the correct `0.0` (initial state).
      Moving stores to `immediate` (which runs before `post_output`) fixes
      this: the captured value is the unmodified forward value at step n.
  - New method `load_bra_fwd_value(sig)`:
    - If `sig` is in `bra_tape_store_var`, emits a `load_table` from the
      corresponding tape array using the current reverse-loop index `i0`.
    - Otherwise falls back to `lower_signal(sig)` (correct for trivially
      re-evaluable signals with no tape requirement).
  - Modified `lower_block_reverse_ad_proj` primal branch: after lowering the
    body signal, calls `ensure_bra_tape_stores` to schedule forward tape
    stores before the reverse loop begins.
  - Modified `propagate_bra_adj`:
    - `Mul` / `Div`: replaced `lower_signal(lhs/rhs)` with
      `load_bra_fwd_value(lhs/rhs)`.
    - `Sin` / `Cos` / `Log`: replaced `lower_signal(x)` with
      `load_bra_fwd_value(x)`.
    - `Exp` / `Sqrt`: replaced `lower_signal(sig)` with
      `load_bra_fwd_value(sig)`.
    - Error message updated from `(B3)` to `(B4)`.
  - Updated doc comment of `propagate_bra_adj` to describe the B4 tape
    dispatch and updated doc comment of `ensure_bra_tape_stores` /
    `load_bra_fwd_value`.

- `crates/compiler/tests/block_reverse_ad.rs`:
  - New test `fir_bra_delay1_mul_tape` — `process = rad(x' * x, x)` with
    `x = 2.0`, BS=8.  This is the canonical B4 tape test:
    - `Delay1(x)` as the lhs of `Mul` is non-trivially re-evaluable →
      triggers tape store during forward loop.
    - Primal: `delay1(x)[n] * x` = 0.0 at n=0, 4.0 for n≥1.
    - Gradient: per-sample adjoint of `x` from the product rule plus the
      anti-causal Delay1 carry = 2.0 at n=0, 4.0 for 1≤n<BS-1, 2.0 at
      n=BS-1.
    - Sum of gradients = 4*(BS-1) = 28, matching the finite-difference
      check of the block loss `L = Σ y[n]`.
  - Full doc comment explaining the per-sample adjoint computation.

## Implement RAD Phase B6 — recursive circuits in BlockReverseAD backward sweep

Landed Phase B6: `SigBlockReverseAD` FIR lowering now correctly handles
Faust feedback loops (`+~f`) as gradient-differentiable circuits.

### Problem solved

After `de_bruijn_to_sym`, a Faust recursive circuit such as `c : +~*(c)`
produces:

```
Proj(0, SYMREC(var, [Add(Input, Mul(c, Delay1(Proj(0, SYMREF(var)))))]))
```

Two structural forms appear inside the BRA body:

- `Proj(slot, SYMREC(var, body_list))` — top-level output node.
- `Proj(slot, SYMREF(var))` — back-reference inside the body, always
  wrapped in `Delay1(...)`.

The pre-B6 code only knew `SigMatch::Rec` (de Bruijn SIGREC), so it
returned an error "Proj over non-Rec group (Unknown)" when it encountered
the SYMREC / SYMREF nodes produced by `de_bruijn_to_sym`.

### TBPTT total-adjoint identity

For TBPTT(BS, BS) the per-sample backward adjoint of the recursive output
is:

```
adj[y[slot][n]] = cotangent[slot][n] + carry_from_step_n+1
```

where `carry_from_step_n+1 = adj[y[slot][n+1]] * ∂y[n+1]/∂y[n]`.

The carry is written by the `Delay1` backward arm at step n+1 and read at
step n.  Only the **block sum** of the per-sample gradient equals the
finite-difference of `L = Σ_n y[n]` (verified analytically and in tests).

### Key implementation choices

1. **Pre-scan (step 3a in `ensure_bra_backward_sweep`)**: before the
   reverse-postorder walk, scan the postorder for every
   `Delay1(Proj(slot, SYMREF))`.  For each: create the carry struct
   field, load it from the struct (reads the value stored at step n+1),
   and accumulate into `adj[body_sigs[slot]]`.  This makes the full
   adjoint `cotangent[n] + carry_from_n+1` available when
   `Proj(slot, SYMREC)` is processed first in the reverse postorder.

2. **Delay1 arm split**: `Delay1(Proj(slot, SYMREF))` now only stores the
   new carry (no redundant load — already done in the pre-scan).
   Non-recursive `Delay1` keeps the original load+store behavior.

3. **Proj arm extended**: handles `SYMREC` (propagate adjoint to
   `body[slot]`) and `SYMREF` (no-op — carry pre-loaded).  Other Proj
   groups still return `UnsupportedSignalNode`.

4. **`collect_bra_postorder`**: SYMREC → recurse into body; SYMREF →
   stop (back-reference cycle guard, no further recursion).

### Files changed

- `crates/transform/src/signal_fir/block_reverse_ad.rs`:

  - Extended `collect_bra_postorder` Proj arm: recurse for SYMREC,

## Fix multi-SYMREC BRA: two independent recursive poles (Phase B6+)

### Context

The user reported that the following DSP failed at compile time:

```faust
p = hslider("p", 0.5, 0.0, 1.0, 0.01);
q = hslider("q", 0.25, 0.0, 1.0, 0.01);
process = rad(((2 : + ~ *(p)), (3 : + ~ *(q))), (p, q));
```

This is a two-body, two-seed BRA circuit: two **independent** SYMREC
groups (`SYMREC_p` for `y_p[n] = 2 + p·y_p[n-1]` and `SYMREC_q` for
`y_q[n] = 3 + q·y_q[n-1]`), each differentiated with respect to its own
feedback coefficient.  The error was:

```
[FRS-SFIR-0004] unbound symbolic recursion variable 45
```

Two independent bugs were identified:

### Bug A — wrong pre-scan carry lookup

In `ensure_bra_backward_sweep` (step 3a), the pre-scan found
`Delay1(Proj(slot, SYMREF(var)))` nodes and then did:

```rust
if let Some(&proj_symrec) = body_sigs.get(slot_usize) {
```

`body_sigs` in the BRA context is the flat BRA body list —
`[Proj(0, SYMREC_p), Proj(0, SYMREC_q)]`.  Both SYMREC groups are
single-output so both have `slot = 0`.  As a result, `body_sigs.get(0)`
always returned `Proj(0, SYMREC_p)`, so both `SYMREF_p` and `SYMREF_q`
were incorrectly seeded into `adj[body_sigs[0]]`.

**Fix**: build a `HashMap<(TreeId /*SYMREC var*/, usize /*slot*/), SigId>`
from `body_sigs` using `match_sym_rec` to extract each SYMREC's var.
Then, in the pre-scan, use `match_sym_ref` to get the SYMREF's var and
look up the map by `(var, slot)` rather than using `slot` as a flat index.
`TreeId` was added to the `tlib` import for the HashMap key.

### Bug B — tape stores crossing recursion contexts

`ensure_bra_tape_stores` was called once with all `body_sigs` (both
bodies).  It built a unified postorder over both bodies and called
`lower_signal(v)` for every tape-needed signal.  For the one-pole case,
`Proj(0, SYMREF_p)` and `Proj(0, SYMREF_q)` are both tape-needed (they
appear as the LHS of `Mul(Proj(0, SYMREF), p/q)` in each body).

When this was called during primal slot 0's lowering (SYMREC_p active),
`lower_signal(Proj(0, SYMREF_q))` failed because `var_q` was not on the
recursion stack → "unbound symbolic recursion variable".

**Fix**: change the call site in `lower_block_reverse_ad_proj` to pass
only `&[body_sigs[index]]` (the single body for the current primal slot).
Inside `ensure_bra_tape_stores`, replace the group-level idempotency guard
(`bra_tape_stores_scheduled`) with a per-signal check:
`if self.bra_tape_store_var.contains_key(&v) { continue; }`.  This way
each primal slot only tapes signals reachable from its own SYMREC body,
staying within the active recursion context.  Signals shared between
bodies are still taped only once via the per-signal guard.

The now-unused `bra_tape_stores_scheduled: HashSet<SigId>` field was
removed from the struct.

### Changes

**`crates/transform/src/signal_fir/module.rs`**

## Fix BRA compute() correctness: primal state reset and UI init placement

### Context

The generated C++ for `process = rad((2 : + ~ *(p)), p)` contained:

```cpp
virtual void instanceResetUserInterface() {
    fHslider0 = 0.5f;
    fBraCarry1 = 0.0f;  // BUG: carry is not a UI parameter
}
virtual void compute(...) {
    fRec60 = 0.0f;       // BUG: primal filter memory reset per block
    fBraCarry1 = 0.0f;   // OK: TBPTT(BS,BS) reset before backward sweep
    for (int i0 = 0; i0 < count; ++i0) { /* forward */ }
    for (int i0 = count-1; i0 >= 0; ...) { /* reverse */ }
}
```

Two bugs:

### Bug A — `fRec<N>` reset in `compute()` preamble

`emit_reverse_time_rec_compute_resets` was designed to reset only
`ReverseTimeRec` (LTI adjoint) carriers before each backward sweep, but
it iterated over ALL values in `rec_array_by_group_index` — which also
contains SYMREC primal carriers allocated when lowering the BRA forward
body.  For a pure BRA circuit (`p` in the feedback coefficient), the SYMREC
primal state `fRec<N>` (the filter memory `y[n-1]`) was being reset to 0 at
the start of every `compute()` call, destroying the continuity of the
feedback filter across host blocks.

**Fix**: added `reverse_time_rec_group_ids: HashSet<u32>` to `RecursionState`.
`allocate_group_arrays` in `RecursionAllocCtx` now inserts `group.as_u32()`
into this set when `group` is a `SigMatch::ReverseTimeRec` node.
`emit_reverse_time_rec_compute_resets` filters `rec_array_by_group_index`
to only those `(group_id, _)` keys present in `reverse_time_rec_group_ids`.
SYMREC primal carriers are now never reset in `compute()`.

### Bug B — `fBraCarry<N>` in `instanceResetUserInterface()`

`ensure_bra_delay1_carry` passed `Some(zero)` to `ensure_named_struct_var`,
which calls `register_reset_init` and puts the variable into
`reset_statements` → `instanceResetUserInterface`.  BRA carry variables are
internal DSP state (TBPTT adjoint memory), not UI-controlled parameters; they
must NOT appear in `instanceResetUserInterface`.

**Fix**: changed `Some(zero)` to `None` in the `ensure_named_struct_var` call.
The `register_clear_init` call below it already correctly places the variable
in `instanceClear`.
