## Loop-copy large constant table init instead of unrolling it

Commit date: 2026-07-19
Source commit: this commit

`dynamic-jack-gtk -interp` crashed with
`[FRS-CGEN-CLIF-0004] define_function 'X::instanceConstants' failed:
Compilation(CodeTooLarge)` on `bela/repeater.dsp` and `misc/inlooper.dsp`.
Both declare a writable `rwtable`/`SIGGEN`-seeded looper buffer (88,200 and
220,500 elements respectively); `register_constant_table_init` in
`crates/transform/src/signal_fir/module/state.rs` emitted one `store_table`
FIR statement per element into `instanceConstants`, so the compiled function
body ballooned to hundreds of thousands of instructions and tripped
Cranelift's per-function code-size guard. Read-only tables were unaffected:
`AccessType::Static` values are materialized as a single JIT data blob
(`jit_data.rs`), never unrolled.

`register_constant_table_init` now keeps the unrolled block for tables at or
under 256 elements (cheap, loop-free code for the common case — small
window/envelope tables), and above that threshold declares a `Static`
companion table holding the same constant values, then copies it into the
`Struct` field with a real `simple_for_loop`, mirroring the existing
`register_clear_recursion_array` pattern and matching what the C++ compiler
always does for array initialization (`generateInitArray` never unrolls).
The fix is at the FIR level, so it benefits every backend, not just
Cranelift. `ensure_wrtbl_table` in `tables.rs` now threads `elem_ty` through
to the new `Static` declaration.

Validated: the 385-test `transform` crate suite and the full workspace build
are green. After the user rebuilt and reinstalled the statically-linked
`dynamic-jack-gtk`, both `repeater.dsp` and `inlooper.dsp` ran cleanly
end-to-end (previously crashing, no longer emitting `CodeTooLarge` or any
other error). A sweep of all 295 `faust/examples/*.dsp` files against
`dynamic-jack-gtk -interp` found this bug plus 8 unrelated pre-existing
failures (7 `stack overflow in eval (depth budget 1024)` on
`physicalModeling/fds/*` and `mi-faust/13_Construction/construction.dsp`, 1
`SIGGEN interpreter: foreign functions/constants/variables not supported` on
`physicalModeling/faust-stk/modalBar.dsp`), neither touched by this change.

## Fix stack overflow in FIR dump walkers on large DSPs

Commit date: 2026-07-19
Source commit: this commit

`dynamic-jack-gtk -interp reverbTank.dsp` crashed with SIGSEGV in a stack
guard page: `fir::dump::fingerprint_node` recursed once per raw arena node,
and since `Cons` list spines make raw tree depth linear in program size, a
large DSP overflowed the host's 8 MiB main-thread stack. The fingerprint is
computed unconditionally for cache identity in the shared
`cranelift-ffi/src/factory.rs` path that every `faust-ffi` factory
(including the interpreter) goes through, so any sufficiently large program
crashed before running. The in-repo `impulse_cranelift` binary had masked
the bug by running on a 256 MiB stack — a workaround external hosts
embedding libfaust do not have.

Both `fingerprint_node` and `dump_node` in `crates/fir/src/dump.rs` are now
iterative with an explicit work stack, preserving byte-identical output
(same preorder, same label assignment), so existing `.ir` cache entries
remain valid. New regression test
`dump_and_fingerprint_survive_deep_trees_on_small_stacks` walks a
100k-deep expression chain on a 512 KiB thread; it aborts with a stack
overflow on the previous recursive code and passes now. Validated
end-to-end by compiling and running `reverbTank.dsp` through
`impulse_cranelift`. The installed `dynamic-jack-gtk` links libfaust
statically and must be relinked to pick up the fix.

## Drop redundant parentheses around C-family `if`/`while` conditions

Commit date: 2026-07-19
Source commit: this commit

The shared C/C++ statement emitter wrapped an already parenthesized rendered
condition in the statement's own parentheses, producing
`if ((dsp->fDSCounter_d0 == 0))` and tripping clang's
`-Wparentheses-equality` on every emitted guard. `unwrap_condition` in
`c_family.rs` now strips exactly one outer layer, and only when the leading
parenthesis closes at the final character — `((a == b) && c)` and
`(a) == (b)` are left untouched. Applied to the three shared statement heads
(`If`, `Control`, `WhileLoop`); the shared walker serves only the C and C++
backends. Unit tests cover the whole-wrapper/partial-wrapper distinction, and
the existing C++ core-statement test now asserts the single-parenthesis
forms.

lanes (`cpp`, `c-ss0`, `cpp-vec0-ss0`, `c-vec0-ss0`) matches the references
with zero differences, and the lane compile logs contain no
`-Wparentheses-equality` occurrence. The emission change is cosmetic by
construction (parentheses only); the oracle arbitrates the numerics.

## Gate ZeroPad on the fire index and fail-close unadopted clocked state

Commit date: 2026-07-19
Source commit: this commit

A user-run C++ impulse oracle caught `upsampling_01_accumulator` emitting 2
where the reference says 1 in `-vec` mode. Reproduced in the repository
harness: scalar passes, all eight `-lv 0/1 x -ss 0..3` vector modes fail
identically. Two distinct vector clock-domain bugs were behind the 
divergences, both invisible to the structural pipeline, which certified the
wrong emissions.

First, the pure vector lowerer treated `ZeroPad(x, h)` as a pass-through, so
an upsampled domain read the outer-rate input on every fire and, e.g.,
accumulated an impulse `factor` times. The lowering now mirrors the scalar
`generateZeroPad` gating: `((vclock_d<N>_fire == h - 1) ? x : 0)`, with the
island resolved from the `CountedUpsampling` clock plan by the ZeroPad's own
signal id, failing closed outside an upsampling island. This fixes
`upsampling_01_accumulator` and `upsampling_05_parallel_state` (8/8 oracle
modes each).

Second, `upsampling_02_feedback` fails differently: its recursion has only
constant inputs, so clock inference correctly types it rate-polymorphic
(bottom environment), and the plan hoists it into an outer-rate loop — its
state then advances once per outer sample instead of once per fire. The
analogous downsampling shape (`downsampling(1 : (+ ~ *(0.5)))`, no committed
fixture) was confirmed equally wrong against the scalar emission. Adopting
such groups into their wrapper's domain is a real design (sharing, nested
domains, storage geometry) and is deliberately not improvised here; instead
the clock plan producer and its independent checker both reject any
Immediate/Delayed dependency from an in-domain consumer to an audio-rate
stateful producer (`UnadoptedStatefulRead`), with one legal crossing: the
`ZeroPad` value operand, which is exactly where propagation zero-stuffs an
outer-rate producer into an upsampled domain. `upsampling_02_feedback` and
the downsampling shape now fall back to scalar and match the reference; the
guard demotes no other corpus DSP.

Qualification: the 16-mode sweep is 96 certified / 35 fallback / 1 error in
every mode (exactly the deliberate `upsampling_02_feedback` demotion; the
versioned baseline and certified list were regenerated). The full 8-mode C++
impulse matrix passes for all six clocked corpus DSPs, 48/48. Byte-identity
against the pre-fix build holds for 188 of 192 certified emissions; the four
differences are the two upsampling DSPs under both loop variants — the added
fire gating, oracle-arbitrated. New regressions:
`upsampling_zero_pad_gating_is_certified_and_bit_exact` (fails on the pre-fix
tree with 2 vs 1) and
`unadopted_stateful_recursion_under_upsampling_falls_back`. The complete
vector-retention gate, fmt, clippy with warnings denied, and the workspace
suite are green.

Lesson recorded: the six clocked DSPs promoted into the certified baseline
during X3 had scalar-side numeric validation only; certification is
structural and never a numeric proof. Promotion into the certified list must
require the vector-mode oracle matrix. Open follow-up: domain adoption for
rate-polymorphic stateful groups under clock wrappers (would recover
`upsampling_02_feedback` and the equivalent downsampling/ondemand shapes as
certified vector code).
