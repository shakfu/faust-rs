## Close the four implementation-design gaps in the vectorization plans

Commit date: 2026-07-12
Source commit: this commit

A readiness audit of the vectorization plans found them ready at the
architecture/acceptance-gate level but missing implementation designs at four
spots where an implementer would have to stop. All four are now specified in
[`vector-mode-signal-level-analysis-cpp-port-plan-2026-07-10-en.md`](../vector-mode-signal-level-analysis-cpp-port-plan-2026-07-10-en.md)
as new subsections 4.3-4.6, referenced from phases P3/P4/P5:

- **4.3 Unified signal-use analysis** — one `signal_dependencies` labelled
  child walk owned by a single function (second walks are a review-rejection
  criterion); one iterative post-order traversal filling one `SignalUseInfo`
  table; the `OccMarkup` port as context-sensitive `OccInfo`
  (per-`UseContext` counts, `multi` derived exactly as
  `hasMultiOccurrences()`); `placement.rs`/`delay/plan.rs` become consumers,
  not recomputation sites; P4 exit compares per-signal against C++ `OccMarkup`.
- **4.4 Stable effect-resource identity** — derivation table: state = owning
  `SigId` (hash-consing makes this semantically exact), recursion state =
  (group, projection index), table = creation-node id, UI = canonical path id,
  output = channel, foreign = declared name+signature with no purity
  inference; identities may never be minted during lowering or scheduling.
- **4.5 Region-aware value cache** — three tables (`control`/`owned`/`inline`)
  with explicit visibility rules; cross-loop `owned` hits resolve through
  **pre-planned transports frozen at P4** (on-demand allocation is a plan bug,
  fail closed) — this is what keeps storage names `-ss`-independent; CSE
  stays per routed region and cannot interfere.
- **4.6 Control and per-domain graphs** — `GraphKey::Control` membership and
  edge rules (Konst/Block signals; wrappers as single parent-graph nodes;
  control references as preconditions, not ordering edges; cross-domain
  immediates only via wrapper or declared external), plus the extended audit.

Echoed in
[`lean-rust-certified-porting-plan-2026-07-11-en.md`](../lean-rust-certified-porting-plan-2026-07-11-en.md):
section 4.7 gains the resource-identity derivation rules; R2D states that
`DecorationCertificate` is the exported projection of the same
`SignalUseInfo` traversal, so exported facts cannot drift from planner facts.

With these, P0/P1/P2 remain immediately codable and the PV slice no longer
has to invent designs mid-flight.

## Mechanize the lockstep extension in the Lean specification

Commit date: 2026-07-12
Source commit: this commit

Extended
[`vector-mode-scheduling-formal-spec.lean`](../vector-mode-scheduling-formal-spec.lean)
with the lockstep bundling layer announced in the port plan. The file still
compiles with bundled Std only, no `sorry`, no axioms.

Added:

- **Structural isomorphism modulo leaves.** `Shape` is the constructor skeleton
  of an `Expr` with leaf payloads erased (literal values, input channels, state
  resources; projection indices and clock annotations stay structural).
  `IsoModuloLeaves a b := a.shape = b.shape`, so the equivalence laws are
  inherited from `Eq`; `isoB` is the decidable checker (the reference for the
  planned Rust shape-hash detector), with `#guard` positives and negatives.
- **`iso_decorations_agree` (proved).** Isomorphic lanes receive the same value
  type, rate, vectorability, and clock — the fact that lets one bundle share a
  single transport element type and placement. Effects are deliberately not
  claimed equal; pairwise commutation handles them.
- **`LoopKind.lockstep (width)`** as a fourth kind (serial in time, parallel
  across lanes).
- **`LockstepObligations`** — the finite bundle gate: ≥ 2 lanes, no duplicates,
  lanes in the plan, and pairwise `Independent` (new `Reaches` reachability
  over plan edges), commuting `loopEffects`, same epoch rank.
- **`LockstepSafe := FissionSafe`** with the lockstep order substituted,
  making explicit that the extension adds no new semantic axiom.

Updated the reader guides
[`docs/vector-mode-scheduling-formal-spec-guide-en.md`](../../docs/vector-mode-scheduling-formal-spec-guide-en.md)
and [`-fr.md`](../../docs/vector-mode-scheduling-formal-spec-guide-fr.md):
new layer-tour subsection 3.7 on lockstep bundling (execution semantics
renumbered 3.8), layer diagram extended, and the proven-vs-promised ledger
gains `iso_decorations_agree` (proven) and `LockstepSafe` (stated). Fixed the
port plan §8.1 wording ("fourth" loop kind, not "third") and noted the
mechanization there.

## Add lockstep vectorization to the vector plans, backed by a measured benchmark

Commit date: 2026-07-12
Source commit: this commit

Explored "vertical" vectorization: k structurally identical, mutually
independent recursive loops (e.g. 4 biquads on 4 outputs) executed in lockstep,
one SIMD lane per instance — the only recursion acceleration compatible with
the bit-exact contract, since each lane runs exactly its scalar IEEE op
sequence (unlike parallel-form IIR restructurings, which reassociate).

New benchmark [`bench_lockstep_biquad.c`](../../tests/bench/bench_lockstep_biquad.c)
(4 independent RBJ peak biquads, DF2T, distinct per-lane coefficients; Apple
M1, clang 14, `-O3`, no fast-math, chunk 512, best of 3):

- V1 four separate scalar loops (current lowering): 4.15 ns/lane-sample;
- V2 plain fusion, 4 scalar chains: **3.69x, bit-exact** — clang SLP-vectorizes
  the fused body on its own, no interleaving/intrinsics needed (C/C++ backends
  only need the plan-level fusion decision);
- V3 lane-innermost C loop + chunk transposes: 3.34x bit-exact (the
  backend-neutral FIR lowering for interpreter/Cranelift/wasm);
- V4k explicit NEON on interleaved buffers: 4.53x ceiling; its 1e-5 drift vs
  scalar demonstrates the FMA-contraction-policy requirement;
- transpose overhead 12-22% ⇒ external `compute` ABI stays planar.
