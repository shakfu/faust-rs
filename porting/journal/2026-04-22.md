## docs(propagate): clarify de Bruijn index handling in forward-mode AD

Commit date: 2026-04-22

Source commit: pending

### Context

The inline comment classifying the `Proj` group node in
`crates/propagate/src/forward_ad.rs` referenced a `SYMREC` match arm and
described the `Other` branch as "already fully separated into primal and
tangent groups by the SYMREC arm above".

That description was stale: the module docstring states that the FAD
transform only ever sees de Bruijn form (`de_bruijn_to_sym` runs in
`signal_prepare`, after FAD), and there is in fact no `SYMREC` arm in
`transform_uncached` ‚Äî symbolic-recursion nodes would simply fall through
to the zero-tangent default. The misleading comment also left the
indexing scheme under-documented: the three distinct notions involved
(static `DEBRUIJNREF` level, traversal counter `debruijn_depth`, and
slot index on `Proj`) were only implied.

### Implementation

Updated `crates/propagate/src/forward_ad.rs`:

- Rewrote the "Projection and recursive groups" section of the module
  docstring to separate the three bookkeeping quantities (level,
  `debruijn_depth`, slot index) and to cover all four `Proj`
  classification cases in a single table.
- Replaced the inline `Proj` classifier comment so it no longer points
  at a non-existent `SYMREC` arm. The `Other` branch is now described
  accurately as a defensive, unreachable identity fallback.

### Verification

- `cargo check -p propagate` ‚Äî clean.

No behavioural change; documentation only.

## refactor(propagate): use typed tlib helpers for de Bruijn nodes in FAD

Commit date: 2026-04-22

Source commit: pending

### Context

`crates/propagate/src/forward_ad.rs` classified `DEBRUIJNREC` and
`DEBRUIJNREF` nodes by pulling the raw `NodeKind::Tag` out of the arena
and comparing its name string against `super::DEBRUIJNREC_TAG` /
`super::DEBRUIJNREF_TAG`. It also rebuilt the interleaved recursion
node by hand with `arena.intern_tag` + `arena.intern`.

Three problems:

- The string-tag comparison duplicates logic already encapsulated by
  `tlib::recursion::{match_de_bruijn_rec, match_de_bruijn_ref}`, which
  is the canonical accessor for these nodes and the one every other
  consumer uses.
- The `Proj` classifier unwrapped the `DEBRUIJNREF` level via
  `tree_to_int(arena, node.children.as_slice()[0]).unwrap_or(1)` ‚Äî
  reaching through the raw child slot with a silent default on
  malformed input. The typed helper returns `Option<i64>` cleanly.
- `SigMatch` deliberately does not model `DEBRUIJNREC`/`DEBRUIJNREF`:
  these nodes live at the `tlib` layer and are meant to disappear at
  `de_bruijn_to_sym` time, before most signal consumers see the DAG.
  FAD runs before that conversion and is the only pass that needs to
  observe them; widening `SigMatch` for a single consumer would force
  every other pass to add a `_ =>` arm.

### Implementation

Updated `crates/propagate/src/forward_ad.rs`:

- Imports: pulled in `de_bruijn_rec`, `match_de_bruijn_rec`,
  `match_de_bruijn_ref` from `tlib`; dropped `tree_to_int` (no longer
  referenced).
- Top of `transform_uncached`: replaced the nested
  `arena.node` / `NodeKind::Tag` / `tag_name` / string-compare ladder
  with two flat calls to `match_de_bruijn_ref` /
  `match_de_bruijn_rec`. The rebuilt FAD recursion node is now
  constructed via the `de_bruijn_rec` builder rather than
  `arena.intern_tag` + `arena.intern`.
- `Proj` classifier: collapsed the four-level nested `if` into
  `if match_de_bruijn_rec ‚Ä¶ else if match_de_bruijn_ref ‚Ä¶ else`. The
  level is now extracted with the typed accessor ‚Äî no more
  `unwrap_or(1)` on a raw child index.
- `super::DEBRUIJNREC_TAG` / `super::DEBRUIJNREF_TAG` are no longer
  referenced from this file; the tag-name strings are fully
  encapsulated in `tlib`.

### Verification

- `cargo check -p propagate` ‚Äî clean.
- `cargo test -p propagate` ‚Äî 3 unit + 45 integration tests pass.

No behavioural change; documentation only.

## refactor(propagate): factor FAD helpers and document seed lifting

Commit date: 2026-04-22

Source commit: pending

### Context

Three residual issues in `crates/propagate/src/forward_ad.rs`:

- The six `FFun` dispatch arms (tanh / sinh / cosh / atanh / asinh /
  acosh) all repeated the same prologue/epilogue: `transform(arg)`,
  `vec_to_list(&[dual.primal])`, fresh `SigBuilder`, `b.ffun(ff, _)`,
  then wrap the computed tangent into a `Dual`. Only the tangent
  formula varied. Roughly 105 lines of boilerplate for six rules.
- The "zero tangent" shape
  `Dual { primal: sig, tangent: SigBuilder::new(self.arena).real(0.0) }`
  appeared six times (Real, Input, sliders, buttons/checkboxes,
  bargraphs, unmatched FFun branches, and the generic `_` fallback),
  obscuring the intent behind a repeated literal.
- A French comment block ("--- LE PATCH CRUCIAL ---") explained the
  seed-lifting step on entry into a `DEBRUIJNREC` body, but the project
  requires English journal/source documentation, the explanation only
  said *how* (restore/shift) not *why* (hash-consing makes
  `DEBRUIJNREF(k)` and `DEBRUIJNREF(k+1)` distinct `SigId`s, so an
  unlifted seed becomes invisible inside the body and its tangent
  collapses to zero). The module docstring made no mention of the
  mechanism.

### Implementation

Updated `crates/propagate/src/forward_ad.rs`:

- Added `ForwardADTransform::zero_tangent(sig)` and replaced every
  literal zero-tangent construction with a call to it.
- Added `ForwardADTransform::ffun_unary_chain(ff, arg, tangent_fn)`
  that owns the transform / list-wrapping / `ffun` rebuild shared by
  every unary foreign-function rule. The callback receives
  `(builder, primal, x_primal, x_tangent)` so rules like `tanh` can
  reuse `primal` for `x' ¬∑ (1 ‚àí tanh¬≤(x))` without a second `ffun`
  call.
- Moved the entire `FFun` block into a dedicated
  `transform_ffun(sig, ff, largs)` method. The `SigMatch::FFun` arm is
  now one line; each rule is 3‚Äì8 lines of tangent formula.
- Replaced the French `// --- LE PATCH CRUCIAL ---` block with an
  English paragraph explaining why the seed must be lifted on entry
  into a `DEBRUIJNREC` body (hash-consing + de Bruijn shift) and
  restored on exit. Added a matching short paragraph in the
  "Dual-number algebra" section of the module docstring so the
  mechanism is discoverable from the top of the file.

### Verification

- `cargo check -p propagate` ‚Äî clean.
- `cargo clippy -p propagate --lib` ‚Äî clean.
- `cargo test -p propagate` ‚Äî 3 unit + 45 integration tests pass.

## fix(propagate): swap select2 argument order for min/max tangents in FAD

Commit date: 2026-04-22

Source commit: pending

### Problem

`SigMatch::Min` and `SigMatch::Max` in `crates/propagate/src/forward_ad.rs`
computed the tangent via `select2(cond, dual_x.tangent, dual_y.tangent)`.

`select2(sel, s0, s1)` follows the Faust convention: `s0` is selected when
`sel = 0` (condition false) and `s1` when `sel = 1` (condition true). The
conditions are:

- `min`: `cond = x < y` ‚Üí true (=1) when x is the minimum ‚Üí must select `x'`
- `max`: `cond = x > y` ‚Üí true (=1) when x is the maximum ‚Üí must select `x'`

The original code had `s0 = dual_x.tangent, s1 = dual_y.tangent`, so a true
condition selected `dual_y.tangent` ‚Äî the derivative of the *non-active*
branch. This is the opposite of what the chain rule requires, meaning the
gradient was negated / mis-attributed whenever x and y differed.

### Fix ‚Äî `crates/propagate/src/forward_ad.rs`

Swapped the two tangent arguments in both arms:

```
// min: cond = (x < y), select x' when cond is true
let tangent = b.select2(cond, dual_y.tangent, dual_x.tangent);
// max: cond = (x > y), select x' when cond is true
let tangent = b.select2(cond, dual_y.tangent, dual_x.tangent);
```

Now when `cond = 1` (the active branch is x), `select2` returns
`dual_x.tangent` as required by the chain rule.

### Verification

- `cargo check -p propagate` ‚Äî clean.
- `cargo test -p propagate` ‚Äî all tests pass.
