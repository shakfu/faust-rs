### Share `create*FromFile/String` FFI boilerplate in interp and cranelift backends
Commit date: 2026-02-26

Reduced duplication between `create...FactoryFromFile` and
`create...FactoryFromString` in both FFI backends without changing file-vs-string
compilation semantics.

What changed:

- `crates/interp-ffi/src/factory.rs`
  - added `create_interp_factory_with_argv(...)`
  - `createCInterpreterDSPFactoryFromFile` and
    `createCInterpreterDSPFactoryFromString` now share:
    - post-`argv` compile-result error handling (`error_msg`)
    - factory allocation
    - cache insertion
  - file constructor still calls `compile_factory_from_file_fastlane(...)`
  - string constructor still calls `compile_factory_from_string_fastlane(...)`
- `crates/cranelift-ffi/src/factory.rs`
  - added `create_cranelift_factory_with_argv(...)`
  - `createCCraneliftDSPFactoryFromFile` and
    `createCCraneliftDSPFactoryFromString` now share:
    - post-`argv` preflight/build error handling (`error_msg`)
    - factory allocation
    - cache insertion
  - file constructor still uses file preflight (`preflight_compile_file_to_cranelift`)
  - string constructor still uses source preflight (`preflight_compile_source_to_cranelift`)

Why:

- removes repeated FFI boilerplate while preserving the important difference
  between file-based and string-based compilation paths
- keeps file import-base behavior intact (`default_import_search_base(path)`)

Validation:

- `cargo fmt --all` ✅
- `cargo clippy -p interp-ffi -p cranelift-ffi --all-targets -- -D warnings` ✅
- `cargo test -p interp-ffi -- --nocapture` ✅
- `cargo test -p cranelift-ffi -- --nocapture` ✅

### Factorize shared FFI helpers and common `-I/-cn` option parsing
Commit date: 2026-02-26

Extended the earlier FFI cache factorization by moving more backend-agnostic
FFI mechanics into `utils`, then wiring both `interp-ffi` and `cranelift-ffi`
to the shared helpers.

What changed:

- `crates/utils/src/lib.rs`
  - shared `UIGlue` / `MetaGlue` `#[repr(C)]` callback-table definitions
  - shared C-string helpers:
    - `alloc_c_string`
    - `free_c_string`
  - shared opaque-pointer helpers:
    - `alloc_opaque<T>`
    - `free_opaque<T>`
  - shared FFI memory/error/argv helpers:
    - `free_c_memory_c_string_only`
    - `write_error_4096`
    - `decode_c_argv`
  - shared C-string argument decoding helpers:
    - `required_c_str_arg`
    - `optional_c_str_arg`
  - shared empty `char**` helper:
    - `null_c_string_array`
  - shared FFI option subset parser:
    - `FfiCompileArgs`
    - `parse_ffi_compile_args`
    - supports `-I <path>` and `-cn <name>` (unknown options ignored)
- `crates/utils/Cargo.toml`
  - local lint override to allow `unsafe_code` for shared raw-pointer FFI helpers
- `crates/interp-ffi/src/types.rs`
  - re-export shared `UIGlue` / `MetaGlue`
  - delegate opaque alloc/free and C-string alloc to shared helpers
- `crates/interp-ffi/src/factory.rs`
  - use shared `write_error_4096`, `decode_c_argv`, `free_c_memory_c_string_only`
  - use shared `required_c_str_arg` / `optional_c_str_arg`
  - use shared `null_c_string_array`
  - local `parse_ffi_compile_args` now delegates to `utils`
- `crates/cranelift-ffi/src/types.rs`
  - re-export shared `UIGlue` / `MetaGlue`
  - delegate opaque alloc/free and C-string alloc to shared helpers
- `crates/cranelift-ffi/src/factory.rs`
  - use shared `write_error_4096`, `decode_c_argv`, `free_c_memory_c_string_only`
  - use shared `required_c_str_arg` / `optional_c_str_arg`
  - use shared `null_c_string_array`
  - file search-path collection now reuses shared `parse_ffi_compile_args`

Notes:

- The shared FFI option parser was intentionally restricted to `-I <path>`
  (space form only) and `-cn <name>`.
- Backend-specific factory/instance semantics, bitcode formats, and runtime
  behavior remain local to each FFI crate.

### `cargo test` fix: expose `rlib` for `cranelift-ffi` examples
Commit date: 2026-02-26

`cargo test` failed because Cargo also compiles crate examples, and the local
untracked example `crates/cranelift-ffi/examples/corpus_scan.rs` imports the
crate as `faust_cranelift`.

Root cause:

- `crates/cranelift-ffi/Cargo.toml` exposed only `cdylib` and `staticlib`
  crate types
- Rust examples/tests need an `rlib` artifact to link the crate as a normal
  Rust dependency

Fix:

- updated `crates/cranelift-ffi/Cargo.toml`
  - `crate-type = [\"rlib\", \"cdylib\", \"staticlib\"]`

Validation:

- `cargo test` ✅ (workspace)
