### Factorize FFI factory cache mechanism into `utils::FactoryCache<T>`
Commit date: 2026-02-26

Factorized the duplicated SHA-keyed factory-cache mechanism used by
`interp-ffi` and `cranelift-ffi` into a shared generic helper in `utils`.

What changed:

- added `utils::FactoryCache<T>` in `crates/utils/src/lib.rs`
  - `insert`
  - `lookup`
  - `remove_by_ptr`
  - `drain`
  - `all_sha_keys`
  - `start_mt` / `stop_mt` compatibility flag
- `crates/interp-ffi/src/cache.rs`
  - now uses `LazyLock<FactoryCache<InterpreterDspFactory>>`
  - keeps backend-local wrapper API unchanged
- `crates/cranelift-ffi/src/cache.rs`
  - now uses `LazyLock<FactoryCache<CraneliftDspFactory>>`
  - keeps backend-local wrapper API unchanged
- added `utils` dependency to:
  - `crates/interp-ffi/Cargo.toml`
  - `crates/cranelift-ffi/Cargo.toml`
- `Cargo.lock` updated

Why:

- removes duplicated cache logic across FFI backends
- centralizes behavior before adding more backend FFI crates
- preserves backend-specific docs/API surfaces through thin wrappers

Validation:

### `cargo test` fix: expose `rlib` for `cranelift-ffi` examples
Commit date: 2026-02-26

`cargo test` failed because Cargo also compiles crate examples, and the local
untracked example `crates/cranelift-ffi/examples/corpus_scan.rs` imports the
crate as `faust_cranelift`.

Root cause:

- `crates/cranelift-ffi/Cargo.toml` exposed only `cdylib` and `staticlib`
  crate types
- Rust examples/tests need an `rlib` artifact to link the crate as a normal
  Rust dependency

Fix:

- updated `crates/cranelift-ffi/Cargo.toml`
  - `crate-type = [\"rlib\", \"cdylib\", \"staticlib\"]`

Validation:

- `cargo test` ✅ (workspace)
