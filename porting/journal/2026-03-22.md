# 2026-03-22

## Fix: Cranelift JIT panics on large DSP (AArch64 branch-offset overflow) — graceful fallback

### Bug

`minimoog-novation.dsp` (and potentially any large DSP) caused an unrecoverable
panic with exit code 101 when compiled via the Cranelift backend:

```
thread 'main' panicked at cranelift-codegen-0.117.2/src/machinst/buffer.rs:1392:
assertion failed: (label_offset - offset) <= kind.max_pos_range()
```

On AArch64, the `B.cond` instruction has a ±1 MiB branch displacement limit.
When the JIT-compiled `compute` function body exceeds this limit, Cranelift's
island-emission logic fails to insert veneers in time (a Cranelift bug), causing
the internal assertion to fire and the process to abort.

### Fix — `crates/codegen/src/backends/cranelift/mod.rs`

- Added `force_stub: bool` parameter to `declare_compute_stub`.  When `true`,
  the subset-check and real lowering are skipped; a `return` stub is emitted
  unconditionally (ignoring `fail_on_subset_gap`).
- Extracted the JIT-creation body of `generate_cranelift_module` into
  `try_generate_cranelift_module(…, force_stub: bool)`.
- `generate_cranelift_module` wraps the first attempt (with `force_stub=false`)
  in `std::panic::catch_unwind(AssertUnwindSafe(…))`.  If Cranelift panics, the
  partial JIT module is dropped during stack unwinding and the function retries
  with `force_stub=true`.  The DSP can still run via the interpreter sidecar.

### Result

`minimoog-novation.dsp` now compiles successfully with
`compute_body_lowered: false` and a warning printed to stderr instead of
crashing the process.

## Fix: Cranelift float comparison with Int32 result type produces silence

### Bug

`harpeautomation.dsp` compiled and ran without error in the Cranelift backend, but
produced only silence (all-zero output). The C++ and interpreter backends were both
correct.

### Root Cause

`lower_binop` in `crates/codegen/src/backends/cranelift/mod.rs` had a dispatch
defect for comparison operators whose FIR result type is `Int32` rather than `Bool`.

FIR can emit `BinOp { op: Lt, lhs: Float32, rhs: Float32, typ: Int32 }` — a
floating-point less-than comparison whose result is represented as an `Int32`
boolean (0 or 1).  The `typ: Int32` field denotes the *result* type, not the
operand type.

The existing code path only invoked the "check actual CLIF operand type and pick
float vs integer comparison" logic when `typ == FirType::Bool`.  For `typ ==
FirType::Int32`, it fell through to a coercion step:

```rust
let l = self.coerce_value_to_fir_type(l, Int32)?;  // fcvt_to_sint: f32 → i32
let r = self.coerce_value_to_fir_type(r, Int32)?;  // fcvt_to_sint: f32 → i32
// then: int_cmp_to_i8(SignedLessThan, i32, i32)
```

`fcvt_to_sint` truncates toward zero, so `0.5 → 0` and `fabs(0.0) → 0`, giving
`0 < 0 = false` instead of `0.0 < 0.5 = true`.

In `harpeautomation.dsp` the trigger-check pattern for each Karplus-Strong string
is `fabs(fTbl987[idx] - note_k) < 0.5`, which is a `FunCall("fabs", ...) < 0.5`
comparison with `typ: Int32`.  The wrong result (always 0) meant no string ever
triggered, producing complete silence.

### Minimal Reproducer

`tests/corpus/rep_65_fabs_trigger.dsp`:

```faust
import("stdfaust.lib");
N = 10;
counter = (+(1)) ~ %(N) : float;
process = (abs(counter - 5.0) < 0.5) : float;
```

Cranelift returned 0 at sample 4 (counter==5); interpreter returned 1.

### Fix

Moved `let lty = self.fb.func.dfg.value_type(l)` before the Bool-branch and
extended the condition to also apply when the op is a comparison and the CLIF
operand type is float:

```rust
let is_cmp_op = matches!(op, Eq | Ne | Lt | Le | Gt | Ge);
if matches!(typ, FirType::Bool) || (is_cmp_op && lty.is_float()) {
    // dispatch float_cmp_to_i8 or int_cmp_to_i8 based on lty
}
```

This ensures that `BinOp { op: Lt, Float32 operands, typ: Int32 }` uses
`float_cmp_to_i8(LessThan, f32, f32)` and returns the correct result.

### Regression Coverage

Added `tests/corpus/rep_65_fabs_trigger.dsp` to `cranelift_interp_runtime_diff_smoke_corpus`.
Also added `rep_62_select2_trigger.dsp`, `rep_63_rwtable.dsp`, and
`rep_64_dynamic_rem.dsp` (all pass) during the bisection investigation.

All 27 `cranelift-ffi` tests pass.

## Fix: `import("stdfaust.lib") + import("demos.lib")` triggers 19 "multiple definitions" parse errors

### Bug

`guitarEffectChain.dsp` (WAC 2017) failed with:

```
parse failed: errors=19, recoveries=0, diagnostics=19
error [FRS-PARSE-0001] multiple definitions of a zero-argument symbol are not allowed
```

All 19 errors pointed to the `<:` (SPLIT) operator position — an apparent red
herring from the parser location tracking.

The file contains:

```faust
import("stdfaust.lib");
import("demos.lib");
process = dm.cubicnl_demo : dm.wah4_demo <: dm.phaser2_demo : ...;
```

### Root Cause

Both `stdfaust.lib` and `demos.lib` define the same 19 library-alias symbols at
their top level:

```faust
ma = library("maths.lib");
ba = library("basics.lib");
de = library("delays.lib");
si = library("signals.lib");
... (19 total)
```

`format_definitions` in `crates/parser/src/lib.rs` groups all definitions by
name across every imported file.  When two imports both provide `ma = …`, the
entry in the `BTreeMap` collects two variants.  `make_definition_from_variants`
then saw `variants_rev.len() > 1` with `expected_arity == 0` and emitted the
error (once per doubly-defined symbol = 19 errors).

The C++ compiler handles this silently via its environment cons-list: later
`import` definitions are prepended and shadow earlier ones without any error.

### Fix — `crates/parser/src/lib.rs`

In `make_definition_from_variants`, replaced the hard error for
`expected_arity == 0` with last-import-wins behaviour: when multiple zero-arity
definitions exist for the same name, use the most-recently-imported one
(`first_body`, which is the first element of `variants_rev.iter().rev()`).

```rust
if expected_arity == 0 {
    // Multiple zero-arity definitions arise when two libraries both define the
    // same alias (e.g. stdfaust.lib and demos.lib both have `ma = library("maths.lib")`).
    // Mirror C++ import-shadowing: last import wins.
    let nil = self.nil();
    return self.make_definition(name, nil, first_body);
}
```

### Result

`guitarEffectChain.dsp` and any DSP that combines `stdfaust.lib` with another
library that re-exports the same standard aliases now compiles correctly.
All 9 parser tests pass.
