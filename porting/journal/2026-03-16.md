### `signal_prepare`: wire algebraic simplification into the pipeline

Commit date: 2026-03-16

Source: `crates/transform/src/signal_prepare.rs`, `crates/normalize/src/simplify.rs`,
`crates/normalize/src/lib.rs`, `crates/transform/Cargo.toml`

#### Context

The C++ Faust compiler runs `simplify()` (in `normalform.cpp`) after `typeAnnotation`
and before FIR lowering.  In faust-rs, `simplify` existed in `crates/normalize` but
was only called inside the normalize crate itself (`simplify_const`), not wired into
the main signal-preparation pipeline.  This meant constant-foldable subexpressions
(e.g. `max(0.25, 0.5)`, `IntCast(Real(1.5))`, `Delay1(x)`) were passed through to
the FIR lowerer without reduction.

#### Changes

1. **`normalize::simplify` made public** â€” Changed `pub(crate) fn simplify` â†’
   `pub fn simplify` in `crates/normalize/src/simplify.rs`, and added
   `pub use simplify::{simplify, simplify_const};` in `lib.rs`.

2. **`normalize` added as dependency of `transform`** â€” Added
   `normalize = { path = "../normalize" }` to `crates/transform/Cargo.toml`.

3. **Pipeline step in `prepare_signals_for_fir`** â€” Inserted between the full-type
   annotation and FIR lowering, mirroring the C++ order:
   ```
   clone â†’ de_bruijn_to_sym â†’ infer_simple_types â†’ promote
       â†’ infer_full_types â†’ simplify â†’ re-infer (simple + full) â†’ FIR
   ```
   After simplification both type maps are recomputed because new nodes introduced
   by simplify won't have entries in the pre-simplification maps.

4. **`simplify_outputs` helper** â€” Thin wrapper that maps `simplify()` over each
   output signal.

#### Downstream fixes required

Wiring `simplify` exposed or altered several downstream assumptions:

**`Delay1` normalization** â€” `simplify` calls `normalize_delay1_term`, converting
`Delay1(x)` â†’ `Delay(x, Int(1))`.  The FIR lowerer's `lower_delay` only called
`lower_delay_state` for `Delay1`; `Delay(sig, Int(1))` fell through to
`lower_fixed_delay`, omitting the single-sample state init.  Fixed by adding a
fast-path in `lower_delay`: when `amount` is `Int(1)`, route to `lower_delay_state`
(same semantics as `Delay1`).

**Constant folding of `max(0.25, 0.5)`** â€” The `pow_min_max` test expected an
`fmax` extern prototype and a `fmax(...)` call in generated code.  After
simplification, `max(0.25, 0.5)` is folded to `Real(0.5)` at compile time.
Updated test to assert `fmax` is absent and the RHS is a float constant.

**`IntCast(Real(1.5))` folded to `Int(1)`** â€” The `promotes_delay_amounts_to_int`
test expected `IntCast(Real(1.5))` to survive into the prepared signals.  After
simplification it becomes `Int(1)`.  Updated test to expect `Int(1)`.

**`normalize_add_term` may reorder operands** â€” Inside `simplify`, additive terms
are reordered.  The `converts_shared_debruijn` test assumed `lhs` was always the
`Proj` side of an `add`.  Updated test to check either side.

**State slot renaming in `fastlane_delay_echo`** â€” The `fRecN`/`fVecN` numbering
depends on node-discovery order during FIR lowering.  Simplification changes the
node graph (fewer intermediate nodes, different sharing), shifting slot numbers
from `fVec4`/`fRec4` to `fVec12`/`fRec12`.  Updated integration test to match
patterns without hard-coded numeric suffixes.

### `fbc_to_cpp`: fix `Loop` indentation â€” remove spurious outer braces

Commit date: 2026-03-16

Source: `crates/codegen/src/backends/interp/fbc_to_cpp.rs`

#### Context

The `Loop` FBC instruction was compiled into:

```cpp
{
    iVec[10] = 0;       // init block (t+1)
    while (true) {      // t+1 â€” same level as init, visually confusing
        ...
    }
}
```

The outer `{â€¦}` wrapper and extra indentation level were unnecessary.
`BlockComp` uses a single shared `ic`/`rc` counter throughout the entire
block compilation, so variable names in the init block (`iI0`, `iI1`, â€¦)
and the loop body are already globally unique â€” no C++ scope is needed to
prevent redeclaration.

#### Change

Removed the outer `{` / `}` and flattened the indentation: the init block
and `while (true) {` are now both emitted at level `t`, with the loop body
at `t+1`:

```cpp
iVec[10] = 0;
while (true) {
    ...
}
```

Verified with `waveform4.fbc` â€” `instanceClear` and the `compute` sample
loop both format correctly.

#### Files changed

| File | Change |
|------|--------|
| `crates/codegen/src/backends/interp/fbc_to_cpp.rs` | Remove outer `{â€¦}` in `Loop` handler; adjust indentation levels; update doc table |

### `signal_fir`: correct waveform lowering â€” cycling index, global static tables, interval bounds

Commit date: 2026-03-16

Source: `crates/transform/src/signal_fir/module.rs`, `crates/fir/src/lib.rs`,
`crates/fir/src/checker.rs`, `crates/codegen/src/backends/cpp/mod.rs`,
`crates/codegen/src/backends/c/mod.rs`

#### Context

Comparing the C++ output of faust-rs against the reference Faust compiler for
`waveform1.dsp`â€“`waveform6.dsp` revealed three independent bugs in the waveform
lowering path:

1. **`SIGWAVEFORM` used as a direct output was not cycling** â€” the generated code
   read `fTbl[0]` every sample instead of cycling through the table with a modular
   index.
2. **Waveform tables were emitted as struct members with a `classInit` fill loop**
   â€” the reference compiler emits them as file-scope `const static` arrays with
   inline initialisers, saving per-instance memory and eliminating the init loop.
3. **`SIGRDTBL` table reads used unconditional modular wrapping** â€” for waveform5/6
   the index interval is `[0, N-1]`, so the bounds check is unnecessary; the
   reference compiler emits direct array access without any clamping.

#### Bug 1 â€” cycling index for bare `SIGWAVEFORM`

`lower_waveform` now generates a named `Int32` struct state slot `iWave{id}`:
- initialised to `0` in `instanceClear` (via `register_clear_init`),
- advanced by `(iWave + 1) % N` deferred to `compute_updates` (appended after the
  sample body, matching the reference order),
- the current value is read first, then the update fires â€” so sample 0 yields
  `table[0]`, sample 1 yields `table[1]`, etc.

#### Bug 2 â€” file-scope `const static` tables

Added a `static_declarations: Vec<FirId>` field to `SignalToFirLower`.
`ensure_waveform_table` and `ensure_wrtbl_table` now call
`b.declare_table(..., AccessType::Static, elem_ty, &values)` (with inline values)
and push to `static_declarations` instead of `struct_declarations`.  No separate
`StoreTable` fill loop is generated.

**FIR layer changes** (`crates/fir/src/lib.rs`):
- `FirMatch::Module` gained a fourth named field `static_decls: FirId`.
- `FirBuilder::module()` accepts a 7th argument; `match_fir` decodes 7 children;
  `fir_children` returns 4 IDs for Module.

**Checker** (`crates/fir/src/checker.rs`):
- `check_phase1` extracts `static_decls` explicitly (was `..`) and runs an M05
  check: the block is passed to `check_globals`, which registers each static table
  name in `symbols.globals`.  Without this, the verifier raised
  `FRS-FIR-0001: use of undeclared variable 'fTblN'`.

**Backends**:
- C++ backend: new `emit_static_tables` helper emits
  `const static type name[N] = {v0,v1,...};` before the class definition.
- C backend: same pattern, emitting `static const type name[N] = {v0,v1,...};`
  before the struct definition.

#### Bug 3 â€” interval-aware table index bounds

New method `table_index_with_bounds(index_fir, index_sig, table_len)` replaces the
unconditional call to `normalized_table_index` in `lower_rdtbl`:

| Index interval | Generated code |
|---|---|
| `[0, N-1]` (proven safe) | direct access â€” no bounds check |
| `[0, hi]` with `hi > N-1` | `min_i(index, N-1)` |
| signed or unknown bounds | `min_i(max_i(index, 0), N-1)` |
| no interval info | fall back to `% N` (modular wrap) |

This makes waveform5/6 emit `iTbl[int(4*...)]` without any clamp, matching the
Faust reference exactly.
