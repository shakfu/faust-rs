## refactor(compiler,xtask): remove SignalFirLane LegacyBridge and legacy lane workflows

Commit date: 2026-04-12

Source commit: pending

### Context

`SignalFirLane::LegacyBridge` had become migration residue rather than an
active production path.

The codebase had already converged on `TransformFastLane` for the meaningful
backend/status workflows, but the compiler facade, CLI, `xtask`, tests, and a
few active docs still carried:

- the `LegacyBridge` enum variant,
- legacy dispatch branches and placeholder FIR bridge code in
  `crates/compiler/src/lib.rs`,
- user-facing `legacy|fast` lane choices,
- and runtime-trace lane-diff workflows whose only purpose was to compare
  against the soon-to-be-removed bridge.

Keeping that scaffolding in place increased maintenance cost and kept stale
public defaults/documentation alive after the transform-owned lowering route
had effectively become the only supported path.

### Implementation

Updated:

- `crates/compiler/src/lib.rs`
- `crates/compiler/src/main.rs`
- `crates/compiler/tests/signal_fir_lane.rs`
- `crates/xtask/src/main.rs`
- `crates/compiler/README.md`
- `crates/xtask/README.md`
- `docs/user-cli-guide-en.md`
- `docs/developer-workflows-en.md`
- `tests/runtime_traces/README.md`

Implemented:

- removed `SignalFirLane::LegacyBridge`, leaving
  `SignalFirLane::TransformFastLane` as the only remaining public route
- switched all remaining compiler facade defaults to the transform fast lane,
  including C/C++ and file-backed WASM artifact helpers
- deleted the legacy FIR bridge lowering/dispatch code paths from
  `crates/compiler/src/lib.rs`
- simplified the CLI lane enum/help so `--signal-fir-lane` only accepts `fast`
- removed `xtask` lane-diff orchestration and simplified runtime-trace lane
  parsing to the transform fast lane only
- pruned legacy-vs-fast regression assertions from `signal_fir_lane` tests,
  keeping fast-lane coverage intact
- refreshed active docs/README files so they no longer describe the removed
  bridge or `legacy|fast` user workflows

### Validation

- `cargo test -p compiler signal_fir_lane --no-run`
- `cargo test -p xtask --no-run`
- `cargo test -p compiler --test signal_fir_lane fastlane_compiles_lowpass_feedback_fixture -- --exact`
- `cargo test -p compiler --test signal_fir_lane default_interp_api_uses_fastlane_runtime_lowering -- --exact`

## fix(cranelift-ffi): make runtime descriptor sample-rate test self-contained in CI

Commit date: 2026-04-12

Source commit: pending

### Context

The GitHub Actions run `24253067210` failed in
`runtime::tests::runtime_descriptor_tracks_sample_rate_fields` on the
`cranelift-ffi` crate.

The test compiled `tests/corpus/rep_61_fmin_sr.dsp`, which imports
`stdfaust.lib`. That dependency happened to resolve on the local development
machine but not on the CI runner, where the test failed with
`Import(UnresolvedImport { name: "stdfaust.lib", ... })`.

This made the test environment-sensitive even though its actual assertion only
checks that FIR lowering exposes the runtime descriptor sample-rate field.

### Implementation

Updated:

- `crates/cranelift-ffi/src/runtime.rs`

Implemented:

- replaced the file-backed `rep_61_fmin_sr.dsp` fixture usage in
  `runtime_descriptor_tracks_sample_rate_fields` with an inline DSP snippet
  that defines `SR` via `fconstant(int fSamplingFreq, <math.h>)`
- kept the same semantic intent by still exercising sample-rate-dependent FIR
  lowering before building the runtime descriptor
- removed the accidental dependency on an installed `stdfaust.lib` tree so the
  test now behaves the same on local machines and GitHub runners

### Validation

- `cargo test -p cranelift-ffi --lib`
- `cargo clippy -p cranelift-ffi --all-targets -- -D warnings`
- `cargo fmt --all -- --check`
